# 01. 개요와 시스템 아키텍처 (SPEC)

**목적**: loupit SPEC(개발 명세서) 문서군의 진입점. (1) 시스템 전체 아키텍처(컴포넌트·경계·책임)를 확정하고, (2) 배포 토폴로지와 요청 라우팅을 고정하며, (3) 빌드타임/런타임 데이터 흐름을 분리 명세하고, (4) 리포지토리 디렉토리 구조·기술 스택 버전을 pin하며, (5) SPEC 문서군 인덱스와 ID 대역을 배정한다. 본 문서는 **SP-ARCH** 대역을 소유한다.

**참조 상위문서**: 브리프 `CONTEXT_BRIEF.md` §2(핵심 결정)·§8(기술 스택)·§9(인프라), PRD `PRD.md`(F1~F8·NFR1~NFR29·AS1~AS14)·`07-비기능-요구사항.md`, FRD `01-개요와-FR규칙.md`(FR-01~FR-08 전역 규약·§5 상태 모델), FRD `11-API.md`(FR-90~FR-96 전송 계약), FLOW `01-사이트맵과-네비.md`(SCR-01~09 라우트 트리).

**상위 추적**: SPEC → (FRD·FLOW·WIREFRAME·PRD·RESEARCH) → 브리프. 각 SP-ARCH 항목은 자신이 구현하는 FR-*/NFR*/F#/D#/UC-*/AS#를 인용한다. 하위 SPEC 문서(SP-DB·SP-API·SP-GEN 등)와 TASK 문서는 본 문서가 확정한 아키텍처 불변식·디렉토리 경로·버전·SPEC 인덱스를 그대로 인용한다.

**전역 전제(FRD `01` §전역 전제 상속)**: 로그인/회원/계정/OAuth/JWT/이메일 인증 없음(NFR16). 프로파일러 영구 제외(§2-3). 서버 측 사용자 데이터 쓰기 없음(NFR20). 사용자 상태는 브라우저 localStorage 한정(NFR17). 비교 계산 100% 클라이언트(§2-8). 회사 등록 = **복지 실데이터 보유 ~96개**(200 아님). 프리셋 = 비교 툴 직접 입력 모드 템플릿용(§4). 밴드 DEC-2(명시 ±5% / 추정 ±20% / 만료 +15% 가산).

## 1. 아키텍처 개요 (한 그림)

loupit은 **정적 콘텐츠 페이지 집합 + 단일 비교 툴 SPA + 슬림 읽기 전용 API**로 구성된 3-티어(정적/앱/데이터) 단일 인스턴스 아키텍처다. 계산은 브라우저에서, 페이지 생성은 빌드타임에, 참조 데이터 서빙만 런타임 API에서 일어난다.

```
                          ┌──────────────────────────── Oracle Cloud ARM (Always Free, 춘천) ────────────────────────────┐
   브라우저 (모바일 웹)     │                                                                                              │
   HTTPS loupit.co         │   ┌─────────────────── Nginx (TLS 종단, Let's Encrypt) ───────────────────┐                  │
        │                  │   │                                                                       │                  │
        │  GET / , /compare │   │  location /            → filesystem  web/  + web/dist/  (정적 직접)   │                  │
        │  /company/{slug}  ├──▶│  location /company/... ─┤  (NFR1: 서버 렌더 0, DB 조회 0)            │                  │
        │  /vs/{a}-{b}      │   │  location /vs/... ...    └──────────────┐                             │                  │
        │  /privacy ...     │   │  location = /sitemap.xml , /robots.txt  │                             │                  │
        │  assets, sitemap  │   │                                        ▼                             │                  │
        │                   │   │                                  [ 정적 파일 ]                       │                  │
        │  GET /api/v1/*    │   │  location /api/v1/  → proxy_pass 127.0.0.1:8000                       │                  │
        └──────────────────┼──▶│                              │                                        │                  │
                           │   └──────────────────────────────┼────────────────────────────────────────┘                  │
                           │                                   ▼                                                            │
                           │                        Uvicorn + FastAPI (systemd, 127.0.0.1:8000)                            │
                           │                        읽기 전용 GET 4종 · aiomysql 풀                                         │
                           │                                   │                                                            │
                           │                                   ▼                                                            │
                           │                        MySQL 8.0 (localhost:3306, utf8mb4)                                     │
                           │                        참조 테이블 5종 (읽기 전용 트래픽)                                       │
                           └────────────────────────────────────────────────────────────────────────────────────────────┘

   빌드타임(배포 파이프라인, 사용자 트래픽과 분리):
      db/schema.sql + db/seed/*.sql  ──(mysql import)──▶  MySQL
      MySQL ──(server.services.reference.build_reference_bundle)──▶ generator ──▶ web/dist/{company,vs,정책}.html + sitemap.xml
```

## 2. 컴포넌트 정의

### [SP-ARCH-1] 시스템 컴포넌트 5종
아키텍처는 정확히 5개 배포 컴포넌트로 구성되며 각 책임 경계는 아래로 고정한다. 컴포넌트 간 유일한 런타임 결합은 `Nginx→Uvicorn(HTTP)`, `Uvicorn→MySQL(TCP/aiomysql)`뿐이다.

| # | 컴포넌트 | 디렉토리 | 책임 | 실행 형태 | 근거 |
| --- | --- | --- | --- | --- | --- |
| C1 | **web (정적 프론트)** | `web/` | 랜딩(SCR-01)·비교 툴 SPA 셸(SCR-02~05)·정적 자산(JS/CSS/폰트/affiliate.json). 무빌드 바닐라 JS. | Nginx가 파일 직접 서빙 | AS3·NFR4·NFR1 |
| C2 | **generator (빌드타임 정적 생성기)** | `generator/` | MySQL 번들 → 회사 상세 ~96(SCR-06)·인기 조합(SCR-07)·정책 4종(SCR-08)·404(SCR-09) HTML + `sitemap.xml`·`robots.txt` 산출 → `web/dist/`. | 빌드타임 파이썬 CLI(사용자 트래픽 무관) | AS2·NFR1·NFR9·F4·F5 |
| C3 | **server (슬림 읽기전용 API)** | `server/` | GET 4종(`health`·`reference/all`·`companies/search`·`companies/{id}`). 인증·쓰기 미들웨어 0. aiomysql 풀. 참조 번들 빌더 소유. | Uvicorn(systemd), 127.0.0.1:8000 | F8·NFR20·NFR27·FR-90 |
| C4 | **db (MySQL 8.0)** | `db/`(스키마·시드), 런타임은 MySQL 서버 | 참조 테이블 5종 저장. utf8mb4. 읽기 전용 트래픽만(런타임 쓰기 0). | MySQL 8.0 서비스, localhost:3306 | D1·NFR29·§4 |
| C5 | **infra (Nginx·systemd·TLS·프로비저닝)** | `infra/` | TLS 종단·정적 서빙·`/api/v1` 프록시·프로세스 관리·인증서 갱신·(선택)IaC. | Nginx·systemd 유닛·certbot·Terraform | AS1·NFR23·NFR28 |

**경계 규칙**: (1) C1은 서버 상태를 갖지 않는다(정적). (2) C2는 런타임 요청 경로에 없다(빌드타임 전용). (3) C3은 사용자 데이터를 쓰지 않는다(GET 전용). (4) C4는 참조 데이터 외 사용자 테이블이 없다(브리프 §4 제거 테이블 목록). (5) 프론트↔API는 **동일 오리진**(`https://loupit.co`, `/api/v1` 하위 프록시)이므로 CORS는 사실상 same-origin이다(FR-96).

- **추적**: F1~F8, NFR1·NFR4·NFR20·NFR23·NFR27·NFR28, AS1·AS2·AS3, FR-01·FR-90, §8·§9.

## 3. 배포 토폴로지

### [SP-ARCH-2] 단일 인스턴스 배포·요청 라우팅
전 컴포넌트를 **하나의 Oracle Cloud Ampere A1(ARM, Always Free) 인스턴스**에 co-locate한다(AS1). 외부 노출은 Nginx의 443(TLS)·80(→443 리다이렉트)뿐이며, Uvicorn·MySQL은 loopback에만 바인딩한다.

| 계층 | 프로세스/포트 | 바인딩 | 관리 | 노출 |
| --- | --- | --- | --- | --- |
| Edge | Nginx :443, :80 | 0.0.0.0 | systemd(nginx) | 공개(TLS, Let's Encrypt) |
| App | Uvicorn :8000 | 127.0.0.1 | systemd(`loupit-api.service`) | 비공개(Nginx만 접근) |
| Data | MySQL :3306 | 127.0.0.1 | systemd(mysqld) | 비공개(App만 접근) |

**Nginx location 라우팅 계약**(상세 지시문은 SP-INFRA 소유, 여기서는 분기 규칙만 고정):

| location | 처리 | 대상 | 근거 |
| --- | --- | --- | --- |
| `= /` | 정적 | `web/index.html` (SCR-01) | FLOW §2 |
| `/compare` | 정적(SPA 셸) | `web/compare/index.html` (try_files → 셸) | SCR-02 |
| `^~ /company/`, `^~ /vs/`, `= /privacy` `= /terms` `= /disclaimer` `= /ads` | 정적 | `web/dist/**` 사전생성 HTML | SCR-06~08 |
| `= /sitemap.xml`, `= /robots.txt` | 정적 | `web/dist/` 산출물 | NFR9·NFR10 |
| `^~ /assets/` | 정적(장기 캐시) | `web/assets/**` | NFR2·NFR5 |
| `^~ /api/v1/` | 프록시 | `proxy_pass http://127.0.0.1:8000;` | FR-96·§8 |
| (그 외/오타) | 정적 404 | `web/dist/404.html` (SCR-09) | FR-E6 |

- **규칙**: (1) `/api/v1/*` 외 모든 경로는 **DB·앱을 거치지 않고** 파일에서 응답한다(NFR1·NFR28: 트래픽·DB 부하 흡수). (2) 정적 HTML은 JS 없이 본문이 읽혀야 한다(NFR12·NFR24 — 생성기 책임, SP-GEN에서 검증). (3) TLS는 전 구간 강제(NFR23), 80→443 리다이렉트. (4) 앱·DB는 공개 포트 미노출(NFR22).
- **추적**: AS1, NFR1·NFR2·NFR22·NFR23·NFR24·NFR28, FR-96, FLOW SCR-01~09, §8·§9.

## 4. 데이터 흐름 (빌드타임 vs 런타임)

### [SP-ARCH-3] 빌드타임 데이터 흐름(시드·정적 생성)
사용자 트래픽과 완전히 분리된 배포 파이프라인 단계다. 순서는 SP-ARCH-9가 소유.

1. **스키마·시드 적재** (SP-DB·SP-SEED): `db/schema.sql`로 참조 테이블 5종 생성 → `db/seed/*.sql`(기업유형 6종, 유형별 프리셋, 회사·별칭 ~96, 상세 복지 SQL 96파일) 적재. 결과: MySQL이 유일한 참조 데이터 원천.
2. **정적 생성** (SP-GEN): generator가 `server.services.reference.build_reference_bundle(conn)`(SP-ARCH-4)로 번들을 얻어 회사 상세(~96)·인기 조합·정책 4종·404 HTML과 `sitemap.xml`·`robots.txt`를 렌더 → `web/dist/`에 기록. 이때 SEO 메타·JSON-LD·canonical·OG를 페이지에 인라인(NFR6~NFR11).

### [SP-ARCH-4] 참조 번들 단일 소스(런타임·빌드타임 공유)
`reference/all` 응답 본문을 만드는 **번들 빌더 함수는 코드 상 단 하나**다: `server/services/reference.py::build_reference_bundle(conn) -> dict`. 런타임 API 라우터(FR-92)와 빌드타임 generator(C2)가 **같은 함수**를 호출해 동일한 데이터 형태를 보장한다. 이로써 (a) 정적 페이지와 SPA가 동일 스키마·값을 공유하고, (b) 생성에 러닝 API가 필요 없다(모노레포 내 `server` 패키지를 `PYTHONPATH`로 import).

- 번들 형태(FRD `02` FR-Dx·브리프 §6): 최상위 키 정확히 3종 — `company_types[]`, `benefit_presets{type_cd:[…]}`, `companies[{…, aliases[], benefits[]}]`. 프로파일러 키(`profiles`/`job_groups`/`questions`) **부재**(NFR16·§2-3).
- 시그니처: `async def build_reference_bundle(conn) -> dict` (런타임은 aiomysql 커넥션, 빌드타임은 동일 인터페이스의 커넥션 주입). 순수 조립(부수효과·쓰기 0).

### [SP-ARCH-5] 런타임 데이터 흐름(읽기 전용)
1. **정적 서빙**: 브라우저의 문서/자산 요청 → Nginx → 파일(§3). 서버 렌더·DB 조회 0(NFR1).
2. **SPA 부팅**(FR-02): `/compare` 셸 로드 → `GET /api/v1/reference/all` **1회** → 메모리 `REF`. 응답 `Cache-Control: public, max-age=3600`(NFR3) → 재방문/재진입은 브라우저 캐시 사용.
3. **검색·상세**(F1·F8): 타이핑 시 `GET /api/v1/companies/search?q=`(≤20건, FR-93). `companies/{id}`는 개별 상세 조회 계약(FR-94)이며, 회사페이지는 사전생성이므로 런타임 호출 대상은 아님(완결성·향후 소비용).
4. **계산**(F3): `REF`+사용자 입력으로 **브라우저에서** 실효연봉·시간가치·워라밸·복지 델타·밴드·판정 산출(§2-8, SP-ENGINE). 서버 호출 0.
5. **지속성**(FR-07): 비교 입력·"최근 비교"를 localStorage에만 저장(NFR16·NFR17). 실패는 try/catch 흡수, 비교는 정상 동작(NFR25).

- **불변식**: 런타임에 서버로 나가는 것은 GET 4종뿐이며 **사용자 데이터 업로드 0**(네트워크 관찰로 검증, SP-ARCH-10 T3·T7). 밴드는 출처 배지가 아니라 금액 신뢰도(02 FR-D9 `amt_source`=stated/estimated/none) 기준(DEC-2, FR-05).
- **추적**: F1·F2·F3·F8, D2·D6, NFR1·NFR3·NFR16·NFR17·NFR25, FR-02·FR-05·FR-06·FR-07·FR-92·FR-93·FR-94, §2-8·§6.
