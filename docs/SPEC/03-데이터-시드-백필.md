## SP-SEED-2 — `COMP_ENG_NM` ↔ 200개 시드 매칭·정규화

**근거**: D2.3(메타는 200개 시드에서 매칭), RESEARCH §8 OI-1(94/96 매칭). 96개 복지 SQL의 `COMP_ENG_NM`(파일 자기등록 키)과 200개 시드(`companies_kospi_1/2.py`·`companies_kosdaq_1/2.py`)의 `id`가 **항상 일치하지 않음**을 실측 확인. 회사 메타(별칭·산업·로고·유형)를 안전하게 붙이려면 조인 키 결정이 필수다.

### SP-SEED-2.1 조인 규칙 (실측 결과)

96개 SQL → 200개 시드 매칭 실측:

| 유형 | 건수 | 처리 |
|------|------|------|
| `eng_nm` 정확 일치(SQL eng = 200-seed `id`) | 79 | 200-seed 레코드에서 별칭·산업·로고 승계 |
| `COMP_NM`만 일치(eng 상이) | 14 | **정식명(`COMP_NM`)으로 조인**해 200-seed 별칭 승계, `eng_nm`은 SQL 값 유지 |
| 매칭 없음(예외/중복) | 3 | `CJ`(교정)·`엔씨소프트`(예외 포함)·`모비스`(중복 제거) — SP-SEED-1 별도 처리 |

- **정본 조인 키 = `COMP_NM`(정식명)**. `TCOMPANY.COMP_NM`은 UNIQUE(SP-DB-3)이고 96개 SQL의 `COMP_NM`과 200-seed `name`이 일치하는 것이 93건(79+14)이라 안정적. `eng_nm`은 파일마다 상이할 수 있어 조인 키로 부적합.
- **정본 `COMP_ENG_NM` = 복지 SQL의 eng 값**. 이유: 각 복지 SQL 내부가 `SET @comp_id = (SELECT COMP_ID FROM TCOMPANY WHERE COMP_ENG_NM='<SQL eng>')`로 동작하므로, 회사 행의 `COMP_ENG_NM`이 SQL eng와 반드시 동일해야 복지 INSERT가 올바른 `@comp_id`에 걸린다. 200-seed `id`는 eng로 채택하지 **않는다**(별칭·산업·로고 승계용으로만 사용).

## SP-SEED-4 — `reference.sql`: 기업유형 6종 + 프리셋 시드

**근거**: D2.1·D2.2, D1.1·D1.5, SP-DB-2·SP-DB-6·SP-DB-9. 소스 = `job_change/server/seed/seed.py`의 `COMPANY_TYPES`·`BEN_PRESETS` 상수(큐레이션, 재게시 아님).

### SP-SEED-4.1 기업유형 6종

`INSERT IGNORE`(UNIQUE `COMP_TP_CD`로 멱등). 확정값:

| COMP_TP_CD | COMP_TP_NM |
|------------|------------|
| large | 대기업 |
| mid | 중견기업 |
| public | 공기업 |
| startup | 스타트업 |
| foreign | 외국계 |
| freelance | 프리랜서 |

- **이력**: 브랜드 축(성장성·안정성)은 2026-07-20 제거됐다. 우선순위는 연봉·워라밸·복지 3종.
  `TCOMPANY_TYPE`의 `GROWTH_RATE_VAL`·`GROWTH_LABEL_NM`·`STABILITY_SCORE_NO` 컬럼이 드랍되어
  본 시드는 코드·명칭 2필드만 채운다. 경위·파급범위: `docs/ANALYSIS-성장성-용어-2026-07-20.md` §6-1 ·
  `docs/HANDOFF-2026-07-19.md` §K.

### SP-SEED-4.2 유형별 복지 프리셋 (총 28행)

소스 `BEN_PRESETS`: large 8·mid 4·public 8·startup 2·foreign 6·freelance 0 = **28행**. 필드 매핑 `{key→BENEFIT_CD, name→BENEFIT_NM, val→BENEFIT_AMT, cat→BENEFIT_CTGR_CD, badge→BADGE_CD, checked→DEFAULT_CHECKED_YN}`, `SORT_ORDER_NO`=배열 인덱스. `COMP_TP_ID`는 `(SELECT COMP_TP_ID FROM TCOMPANY_TYPE WHERE COMP_TP_CD=?)`로 파생.

```sql
SET NAMES utf8mb4;
-- 기업유형 6종
INSERT IGNORE INTO TCOMPANY_TYPE (COMP_TP_CD, COMP_TP_NM) VALUES
  ('large','대기업'),
  ('mid','중견기업'),
  ('public','공기업'),
  ('startup','스타트업'),
  ('foreign','외국계'),
  ('freelance','프리랜서');

-- 프리셋: 멱등 full-refresh (TBENEFIT_PRESET에 자연 UNIQUE 부재 → 재실행 시 중복 방지)
DELETE FROM TBENEFIT_PRESET;
INSERT INTO TBENEFIT_PRESET
  (COMP_TP_ID, BENEFIT_CD, BENEFIT_NM, BENEFIT_AMT, BENEFIT_CTGR_CD, BADGE_CD, DEFAULT_CHECKED_YN, SORT_ORDER_NO)
VALUES
  ((SELECT COMP_TP_ID FROM TCOMPANY_TYPE WHERE COMP_TP_CD='large'),'meal','식대 지원 (3식)',360,'perks','est',TRUE,0),
  -- … BEN_PRESETS 순서대로 large 8 / mid 4 / public 8 / startup 2 / foreign 6 …
  ((SELECT COMP_TP_ID FROM TCOMPANY_TYPE WHERE COMP_TP_CD='foreign'),'edu','교육비 (도서, 세미나)',200,'growth','est',TRUE,5);
```

- **멱등성 주의(SP-SEED-10)**: `TBENEFIT_PRESET`은 스키마상 `(COMP_TP_ID,BENEFIT_CD)` UNIQUE가 없어(SP-DB-6) `INSERT IGNORE`가 중복을 막지 못한다. 프리셋은 소량 큐레이션 상수이므로 **`DELETE FROM TBENEFIT_PRESET;` 후 전량 재삽입**(full-refresh)으로 멱등을 보장한다. freelance는 프리셋 0행(정상).
- `reference.sql`은 `reference_gen.py`(선택)로 `seed.py` 상수에서 생성하거나 수기 유지. 값 집합은 SP-DB-9(카테고리 9종·배지 2종)를 위반하지 않아야 하며 SP-SEED-11 DC-5/DC-6로 검증.

```
        # eng_nm='cj' 유지(복지 INSERT의 @comp_id 조회 키 불변)
    write(DST/basename(f), text)
assert count(DST/*.sql) == 95
```

- `replace_company_nm`은 `INSERT IGNORE INTO TCOMPANY (...) VALUES ('cj', 'CJ그룹',` 튜플의 두 번째 리터럴만 치환(정규식 앵커 `VALUES \('cj', '\KCJ그룹`). 헤더 주석은 참고용이라 무해하나 함께 갱신 권장.
- 그 외 94개(모비스 제외 나머지)는 **바이트 무변경 복사**. 파일별 헤더(`출처: AI 파싱 (날짜)` / `URL:`)는 백필(SP-SEED-8)의 프로버넌스 소스이므로 보존 필수.

### SP-SEED-5.2 각 복지 SQL의 포맷 계약(불변, RESEARCH §3[8])

재이식 SQL은 아래 3부 구조를 유지한다. 회사 자기등록(6컬럼) + est 삭제(official 보존) + 복지 upsert.

```sql
INSERT IGNORE INTO TCOMPANY (COMP_ENG_NM, COMP_NM, COMP_TP_ID, INDUSTRY_NM, LOGO_NM, CAREERS_BENEFIT_URL)
VALUES ('<eng>', '<정식명>', (SELECT COMP_TP_ID FROM TCOMPANY_TYPE WHERE COMP_TP_CD='<type>'), '<산업>', '<로고>', <url|NULL>);
SET @comp_id = (SELECT COMP_ID FROM TCOMPANY WHERE COMP_ENG_NM='<eng>');
DELETE FROM TCOMPANY_BENEFIT WHERE COMP_ID=@comp_id AND BADGE_CD='est';   -- official 보존
INSERT INTO TCOMPANY_BENEFIT (COMP_ID, BENEFIT_CD, BENEFIT_NM, BENEFIT_AMT, BENEFIT_CTGR_CD, BADGE_CD, NOTE_CTNT, QUAL_YN, QUAL_DESC_CTNT, SORT_ORDER_NO)
VALUES (@comp_id, …, 'est', …) ON DUPLICATE KEY UPDATE …;
```

- **`AMT_SOURCE_CD` 미기재**: 레거시 SQL은 이 컬럼을 INSERT 목록에 넣지 않는다 → 컬럼 DEFAULT `'estimated'`(SP-DB-7.2)로 로드. 백필(SP-SEED-8)이 `stated`/`none`으로 정밀화.
- **회사 메타는 SQL이 6컬럼만 자기등록**(별칭·`WORK_STYLE_VAL` 없음) → SP-SEED-6이 보강. 단계 4가 3 뒤이므로 `eng_nm` 이미 존재.
- CJ 교정 후에도 `eng_nm='cj'` 유지 → `@comp_id` 조회 정상. `COMP_NM`은 `companies.py`가 아닌 이 SQL의 교정 리터럴이 정본(테스트 DC-15).

```python
def apply_company_meta(cur, meta: dict):
    cur.execute("SELECT COMP_ID, COMP_ENG_NM FROM TCOMPANY")
    id_of = {eng: cid for cid, eng in cur.fetchall()}
    for eng, m in meta.items():
        cid = id_of.get(eng)
```

## SP-SEED-8 — DEC-2 백필 오케스트레이션 (`backfill_dec2.py`)

**근거**: DEC-2, RESEARCH §4·§6 CHANGE·§7 Phase 0, OI-3·OI-6, D3. SP-DB-13이 **순수 SQL 판본**을 정의한다. 본 절은 로더가 실행하는 **프로버넌스 정밀 판본**을 확정한다(파일 헤더의 수집일·실 URL을 반영 → SP-DB-13의 평면 날짜/URL을 정밀화). 두 판본은 **동일 불변식**(DC-13/14, SP-DB-16.3)을 만족해야 한다.

### SP-SEED-8.1 출처 신뢰도 승격 (`BADGE_CD → official`)

- 96개(=95 등록분) 로드된 `BADGE_CD='est'` 복지 전량을 `official`로 승격(DEC-2). `UPDATE TCOMPANY_BENEFIT SET BADGE_CD='official' WHERE BADGE_CD='est'`.
- 재실행 시 대상 0건(멱등). 단, 재시드로 est가 재유입되면 백필 재적용 필요(SP-SEED-10).

### SP-SEED-8.2 금액 신뢰도 도출 `AMT_SOURCE_CD` (OI-6 확정 규칙)

로더가 각 복지 행에 대해 결정. 소스 실측(‘(추정)’ 522회·‘환산’ 8회·정성 NULL 697행)에 근거한 **결정적 규칙**:

```
def derive_amt_source(benefit_amt, qual_yn, note_ctnt) -> str:
    if qual_yn or benefit_amt is None:              # 정성/금액없음
        return "none"
    n = note_ctnt or ""
    if ("추정" in n) or ("환산" in n) or (n == ""):  # 앵커/계산/근거없음 → 보수적 넓은 밴드
        return "estimated"
    return "stated"                                  # note에 명시 금액 근거 있고 추정 표기 없음
```

- 규칙 요지: 정성→`none` / (추정·환산 표기 있음 OR note 없음)→`estimated` / note 있고 추정 표기 없음→`stated`.
- 삼성전자 예시(`'(추정)'`, `'일 18,000원 x 240일 환산'`)는 모두 `estimated`로 귀결 → ±20% 밴드 유지(DEC-2 취지: 공식 출처라도 추정 금액은 넓은 밴드). CJ `welfare_point`(note "연간 1,000p(100만원)", 추정 표기 없음)는 `stated`.
- 순수 SQL 등가(SP-DB-13.2): `NOTE_CTNT LIKE '%추정%' OR LIKE '%환산%' OR IS NULL` → estimated 등. 두 구현 결과는 동치.

### SP-SEED-8.3 출처 유형·URL (`BADGE_SRC_CD`·`BADGE_SRC_URL_CTNT`)

파일 헤더 `URL:` 실측(실 http 5·`수동 입력`류 91)을 반영:

| 조건 | `BADGE_SRC_CD` | `BADGE_SRC_URL_CTNT` |
|------|----------------|----------------------|
| 헤더 `URL: http…`(실 URL) 존재 | `scrape_official` | 헤더 URL |
| `URL: 수동 입력`류 | `ai_parse` | NULL(Phase 2에서 재발견 복원) |

- 값이 이미 존재하면 보존(멱등). SP-DB-13은 전량 `ai_parse` 평면 처리 → 본 절이 5건을 `scrape_official`로 정밀화(상위 호환, DC-8 위반 없음).

### SP-SEED-8.4 신선도·만료 (`VERIFIED_DTM`·`EXPIRES_DTM`, TTL=OI-3 확정)

- `VERIFIED_DTM` = 파일 헤더 `출처: AI 파싱 (YYYY-MM-DD)`의 수집일(전 파일 존재). 파싱 실패 시 시드 배치일 `2026-07-10`로 폴백.
- `EXPIRES_DTM` = `VERIFIED_DTM + 카테고리 TTL`. **OI-3 확정값(RESEARCH §4.3 채택, `benefit_service.py`와의 상충 종결)**:

| 카테고리 | TTL(개월) |
|----------|-----------|
| compensation | 6 |
| perks | 9 |
| time_off, flexibility | 12 |
| work_env, health, growth, leisure, family | 18 |

```python
TTL_MONTHS = {"compensation":6, "perks":9, "time_off":12, "flexibility":12}   # 그 외 18
def expires(verified_dtm, ctgr):
    return verified_dtm + relativedelta(months=TTL_MONTHS.get(ctgr, 18))
```

- SP-DB-13의 `CASE BENEFIT_CTGR_CD … MONTH` 표와 동일. `EXPIRES_DTM > VERIFIED_DTM` 보장(DC-14).
- 만료 상태(경과)는 클라이언트가 `EXPIRES_DTM`으로 판정, 밴드 +15% 확대(D3.9, SPEC 프론트). 백필은 근거만 저장.

## SP-SEED-9 — 시드 오케스트레이터 (`load.py`)

**근거**: SP-SEED-3 실행 순서, D4.2(빌드타임 재시드). 단일 엔트리포인트.

```python
# server/seed/load.py
def main(fresh: bool = False):
    """
    fresh=True  : DROP+CREATE 후 전체 재시드(테스트/클린 재빌드)
    fresh=False : 멱등 재적용(운영 재시드 기본)
    """
    conn = connect()                                  # aiomysql 아님, 시드는 동기 pymysql
    with conn:
        cur = conn.cursor()
        run_sql_file(cur, "server/db/schema.sql")             # 1
        run_sql_file(cur, "server/seed/reference.sql")        # 2
        for f in sorted(glob("server/seed/benefit/sql/*.sql")): # 3
            run_sql_file(cur, f)                              #   (95개)
        apply_company_meta(cur, build_company_meta())         # 4
        stats = backfill(cur)                                 # 5
        conn.commit()
    verify_counts(stats)                              # 하한 assert(선택) — 상세는 pytest
    print(f"seed done: companies={count_companies()}, benefits={count_benefits()}, {stats}")
```

- `--fresh`는 `DROP TABLE IF EXISTS`(FK 역순) → `schema.sql`. 운영 재시드는 `--fresh` 없이 멱등 경로.
- 종료 코드 0 = 성공. 예외 시 `conn.rollback()` 후 비0 종료(부분 시드 방지, 단일 트랜잭션).

## SP-SEED-10 — 멱등성 규약

**근거**: SP-DB-1.9(반복 적용 안전), RESEARCH §3[8]. 재빌드가 데이터 갱신 경로(D4.2)이므로 멱등이 필수.

| 단계 | 멱등 기전 | 주의 |
|------|-----------|------|
| 기업유형 | `INSERT IGNORE`(UNIQUE `COMP_TP_CD`) | 안전 |
| 프리셋 | **`DELETE`+재삽입 full-refresh** | 스키마에 `(COMP_TP_ID,BENEFIT_CD)` UNIQUE 부재(SP-DB-6) → `INSERT IGNORE` 부적합. full-refresh 필수(SP-SEED-4.2) |
| 회사 등록 | `INSERT IGNORE TCOMPANY`(UNIQUE eng/nm) | 안전 |
| 복지 upsert | `DELETE … BADGE_CD='est'`(official 보존) + `INSERT … ON DUP KEY`(`uq_comp_benefit`) | 아래 **est↔official 순환** 주의 |
| 별칭 | `INSERT IGNORE`(`uq_comp_alias`) | 안전 |
| work_style | `UPDATE`(값 고정) | 안전 |
| 백필 | `official` 승격 대상=est만 / `COALESCE`·조건부 UPDATE | 재시드로 est 재유입 시 백필 재실행 필수 |

**est↔official 순환 불변식(핵심)**: 복지 SQL은 행을 `BADGE_CD='est'`로 재삽입하며, `ON DUPLICATE KEY UPDATE BADGE_CD=VALUES(BADGE_CD)`가 기존 `official` 행을 다시 `est`로 되돌린다. 따라서:
- **파이프라인 단위(단계 3+5)로만 멱등**하다. 단계 3만 재실행(백필 없이)하면 데이터가 `est` 중간상태로 남는다 → **최종 상태로 취급 금지**.
- `load.py`는 항상 단계 3 뒤 단계 5(백필)를 실행한다. 부분 실행 금지. 전체 재실행 결과는 결정적(동일 최종 상태): 모든 등록 복지 `official` + `amt_source` 규칙적용 + `verified/expires` 채움.
- `DELETE … BADGE_CD='est'`는 재실행 시 est 행이 없으면(이미 official) no-op → 기존 official 데이터 파괴 없음. 재삽입 후 백필이 다시 승격.
