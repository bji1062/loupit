# SPEC 11 — 인프라·배포·운영

**영역 ID 대역**: `SP-INFRA`
**저장 경로**: `docs/SPEC/11-인프라-배포.md`
**컴포넌트**: C5 (infra) — `infra/` (SP-ARCH-1·SP-ARCH-6)

**목적**: loupit(C1~C4)을 **Oracle Cloud Ampere A1(ARM64, Always Free, 춘천)** 단일 인스턴스에 배포·운영하기 위한 인프라 계약을 개발자가 추측 없이 그대로 구성할 수준으로 확정한다. (1) 배포 토폴로지·포트·바인딩, (2) 인스턴스 프로비저닝(OS·패키지·디렉토리·권한), (3) Nginx 설정(정적 서빙 + `/api/v1` 프록시 + SSL Let's Encrypt + `gzip_static` + 캐시 헤더 + HTTP/2 + HSTS), (4) TLS·인증서 자동갱신, (5) systemd 서비스 유닛(uvicorn 자동재시작·프로세스 하드닝), (6) MySQL 설정(utf8mb4·리소스 최적화·loopback 바인딩·읽기전용 계정), (7) 환경변수·비밀키 관리(노출 금지), (8) 방화벽·보안(SSH 제한·MySQL/uvicorn 외부차단), (9) 빌드/릴리스 플로우(시드→생성→배포 순서), (10) 로그·백업·운영, (11) 헬스체크·스모크 테스트 명세를 소유한다.

**상위 추적**: CONTEXT_BRIEF §8(기술 스택)·§9(인프라: OCI ARM + Nginx SSL + systemd, 정적=Nginx / `/api/v1`=Uvicorn 프록시), PRD `07-비기능-요구사항.md`(**NFR1·NFR3·NFR5·NFR12·NFR22·NFR23·NFR27·NFR28·NFR29**), PRD `09-제약과-가정.md`(**AS1**(Always Free 단일 구성)·**AS2**(빌드타임 정적 생성)·**AS4**(슬림 read-only)), SPEC `01-개요와-아키텍처.md`(**SP-ARCH-2**(토폴로지·포트)·**SP-ARCH-6**(디렉토리)·**SP-ARCH-7**(버전 pin)·**SP-ARCH-9**(릴리스 파이프라인 순서)·**SP-ARCH-10** T8(배포 스모크)), SPEC `03-데이터-시드-백필.md`(**SP-SEED-9** 로더 진입점), SPEC `07-정적-생성기.md`(**SP-GEN-1.4** CLI·**SP-GEN-11** 릴리스 산출물 gzip/원자적 스왑), SPEC `04-백엔드-API.md`(**SP-API-2** 설정 키·`server.main:app`). 하위 TASK 문서는 `SP-INFRA-N` ID를 인용한다.

**범위 경계**: 본 문서는 **호스트 OS·엣지·프로세스·릴리스 오케스트레이션**만 소유한다. DB DDL은 SP-DB, 시드 로직은 SP-SEED, 생성기 CLI·스왑 규격은 SP-GEN, FastAPI 앱·설정 키 정의는 SP-API가 소유하며, 본 문서는 그것들을 **호출·배치·기동**하는 계약(systemd 유닛·nginx conf·release.sh·my.cnf·방화벽)을 확정한다. 프로파일러(가치관 진단 설문)와 익명 서버쓰기는 영구 제외이나, 로그인·회원·계정은 복지 등록·수정 기여(SC14, 문서 09/13·SP-AUTH)에 한해 In-scope다. SC14 기여 흐름의 **메일 발송 rate-limit 존(`loupit_mail`)·`X-Loupit-Client` CSRF 게이트·SMTP 설정 키·런타임 DB 쓰기 그랜트**에 대한 인프라 명세는 **본 문서 §3.4·§6.2a·§7에서 재명세 완료**(2026-07-22, SPEC/13 SP-AUTH-11·12·4·9 대응 — 구 '재명세 대기' 해소). 세션은 앱계층 DB 테이블(`TSESSION`, SP-AUTH-4)·라우트 의존성으로 구현되어 **외부 세션 스토어 등 신규 인프라 컴포넌트는 없다**(INV-9) — 단 런타임 DB 계정은 익명 SELECT-only에서 **참여 테이블 스코프 쓰기로 확장**된다(§6.2a). 익명 읽기 흐름의 인프라 요소는 로그인 없이 그대로다(§2-1·§2-4, NFR16·NFR20).

**전역 불변식(SP-ARCH-10 상속)**: INV-3(정적 경로는 앱·DB 미경유) · INV-7(프론트↔API 동일 오리진 `/api/v1` 프록시, **앱(8000)·DB(3306)는 공개 포트 미노출**). 본 문서는 이 둘을 인프라 설정으로 강제한다.

> **문서 번호 주석**: SP-ARCH-8 인덱스는 본 영역을 "12 인프라·배포·운영(SP-INFRA)"으로 등재하나, 배정된 저장 경로는 `docs/SPEC/11-인프라-배포.md`다(경로 우선). ID 대역 `SP-INFRA`는 동일하다.

## SP-INFRA-1 — 배포 토폴로지·포트·바인딩

**근거**: SP-ARCH-2, AS1, NFR22·NFR23·NFR28, INV-7.

전 컴포넌트를 **단일 Ampere A1 인스턴스에 co-locate**한다. 외부로 노출되는 것은 Nginx의 `443`(TLS)·`80`(→443 리다이렉트 + ACME challenge)뿐이며, Uvicorn·MySQL은 **loopback(127.0.0.1)에만 바인딩**하여 물리적으로 외부에서 접근 불가하게 한다(방화벽 + 바인딩 이중 방어).

### [SP-INFRA-1.1] 포트·바인딩·노출 매트릭스

| 계층 | 프로세스 | 포트 | 바인딩 | 관리 유닛 | 외부 노출 | 근거 |
| --- | --- | --- | --- | --- | --- | --- |
| Edge | Nginx | `80`, `443` | `0.0.0.0` / `[::]` | `nginx.service` | **공개**(TLS) | NFR23 |
| App | Uvicorn(FastAPI) | `8000` | `127.0.0.1` | `loupit-api.service` | 비공개(Nginx만) | INV-7·NFR22 |
| Data | MySQL 8.0 | `3306` | `127.0.0.1` | `mysql.service` | 비공개(App만) | INV-7·NFR22 |
| Ops | SSH(sshd) | `22` | `0.0.0.0` | `ssh.service` | **제한 공개**(관리 IP·키만) | SP-INFRA-8 |

### [SP-INFRA-1.2] 요청 흐름(런타임)

```
브라우저 ──HTTPS:443──▶ Nginx(TLS 종단)
    ├─ 문서/정적자산/정적HTML(회사·조합·정책·sitemap·robots) ─▶ 파일시스템(web/, web/dist/)   [DB·앱 미경유 = INV-3]
    └─ /api/v1/*  ──proxy_pass──▶ 127.0.0.1:8000 (Uvicorn) ──aiomysql──▶ 127.0.0.1:3306 (MySQL)
브라우저 ──HTTP:80──▶ Nginx ── 301 https (단, /.well-known/acme-challenge/* 는 통과)
```

- **불변식 강제**: `/api/v1/` 외 모든 경로는 파일에서 응답(NFR1·NFR28). 앱·DB는 공개 IP에 바인딩되지 않으므로, 방화벽 실수가 있어도 `127.0.0.1` 바인딩이 최종 방어선이다(NFR22).
- **추적**: SP-ARCH-2, INV-3·INV-7, NFR22·NFR23·NFR28.

## SP-INFRA-3 — Nginx 설정

**근거**: CONTEXT_BRIEF §9, NFR1·NFR3·NFR5·NFR12·NFR23·NFR28·**NFR31**(발송 리밋), SP-ARCH-2 라우팅 계약, SP-GEN-11(gzip 사전압축·산출 트리), **SP-AUTH-11·12·FR-112·FR-113**(SC14 메일 존·CSRF 게이트 — §3.4). 레거시 `job_change/server/deploy/nginx.conf` 계승(인증·SPA-only fallback 제거, 정적 다중 root 반영).

### [SP-INFRA-3.1] 라우팅 계약(SP-ARCH-2 확정 구현)

| location | 처리 | 대상 파일 | 캐시 정책 | 근거 |
| --- | --- | --- | --- | --- |
| `^~ /.well-known/acme-challenge/` (:80) | 정적 | `web/.well-known/…` | no-store | SP-INFRA-4 |
| `location / ` (:80) | 301 | → `https://$host$request_uri` | — | NFR23 |
| `= /` | 정적 | `web/index.html`(SCR-01) | `no-cache, must-revalidate` | FLOW §2 |
| `= /compare`, `= /compare/` | 정적(SPA 셸) | `web/compare/index.html`(SCR-02~05) | `no-cache, must-revalidate` | SCR-02 |
| `^~ /company/` | 정적 | `web/dist/company/{slug}.html`(SCR-06) | `public, max-age=300`(단명, 재생성 반영) | NFR1·F4 |
| `^~ /vs/` | 정적 | `web/dist/vs/{a}-{b}.html`(SCR-07) | `public, max-age=300` | NFR1·F5 |
| `= /privacy` `= /terms` `= /disclaimer` `= /ads` | 정적 | `web/dist/{name}.html`(SCR-08) | `public, max-age=300` | F7 |
| `= /sitemap.xml` `= /robots.txt` | 정적 | `web/dist/…` | `public, max-age=3600` | NFR9·NFR10 |
| `^~ /assets/v2/` | 정적(재검증) | `web/assets/**`(alias) | `no-cache`(ETag/IMS 304) | NFR2·NFR5 |
| `/assets/`(구경로, 확장자 매칭) | 정적(하위호환) | `web/assets/**` | `no-cache`(구 HTML 캐시분용) | NFR2 |
| `^~ /api/v1/` | 프록시 | `http://127.0.0.1:8000` | (앱이 헤더 설정) | FR-96·§8 |
| `= /api/v1/members/login-code` · `= /api/v1/employment/verify-code`(SC14) | 프록시(메일 발송) | `127.0.0.1:8000` | (앱이 설정) | `loupit_mail`·`X-Loupit-Client`(§3.4) |
| SC14 기여 쓰기(`^~ /api/v1/` 하위 POST/PUT/DELETE) | 프록시 | `127.0.0.1:8000` | — | Layer A `X-Loupit-Client` 게이트 상속·§3.4 |
| 그 외/오타 | 정적 404 | `web/dist/404.html`(SCR-09) | no-store | FR-E6 |

- **정적 다중 root**: 프론트 셸·자산은 `web/`, 생성물은 `web/dist/`. location별 `root` 오버라이드로 분리한다(빌드타임/무빌드 경계 유지, 병합/심링크 불필요).
- **동적 페이지 단명 캐시(`max-age=300`)**: 회사/조합/정책 HTML은 재시드·재생성으로 갱신되므로(AS2) 5분 단명 캐시로 갱신 반영과 부하 흡수를 절충.
- **자산 캐시 = v2 세대 + no-cache 재검증(2026-07-19 개정)**: 무빌드(AS3)라 콘텐츠 해시 URL이 없고, ES모듈 하위 import(`./ui.js`)는 `?v=` 쿼리 스탬프로 URL을 바꿀 수 없다 — 따라서 종전의 "1년 immutable + ?v= 스탬프" 계획(§9.3 구판, stamp_assets.sh)은 폐기. 공개 자산 URL은 `/assets/v2/…`(nginx alias → `web/assets/`) 하나의 세대로 고정하고 `no-cache`로 매 요청 재검증(변경 없으면 304)한다. 상대 import가 importer URL을 상속하므로 모듈 전체가 자동으로 같은 세대다. 구 `/assets/` 경로는 하위호환으로 no-cache 서빙만 유지. **불변식: `web/assets`에 `.gz` 사전압축 금지**(gzip_static은 원본과 신선도 비교가 없어 no-cache를 무력화 — v2 location은 `gzip_static off`, 압축은 런타임 gzip 전용). 재방문 시 자산별 304 왕복이 생기지만 HTTP/2 + 소형 자산 ~11개라 비용은 무시 가능. 한계(정직 서술): bfcache 복원은 재검증 없이 구 JS 힙을 되살릴 수 있고, 배포는 파일 단위 갱신이라 배포 순간 로드된 페이지는 신구 모듈 혼합 그래프가 될 수 있다(수 초 창).

### [SP-INFRA-3.2] `infra/nginx/loupit.conf`(정본 = 실파일, 아래는 골격 예시)

> **정본·드리프트 주석(2026-07-22, SC14 ②)**: 아래 인라인 conf는 라우팅 **골격 예시**이며, 운영 정본은 실파일 `infra/nginx/loupit.conf`다(§9.2 release.sh와 동일 원칙 — 전문 인용은 드리프트 원천). 실파일은 아래 골격에 더해 **감사 2026-07-20/21 계층**을 이미 싣고 있으며, 이들이 §3.4 SC14 계약의 전제다:
> - **(a) Layer A** — `^~ /api/v1/`에 `if ($http_x_loupit_client = "") { return 403; }`. 익명 데이터 GET의 벌크 스크래핑 방어이자, **`/api/v1/` 전역 강제라 SC14 상태변경 쓰기의 CSRF 헤더 요구(FR-113)를 엣지에서 이미 충족**한다(§3.4.3).
> - **(b) Layer B** — `$loupit_bad_bot` UA map(`conf.d/loupit-botdefense.conf`, 검색·AdSense 크롤러 화이트리스트 0) + 서버레벨 `if ($loupit_bad_bot) { return 403; }`.
> - **(c) rate-limit** — `loupit_api`(300r/m)·`loupit_log`(20r/m) 존(`conf.d/loupit-limits.conf`) + `limit_req_status 429` + `= /api/v1/comparisons/log`·`= /api/v1/health` 전용 블록(§3.4가 `loupit_mail` 존과 메일 발송 `=` 블록을 **여기에 추가**).
> - **(d)** `reference/all`(~600KB) 절단 방지 `proxy_buffers 16 64k`(2026-07-20 실장애 대응).
> - **(e)** `^~ /dist/`·`/dist.prev/` → 404(canonical 중복 제거), `= /companies`(회사 인덱스), `= /ads.txt`(AdSense).
> - **(f)** 자산 `no-cache` 재검증 + `/assets/v2/` 세대(구판 `immutable`·`?v=` 스탬프 폐기, §3.1·§9.3).
>
> 따라서 아래 예시(단순화판)를 **그대로 배포하지 말 것** — 실파일과 대조하고, SC14 참여 경로(메일 존·CSRF)의 정밀 계약은 **§3.4**가 소유한다. (§3.2 전면 실파일 정합은 별도 정리 대상 — 본 ②의 범위는 SC14 델타.)

```nginx

server {

    ssl_certificate_key /etc/letsencrypt/live/jobcho.wiki/privkey.pem;
    return 301 https://jobcho.wiki$request_uri;
}

# ── HTTPS :443 (apex) ──
server {
    listen 443 ssl;
    http2 on;                       # nginx ≥1.25 지시자 형태(구버전은 `listen 443 ssl http2;`)
    listen [::]:443 ssl;
    server_name jobcho.wiki;

    ssl_certificate     /etc/letsencrypt/live/jobcho.wiki/fullchain.pem;

    ssl_session_cache shared:MozSSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;
    # OCSP stapling 미사용 — Let's Encrypt 2025 OCSP 종료, 신규 인증서에 responder URL 없음

    # 정적 기본 루트 = 프론트 셸·자산
    root /home/ubuntu/loupit/web;
    index index.html;
    charset utf-8;                  # NFR29

    # ── gzip: 런타임 + 사전압축(gzip_static) ──
    gzip on;

    # 랜딩

        include /etc/nginx/snippets/loupit-security.conf;
        add_header Cache-Control "no-cache, must-revalidate" always;
        try_files /index.html =404;
    }

    # 비교 툴 SPA 셸
    location = /compare  { return 301 /compare/; }

    # 정적 자산(해시 버전 → 1년 immutable). 확장자 매칭.
    location ~* \.(?:css|js|woff2|svg|png|jpg|webp|ico)$ {

        add_header Cache-Control "public, max-age=31536000, immutable" always;
        access_log off;
        try_files $uri =404;
    }

    # 생성물: 회사 상세 ~96 (web/dist/company/{slug}.html)
    location ^~ /company/ {
        root /home/ubuntu/loupit/web/dist;

        add_header Cache-Control "public, max-age=300" always;
        try_files $uri $uri.html /404.html;
    }

    # 생성물: 인기 조합 (web/dist/vs/{a}-{b}.html)
    location ^~ /vs/ {
        root /home/ubuntu/loupit/web/dist;
        include /etc/nginx/snippets/loupit-security.conf;
        add_header Cache-Control "public, max-age=300" always;

    }

    # 생성물: 정책 4종
    location = /privacy    { root /home/ubuntu/loupit/web/dist; include /etc/nginx/snippets/loupit-security.conf; add_header Cache-Control "public, max-age=300" always; try_files /privacy.html    =404; }

    # ── API 프록시 (익명 읽기 GET 5종 + 익명 비교 로그 POST 1종; SC14 기여 쓰기는 별도) ──

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 3s;
        proxy_read_timeout 15s;
        proxy_buffering on;
        # Cache-Control은 앱이 설정(reference/all: public, max-age=3600 — FR-92). Nginx add_header 미개입.
    }

    # 그 외 경로 → 정적 404(생성물)
    location / {
        include /etc/nginx/snippets/loupit-security.conf;
        try_files $uri $uri/ =404;
    }
    error_page 404 /404.html;
    location = /404.html {
        root /home/ubuntu/loupit/web/dist;
        include /etc/nginx/snippets/loupit-security.conf;
        add_header Cache-Control "no-store" always;
        internal;
    }
}
```

### [SP-INFRA-3.3] `infra/nginx/snippets/loupit-security.conf`(정본)

`add_header`는 하위 location에 자체 `add_header`가 있으면 상속되지 않으므로, 공통 헤더를 `include`로 각 location에 주입한다(레거시가 헤더를 매 location 중복 선언한 문제를 include로 해소).

```nginx
# 보안·전송 헤더 (HSTS 포함) — 각 location에서 include
# HSTS max-age=15768000(6개월). 운영 안정 후 31536000 + includeSubDomains + preload로 격상 권장.
add_header Strict-Transport-Security "max-age=15768000; includeSubDomains" always;   # NFR23
add_header X-Content-Type-Options    "nosniff" always;
add_header X-Frame-Options           "DENY" always;
add_header Referrer-Policy           "strict-origin-when-cross-origin" always;
add_header Permissions-Policy        "geolocation=(), microphone=(), camera=()" always;
# CSP는 AdSense/affiliate 도메인 허용이 필요하므로 SP-ADS(09)와 합의 후 확정. 초기엔 미설정(과차단 방지).
```

- **CSP 주의**: AdSense는 다수 서드파티 스크립트/프레임을 로드하므로 엄격한 CSP는 광고를 깨뜨린다. CSP 정본은 SP-ADS(09)의 허용 도메인 목록 확정 후 별도 스니펫으로 추가한다(본 문서는 자리만 명시).
- **HTTP/2**: `http2 on;`(nginx ≥1.25) 또는 `listen 443 ssl http2;`(≤1.24). 설치 버전에 맞춰 택1(SP-ARCH-7: Nginx ≥1.24).
- **추적**: NFR2·NFR3·NFR5·NFR12·NFR23·NFR28·NFR29, SP-ARCH-2, SP-GEN-11.

### [SP-INFRA-3.4] SC14 참여 경로 nginx 계약 (`loupit_mail` 존 · `X-Loupit-Client` CSRF 게이트)

**근거**: SP-AUTH-11·12(메일러·CSRF·발송/시도 리밋), FR-112(발송 rate limit)·FR-113(CSRF 헤더·세션 전송), NFR31, INV-9. 실파일 `infra/nginx/loupit.conf`(Layer A/B·limits — §3.2 드리프트 주석)와 `conf.d/loupit-limits.conf`에 **추가**된다(재작성 아님, 기존 존·게이트와 동거).

SC14 기여 경로는 익명 읽기 인프라 위에 두 계약을 얹는다: (1) 메일 발송 엔드포인트에 전용 rate-limit 존(`loupit_mail`), (2) 상태변경 쓰기에 `X-Loupit-Client` 헤더 게이트 — 후자는 **이미 존재하는 Layer A가 `/api/v1/` 전역에 강제**하므로 SC14 쓰기는 이를 상속한다. 세션·코드 시도·편집 상한은 앱계층(SP-AUTH-12: 쿠키 속성·`code_max_attempts`=5·`daily_edit_limit`=30)이 소유하고, nginx는 **거친 남용 상한**만 담당한다(nginx 게이트 + 앱 이중검사, INV-9).

#### [SP-INFRA-3.4.1] `loupit_mail` rate-limit 존 (`conf.d/loupit-limits.conf`에 추가)

메일 발송(`POST /members/login-code` FR-102 · `POST /employment/verify-code` FR-105)은 계정당 실비용(SMTP 전송)·남용 벡터(스팸·계정 열거 프로빙)라 일반 `loupit_api`(300r/m)보다 강한 존을 둔다. `limit_req_zone`은 **http 컨텍스트 전용** 지시자라 `server{}` 안에 둘 수 없어 기존 두 존과 같은 `loupit-limits.conf`(conf.d)에 정의한다(동일 관례 — conf.d는 sites-enabled보다 먼저 include되어 존이 사용보다 앞서 로드됨).

```nginx
# conf.d/loupit-limits.conf 의 loupit_api·loupit_log 옆에 추가 (SP-AUTH-12, FR-112)
# 메일 발송 전용: 3r/m(≈20s당 1건). 키=$binary_remote_addr(IP). 정상 로그인/인증은 세션당
# 소수라 무영향, 자동 스팸·열거 루프는 상한. 계정별 정밀 제어는 앱 재전송 쿨다운(60s)이 담당.
limit_req_zone $binary_remote_addr zone=loupit_mail:10m rate=3r/m;
```

#### [SP-INFRA-3.4.2] 메일 발송 전용 location (`=` 정확매칭 — `^~ /api/v1/`보다 우선)

기존 `= /api/v1/comparisons/log`·`= /api/v1/health` 블록과 동일 관례로 메일 발송 두 엔드포인트에 전용 `=` 블록을 둔다. `=` location은 상위 `^~ /api/v1/`의 `proxy_*`·헤더 게이트를 **상속하지 않으므로 명시 반복**한다(로그/헬스 블록과 동기 유지 규칙 그대로). 로그 POST(`sendBeacon`이라 커스텀 헤더 불가 → 게이트 예외)와 달리, 메일 발송은 프론트 `apiSend`(일반 fetch)가 `X-Loupit-Client`를 붙이므로 **게이트를 재선언**한다.

```nginx
# 메일 발송(SC14) — 일반 리밋 + 강한 메일 리밋 이중 적용. CSRF/스크래핑 헤더 게이트 재선언(= 미상속).
location = /api/v1/members/login-code {
    include /etc/nginx/snippets/loupit-security.conf;
    if ($http_x_loupit_client = "") { return 403; }   # Layer A 재선언(= 는 ^~ 미상속)
    limit_req zone=loupit_api  burst=60 nodelay;       # 일반 리밋(로그/헬스 블록과 동일)
    limit_req zone=loupit_mail burst=2  nodelay;       # 3r/m + 소버스트(정상 재시도 흡수)
    proxy_pass http://127.0.0.1:8000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_connect_timeout 3s;
    proxy_read_timeout 15s;
    proxy_buffering on;
}
location = /api/v1/employment/verify-code {
    include /etc/nginx/snippets/loupit-security.conf;
    if ($http_x_loupit_client = "") { return 403; }
    limit_req zone=loupit_api  burst=60 nodelay;
    limit_req zone=loupit_mail burst=5  nodelay;       # 3r/m + 큰 버스트: 회사 NAT 동시 온보딩 흡수(§3.4.4 #1)
    proxy_pass http://127.0.0.1:8000;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_connect_timeout 3s;
    proxy_read_timeout 15s;
    proxy_buffering on;
}
```

- **응답은 소형**(균일 204/오류 envelope)이라 `reference/all`용 대형 `proxy_buffers`(§3.2 실파일 (d))는 불필요 — 로그/헬스 `=` 블록과 동일하게 생략한다.
- **`=` 우선순위**: nginx는 정확매칭(`=`)을 접두(`^~`)보다 먼저 확정하므로, 두 엔드포인트는 위 블록으로 매칭되고 아래 `^~ /api/v1/`(일반)로 내려가지 않는다.

#### [SP-INFRA-3.4.3] SC14 쓰기·인증 라우트의 `X-Loupit-Client` 게이트

메일 발송 2종을 제외한 SC14 라우트(`POST /members/login`·`POST /members/logout`·`POST/PUT /companies/{id}/benefits`·`POST /employment/verify|requests`·`GET/PUT/DELETE /members/me`·`GET /companies/{id}/edits`)는 별도 `=` 블록이 없어 **기존 `^~ /api/v1/` 블록으로 매칭**되고, 그 안의 Layer A(`if ($http_x_loupit_client = "") return 403`)가 이미 헤더를 강제한다(FR-100 12라우트 중 메일 2종 제외 10종 전부 커버). 따라서:

- **상태변경 쓰기(POST/PUT/DELETE)의 CSRF 헤더 요구(FR-113)는 엣지에서 이미 충족**된다 — 프론트 `apiSend`가 헤더를 붙이고, 헤더 없는 크로스사이트/맨-curl 쓰기는 403. 앱계층 `require_member`·CSRF 재검사가 이중 방어(INV-9, **미들웨어 아님** — `app.user_middleware==['CORSMiddleware']` 불변).
- **경로변수 라우트**(`/companies/{id}/...`)는 `^~ /api/v1/`의 `loupit_api`(300r/m)만 받고, 세밀한 편집 상한은 앱 `daily_edit_limit`=30(429)이 담당(SP-AUTH-12, FR-112).
- **`GET /companies/{id}/edits`**(공개 열람, 세션 불요 FR-110)도 `^~ /api/v1/` 경유라 `X-Loupit-Client`를 요구한다 — 사이트 UI(`apiFetch`)가 헤더를 붙여 정상 열람은 통과하고, 익명 벌크 스크래핑만 차단(익명 GET 정책과 정합). 응답 캐시는 앱이 `no-store`(FR-110).
- **`GET /members/me`**(FR-104)는 credentialed(쿠키 + 헤더) — Layer A 통과 후 앱 세션 검증. `Path=/api/v1` 쿠키라 정적 경로엔 미전송(NFR16, SP-AUTH-12 규칙2).

메서드 단위로 GET을 게이트에서 빼지 않는 이유: 익명 데이터 GET에도 Layer A를 적용하는 것이 기존 스크래핑 방어 정책이며 SC14가 이를 바꾸지 않는다(프론트가 모든 `/api/v1/` 요청에 헤더를 붙이므로 사용자 영향 0).

#### [SP-INFRA-3.4.4] 검증·한계(정직 서술)

1. `loupit_mail` 키는 `$binary_remote_addr`(IP)다. 공유 NAT 뒤 다수 사용자가 동시에 코드를 요청하면 오탐 가능성이 있다(3r/m/IP). 특히 **재직 인증(`verify-code`)은 구조적으로 회사 egress IP에 집중**된다 — 회사 도메인 이메일로 코드를 받으므로 요청자가 그 회사 네트워크에 있을 확률이 높고 동시 온보딩 시 한 NAT를 공유한다. 그래서 `verify-code`는 버스트를 크게(5) 잡아 소규모 동시 온보딩을 흡수하고 로그인(`login-code`)은 버스트 2로 조인다. 어느 경로든 계정별 정밀 제어는 앱 쿨다운(`mail_resend_cooldown_sec`=60)이 담당하며, 초과 시 `429`는 우아하게 저하된다(앱 미발송·사용자 잠시 후 재시도). 정상 사용은 IP당 분당 3건 내외라 지속 트래픽엔 무영향이다.
2. `X-Loupit-Client` 값은 비밀이 아니다(공개 JS 노출). CSRF 방어의 본질은 **커스텀 헤더는 크로스오리진에서 preflight 없이 못 붙고**, preflight는 CORS 허용목록 + `allow_credentials=false`에서 실패한다는 것이다(SP-AUTH-12 규칙1). 엣지 헤더 게이트는 여기에 "게으른 봇·맨-curl 차단"을 더하는 층이다(정직한 한계).
3. nginx `429`(리밋)·`403`(헤더)는 앱에 닿기 전 차단이라 앱 로그에 남지 않는다. 앱계층 재검사(403/429)가 정본 감사 로그를 남긴다.
4. 이 계약은 SC14 배포 시 유효하다. 현 익명 배포엔 메일 발송·SC14 쓰기 라우트가 없어 `loupit_mail` 존·메일 `=` 블록은 **비활성(추가되어도 no-op)**이며, Layer A는 이미 익명 목적으로 라이브다(배포는 AdSense 게이트 홀드, 핸드오프 §E·SPEC/09).

**추적**: SC14 / FR-102·FR-105·FR-112·FR-113 / NFR31·INV-9 / SP-AUTH-11·12 / SP-INFRA-3.1·3.2(실파일 Layer A·limits) · 스모크 SM-16·SM-17(§11.2).

## SP-INFRA-5 — systemd 서비스 유닛 (uvicorn 자동재시작)

**근거**: CONTEXT_BRIEF §9(systemd), SP-ARCH-2(App 계층), NFR27(슬림 상시 서비스), SP-API-2(`server.main:app`·설정 키). 레거시 `job_change/server/deploy/jobchoice.service` 계승(모듈 경로·EnvironmentFile·하드닝 추가).

### [SP-INFRA-5.1] `infra/systemd/loupit-api.service`(정본)

```ini
[Unit]
Description=loupit read-only API (FastAPI/Uvicorn)
After=network.target mysql.service
Wants=mysql.service

[Service]
Type=simple
User=ubuntu
Group=ubuntu
# 리포 루트에서 실행 → `server` 패키지 import 가능(server.config 등, SP-API-1)
WorkingDirectory=/home/ubuntu/loupit
Environment=PATH=/home/ubuntu/loupit/server/venv/bin
# 비밀·설정은 파일에서 주입(웹 서빙 경로 밖). '-'로 부재 허용(개발). SP-INFRA-7·SP-API-2
EnvironmentFile=-/home/ubuntu/loupit/server/.env
ExecStart=/home/ubuntu/loupit/server/venv/bin/uvicorn server.main:app --host 127.0.0.1 --port 8000
ExecReload=/bin/kill -HUP $MAINPID
Restart=always
RestartSec=5
TimeoutStopSec=10
KillSignal=SIGINT

# ── 프로세스 하드닝(읽기전용 서비스에 맞춤) ──
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ProtectHome=read-only            # /home는 읽기·실행만(venv 실행 가능, 쓰기 불가)
ProtectKernelTunables=true
ProtectControlGroups=true
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX
RestrictNamespaces=true
LockPersonality=true

# 로그 → journald
StandardOutput=journal
StandardError=journal
SyslogIdentifier=loupit-api

[Install]
WantedBy=multi-user.target
```

### [SP-INFRA-5.2] 배치·기동·검증

```bash
sudo cp infra/systemd/loupit-api.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now loupit-api
systemctl is-active loupit-api          # → active
curl -s 127.0.0.1:8000/api/v1/health    # → {"status":"ok"} (FR-91)
```

- **자동재시작 계약**: `Restart=always`·`RestartSec=5` — 앱 크래시·OOM 시 5초 후 재기동. `WorkingDirectory=리포 루트`가 핵심(레거시는 `server/`였으나 loupit은 `server.` 패키지 import 형태라 루트에서 실행).
- **바인딩 강제**: `--host 127.0.0.1`로 loopback 고정(INV-7·NFR22). `0.0.0.0` 금지.
- **EnvironmentFile vs dotenv**: systemd가 `server/.env`의 `KEY=VALUE`를 프로세스 환경으로 주입한다. `pydantic-settings`는 프로세스 환경을 우선 읽으므로 CWD와 무관하게 동작한다(SP-API-2의 `env_file`은 개발용 폴백).
- **추적**: §9, SP-ARCH-2, SP-API-2, NFR22·NFR27.

## SP-INFRA-6 — MySQL 설정 (utf8mb4 · 리소스 최적화 · loopback)

**근거**: NFR29(utf8mb4)·NFR27(Always Free 자원)·NFR22(외부차단), SP-DB(문자셋·ngram)·SP-API-2(계정), INV-7.

### [SP-INFRA-6.1] `/etc/mysql/mysql.conf.d/loupit.cnf`(정본)

읽기 위주 소규모 참조 DB(회사 ~96 · 복지 행 ~1,330, SP-SEED)에 맞춘 보수적 튜닝. 비밀값은 포함하지 않는다.

```ini
[mysqld]
# ── 네트워크: loopback 전용(외부차단, INV-7·NFR22) ──
bind-address           = 127.0.0.1
skip-name-resolve      = ON

# ── 문자셋(NFR29): utf8mb4 전면 ──
character-set-server   = utf8mb4
collation-server       = utf8mb4_0900_ai_ci

# ── FULLTEXT ngram(한글 부분검색, SP-DB-1.8) ──
ngram_token_size       = 2          # 비동적 → 설정파일 필수. 기본 2 명시

# ── 리소스 최적화(6GB RAM · nginx/uvicorn 공존, NFR27) ──
innodb_buffer_pool_size        = 512M   # 데이터셋 극소 → 512M로 전체 캐시. RAM 상향 시 재조정
innodb_buffer_pool_instances   = 1
innodb_log_file_size           = 128M
innodb_flush_method            = O_DIRECT
innodb_flush_log_at_trx_commit = 2      # 쓰기는 배치 시드 뿐(재현 가능) → 지연 flush 허용
max_connections                = 50     # uvicorn 풀 max=10(SP-API-2)로 충분
table_open_cache               = 400
thread_cache_size              = 16
performance_schema             = OFF    # 상시 메모리 절감(운영 read-only). 진단 시 임시 ON

# ── 로깅 ──
slow_query_log         = ON
slow_query_log_file    = /var/log/mysql/slow.log
long_query_time        = 1
```

- 적용: `sudo systemctl restart mysql` → `SHOW VARIABLES LIKE 'character_set_server';` == `utf8mb4`, `bind_address` == `127.0.0.1`.
- **`innodb_flush_log_at_trx_commit=2` 근거**: 런타임은 읽기 전용(NFR20)이라 내구성 손실 위험 있는 쓰기가 없다. 유일한 쓰기는 재시드(SP-SEED)이며 소스에서 언제든 재현 가능하므로 지연 flush가 안전(성능·SSD 수명 이점).

### [SP-INFRA-6.2] DB·계정 프로비저닝(최소권한, NFR22)

**현 익명 배포**의 런타임 계정(`loupit`)은 **전 테이블 `SELECT` + 익명 비교 로그(`TCOMPARE_LOG`) 한정 INSERT·DELETE**만 가진다(defense-in-depth·최소권한, NFR20 — INSERT는 POST `/comparisons/log`, DELETE는 보존 퍼지, INV-1 개정 2026-07-14). 시드/DDL은 별도 특권 계정으로 빌드타임에만 사용. **SC14 기여 배포**는 이 런타임 계정에 **참여 테이블 한정 쓰기**를 추가한다(참조 테이블은 SELECT 유지 — §6.2a).

> **스키마명 = 대문자 `LOUPIT`**(실서빙 스키마, `server/.env` `DB_NAME=LOUPIT`, `schema.sql`). 배포 호스트가 `lower_case_table_names=0`이라 `loupit`≠`LOUPIT`이므로 **소문자 프로비저닝 금지**(소문자로 GRANT하면 서버 접속 스키마와 어긋나 GRANT은 조용히 성공하되 런타임은 여전히 `command denied` — 2026-07-17 감사 #6·#13). **정본 = `infra/mysql/provision_accounts.sql`**(실파일; MySQL 사용자명 `'loupit'`은 소문자, 스키마 `LOUPIT`은 대문자 — 구분 유지). 그 파일은 라이브 드리프트(#13: 현재 단일 `APP_LOUPIT` GRANT ALL 계정이 런타임·DDL 겸용, 최소권한 미이행)도 정직히 기록한다 — 본 절·§6.2a는 도달 목표(최소권한) 상태를 정의한다.

```sql
CREATE DATABASE IF NOT EXISTS LOUPIT
  CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;

-- 런타임(앱): 전 테이블 읽기 + 익명 비교 로그 한정 쓰기. server/.env엔 이 자격만.
CREATE USER IF NOT EXISTS 'loupit'@'127.0.0.1' IDENTIFIED BY '<RUNTIME_PW>';
GRANT SELECT ON LOUPIT.* TO 'loupit'@'127.0.0.1';
GRANT INSERT, DELETE ON LOUPIT.TCOMPARE_LOG TO 'loupit'@'127.0.0.1'; -- 익명 로그 쓰기·보존 퍼지(SC14 참여 쓰기는 §6.2a)

-- 시드/DDL(빌드타임 전용, .env와 분리 보관): DDL+DML — release.sh·backup.sh가 사용
CREATE USER IF NOT EXISTS 'loupit_seed'@'127.0.0.1' IDENTIFIED BY '<SEED_PW>';
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE, ALTER, DROP, INDEX, REFERENCES
  ON LOUPIT.* TO 'loupit_seed'@'127.0.0.1';
FLUSH PRIVILEGES;
```

- 계정 인증 플러그인이 `caching_sha2_password`면 aiomysql에 `cryptography`가 필요(SP-ARCH-7 조건부). `mysql_native_password`면 생략 가능.
- 런타임 `.env`에는 **`loupit` 자격증명만** 넣는다(익명 배포=SELECT+로그쓰기, SC14 배포=+참여 쓰기 §6.2a). `loupit_seed` 자격은 배포 오퍼레이터의 릴리스 환경에만 두고 서버 상주 `.env`에 저장하지 않는다.
- **추적**: NFR20·NFR22·NFR29, SP-DB, SP-SEED-9, SP-API-2.

### [SP-INFRA-6.2a] SC14 런타임 쓰기 그랜트 · 운영자 CLI 계정 (기여 배포 시)

**근거**: SP-AUTH-4·5·6·7·8·9·10(참여 쓰기), SP-DB-17(7테이블), FR-115(운영자 CLI), NFR20·NFR22. **SC14 재명세(2026-07-22)** — 최소권한 목표 설계(§6.2: 전 테이블 SELECT + `TCOMPARE_LOG` 로그 쓰기)에서는 SC14 기여가 기동해도 **첫 참여 쓰기**에서 `command denied`로 전 기여가 실패하므로, 참여 테이블 한정 쓰기를 익명 `TCOMPARE_LOG` 쓰기 위에 증분으로 반드시 추가한다.

SC14 기여 흐름은 런타임 앱(`loupit` 계정, SP-AUTH-4 `database.py`의 `execute`/`transaction` 헬퍼)이 **참여 테이블에 쓴다** — 세션 발급/폐기(`TSESSION`)·로그인/인증 코드(`TMEMBER`·`TAUTH_CODE`)·재직 인증(`TEMPLOY_VERIFICATION`·`TEMPLOY_VRF_REQUEST`)·복지 편집·이력(`TCOMPANY_BENEFIT`·`TBENEFIT_EDIT_LOG`). 아래를 **테이블 스코프**로 추가하되 참조/열람 테이블은 SELECT-only를 유지해 최소권한을 보존한다(전역 `GRANT ... ON LOUPIT.*` 금지 — 손상된 앱이 참조 데이터를 훼손 못 하게):

```sql
-- SC14 기여 배포 시 런타임 'loupit' 계정에 참여 테이블 한정 쓰기 추가 (스키마 LOUPIT 대문자; 참조 테이블은 위 GRANT SELECT 그대로)
GRANT INSERT, UPDATE, DELETE ON LOUPIT.TSESSION             TO 'loupit'@'127.0.0.1'; -- 발급(INSERT)·폐기 REVOKED_DTM(UPDATE)·만료 retention purge(DELETE, session.py)
GRANT INSERT, UPDATE, DELETE ON LOUPIT.TAUTH_CODE           TO 'loupit'@'127.0.0.1'; -- 발급(INSERT)·시도갱신 ATTEMPT_CNT·무효화 CONSUMED_DTM(UPDATE)·purge(DELETE)
GRANT INSERT, UPDATE         ON LOUPIT.TMEMBER              TO 'loupit'@'127.0.0.1'; -- 가입(INSERT)·닉네임·탈퇴 LOGIN_EMAIL_NM=NULL(UPDATE). DELETE 불요(이력 존치)
GRANT INSERT, UPDATE         ON LOUPIT.TEMPLOY_VERIFICATION TO 'loupit'@'127.0.0.1'; -- 생성(INSERT)·탈퇴 폐기 REVOKED_DTM·HMAC 파기 NULL(UPDATE). DELETE 불요(소프트 폐기, P7)
GRANT INSERT                 ON LOUPIT.TEMPLOY_VRF_REQUEST  TO 'loupit'@'127.0.0.1'; -- 수동 승인 요청 제출(승인은 운영자 CLI)
GRANT INSERT                 ON LOUPIT.TBENEFIT_EDIT_LOG    TO 'loupit'@'127.0.0.1'; -- append-only(UPDATE/DELETE 미부여, SC-4)
GRANT INSERT, UPDATE         ON LOUPIT.TCOMPANY_BENEFIT     TO 'loupit'@'127.0.0.1'; -- 복지 등록·수정(사용자 DELETE 없음, FR-115)
FLUSH PRIVILEGES;
```

- **`TCOMPANY_EMAIL_DOMAIN`은 SELECT-only 유지** — 도메인 화이트리스트는 시드/운영자 관리(런타임 무쓰기).
- **`TBENEFIT_EDIT_LOG` = INSERT만**(UPDATE/DELETE 미부여) — 편집 이력 불변(SC-4·SP-AUTH-10).
- **`TCOMPANY_BENEFIT` DELETE 미부여** — v1 사용자 삭제 없음(운영자 CLI 전용, FR-115).
- **소프트 폐기 원칙**: 로그아웃·탈퇴·재직 인증 폐기·코드 무효화는 전부 `REVOKED_DTM`/`CONSUMED_DTM`/`LOGIN_EMAIL_NM=NULL`(UPDATE)다(SP-AUTH-4·6·7). 하드 `DELETE`는 세션·코드의 **만료 retention purge**(session.py, SP-AUTH-4)뿐이라 `TSESSION`·`TAUTH_CODE`에만 부여한다 — `TMEMBER`·`TEMPLOY_VERIFICATION`·`TBENEFIT_EDIT_LOG`는 감사·이력 무결성으로 DELETE 없음.
- **운영자 CLI(`python -m server.ops`, SP-AUTH-8·FR-115)**: 재직 수동 승인(INSERT `TEMPLOY_VERIFICATION`)·복지 삭제(DELETE `TCOMPANY_BENEFIT` + INSERT `TBENEFIT_EDIT_LOG`) 등 특권 DML은 런타임 `loupit`가 아니라 **`loupit_seed`(DML) 자격**으로 실행한다(운영자 릴리스 환경 자격, 서버 상주 `.env` 미저장). **운영자 인증 = 제한된 SSH/셸 접근**(SP-INFRA-8)으로 대체하며 앱 세션·별도 인증을 두지 않는다(FR-115 규칙3).
- 이 그랜트는 **SC14 배포 시** 적용(AdSense 게이트 후, 핸드오프 §E). 현 익명 배포는 참여 테이블 자체가 없어 무효(no-op).
- **추적(SC14 증분)**: SP-DB-17 · SP-AUTH-4·5·6·7·8·9·10 / FR-115 / NFR20·NFR22 / SP-INFRA-8(운영자 셸).

### [SP-INFRA-7.1] `server/.env`(비상주 비밀, `600`)

현 익명 배포는 SP-API-2 키만 포함한다. 비밀번호·소셜·JWT 키는 **영구 정의 금지**(부재 자체가 계약, NFR16·SC10). **SC14 기여 배포**는 아래 **메일러·세션 TTL·pepper 키(SP-AUTH-2)를 추가**한다 — `SMTP_*`·`SESSION_*`·`*_PEPPER`는 SC14로 정당한 키가 되어 NFR16 금지 스코프(익명 배포)에서 **해제**된다(config 금지 substring 재명세 T10·SP-AUTH-2 규칙2와 정합: 금지목록을 넓히는 게 아니라 좁히는 개정). ## SP-INFRA-8 — 방화벽 · 보안 (SSH 제한 · MySQL/uvicorn 외부차단)

**근거**: NFR22·NFR23, INV-7, AS1.

### [SP-INFRA-8.1] 2계층 네트워크 차단

| 계층 | 위치 | 정책 |
| --- | --- | --- |
| 클라우드 | OCI **Security List / NSG**(VCN ingress) | 인바운드 허용 = `22/tcp`(관리 소스 CIDR 한정), `80/tcp`(0.0.0.0/0), `443/tcp`(0.0.0.0/0). **`3306`·`8000` 규칙 부재(암묵 차단)** |
| 호스트 | `iptables`/`nftables`(Ubuntu 기본 + 명시 규칙) | 동일: `22`(관리 IP), `80`, `443` allow, established 유지, 그 외 drop |
| 바인딩 | 프로세스 | MySQL·uvicorn = `127.0.0.1`(SP-INFRA-5·6) — 방화벽 실수와 무관한 최종 방어선 |

- OCI Ubuntu 이미지는 기본 `iptables` 규칙을 탑재한다. 신규 포트(80/443)는 OCI Security List **와** 호스트 방화벽 **양쪽**에 열어야 트래픽이 통과한다. `3306`/`8000`은 어느 쪽에도 열지 않는다(외부차단 = INV-7·NFR22).
- 검증: 외부 호스트에서 `nc -vz <PUBLIC_IP> 3306` / `8000` → 연결 실패/타임아웃(SP-INFRA-11 SM-8).

### [SP-INFRA-9.3] 프론트 자산 캐시 계약 (2026-07-19 개정 — `stamp_assets.sh` 폐기)

**구판의 결함**: `?v=` sha8 스탬프 + 1년 immutable 계획은 두 번 실패했다. (a) HTML 측 `?v=PLACEHOLDER` 규약(SP-FE 위임)이 끝내 구현되지 않아 sed 치환이 조용한 무동작이었고(TASK/11 기지 이슈 #1), (b) 구현됐더라도 ES모듈 하위 import(`./ui.js` 등)는 진입점 쿼리로 URL이 바뀌지 않아 immutable 캐시를 못 뚫는다. 그 결과 2026-07-17 컷오버 이후 방문자에게 구 자산이 최대 1년 잔존하는 사고가 실제 발생했다(2026-07-19 배포에서 확인).

**개정 계약** (구현: `infra/nginx/loupit.conf`의 `^~ /assets/v2/` 블록):

1. 공개 자산 URL은 `/assets/v2/…` 단일 세대(nginx `alias` → `web/assets/`, 파일 이동 없음). HTML·CSS `url()`·JS 절대경로 fetch가 전부 이 세대를 참조하고, ES모듈 상대 import는 importer URL을 상속하므로 모듈 전체가 자동으로 같은 세대다.
2. 캐시는 `Cache-Control: no-cache` — 매 요청 조건부 재검증(변경 없으면 304). 릴리스에 자산 전용 단계가 필요 없다(파일 저장 = 배포).
3. 구 `/assets/` 경로(확장자 매칭)는 하위호환으로 no-cache 서빙 유지 — v2 이전 HTML을 캐시한 방문자·크롤러용. 기오염(immutable) 사용자 구제는 URL 세대 이동이 전담한다.
4. **불변식: `web/assets`에 `.gz` 사전압축 금지.** `gzip_static`은 사이드카와 원본의 신선도를 비교하지 않아 no-cache 재검증을 조용히 무력화한다(v2 location은 `gzip_static off`, 압축은 런타임 gzip 전용). 생성물 `web/dist`의 `.html.gz`(SP-GEN 소유)는 매 빌드 재생성이므로 무관.
5. 라이브 검증: smoke.sh SM-9(200+gzip 동시 검사 — 404 폴백도 gzip이라 상태코드 필수)·SM-15a~d(v2 200+no-cache, 구경로 강등, HTML 참조).

- **추적**: SP-ARCH-9, SP-GEN-1.4·11, AS2·AS3, NFR2·NFR5, 감사 2026-07-17 미해소 ⑤.

### [SP-INFRA-10.2] 백업 (`infra/deploy/backup.sh`)

참조 DB는 소스(시드 SQL, `job_change` 큐레이션)에서 재현 가능하므로 백업은 보조 안전장치다. 그래도 일일 mysqldump로 회복 시간 단축.

```bash
#!/usr/bin/env bash
set -euo pipefail
TS="$(date +%Y%m%d)"
OUT="/var/backups/loupit/loupit-${TS}.sql.gz"
mkdir -p /var/backups/loupit
mysqldump --single-transaction --default-character-set=utf8mb4 \
  -u loupit_seed -p"${SEED_PW:?}" loupit | gzip -9 > "${OUT}"
# 14일 보관
find /var/backups/loupit -name 'loupit-*.sql.gz' -mtime +14 -delete
```

- systemd timer(`loupit-backup.timer`, 일 1회) 또는 cron으로 스케줄. `SEED_PW`는 오퍼레이터 환경변수(서버 상주 `.env`에 미저장, SP-INFRA-6.2).
- **추적**: NFR27, AS1·AS13.

### [SP-INFRA-11.2] 라이브 스모크 (`smoke.sh`)

기본 대상 `BASE=https://jobcho.wiki`(로컬 스택은 `http://127.0.0.1`). 각 케이스는 종료코드로 성공/실패를 판정한다.

| ID | 케이스 | 명령(요지) | 기대 | 근거 |
| --- | --- | --- | --- | --- |
| SM-1 | 랜딩 정적 | `curl -sI $BASE/` | `200`, `content-type: text/html`(비-JS 본문 존재) | NFR1·NFR12 |
| SM-2 | HTTP→HTTPS | `curl -sI http://jobcho.wiki/` | `301` → `https://jobcho.wiki/` | NFR23 |
| SM-3 | API 헬스 | `curl -s $BASE/api/v1/health` | `200`, `{"status":"ok"}` | FR-91·T8 |
| SM-4 | 참조 번들 캐시헤더 | `curl -sI $BASE/api/v1/reference/all` | `200`, `cache-control: public, max-age=3600` | NFR3·FR-92·T8 |
| SM-5 | 회사 상세 정적 | `curl -sI $BASE/company/<slug>` | `200`(정적, `/api/v1` 미호출) | NFR1·F4·T8 |
| SM-6 | 정책 페이지 | `curl -sI $BASE/privacy` | `200` | F7 |
| SM-7 | HTTP/2 | `curl -sI --http2 $BASE/` | `HTTP/2 200` | §9(HTTP2) |
| SM-8 | HSTS 헤더 | `curl -sI $BASE/ \| grep -i strict-transport-security` | `max-age=15768000; includeSubDomains` | NFR23 |
| SM-9 | gzip_static | `curl -sI -H 'Accept-Encoding: gzip' $BASE/assets/js/app.js?v=<h>` | `content-encoding: gzip` | NFR2·SP-GEN-11 |
| SM-10 | TLS 유효성 | `echo \| openssl s_client -connect jobcho.wiki:443 -servername jobcho.wiki 2>/dev/null \| openssl x509 -noout -dates` | 유효기간 내, 체인 정상 | NFR23 |
| SM-11 | MySQL 외부차단 | (외부 호스트)`nc -vz <PUBLIC_IP> 3306` | 연결 실패/타임아웃 | NFR22·INV-7 |
| SM-12 | uvicorn 외부차단 | (외부 호스트)`nc -vz <PUBLIC_IP> 8000` | 연결 실패/타임아웃 | NFR22·INV-7 |
| SM-13 | 404 처리 | `curl -sI $BASE/nonexistent-xyz` | `404`(정적 `404.html`) | FR-E6 |
| SM-14 | 앱 자동재시작 | `sudo pkill -f 'uvicorn server.main'; sleep 7; curl -s $BASE/api/v1/health` | `RestartSec=5` 후 `200` | SP-INFRA-5 |
| SM-16 | CSRF 게이트(SC14) | `curl -s -o /dev/null -w '%{http_code}' -X POST $BASE/api/v1/members/login-code -d '{}'`(헤더 없음) | `403`(Layer A·§3.4.2) | FR-113·SP-AUTH-12 |
| SM-17 | 메일 발송 리밋(SC14) | 헤더 포함 `POST /members/login-code` 5회 연속(>3r/m) | 초과분 `429`(`loupit_mail`·§3.4.1) | FR-112·NFR31 |

- **SM-16은 조건부 아님 — 오늘도 통과**: 헤더 없는 POST는 현 익명 배포에서 `=` 블록이 없어 `^~ /api/v1/`로 매칭되고 Layer A(`if ($http_x_loupit_client = "")`)가 앱 도달 전 `403`을 반환한다(SC14 배포 후엔 `=` 블록의 재선언 게이트가 동일 `403`). 즉 SC14 라우트 부재와 무관하게 CSRF 헤더 게이트를 검증하므로 스킵하지 않는다(404가 아니라 403).
- **SM-17만 SC14-조건부**: 헤더를 포함하면 게이트를 통과해 앱 라우트에 도달하는데, 현 익명 배포엔 그 라우트가 없어 `404`다 — SC14 기여 배포(AdSense 게이트 후, 핸드오프 §E) 이후에야 `loupit_mail` `429`를 검증할 수 있다. 그때까지 `smoke.sh`는 **헤더 포함 프로브의 `404`**로 부재를 감지해 SM-17을 스킵한다. 단 `loupit_mail` 존·메일 `=` 블록을 앱 라우트보다 먼저 배포하면 5연속 요청이 404/429 혼재가 되므로, 존·블록은 **앱 라우트와 함께 배포**한다(선배포 금지). (자산 v2 검증 SM-9·SM-15a~d는 §9.3이 소유.)

### [SP-INFRA-11.3] `infra/deploy/smoke.sh`(골격)

```bash
#!/usr/bin/env bash
# SP-INFRA-11.2 라이브 스모크. 실패 시 비0 종료(릴리스 게이트).
set -uo pipefail
BASE="${BASE:-https://jobcho.wiki}"
fail=0
chk(){ # $1=이름 $2=조건(0/1 반환 명령)
  if eval "$2" >/dev/null 2>&1; then echo "  OK  $1"; else echo "  FAIL $1" >&2; fail=1; fi
}
code(){ curl -s -o /dev/null -w '%{http_code}' "$1"; }

chk "SM-1 landing 200"      "[ \"$(code $BASE/)\" = 200 ]"
chk "SM-2 http->https 301"  "[ \"$(curl -s -o /dev/null -w '%{http_code}' http://jobcho.wiki/)\" = 301 ]"
chk "SM-3 health ok"        "curl -s $BASE/api/v1/health | grep -q '\"status\":\"ok\"'"
chk "SM-4 ref cache-header" "curl -sI $BASE/api/v1/reference/all | grep -qi 'cache-control: public, max-age=3600'"
chk "SM-6 privacy 200"      "[ \"$(code $BASE/privacy)\" = 200 ]"
chk "SM-7 http2"            "curl -sI --http2 $BASE/ | grep -qi '^HTTP/2 200'"
chk "SM-8 hsts"             "curl -sI $BASE/ | grep -qi 'strict-transport-security'"
chk "SM-13 404"             "[ \"$(code $BASE/nonexistent-xyz)\" = 404 ]"
[ "$fail" = 0 ] && echo "SMOKE PASS" || { echo "SMOKE FAIL" >&2; exit 1; }
```

- **도구**: `curl`·`openssl`·`nc`(스모크), `nginx -t`·`systemd-analyze verify`(정적), `git grep`(시크릿 스캔). 외부차단(SM-11/12)은 외부 호스트에서 수행(로컬에선 loopback이라 열려 보임).
- **릴리스 게이트**: CFG-1~6 + SM-1~14 통과가 배포 완료 조건(§9.1 단계 4·7과 결합). SP-ARCH-10 T8을 SM-1·3·4·5로 구체화·확장한다.
- **추적**: 브리프 §10, SP-ARCH-10 T8, NFR1·NFR3·NFR12·NFR22·NFR23·NFR28·NFR29.
