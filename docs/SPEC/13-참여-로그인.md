# SPEC 13 — 참여·로그인·재직인증 (SC14)

> **신규(2026-07-21)**: "로그인 없음 = 영구 불변식"의 공식 개정. 익명 열람·비교는 불변(익명 GET·무쿠키·무세션)이고, **로그인 + 재직 인증은 복지 등록·수정 기여(SC14)에만** 요구한다. 경위·범위: `docs/PRD/04-범위.md`(SC14)·`docs/HANDOFF-로그인기능-ML-A.md`.

**영역 ID 대역**: `SP-AUTH`
**저장 경로**: `docs/SPEC/13-참여-로그인.md`
**목적**: loupit **참여(SC14)** 기능 — 무비밀번호 이메일 코드 로그인, 계정 관리·탈퇴, 회사 도메인 이메일 재직 인증, 복지 등록·수정, 편집 이력 공개 열람 — 의 **앱계층 구현 계약**을 개발자가 추측 없이 그대로 만들 수 있는 수준으로 확정한다. (1) 라우터/서비스/의존성/CLI 모듈 구조, (2) 세션 발급·검증·폐기(불투명 토큰·SHA-256 해시·쿠키·TTL·`require_member` 의존성), (3) 무비밀번호 코드 로그인 흐름과 닉네임 자동 생성, (4) 재직 인증(도메인 화이트리스트 매칭·HMAC·원문 파기·`require_employment`·수동 승인 폴백), (5) 복지 편집(배지 시맨틱 강제·낙관적 동시성 `base_dtm`·원자 트랜잭션·편집 이력 append), (6) 편집 이력 공개 조회, (7) 메일러 추상화(Console/SMTP)·발송/시도 리밋·CSRF(`X-Loupit-Client`), (8) `config` 필드·금지 substring 재명세, (9) 운영자 CLI(`server/ops`), (10) pytest + httpx 테스트 명세를 소유한다. **참여 7테이블 DDL은 소유하지 않고 SP-DB(SPEC 02)로 미루며, 컬럼 이름만 규약대로 참조한다.**

**상위 추적**: CONTEXT_BRIEF §2-1·§2-3·§2-4(비로그인·프로파일러 제외·서버 무쓰기 — SC14 예외 조정)·§8(스택)·§9(인프라), PRD `04-범위.md`(**SC14**)·`07-비기능-요구사항.md`(**NFR16·17·20·21·30·31**), FRD `13-참여-API.md`(**FR-100~FR-115** 전송 계약)·`01-개요와-FR규칙.md`(**FR-01** 전역 불변식·기여 예외), USECASE `09-참여.md`(**UC-70~UC-77**)·`01-액터와-개요.md`(**액터 A6** 인증 기여자), SPEC `01-개요와-아키텍처.md`(**INV-1·INV-8·INV-9·T2·T9·T10**)·`02-데이터베이스-스키마.md`(참여 7테이블 **DDL 정본**·`TCOMPANY_BENEFIT` 배지 컬럼)·`05-비교-계산엔진.md`(밴드 불변, **INV-5**)·`09-정책-페이지.md`(약관 **T5**(SP-POL 조항, ≠ SP-ARCH 검증지점 T5)·개인정보 **P7** — SPEC/09 SP-POL-4·SP-POL-3에 **신설 완료**)·`11-인프라-배포.md`(nginx `loupit_mail` 존·운영자 도구 — 현 SPEC/11 미정의, **SP-INFRA 재명세 대기**). 하위 TASK 문서는 `SP-AUTH-N` ID를 인용한다.

**범위 경계**: 본 문서는 SC14 기여 경로의 **런타임 앱계층·CLI 구현 계약**만 소유한다 — 세션 발급/검증/폐기, 코드·토큰·회사 이메일의 해시/HMAC 알고리즘, 라우트 의존성(`require_member`/`require_employment`), 메일러, `config` 필드, CSRF, 낙관적 동시성, 편집 이력 트랜잭션, 운영자 CLI. **참여 7테이블·`TCOMPANY_BENEFIT` 신규 배지값의 DDL 정본은 SP-DB(SPEC 02, ML-B)**, 밴드·판정·계산은 SP-ENGINE(`calc.js` 불변, 표시 계층만 확장), 약관 T5·개인정보처리방침 P7 **문안**은 SP-POL, nginx rate limit·메일 발송 존·백업/게이트는 SP-INFRA·SP-TEST가 소유한다(대역-문서 1:1, 중복 정의 금지 — 단 현 SPEC/11은 `SMTP_*`를 금지 키로 두고 `loupit_mail` 존이 없어 **SC14용 SP-INFRA 재명세 대기**, SP-DB·SP-POL과 동일 취급). 익명 열람·비교(F1~F8, FR-1x~9x)는 로그인 없이 그대로이며 본 문서 대상이 아니다.

**전역 불변식(SP-ARCH-10 상속)**: **INV-1**(익명 API 표면 = GET 5종 + 익명 비교 로그 POST 1종 + **열거된 SC14 기여 쓰기 라우트**, 그 외 쓰기 0, 인증/세션 미들웨어 0) · **INV-8**(회원 PII 최소 = 로그인 이메일 + 닉네임뿐, 회사 이메일·인증 코드·세션 토큰 원문 저장 0(해시/HMAC만), 검증 직후 원문 파기) · **INV-9**(세션 = 불투명 랜덤 토큰(DB엔 SHA-256 해시만) + `HttpOnly·Secure·SameSite=Lax` 쿠키, JWT·OAuth·비밀번호 미사용, 세션·재직 검증은 **미들웨어 아닌 라우트 의존성** — `app.user_middleware == ['CORSMiddleware']`).

## SP-AUTH-1 — 패키지·모듈 구조

**근거**: SP-ARCH-6(디렉토리·레거시 참조 L188: 라우터 `member`/`employment`/`benefit_edit` 신규 추가, `auth` 금지), FR-100(쓰기 라우트 한정 열거), T10(라우터 allowlist·`auth` 모듈 부재), FR-115(운영자 CLI `python -m server.ops`).

익명 읽기 전용 서버(SP-API)에 **기여(SC14) 앱계층**을 얹는다. 신규 모듈은 라우터 3종·서비스·의존성·메일러·운영자 CLI뿐이며, **인증/세션 미들웨어는 추가하지 않는다**(INV-9). 라우터 파일명 `auth`는 **금지**(레거시 잔재 방지, T10) — 로그인 라우터는 `member.py`를 쓴다.

```
server/
├─ main.py                    # SP-API-5  FastAPI 앱·lifespan. SC14 라우터 3종 추가 등록. 미들웨어 = CORS 1종 불변(INV-9)
├─ config.py                  # SP-AUTH-2  기존 + 메일·세션 TTL·HMAC pepper·mailer_mode. 금지 substring 재명세(T10)

├─ mailer.py                  # SP-AUTH-11  Mailer 추상화: ConsoleMailer | SmtpMailer (mailer_mode로 선택)
├─ routers/
│  ├─ health.py               # SP-API-8   (FR-91)
│  ├─ reference.py            # SP-API-9   (FR-92)

│  ├─ member.py               # SP-AUTH-5·6  로그인 코드·로그인·me·로그아웃·탈퇴 (FR-102~104). ← 파일명 auth 금지
│  ├─ employment.py           # SP-AUTH-7·8  재직 코드·재직 인증·수동 승인 요청 (FR-105~107)
│  └─ benefit_edit.py         # SP-AUTH-9·10 복지 등록·수정·편집 이력 (FR-108~110)
```

**검증·규칙**
1. `main.py`는 SC14 라우터 3종(`member`·`employment`·`benefit_edit`)을 `include_router`로 등록하되 **미들웨어는 추가하지 않는다** — `app.user_middleware == ['CORSMiddleware']` 불변(T2·T10, INV-9).
2. 라우터 모듈 allowlist(test_package 재명세)에 `member`·`employment`·`benefit_edit` **추가**, `FORBIDDEN_MODULE_NAMES`에 **`auth` 유지**(라우터명 `auth` 금지 → `member.py` 사용, T10).
3. 세션·재직 검증은 `deps.py`의 `require_member`·`require_employment(comp_id)` **의존성으로만** 주입한다(미들웨어 0, INV-9). 익명 GET 라우터의 dependant 트리에 `require_member`가 없어야 한다(T2·T10).
4. `ops.py`는 런타임 API와 분리된 **동기 CLI**(pymysql). 사용자 대면 DELETE 라우트는 두지 않는다(복지 삭제는 CLI 전용, FR-100·FR-115).

**추적**: SC14 / UC-70~77(전제) / INV-1·INV-9 / T2·T10 / FR-100·FR-115 / SP-ARCH-6·SP-API-5·SP-INFRA.

## SP-AUTH-2 — 설정 (`config.py`) 확장·금지 substring 재명세

**근거**: SP-ARCH-6 L153(config = DB·CORS·캐시 TTL + 메일·세션 TTL·해시 pepper, JWT/OAuth/비밀번호 키 없음), T10(`Settings` 필드명 금지 substring), FR-101(세션 TTL)·FR-102·FR-112(발송·시도 상한)·NFR30·NFR31.

기존 `Settings`(SP-API-2)에 참여 필드를 **추가**한다. 신규 필드는 모두 환경변수 주입이며, 어떤 필드명도 `jwt·oauth·password_reset·social` 부분문자열을 포함하지 않는다.

```python
# server/config.py  (SP-API-2 Settings 확장)
class Settings(BaseSettings):
    # ... (기존 DB·CORS·캐시 TTL 필드, SP-API-2) ...

    # --- 메일러 (SP-AUTH-11) ---
    mailer_mode: str = "console"        # ∈ {console, smtp}. 개발/미설정 기본 console(실발송 없음)
    smtp_host: str = ""                 # 예: smtp.naver.com
    smtp_port: int = 587
    smtp_user: str = ""                 # 비어있으면 mailer_mode=smtp라도 console로 폴백(SP-AUTH-11 규칙3)
    smtp_pass: str = ""
    smtp_from: str = ""                 # 발신 주소(표시명 포함 가능)

    # --- 세션·코드 TTL·시도 상한 (SP-AUTH-4·5, FR-101·112) ---
    session_ttl_days: int = 30          # 세션 만료(FR-101)
    login_code_ttl_min: int = 5         # 로그인/인증 코드 만료(FR-102·105, NFR31)
    code_max_attempts: int = 5          # 코드 검증 시도 상한(FR-112, NFR31)
    mail_resend_cooldown_sec: int = 60  # 재전송 쿨다운(FR-112)
    daily_edit_limit: int = 30          # 계정·회사당 일일 복지 편집 상한(FR-108·112)
    employ_vrf_ttl_days: int = 365      # 재직 인증 만료(FR-106)

    # --- 비밀값 pepper (SP-AUTH-4·7, NFR30) ---
    session_hash_pepper: str = ""       # 세션 토큰 SHA-256 pepper(선택). 미설정 시 순수 SHA-256
    comp_email_hmac_pepper: str = ""    # 회사 이메일 HMAC 키(필수, 미설정 시 기동 실패 — SP-AUTH-7)
```

**검증·규칙**
1. **금지 substring 정본(T10, test_config 재명세)**: `Settings` 필드명에 `jwt`·`oauth`·`password_reset`·`social` 부분문자열이 **부재**해야 한다(어서션 원문 유지). 위 신규 필드는 어느 것도 이 4종을 포함하지 않는다.
2. **구 금지목록 정정**: 구 regime(로그인 영구제외)에서 금지했던 `smtp`·`session`·`secret`·`mail`·`pepper` substring은 **금지목록에서 제거**한다 — SC14로 정당한 필드가 되었기 때문이다(핸드오프 §C, T10). 이는 금지 substring을 넓히는 것이 아니라 **좁히는** 개정이다.
3. 신규 필드 positive test: 위 필드가 `Settings`에 존재하고 기본값·타입이 위와 일치함을 검증한다(AU-5).
4. `comp_email_hmac_pepper` 미설정 시 앱은 재직 인증 경로를 활성화하지 않고 **기동 시 명확히 실패**한다(약한 HMAC 키 금지, NFR30).
5. `mailer_mode`·SMTP 값은 로그에 남기지 않는다(비밀값 노출 방지).

**추적**: SC14 / UC-70·71·73(전제) / INV-8·INV-9·NFR30·NFR31 / T10 / FR-101·FR-102·FR-112 / SP-API-2·SP-AUTH-4·11.

## SP-AUTH-3 — 참여 7테이블 컬럼 계약 (DDL은 SP-DB)

**근거**: SP-ARCH C4/L57·L60(기여용 회원·세션·인증·편집이력 7종을 C4가 둔다), INV-8·T9(PII 컬럼·해시 at-rest), FR-111(인증 데이터 at-rest), NFR30. **DDL 정본은 SP-DB(SPEC 02, ML-B)** — 본 절은 앱계층이 참조하는 **컬럼 이름·해시 컬럼·감사·인덱스 요건**만 확정한다.

**소유 경계 확정(중요)**: SPEC/01 원문은 7테이블 DDL 소유 문서를 완전히 결정하지 않는다(SP-DB 행 L233은 "DDL 5테이블"만 열거). 본 문서는 이를 **명시적으로 확정**한다 — **7테이블 CREATE/제약/인덱스 DDL 정본 = SP-DB(SPEC 02)**, SP-AUTH는 컬럼 이름을 SP-DB 규약대로 참조할 뿐 DDL을 정의하지 않는다. SP-DB(ML-B)는 `REMOVED_TABLES`에서 `TMEMBER`를 제거하고 아래 7종을 신규 소유한다(현재 SPEC/02는 `TMEMBER`를 "영구 제외 제거 테이블"로 명시 — SC14 재명세 필수).

**명명 규약(SP-DB 상속)**: 테이블 접두 `T`, 컬럼 대문자 스네이크 + 분류어 접미(`_ID`/`_CD`/`_NM`/`_DTM`/`_VAL`/`_YN`/`_CTNT`/`_NO`), PK `*_ID INT AUTO_INCREMENT PRIMARY KEY`, FK `INT`, ENUM 금지(VARCHAR+COMMENT), 전 컬럼 COMMENT, `ENGINE=InnoDB CHARSET=utf8mb4`, `IF NOT EXISTS`, 감사 4종 말미(`INS_ID`·`INS_DTM`·`MOD_ID`·`MOD_DTM`). **접미 예외**: 코드 검증 시도 횟수는 FRD/13이 확정한 `ATTEMPT_CNT`를 쓴다 — `_CNT`(횟수)는 기존 SP-DB 접미 규약에 없던 신규 분류어로, **SP-DB(ML-B)가 참여 도메인용으로 비준**한다(구 regime은 카운트에 `_NO` 사용).

| 테이블 | 성격 | 핵심 컬럼(앱 참조) | 비밀값 처리 |
|--------|------|--------------------|-------------|
| `TMEMBER` | 회원(무비밀번호·PII 최소) | `MBR_ID`·`LOGIN_EMAIL_NM`(UNIQUE, 탈퇴 시 NULL)·`NICKNAME_NM`(UNIQUE)·`STATUS_CD`·감사4 | PII = 이메일·닉네임뿐(T9). **비밀번호 컬럼 부재** |
| `TSESSION` | 세션(불투명 토큰) | `SESSION_ID`·`MBR_ID`(FK)·`TOKEN_HASH_VAL`(CHAR(64), UNIQUE)·`EXPIRES_DTM`·`REVOKED_DTM`·감사4 | 토큰 원문 컬럼 **부재**, SHA-256 해시만(T9) |
| `TAUTH_CODE` | 로그인·인증 코드 | `AUTH_CODE_ID`·`PURPOSE_CD`(∈{login, employ_verify})·`CODE_HASH_VAL`·`TARGET_HASH_VAL`(이메일/회사이메일 조회키)·`COMP_ID`(널)·`MBR_ID`(널)·`ATTEMPT_CNT`·`EXPIRES_DTM`·`CONSUMED_DTM`·감사4 | 코드·이메일 원문 컬럼 **부재**, 해시만(T9) |
| `TCOMPANY_EMAIL_DOMAIN` | 회사↔도메인 화이트리스트 | `DOMAIN_ID`·`COMP_ID`(FK)·`EMAIL_DOMAIN_NM`(예: `samsung.com`)·`ACTIVE_YN`·감사4 | 비밀값 아님 |
| `TEMPLOY_VERIFICATION` | 재직 인증(회사 이메일 HMAC) | `EMPLOY_VRF_ID`·`MBR_ID`(FK)·`COMP_ID`(FK, `ON DELETE SET NULL` IDOR 근거)·`VRF_METHOD_CD`(∈{domain, manual})·`COMP_EMAIL_HASH_VAL`(HMAC, UNIQUE)·`EXPIRES_DTM`·`REVOKED_DTM`·감사4 | 회사 이메일 원문 컬럼 **부재**, HMAC만(T9·NFR30) |

| `TBENEFIT_EDIT_LOG` | 편집 이력(append-only) | `EDIT_LOG_ID`·`BENEFIT_ID`(FK, `ON DELETE CASCADE`)·`COMP_ID`·`ACTOR_MBR_ID`(FK, `ON DELETE SET NULL`)·`EDIT_TYPE_CD`(∈{create, update, delete})·`BEFORE_VAL`(JSON)·`AFTER_VAL`(JSON)·`EDIT_NOTE_CTNT`·`INS_DTM` | 식별 PII 없음(닉네임 조인으로 표시). MOD 없음(불변 로그) |

**검증·규칙**
1. **원문 컬럼 부재(T9·AU-4)**: `TSESSION`·`TAUTH_CODE`·`TEMPLOY_VERIFICATION`에 원문 토큰·코드·회사 이메일 컬럼이 **없고** `*_HASH_VAL`만 존재한다.
2. **PII 컬럼 정확집합(T9·AU-3)**: `TMEMBER`의 PII 컬럼은 `LOGIN_EMAIL_NM`·`NICKNAME_NM` 2종뿐이다. 그 외 이메일/전화/실명 컬럼 부재.
3. `TCOMPARE_LOG`(익명 4컬럼)는 **불변**이며 참여 테이블과 분리된다(INV-8).
4. `TBENEFIT_EDIT_LOG`는 감사 4종이 아니라 `INS_DTM` 하나만 두는 **불변 append-only** 로그다. 복합 인덱스 `(COMP_ID, INS_DTM)`·`(ACTOR_MBR_ID, INS_DTM)`은 SP-DB가 확정한다.
5. `TEMPLOY_VERIFICATION.COMP_EMAIL_HASH_VAL`은 UNIQUE로 **한 회사 이메일 = 한 계정**을 강제한다(중복 인증 차단, FR-106 409).

## SP-AUTH-4 — 세션 발급·검증·의존성 (`services/session.py`·`deps.py`)

**근거**: INV-9·T2·T10(세션 = 라우트 의존성·미들웨어 아님), FR-100·FR-101(세션 계약), NFR17·NFR30. **블루프린트 선례 없음**(job_change는 JWT 무상태) — DB 기반 세션은 loupit 신규 설계.

발급은 불투명 랜덤 토큰(`secrets`)을 만들어 **DB엔 SHA-256 해시만** 저장하고, 원문은 쿠키로만 전달한다. 검증은 미들웨어가 아니라 `Depends(require_member)`로 주입한다.

```python
# server/services/session.py
import secrets, hashlib
from datetime import datetime, timedelta, timezone

_COOKIE_NAME = "loupit_sid"                          # FR-101

def _hash_token(raw: str) -> str:                    # DB 조회키
    pepper = get_settings().session_hash_pepper.encode()
    return hashlib.sha256(pepper + raw.encode()).hexdigest()   # CHAR(64)

async def issue_session(mbr_id: int) -> str:
    """불투명 토큰 발급. DB엔 해시만 저장, 원문 반환(쿠키 전용)."""
    raw = secrets.token_urlsafe(48)                  # 원문(무저장)
    expires = datetime.now(timezone.utc) + timedelta(days=get_settings().session_ttl_days)
    await db.execute(
        "INSERT INTO TSESSION (MBR_ID, TOKEN_HASH_VAL, EXPIRES_DTM, INS_ID) VALUES (%s,%s,%s,%s)",
        (mbr_id, _hash_token(raw), expires, mbr_id))
    return raw                                        # → Set-Cookie(SP-AUTH-12)

async def resolve_session(raw: str | None) -> dict | None:
    if not raw: return None
    row = await db.fetch_one(
        "SELECT MBR_ID FROM TSESSION "
        "WHERE TOKEN_HASH_VAL=%s AND REVOKED_DTM IS NULL AND EXPIRES_DTM > UTC_TIMESTAMP()",
        (_hash_token(raw),))
    return row                                        # 미만료·미폐기만

async def revoke_session(raw: str) -> None:          # 로그아웃(FR-104)
    await db.execute("UPDATE TSESSION SET REVOKED_DTM=UTC_TIMESTAMP() WHERE TOKEN_HASH_VAL=%s",
                     (_hash_token(raw),))
```

```python
# server/deps.py  — 라우트 의존성(미들웨어 아님, INV-9)
from fastapi import Cookie, Depends, HTTPException

async def require_member(loupit_sid: str | None = Cookie(default=None)) -> dict:
    m = await session.resolve_session(loupit_sid)
    if not m: raise HTTPException(401, "로그인이 필요합니다.")     # FR-114
    return m                                          # {"MBR_ID": ...}

async def require_employment(comp_id: int, member: dict = Depends(require_member)) -> dict:
    v = await employment.active_verification(member["MBR_ID"], comp_id)
    if not v: raise HTTPException(403, "해당 회사 재직 인증이 필요합니다.")  # FR-108/109
    return v
```

**검증·규칙**
1. DB에 세션 토큰 **원문 컬럼 부재** — `TOKEN_HASH_VAL`(SHA-256, CHAR(64)) 해시만(T9·AU-4, INV-8).
2. 세션 검증은 `Depends`로만 이뤄지고 **미들웨어 미등록** — `app.user_middleware == ['CORSMiddleware']`(T2·T10, INV-9). 익명 GET 라우터는 `require_member`를 의존하지 않는다.
3. 세션 TTL = `session_ttl_days`(기본 30). `lifespan`의 `_retention_scheduler`가 만료·폐기 세션과 만료 코드(`TAUTH_CODE`)를 주기 퍼지한다.
4. `require_employment(comp_id)`는 경로변수 `comp_id`에 대해 **활성(미만료·미폐기) 재직 인증**을 요구한다(IDOR 방어 — 인증한 회사만).
5. 로그아웃·탈퇴는 세션을 폐기(`REVOKED_DTM`)하고 쿠키를 삭제한다(SP-AUTH-6).

**추적**: SC14 / UC-71·72 / INV-8·INV-9·NFR17·NFR30 / T2·T9·T10 / FR-100·FR-101 / SP-AUTH-3·6·12.

## SP-AUTH-5 — 무비밀번호 로그인 흐름 (`member.py`·`services/auth_code.py`)

**근거**: FR-102(코드 발송)·FR-103(코드 검증·세션 발급), UC-70·71, NFR30·NFR31. 로그인은 **무비밀번호 이메일 6자리 코드**뿐 — 비밀번호·OAuth·소셜 미존재(SC10).

```python
# server/services/auth_code.py
import secrets, hashlib

def _gen_code() -> str:
    return f"{secrets.randbelow(1_000_000):06d}"      # 6자리, 앞자리 0 보존

def _hash_code(code: str, target: str) -> str:        # target = 이메일(정규화)로 스코프
    return hashlib.sha256(f"{target}:{code}".encode()).hexdigest()

async def issue_login_code(email: str) -> None:
    """코드 해시만 저장(+5분), 원문은 메일로만. 계정 유무 무관 균일(호출측 204)."""
    code = _gen_code()
    await db.execute(
        "INSERT INTO TAUTH_CODE (PURPOSE_CD, CODE_HASH_VAL, TARGET_HASH_VAL, EXPIRES_DTM, ATTEMPT_CNT) "
        "VALUES ('login', %s, %s, UTC_TIMESTAMP() + INTERVAL %s MINUTE, 0)",
        (_hash_code(code, email), _hash_email(email), settings.login_code_ttl_min))
    await mailer.send_login_code(email, code)          # 원문은 여기서 소멸(무저장)
```

**흐름**
- **`POST /api/v1/members/login-code`** (FR-102): `{email}` → 형식 검증(422) → `issue_login_code` → **균일 204**(계정 열거 차단). rate limit·쿨다운은 SP-AUTH-12.
- **`POST /api/v1/members/login`** (FR-103): `{email, code}` → `TAUTH_CODE`(purpose=login, 미소비·미만료) 조회 → `ATTEMPT_CNT` 증가·상한(5) 검사 → 해시 대조. 성공 시 코드 소비(`CONSUMED_DTM`), 계정 조회 또는 **신규 생성**(닉네임 `직장인-{랜덤6}` 자동), `issue_session` 후 `Set-Cookie` + `200 {"nickname","is_new"}`.

**검증·규칙**
1. 코드·이메일 원문 무저장 — `CODE_HASH_VAL`·`TARGET_HASH_VAL`만, 만료 +`login_code_ttl_min`(FR-111·T9).
2. 계정 유무와 무관하게 **동일 204**(계정 열거 차단, NFR31·FR-112).
3. 검증 실패 매트릭스: 코드 불일치 401 · 만료 410 · 시도 상한 초과 429(코드 무효화). 성공 코드는 재사용 불가(`CONSUMED_DTM`).
4. 신규 계정 저장 PII = 로그인 이메일 + 자동 닉네임뿐(INV-8·T9). 2단계 가입 없음(닉네임은 마이페이지에서 변경).
5. 응답 본문·로그에 코드·세션 토큰 원문·이메일 미출력(PII 로그 grep 게이트).

**추적**: SC14 / UC-70·71 / INV-8·INV-9·NFR16·NFR30·NFR31 / FR-102·FR-103·FR-111·FR-112 / SP-AUTH-4·11·12.

## SP-AUTH-7 — 재직 인증 (`employment.py`·`services/employment.py`)

**근거**: FR-105(코드 발송)·FR-106(인증 생성), UC-73, INV-8·NFR30·NFR31. **블루프린트 선례 없음** — 회사↔도메인 **화이트리스트 매칭**·HMAC·원문 파기는 loupit 신규 설계(job_change는 개인도메인 블록리스트 + 평문 토큰).

```python
# server/services/employment.py
import hmac, hashlib

def _normalize_domain(email: str) -> str:
    return email.rsplit("@", 1)[-1].lower()

async def domain_matches(comp_id: int, email: str) -> bool:
    """입력 이메일 도메인 ∈ 해당 회사 등록 도메인(화이트리스트)."""
    dom = _normalize_domain(email)
    return bool(await db.fetch_one(
        "SELECT 1 FROM TCOMPANY_EMAIL_DOMAIN "
        "WHERE COMP_ID=%s AND EMAIL_DOMAIN_NM=%s AND ACTIVE_YN=TRUE", (comp_id, dom)))

def _hmac_email(email: str) -> str:                   # 중복 인증 차단용(원문 무저장)
    key = get_settings().comp_email_hmac_pepper.encode()
    return hmac.new(key, email.lower().encode(), hashlib.sha256).hexdigest()
```

- **`POST /employment/verify-code`** (FR-105, `require_member`): `{comp_id, company_email}` → `domain_matches` → 일치 시 코드 해시 저장(`TAUTH_CODE` purpose=employ_verify, `COMP_ID`·`MBR_ID` 연계)·회사 이메일로 발송 → 204. 미등록 도메인 → 409 `manual_required`(SP-AUTH-8 유도). 도메인 불일치 → 422.
- **`POST /employment/verify`** (FR-106, `require_member`): `{comp_id, company_email, code}` 검증 성공 시 `TEMPLOY_VERIFICATION` 생성 — **회사 이메일 원문 파기**, `COMP_EMAIL_HASH_VAL`(HMAC)만, `VRF_METHOD_CD='domain'`, 만료 +`employ_vrf_ttl_days`(365) → 201. 해당 회사 편집 권한 활성화.

**검증·규칙**
1. 재직 인증 후 회사 이메일 **원문 컬럼 부재** — HMAC(`COMP_EMAIL_HASH_VAL`)만(T9·AU-4, INV-8·NFR30).
2. `COMP_EMAIL_HASH_VAL` UNIQUE → 같은 회사 이메일이 타 계정에 기인증이면 409(한 회사 이메일 = 한 계정).
3. 코드 검증 실패 매트릭스 = SP-AUTH-5와 동일(401/410/429). 이미 인증됨 → 409 `already_verified`.
4. 인증 만료 +365일. 만료 시 재인증 필요(퇴사자 자연 차단). `COMP_ID`는 `ON DELETE SET NULL`(회사 삭제 시 orphan 방지, IDOR 근거).
5. 발송·시도 상한은 SP-AUTH-12(FR-112).

**추적**: SC14 / UC-73 / INV-8·NFR30·NFR31 / FR-105·FR-106 / SP-AUTH-4·8·11·12 / SP-DB.

```python
    async with db.transaction() as tx:
        row = await tx.fetch_one("SELECT * FROM TCOMPANY_BENEFIT "
                                 "WHERE BENEFIT_ID=%s AND COMP_ID=%s FOR UPDATE", (benefit_id, comp_id))
        if not row: return None                        # → 404
        if iso(row["MOD_DTM"]) != payload.base_dtm:    # 낙관적 동시성(CAS)
            return {"conflict": True, "current_benefit": row}   # → 409 + 현재행 동봉
        await tx.execute("UPDATE TCOMPANY_BENEFIT SET ..., "
                         "BADGE_CD='verified', MOD_ID=%s WHERE BENEFIT_ID=%s", (...))  # official→verified 강등
        await _append_log(tx, benefit_id, comp_id, member["MBR_ID"], "update",
                          before=_snapshot(row), after=_snapshot(payload), note=payload.edit_note)
    return {...}
```

- **`POST /companies/{comp_id}/benefits`** (FR-108, `require_employment(comp_id)` + CSRF): 배지 강제(`BADGE_CD='verified'`·`BADGE_SRC_CD='user_report'`·만료 +18개월). `AMT_SOURCE_CD`는 `QUAL_YN` 분기 — 금액 행 `estimated`, 정성 행 `none`(FR-108의 강제값 `estimated`는 금액 기준, 정성은 DC-9에 따라 `none`). 정성이면 `QUAL_YN=true`·금액 NULL. 201 + `{benefit, benefits[]}`. 참조 캐시 무효화.
- **`PUT /companies/{comp_id}/benefits/{benefit_id}`** (FR-109): `base_dtm`(읽은 행 `MOD_DTM`) 일치 시 갱신, 불일치 시 **409 + 현재 행 동봉**. official 행을 사용자가 수정하면 `verified`로 강등.

**검증·규칙**
1. 배지는 **서버 강제** — 사용자가 `official`·`stated` 지정 불가. `BADGE_CD='verified'`는 SP-DB가 비준할 **신규 제3값**(기존 `official`/`est` 게이트 불변, `est` 재사용 금지). SP-DB(ML-B)가 `TCOMPANY_BENEFIT.BADGE_CD` COMMENT에 `verified`를 추가한다. `AMT_SOURCE_CD`도 서버가 `QUAL_YN`에 따라 강제한다 — 금액 행 `estimated`, 정성 행 `none`(DC-9 불변식·밴드 근거). 사용자는 `amt_source`를 지정하지 못한다.
2. 본체 + 편집 이력은 **원자 트랜잭션**(`db.transaction()`, `SELECT ... FOR UPDATE`).
3. **밴드·계산 불변(INV-5)**: `AMT_SOURCE_CD`가 밴드를 결정하고 `BADGE_CD`는 표시 전용 — `calc.js`/SP-ENGINE 무변경(표시 계층만 확장). `verified` 배지는 밴드에 영향 없음.
4. `QUAL_YN=true ⇒ BENEFIT_AMT IS NULL AND AMT_SOURCE_CD='none'` 불변식은 서버가 강제하며, 클라이언트가 위반 조합(정성인데 금액 존재 등)을 보내면 **422**로 거부한다(SP-DB DC-9, 테스트 AB-6). 금액 `>= 0`.
5. 권한/CSRF: 세션·재직 인증 미보유 401/403, CSRF 헤더 부재 403(SP-AUTH-12). 동일 회사·코드 중복 409, 일일 편집 상한(`daily_edit_limit`=30) 초과 429.
6. 응답 `benefits[]`로 클라이언트 REF 인메모리 치환·재렌더. 정적 페이지 반영은 야간 재생성(신선도 고지). v1 사용자 DELETE 없음(CLI 전용).

**추적**: SC14 / UC-75·76 / INV-1·INV-5·INV-8·NFR21 / FR-108·FR-109 / SP-AUTH-4·7·10·12 / SP-DB(배지값·편집로그)·SP-ENGINE(밴드 불변).

## SP-AUTH-11 — 메일러 추상화 (`mailer.py`)

**근거**: FR-102·FR-105(코드 발송), NFR31, SP-INFRA(SMTP·메일 존 — 현 SPEC/11은 `SMTP_*` 금지·`loupit_mail` 존 부재, **SC14 재명세 대기**). **블루프린트 선례 없음** — job_change는 SMTP 직결 + 미설정 시 조용히 skip뿐. loupit은 Console/SMTP **인터페이스 분리**를 신규 설계.

신규 의존성 0 — stdlib `smtplib` + `asyncio.to_thread`(블로킹 회피). `mailer_mode`로 구현을 선택한다.

```python
# server/mailer.py
class Mailer(Protocol):
    async def send_login_code(self, email: str, code: str) -> None: ...
    async def send_employ_code(self, email: str, code: str, comp_nm: str) -> None: ...

class ConsoleMailer:                      # 개발·미설정 기본. 실발송 없음, 코드 원문 stdout(운영 금지)
    async def send_login_code(self, email, code): log(f"[login-code] {mask(email)} → {code}")

class SmtpMailer:                         # 운영. smtplib + starttls + login, asyncio.to_thread
    async def send_login_code(self, email, code):
        await asyncio.to_thread(self._send, email, "loupit 로그인 코드", _body(code))

def get_mailer() -> Mailer:               # mailer_mode + SMTP 설정 유무로 선택
    s = get_settings()
    if s.mailer_mode == "smtp" and s.smtp_user:
        return SmtpMailer()
    return ConsoleMailer()                # 폴백(운영 미설정 시 실발송 금지)
```

**검증·규칙**
1. `mailer_mode='smtp'`이되 `smtp_user` 미설정이면 **ConsoleMailer로 폴백**(운영 배포 전 실발송 방지). 운영은 `smtp_*` 필수.
2. **운영 로그·응답에 코드·이메일 원문 미출력**(PII 로그 grep 게이트, SP-AUTH-5·12). 예외: `ConsoleMailer`는 **개발 전용**으로 코드를 stdout에 표시하나 운영 로그가 아니며 게이트 대상 외다(운영 배포는 `SmtpMailer`). SMTP 자격증명·비밀값 로그 금지.
3. 발송은 `asyncio.to_thread`로 이벤트 루프 블로킹 회피(stdlib `smtplib`, 신규 의존성 0).
4. 발송 rate limit·쿨다운은 SP-AUTH-12(nginx `loupit_mail` 존 + 앱 쿨다운).

**추적**: SC14 / UC-70·73(전제) / NFR31 / FR-102·FR-105 / SP-AUTH-5·7·12 / SP-INFRA(SMTP·메일 존).

## SP-AUTH-12 — CSRF·전송 헤더·발송/시도 리밋

**근거**: FR-113(CSRF·세션 전송)·FR-112(발송·시도 리밋), INV-9·NFR16·NFR31, SP-INFRA(nginx 헤더 게이트·리밋 존).

**CSRF·쿠키·CORS**
- 세션 쿠키 `loupit_sid`: `HttpOnly; Secure; SameSite=Lax; Path=/api/v1; Max-Age=<session_ttl_days>`.
- 상태변경(POST/PUT/DELETE) 요청은 커스텀 헤더 `X-Loupit-Client` 필수 — 없으면 403. **nginx 게이트 + 앱 레벨 이중 검사**.
- CORS `Access-Control-Allow-Credentials`는 미설정(false) 유지. 프로덕션은 nginx 동일 오리진 프록시라 쿠키에 CORS 불관여.
- 프론트: 익명 GET(`apiFetch`)은 `credentials:'omit'`(무쿠키), 기여 쓰기(`apiSend`)·`GET /members/me`만 credentialed + `X-Loupit-Client`.

**발송·시도 리밋**
- 메일 발송: nginx rate limit(`loupit_mail` 존, 예 3r/m) + 앱 재전송 쿨다운(`mail_resend_cooldown_sec`)·시간당 상한.
- 코드 검증: 시도 상한(`code_max_attempts`=5, `ATTEMPT_CNT`)·만료(`login_code_ttl_min`=5). 초과 시 코드 무효화·429.
- 계정 열거 차단: 로그인 코드 요청(FR-102)은 계정 유무 무관 균일 204.
- 경로변수 라우트(`/companies/{id}/...`)는 nginx `^~` 블록 경유 → 앱 레벨 상한(`daily_edit_limit`=30)으로 방어.

**검증·규칙**
1. CSRF 헤더 부재 쓰기는 403(FR-113). `SameSite=Lax`로 top-level 이외 크로스사이트 전송 차단. 크로스오리진은 커스텀 헤더를 preflight 없이 못 붙이며, preflight는 CORS 허용목록 + `allow_credentials=false`에서 실패한다.
2. `Path=/api/v1`로 익명 정적 경로에 쿠키 미전송(무쿠키 익명 GET 보존, NFR16).
3. 로그·응답에 이메일·코드·토큰 원문 미출력(NFR31·INV-8).
4. `app.user_middleware == ['CORSMiddleware']` 불변 — CSRF는 미들웨어가 아니라 라우트 의존성/헤더 검사로 구현(INV-9).

**추적**: SC14 / UC-75·76(전제) / INV-9·NFR16·NFR31 / FR-112·FR-113 / SP-AUTH-4·5·7·9·11 / SP-INFRA(nginx 게이트·리밋 — **재명세 대기**: `loupit_mail` 존·`X-Loupit-Client` 게이트 신설 필요).

## SP-AUTH-13 — 공통 상태코드·오류 응답 계약

**근거**: FR-114(상태코드 매트릭스·envelope), NFR16·NFR26. 익명 API의 SP-API-12(FR-95)를 참여 쓰기로 확장한다.

| 상태코드 | 발생 조건 | 적용 |
|----------|-----------|------|
| **200/201/202/204** | 정상(조회·생성·큐등록·무본문) | 전 엔드포인트 |
| **401 Unauthorized** | 세션 없음·만료, 코드 불일치 | 세션 필요 라우트·로그인 |
| **403 Forbidden** | 재직 인증 미보유, CSRF 헤더 부재 | 복지 편집·재직 필요 |
| **404 Not Found** | 미존재 `comp_id`(편집 이력) | edits |
| **405 Method Not Allowed** | 열거 외 메서드(`Allow` 헤더) | 전 라우트 |
| **409 Conflict** | 닉네임 중복·HMAC 중복·`base_dtm` 충돌·중복 pending·`manual_required`·`already_verified` | me·verify·복지 |
| **410 Gone** | 코드 만료(+`login_code_ttl_min`) | 로그인·재직 검증 |
| **422 Unprocessable Entity** | 형식 오류·도메인 불일치·`QUAL_YN`/`BENEFIT_AMT` 불변식 위반 | 발송·검증·편집 |
| **429 Too Many Requests** | rate limit·시도·편집 상한 | 발송·검증·편집 |
| **500** | 예기치 못한 서버 오류(민감정보 미노출) | 전 엔드포인트 |

**오류 envelope**: `{"detail": "<메시지>"}`(명시) / 422는 FastAPI 검증 배열. `Cache-Control: no-store`.

**검증·규칙**
1. 409 충돌(특히 FR-109 `base_dtm`)은 **현재 상태를 동봉**(`current_benefit`)해 클라이언트 재조정을 돕는다.
2. 오류 응답에 PII·비밀값·스택·SQL·이메일·코드 원문 미포함(서버 로그만, INV-8).
3. 모든 비-2xx는 클라이언트가 방어적으로 흡수(무크래시, NFR26 정합).

**추적**: SC14 / UC-70~77 / NFR16·NFR26 / FR-114·FR-95(익명 확장) / SP-API-12.
