# SPEC 02 — 데이터베이스 스키마 (DDL)

> **이력**: 브랜드 축(성장성·안정성)은 2026-07-20 제거됐다. 우선순위는 연봉·워라밸·복지 3종.
> 경위·파급범위: `docs/ANALYSIS-성장성-용어-2026-07-20.md` §6-1 · `docs/HANDOFF-2026-07-19.md` §K.


**영역 ID 대역**: `SP-DB`
**저장 경로**: `docs/SPEC/02-데이터베이스-스키마.md`
**목적**: loupit MySQL 8.0(utf8mb4) 참조 데이터베이스의 **유지 참조 테이블 5종**에 대한 완성 DDL(컬럼·타입·한국어 COMMENT·PK·FK·인덱스·UNIQUE·FULLTEXT), DEC-2 신뢰도 디커플링 컬럼(`AMT_SOURCE_CD` 신규 + 밴드 근거 컬럼), 제거 테이블·제거 컬럼 목록, DDL/마이그레이션 파일 위치와 적용 순서, 그리고 스키마 로드·제약·데이터 계약 검증을 위한 테스트 명세를 개발자가 추측 없이 구현할 수 있는 수준으로 확정한다.

**상위 추적**: 본 SPEC은 CONTEXT_BRIEF §4(유지 데이터 모델)·§5(보존 실데이터), PRD `08-데이터-요구사항` **D1~D6**, FRD `02-데이터-계약` **FR-D1~FR-D11**(특히 FR-D5 benefit 필드·FR-D9 디커플링), RESEARCH `benefit-scraping` **§4(신뢰도 모델·DEC-2)**·§8(오픈이슈)에 근거한다. 하위 문서(SPEC 백엔드/시드·TASK)는 `SP-DB-N` ID를 인용한다.

**범위 경계**: 본 문서는 **DB 스키마(DDL) + 제약 + 백필 마이그레이션 + 스키마 테스트**만 확정한다. API 라우팅·SQL 쿼리·캐시는 SPEC 백엔드 영역, 시드 로더 구현·96개 SQL 재이식 절차는 SPEC 시드 영역, 클라이언트 계산·`value_source` 프로버넌스는 SPEC 프론트 영역에서 다룬다(중복 서술 금지). 프로파일러·익명 서버측 사용자 쓰기 테이블은 영구 제외이며 본(익명 참조) 스키마에 존재하지 않는다(CONTEXT_BRIEF §2-1·§2-3, D1.8, SP-DB-14). 다만 로그인/회원 등 SC14 복지 등록·수정 기여 참여 테이블(7종)은 영구 제외가 아니라 SP-DB(본 문서, ML-B)가 신규 소유 예정이며, 그 DDL 정본은 SPEC/13(SP-AUTH)이 본 영역에 배정한다(서술만 반영, 실 DDL은 ML-B 소관·별도 추적).

## SP-DB-1 — 명명 규칙·공통 규약 (전 테이블 적용)

기존 job_change 스키마 명명 규칙을 그대로 계승한다(CONTEXT_BRIEF §4, D1 개요). 신규 컬럼도 동일 규칙을 따른다.

| # | 규약 | 규칙 |
|---|------|------|
| SP-DB-1.1 | 테이블 접두사 | 모든 테이블은 `T` 접두. 대문자. (예: `TCOMPANY`, `TCOMPANY_BENEFIT`) |
| SP-DB-1.2 | 컬럼 대문자·분류어 종결 | 전 컬럼 대문자 스네이크. 분류어 접미사 사용: `_ID`(PK/FK) · `_CD`(코드) · `_NM`(이름/라벨) · `_AMT`(만원 금액) · `_VAL`(JSON/값) · `_URL`(URL) · `_DTM`(일시) · `_NO`(수치/순번/점수) · `_CNT`(횟수 — 참여 도메인 비준, SP-DB-17) · `_YN`(불리언) · `_CTNT`(서술 텍스트) |
| SP-DB-1.3 | PK | 모든 PK는 `INT AUTO_INCREMENT PRIMARY KEY`. FK는 `INT` |
| SP-DB-1.4 | 감사 컬럼 4종 | 전 테이블 공통 말미 4컬럼: `INS_ID INT`(입력자), `INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP`(입력일시), `MOD_ID INT`(수정자), `MOD_DTM TIMESTAMP NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP`(수정일시). API 응답 미노출(FR 공통 규약) |
| SP-DB-1.5 | 금액 단위 | 모든 금액 컬럼(`*_AMT`)은 **연간 만원 단위 정수**(`INT`). 금액화 불가(정성) 항목은 `NULL`(D3.6, FR-D5) |
| SP-DB-1.6 | ENUM 금지 | 코드 컬럼은 `ENUM`을 쓰지 않고 `VARCHAR(n)` + COMMENT에 허용 값 집합 명시. 값 집합 강제는 시드/앱 계층 + 데이터 린트 테스트(SP-DB-16)로 수행 |
| SP-DB-1.7 | COMMENT 필수 | 전 컬럼·전 테이블에 한국어 `COMMENT` 부착 |
| SP-DB-1.8 | 문자셋/엔진 | `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4`. 세션 시작 시 `SET NAMES utf8mb4;`. FULLTEXT는 한글 대응 위해 `WITH PARSER ngram` |
| SP-DB-1.9 | 읽기 전용 | 런타임 쓰기 경로 없음. 데이터 변경은 **빌드타임 재시드**만(D4.2). 스키마·시드는 반복 적용 안전(`CREATE TABLE IF NOT EXISTS`, `INSERT IGNORE`, `INSERT … ON DUPLICATE KEY UPDATE`) |

## SP-DB-2 — `TCOMPANY_TYPE` (기업유형 마스터) DDL

**근거**: D1.1, FR-D2. 기업유형 6종({`large`,`startup`,`mid`,`foreign`,`public`,`freelance`}). 회사 분류·프리셋 그룹핑의 유형 소스.

```sql
CREATE TABLE IF NOT EXISTS TCOMPANY_TYPE (
  COMP_TP_ID          INT AUTO_INCREMENT PRIMARY KEY COMMENT '기업유형 PK',
  COMP_TP_CD          VARCHAR(20)  NOT NULL UNIQUE
                      COMMENT '기업유형 코드 (large, startup, mid, foreign, public, freelance)',
  COMP_TP_NM          VARCHAR(20)  NOT NULL COMMENT '표시 라벨 (대기업, 스타트업, 중견기업, 외국계, 공기업, 프리랜서)',
  INS_ID  INT COMMENT '입력자 ID',
  INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '입력 일시',
  MOD_ID  INT COMMENT '수정자 ID',
  MOD_DTM TIMESTAMP NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='기업유형 6종 마스터';
```

- `COMP_TP_CD` 값 집합 = 6종. `UNIQUE` 강제로 유형당 1행(FR-D2 유일성).
- 총 7컬럼: `COMP_TP_ID`/`COMP_TP_CD`/`COMP_TP_NM` + 감사 4종(SP-DB-1.4). 유형 자체에 지표성 수치 컬럼을 두지 않는다.

## SP-DB-3 — `TCOMPANY` (회사 마스터) DDL

**근거**: D1.2, D1.6, FR-D4. 회사 ~96개(복지 데이터 보유 회사만 등록, D2.6). 영문 식별자·정식명 유일, `COMP_NM` FULLTEXT.

```sql
CREATE TABLE IF NOT EXISTS TCOMPANY (
  COMP_ID              INT AUTO_INCREMENT PRIMARY KEY COMMENT '회사 PK',
  COMP_ENG_NM          VARCHAR(30)  NOT NULL UNIQUE
                       COMMENT '회사 영문 식별명 (samsung_elec, cj 등 — URL/로고 매핑 키, 과거 id)',
  COMP_NM              VARCHAR(100) NOT NULL UNIQUE COMMENT '회사 정식 명칭 (삼성전자 등)',
  COMP_TP_ID           INT          NOT NULL COMMENT '기업유형 FK (TCOMPANY_TYPE.COMP_TP_ID)',
  INDUSTRY_NM          VARCHAR(50)  COMMENT '산업 분류 (전자/반도체, 핀테크 등)',
  LOGO_NM              VARCHAR(10)  COMMENT '로고 약어 (S, CJ, T 등)',
  WORK_STYLE_VAL       JSON         COMMENT '근무 형태 {remote, flex, unlimitedPTO, refreshLeave, overtime}',
  CAREERS_BENEFIT_URL  VARCHAR(500) COMMENT '공식 채용/복지 페이지 URL (출처 아웃링크)',
  INS_ID  INT COMMENT '입력자 ID',
  INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '입력 일시',
  MOD_ID  INT COMMENT '수정자 ID',
  MOD_DTM TIMESTAMP NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
  FOREIGN KEY (COMP_TP_ID) REFERENCES TCOMPANY_TYPE(COMP_TP_ID),
  FULLTEXT INDEX idx_comp_nm (COMP_NM) WITH PARSER ngram
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='회사 마스터 (복지 데이터 보유 ~96개만 등록)';
```

- `COMP_ENG_NM`·`COMP_NM` 각각 `UNIQUE`(D1.2). 회사 등록 예외 2건 반영: **엔씨소프트** 포함, **CJ**는 `COMP_NM='CJ올리브네트웍스'`로 등록(D2.7 — 시드 영역에서 96개 SQL 값 교정).
- `WORK_STYLE_VAL JSON`: 키 5종(`remote`,`flex`,`unlimitedPTO`,`refreshLeave`,`overtime`, D1.6/FR-D4.1). `NULL` 허용.
- `FULLTEXT … WITH PARSER ngram`: 한글 부분 검색 정확도 확보(utf8mb4 ngram). **MVP 검색(`/companies/search`)은 `LIKE '%q%'` 부분일치**를 사용하며(FR-D6, §6), FULLTEXT는 D1.2 요구 충족 + 향후 확장 예비. LIKE 부분일치는 별칭 인덱스(SP-DB-4)와 함께 커버.
- FK `COMP_TP_ID → TCOMPANY_TYPE`: `ON DELETE` 미지정(참조 유형은 삭제 대상 아님, 기본 `RESTRICT`).

## SP-DB-4 — `TCOMPANY_ALIAS` (검색 별칭) DDL

**근거**: D1.3, FR-D4(`aliases[]`). 회사당 1건 이상(D2.6). 이름/별칭 LIKE 검색의 별칭 축.

```sql
CREATE TABLE IF NOT EXISTS TCOMPANY_ALIAS (
  ALIAS_ID  INT AUTO_INCREMENT PRIMARY KEY COMMENT '별칭 PK',
  COMP_ID   INT          NOT NULL COMMENT '회사 FK (TCOMPANY.COMP_ID)',
  ALIAS_NM  VARCHAR(100) NOT NULL COMMENT '검색용 별칭 (삼성, 삼성전자, samsung 등)',
  INS_ID  INT COMMENT '입력자 ID',
  INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '입력 일시',
  MOD_ID  INT COMMENT '수정자 ID',
  MOD_DTM TIMESTAMP NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
  FOREIGN KEY (COMP_ID) REFERENCES TCOMPANY(COMP_ID) ON DELETE CASCADE,
  UNIQUE KEY uq_comp_alias (COMP_ID, ALIAS_NM),
  INDEX idx_alias_nm (ALIAS_NM)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='검색 별칭 (회사당 1건 이상)';
```

- `UNIQUE(COMP_ID, ALIAS_NM)`(D1.3): 동일 회사 내 중복 별칭 금지.
- `INDEX idx_alias_nm (ALIAS_NM)`: 별칭 접두 LIKE(`ALIAS_NM LIKE 'q%'`) 가속. 재시드 시 회사별 별칭 재삽입은 `INSERT IGNORE`로 멱등.
- FK `ON DELETE CASCADE`: 회사 삭제 시 별칭 동반 삭제(재시드 정합).

## SP-DB-5 — `TCOMPANY_BENEFIT` (회사별 복지) DDL

**근거**: D1.4, D3(배지·신뢰도), FR-D5(필드 계약)·FR-D9(디커플링), RESEARCH §4·DEC-2. **핵심 변경 2건**: (1) 신규 컬럼 `AMT_SOURCE_CD`(금액 신뢰도, DEC-2) 추가(SP-DB-7), (2) 제거 컬럼 `VERIFIED_BY_ID`(FK→TMEMBER, 회원 테이블 제거로 무효) 삭제(SP-DB-15).

```sql
CREATE TABLE IF NOT EXISTS TCOMPANY_BENEFIT (
  BENEFIT_ID          INT AUTO_INCREMENT PRIMARY KEY COMMENT '복지항목 PK',
  COMP_ID             INT          NOT NULL COMMENT '회사 FK (TCOMPANY.COMP_ID)',
  BENEFIT_CD          VARCHAR(30)  NOT NULL COMMENT '복지 코드 (meal, transport, welfare_point, health_check 등, 회사 내 유니크)',
  BENEFIT_NM          VARCHAR(100) NOT NULL COMMENT '복지 표시명 (식대, 통근버스 등)',
  BENEFIT_AMT         INT          DEFAULT NULL COMMENT '연간 환산 금액 (만원). 정성 항목은 NULL, 음수 불가',
  BENEFIT_CTGR_CD     VARCHAR(20)  NOT NULL
                      COMMENT '복지 카테고리 9종 (compensation, flexibility, work_env, time_off, health, family, growth, leisure, perks)',
  BADGE_CD            VARCHAR(10)  NOT NULL DEFAULT 'est'
                      COMMENT '출처 신뢰도 배지 — 복지 존재의 확실성 (official: 공식 확인, est: 추정, verified: 사용자 재직 인증 기여 SC14·FR-108·SP-DB-17.8)',
  AMT_SOURCE_CD       VARCHAR(10)  NOT NULL DEFAULT 'estimated'
                      COMMENT '금액 신뢰도 (stated: 공식 명시금액, estimated: 앵커 추정, none: 정성/금액없음) — DEC-2, BADGE_CD와 독립. 불확실성 밴드 근거. API JSON 필드명 amt_source',
  BADGE_SRC_CD        VARCHAR(20)  DEFAULT NULL
                      COMMENT '출처 유형 (scrape_official, scrape_fallback, ai_parse, manual, user_report)',
  BADGE_SRC_URL_CTNT  VARCHAR(500) DEFAULT NULL COMMENT '출처 URL (공식 페이지 등 근거 링크, 출처표기 의무 핵심)',
  VERIFIED_DTM        DATETIME     DEFAULT NULL COMMENT '마지막 출처 재확인 시점 (신선도 기준, 감사 DTM과 별개)',
  EXPIRES_DTM         DATETIME     DEFAULT NULL COMMENT '유효 만료 시점 (VERIFIED_DTM + TTL — DG-1 확정: 균일 18개월). 경과 시 만료 취급·밴드 확대',
  NOTE_CTNT           VARCHAR(200) COMMENT '비고/상세 설명 (예: 일 18,000원 x 240일 환산)',
  QUAL_YN             BOOLEAN      NOT NULL DEFAULT FALSE COMMENT '정성적 복지 여부 (TRUE: 금액 환산 불가 → BENEFIT_AMT NULL, AMT_SOURCE_CD none)',
  QUAL_DESC_CTNT      VARCHAR(500) COMMENT '정성적 복지 상세 텍스트 (짧은 발췌만, 원문 복제 금지)',
  SORT_ORDER_NO       SMALLINT     DEFAULT 0 COMMENT '표시 정렬 순서',
  INS_ID  INT COMMENT '입력자 ID',
  INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '입력 일시',
  MOD_ID  INT COMMENT '수정자 ID',
  MOD_DTM TIMESTAMP NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
  FOREIGN KEY (COMP_ID) REFERENCES TCOMPANY(COMP_ID) ON DELETE CASCADE,
  UNIQUE KEY uq_comp_benefit (COMP_ID, BENEFIT_CD),
  INDEX idx_benefit_comp (COMP_ID, SORT_ORDER_NO)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='회사별 복지 (배지·출처·만료·금액신뢰도 메타 포함)';
```

**컬럼 계약 요약**(FR-D5 ↔ DB):

| DB 컬럼 | API JSON 필드 | 값 집합/규칙 |
|---------|---------------|--------------|
| `BENEFIT_CD` | `benefit_cd` | 회사 내 유니크(`uq_comp_benefit`) |
| `BENEFIT_AMT` | `benefit_amt` | 만원 정수 \| NULL. 음수 불가. `qual_yn` 시 NULL |
| `BENEFIT_CTGR_CD` | `benefit_ctgr_cd` | 9종 집합 |
| `BADGE_CD` | `badge_cd` | {`official`,`est`,`verified`} — 출처 신뢰도(verified=SC14 사용자 기여, SP-DB-17.8) |
| `AMT_SOURCE_CD` | `amt_source` | {`stated`,`estimated`,`none`} — 금액 신뢰도(신규, SP-DB-7) |
| `BADGE_SRC_CD` | `badge_src_cd` | {`scrape_official`,`scrape_fallback`,`ai_parse`,`manual`,`user_report`} |
| `BADGE_SRC_URL_CTNT` | `badge_src_url_ctnt` | URL \| NULL |
| `VERIFIED_DTM`/`EXPIRES_DTM` | `verified_dtm`/`expires_dtm` | ISO8601 \| NULL |
| `QUAL_YN` | `qual_yn` | `BOOLEAN` → JSON bool |
| `QUAL_DESC_CTNT`/`NOTE_CTNT`/`SORT_ORDER_NO` | `qual_desc_ctnt`/`note_ctnt`/`sort_order_no` | — |

- **컬럼명 예외(SP-DB-1.6 명시 처리)**: `AMT_SOURCE_CD`(코드 → `_CD` 접미 규칙 준수)는 API 직렬화 시 FR-D5가 고정한 와이어 이름 `amt_source`로 매핑한다(`_CD` 접미 탈락). FRD가 DB명은 SPEC 확정으로 남기고 와이어명만 고정했으므로, DB는 명명 규칙을, 와이어는 계약을 각각 만족. 이 한 필드에 한해 명시적 필드맵을 API 직렬화기가 사용한다(SPEC 백엔드 영역이 인용).
- `idx_benefit_comp (COMP_ID, SORT_ORDER_NO)`: 회사 상세·번들에서 회사별 복지를 정렬순 조회(커버링).
- FK `ON DELETE CASCADE`: 회사 삭제 시 복지 동반 삭제. `uq_comp_benefit`로 96개 SQL의 `ON DUPLICATE KEY UPDATE` 멱등 재시드 지원(RESEARCH §3 [8]).

## SP-DB-6 — `TBENEFIT_PRESET` (기업유형별 기본복지) DDL

**근거**: D1.5, FR-D3, CONTEXT_BRIEF §4(역할 변경). **역할 = 비교 툴 직접 입력 모드 템플릿**(회사페이지 폴백 아님, UC-15/21). benefit 필드 + 프리셋 전용 `DEFAULT_CHECKED_YN`.

```sql
CREATE TABLE IF NOT EXISTS TBENEFIT_PRESET (
  PRESET_ID           INT AUTO_INCREMENT PRIMARY KEY COMMENT '프리셋 PK',
  COMP_TP_ID          INT          NOT NULL COMMENT '기업유형 FK (TCOMPANY_TYPE.COMP_TP_ID)',
  BENEFIT_CD          VARCHAR(30)  NOT NULL COMMENT '복지 코드',
  BENEFIT_NM          VARCHAR(100) NOT NULL COMMENT '복지 표시명',
  BENEFIT_AMT         INT          DEFAULT NULL COMMENT '연간 환산 금액 (만원, 정성 항목은 NULL)',
  BENEFIT_CTGR_CD     VARCHAR(20)  NOT NULL
                      COMMENT '복지 카테고리 9종 (compensation, flexibility, work_env, time_off, health, family, growth, leisure, perks)',
  BADGE_CD            VARCHAR(10)  NOT NULL DEFAULT 'est' COMMENT '데이터 신뢰도 (프리셋은 통상 est: 공식 확인 아님, D3.10)',

  SORT_ORDER_NO       SMALLINT     DEFAULT 0 COMMENT '표시 정렬 순서',
  INS_ID  INT COMMENT '입력자 ID',
  INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '입력 일시',
  MOD_ID  INT COMMENT '수정자 ID',
  MOD_DTM TIMESTAMP NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
  FOREIGN KEY (COMP_TP_ID) REFERENCES TCOMPANY_TYPE(COMP_TP_ID),
  INDEX idx_preset_type (COMP_TP_ID, SORT_ORDER_NO)

```

- 프리셋에는 `AMT_SOURCE_CD`·배지 출처/만료 메타를 **두지 않는다**: 프리셋은 실회사 관측이 아니라 유형 기반 템플릿이므로 출처·만료 개념이 없다. 금액 신뢰도가 필요하면 클라이언트가 프리셋 유래 항목을 `est`/추정으로 취급(FR-D3, D3.10).
- `benefit_presets{comp_tp_cd:[…]}` 번들 직렬화 시 `COMP_TP_ID`는 조인으로 `comp_tp_cd`를 파생(FR-D3, 부록 대응).
- `idx_preset_type (COMP_TP_ID, SORT_ORDER_NO)`: 유형별 정렬 조회.

## SP-DB-7 — DEC-2 신뢰도 디커플링: 신규 컬럼 `AMT_SOURCE_CD` + 밴드 근거 컬럼

**근거**: DEC-2(RESEARCH §0·§4), FR-D5·FR-D9, D3.9. 본 SPEC은 CONTEXT_BRIEF 지시대로 **불확실성 밴드 기준을 금액 신뢰도(`AMT_SOURCE_CD`)로 확정**한다(stated ±5% / estimated ±20% / 만료 +15%). 이로써 RESEARCH OI-6(밴드 기준 상충)를 `amt_source` 기준으로 종결한다.

### SP-DB-7.1 두 축의 독립 (한 컬럼으로 합치지 않음)

| 축 | 컬럼 | 값 집합 | 의미 |
|----|------|---------|------|
| 출처 신뢰도 | `BADGE_CD` | {`official`,`est`} | 이 회사가 이 복지를 제공한다는 **사실의 확실성** |
| 금액 신뢰도 | **`AMT_SOURCE_CD`(신규)** | {`stated`,`estimated`,`none`} | 금액 수치가 **공식 명시값인지 추정인지** |

- 두 축은 독립: `BADGE_CD='official'` ∧ `AMT_SOURCE_CD='estimated'` 조합이 **유효**하며, 96개 큐레이션 복지 다수가 이 상태(공식 출처·앵커 추정 금액)로 표기된다(DEC-2 핵심, FR-D9). 밴드는 오직 `AMT_SOURCE_CD`(+만료)로 산정한다.
- **값 출처(프로버넌스) `value_source`**({`real`,`preset`,`user`}, FR-D8)는 클라이언트 런타임 파생 상태로 **DB 컬럼·API 필드가 아니다**. 본 스키마에 컬럼을 두지 않는다(SPEC 프론트 영역 소관). `AMT_SOURCE_CD`와 이름·의미·값 집합이 다른 별개 축.

### SP-DB-7.2 `AMT_SOURCE_CD` 신규 추가 명시

- **신규 컬럼**: 기존 job_change 스키마에 없던 신규 개념. `TCOMPANY_BENEFIT`에 `NOT NULL DEFAULT 'estimated'`로 추가(SP-DB-5). `DEFAULT 'estimated'`는 (a) `AMT_SOURCE_CD`를 포함하지 않는 레거시 96개 SQL의 `INSERT`가 실패 없이 로드되게 하고(보수적 넓은 밴드), (b) 이후 백필(SP-DB-13)이 `stated`/`none`으로 정밀화하도록 한다.
- 그린필드 신규 DB에서는 `schema.sql`에 컬럼이 처음부터 존재하므로 별도 `ALTER`가 불필요하다. (기존 job_change DB를 승계 이관하는 경우에 한해 `ALTER TABLE … ADD COLUMN AMT_SOURCE_CD …`가 필요하나, loupit은 신규 스키마이므로 `schema.sql` 인라인 정의가 정본이다.)

### SP-DB-7.3 불확실성 밴드 근거 컬럼 (DB는 근거만 저장, 계산은 100% 클라이언트)

밴드 계산 자체는 클라이언트가 수행한다(§2-8). DB는 **밴드 산정 근거 컬럼**만 보관한다:

| 근거 컬럼 | 밴드 규칙(클라이언트, DEC-2 확정) |
|-----------|-----------------------------------|
| `AMT_SOURCE_CD='stated'` | ±5% |
| `AMT_SOURCE_CD='estimated'` | ±20% |
| `AMT_SOURCE_CD='none'` (정성) | 금액 밴드 미적용 |
| `EXPIRES_DTM` 경과(만료) | +15% 가산 확대 + "재확인 필요" 표기 |
| `BADGE_CD` × `EXPIRES_DTM` | 표시 배지 파생(official·미만료→공식 / official·만료→만료 / 그 외→추정, FR-D9.1) |

- DB에 계산된 밴드값·파생 배지를 저장하는 컬럼은 두지 않는다(파생은 클라이언트 순수함수, SPEC 프론트 영역). `AMT_SOURCE_CD`·`EXPIRES_DTM`·`BADGE_CD`가 근거 컬럼의 전부다.

## SP-DB-13 — DEC-2 백필 마이그레이션 (`20260710_backfill_dec2.sql`)

**근거**: RESEARCH §6 CHANGE(96개 백필)·§4.1(채움 규칙)·§4.3(TTL), DEC-2, OI-6/OI-3. 96개 로드(SP-DB-10 단계3) 이후 실행. 멱등(재실행 안전).

```sql
-- ── DEC-2 백필: 출처 신뢰도(official) 승격 + 금액 신뢰도(amt_source) 정밀화 + 신선도/만료 ──
SET NAMES utf8mb4;

-- 1) 출처 신뢰도: 96개 큐레이션 복지는 공식 페이지 기반 → official 승격 (DEC-2).
--    출처 유형 기본값 ai_parse (기존 backfill 관례), 이미 값이 있으면 보존.
UPDATE TCOMPANY_BENEFIT
   SET BADGE_CD = 'official'
 WHERE BADGE_CD = 'est';
UPDATE TCOMPANY_BENEFIT
   SET BADGE_SRC_CD = 'ai_parse'
 WHERE BADGE_SRC_CD IS NULL;

-- 2) 금액 신뢰도(amt_source) 도출 (OI-6 규칙):
--    (a) 정성 복지 → none
UPDATE TCOMPANY_BENEFIT
   SET AMT_SOURCE_CD = 'none'
 WHERE QUAL_YN = TRUE OR BENEFIT_AMT IS NULL;
--    (b) 금액 있고 '(추정)'/'추정'/환산 표기 → estimated (앵커 추정)
UPDATE TCOMPANY_BENEFIT
   SET AMT_SOURCE_CD = 'estimated'
 WHERE BENEFIT_AMT IS NOT NULL
   AND (NOTE_CTNT LIKE '%추정%' OR NOTE_CTNT LIKE '%환산%' OR NOTE_CTNT IS NULL);
--    (c) 금액 있고 공식 명시(추정 표기 없음) → stated
UPDATE TCOMPANY_BENEFIT
   SET AMT_SOURCE_CD = 'stated'
 WHERE BENEFIT_AMT IS NOT NULL
   AND NOTE_CTNT IS NOT NULL
   AND NOTE_CTNT NOT LIKE '%추정%'
   AND NOTE_CTNT NOT LIKE '%환산%';

-- 3) 신선도/만료: VERIFIED_DTM = 수집일(미기재분 시드 배치일로 통일),
--    EXPIRES_DTM = VERIFIED_DTM + 카테고리 TTL (RESEARCH §4.3 권장값, OI-3 확정).
UPDATE TCOMPANY_BENEFIT
   SET VERIFIED_DTM = COALESCE(VERIFIED_DTM, '2026-07-10 00:00:00');
UPDATE TCOMPANY_BENEFIT
   SET EXPIRES_DTM = DATE_ADD(VERIFIED_DTM, INTERVAL
        CASE BENEFIT_CTGR_CD
          WHEN 'compensation' THEN 6
          WHEN 'perks'        THEN 9
          WHEN 'time_off'     THEN 12
          WHEN 'flexibility'  THEN 12
          ELSE 18                         -- work_env/health/growth/leisure/family
        END MONTH)
 WHERE EXPIRES_DTM IS NULL;
```

- **OI-3(TTL) 확정**: RESEARCH §4.3 권장값을 채택(compensation 6M / perks 9M / time_off·flexibility 12M / 그 외 18M). 기존 `benefit_service.py` TTL과의 상충은 위 값으로 종결.

- `BADGE_SRC_URL_CTNT`는 96개 SQL의 `CAREERS_BENEFIT_URL='수동 입력'` 유실분 재발견 후 보정(RESEARCH §6, Phase 2). MVP 백필은 회사 URL 승계 범위까지.
- **멱등성**: (1)은 `BADGE_CD='est'` 대상만 승격하므로 재실행 시 no-op. (2)~(3)은 조건부/`COALESCE`로 안전. 단, 재시드로 `est` 행이 다시 들어오면 백필을 재적용해야 한다(SP-DB-10 순서 준수).
- **`verified` 불간섭(SC14)**: 본 백필은 `est`→`official`만 승격하며 `BADGE_CD='verified'`(사용자 재직 인증 기여, SP-DB-17.8) 행은 건드리지 않는다. verified는 큐레이션 백필이 아니라 기여 쓰기 경로(SP-AUTH-9)로만 생성된다.

## SP-DB-17 — 참여 7테이블 DDL (SC14 기여) 정본

**근거**: SC14, SPEC/13 **SP-AUTH-3**(컬럼 계약 원천)이 "7테이블 CREATE/제약/인덱스 DDL 정본 = SP-DB(본 문서)"로 명시 배정. INV-8(PII 최소)·T9(원문 컬럼 부재)·NFR30(해시 at-rest), FRD/13 FR-100~115. **본 절이 참여 7테이블 DDL의 유일 정본**이며, SP-AUTH는 컬럼 이름을 참조만 한다.

**소유 확정**: 이 7종은 SP-DB-11(제거 테이블)에서 **해제**된다 — `TMEMBER`는 영구 제외가 아니라 본 절이 신규 소유한다. `TSOCIAL_ACCOUNT`·`TEMAIL_VERIFICATION`은 무소셜·무비밀번호이므로 SP-DB-11에 영구 제거로 잔류(별개 계열).

**명명·공통(SP-DB-1 상속 + 증분)**: SP-DB-1.1~1.8을 그대로 계승하되 아래 2개를 **참여 도메인용으로 비준**한다.
- **접미 `_CNT`(횟수) 비준**: 코드 검증 시도 횟수 `ATTEMPT_CNT`처럼 "횟수"를 세는 컬럼은 신규 분류어 `_CNT`를 쓴다(구 regime은 순번·점수에 `_NO`만 사용, SP-DB-1.2에 `_CNT` 추가). `_NO`(순번/점수)와 의미 구분.
- **감사 4종 예외(append-only)**: `TBENEFIT_EDIT_LOG`는 불변 로그이므로 감사 4종 대신 `INS_DTM` 하나만 둔다(수정 없음, SP-AUTH-3 규칙4).
- **비밀값 = 해시만**: 토큰·코드·회사 이메일은 원문 컬럼을 두지 않고 `*_HASH_VAL`(SHA-256 hex `CHAR(64)` 또는 HMAC)만 저장한다(T9·AU-4·NFR30). `TMEMBER` PII는 `LOGIN_EMAIL_NM`·`NICKNAME_NM` 2종뿐, 비밀번호 컬럼 부재(T9·AU-3).

**생성 순서(FK 위상, conftest `TABLE_CREATE_ORDER`·`run_tests.sh` 백업 대상)**: ① `TMEMBER` → ② `TCOMPANY_EMAIL_DOMAIN`(FK `TCOMPANY`) → ③ `TSESSION`(FK `TMEMBER`) → ④ `TAUTH_CODE`(FK `TMEMBER`·`TCOMPANY` 널) → ⑤ `TEMPLOY_VERIFICATION`(FK `TMEMBER`·`TCOMPANY`) → ⑥ `TEMPLOY_VRF_REQUEST`(FK `TMEMBER`·`TCOMPANY`) → ⑦ `TBENEFIT_EDIT_LOG`(FK `TCOMPANY_BENEFIT`·`TMEMBER`). `TCOMPANY`·`TCOMPANY_BENEFIT`는 참조 대상이므로 이보다 선행 생성(SP-DB-3·5).

### SP-DB-17.1 `TMEMBER` (회원 — 무비밀번호·PII 최소)

```sql
CREATE TABLE IF NOT EXISTS TMEMBER (
  MBR_ID          INT AUTO_INCREMENT PRIMARY KEY COMMENT '회원 PK',
  LOGIN_EMAIL_NM  VARCHAR(255) DEFAULT NULL COMMENT '로그인 이메일 (UNIQUE, 탈퇴 시 NULL 파기 — INV-8). PII 2종 중 1',
  NICKNAME_NM     VARCHAR(30)  NOT NULL COMMENT '공개 닉네임 (UNIQUE, 편집이력 표시용). PII 2종 중 2',
  STATUS_CD       VARCHAR(12)  NOT NULL DEFAULT 'active' COMMENT '회원 상태 (active, withdrawn) — ENUM 금지·값집합 SP-DB-17.8',
  INS_ID  INT COMMENT '입력자 ID',
  INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '입력 일시',
  MOD_ID  INT COMMENT '수정자 ID',
  MOD_DTM TIMESTAMP NULL DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '수정 일시',
  UNIQUE KEY uq_member_email    (LOGIN_EMAIL_NM),
  UNIQUE KEY uq_member_nickname (NICKNAME_NM)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='회원 (무비밀번호·PII=이메일·닉네임 2종뿐, T9·AU-3)';
```

- **PII 정확집합(AU-3)**: 데이터 컬럼은 `LOGIN_EMAIL_NM`·`NICKNAME_NM` 2종뿐. 비밀번호·실명·전화·주소 컬럼 **부재**. `MBR_ID`·`STATUS_CD`·감사4는 비-PII.
- `LOGIN_EMAIL_NM`은 탈퇴 시 `NULL`로 파기(닉네임·편집이력 존치, 약관 T5·개인정보 P7). `UNIQUE`는 MySQL에서 다중 NULL 허용이라 탈퇴 다건과 양립.

### SP-DB-17.2 `TSESSION` (세션 — 불투명 토큰 SHA-256 해시)

```sql
CREATE TABLE IF NOT EXISTS TSESSION (
  SESSION_ID      INT AUTO_INCREMENT PRIMARY KEY COMMENT '세션 PK',
  MBR_ID          INT          NOT NULL COMMENT '회원 FK (TMEMBER.MBR_ID)',
  TOKEN_HASH_VAL  CHAR(64)     NOT NULL COMMENT '세션 토큰 SHA-256 해시 (원문 무저장·쿠키 전용, T9). UNIQUE 조회키',
  EXPIRES_DTM     DATETIME     NOT NULL COMMENT '만료 일시 (INS + session_ttl_days)',
  REVOKED_DTM     DATETIME     DEFAULT NULL COMMENT '폐기 일시 (로그아웃·강제만료 시 세팅, FR-104)',
  INS_ID  INT COMMENT '입력자 ID',
  INS_DTM TIMESTAMP DEFAULT CURRENT_TIMESTAMP COMMENT '입력 일시',
  MOD_ID  INT COMMENT '수정자 ID',

  FOREIGN KEY (MBR_ID) REFERENCES TMEMBER(MBR_ID) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='세션 (SHA-256 해시만·원문 토큰 컬럼 부재, T9·AU-4)';
```

- 원문 토큰 컬럼 **부재**(AU-4). 검증은 `TOKEN_HASH_VAL=%s AND REVOKED_DTM IS NULL AND EXPIRES_DTM > UTC_TIMESTAMP()`(SP-AUTH-4). 회원 탈퇴 시 세션 `CASCADE` 정리.

### SP-DB-17.9 참여 도메인 delta·테스트 영향 (ML-B)

- **SP-DB-11 해제**: `TMEMBER` 제거 목록에서 삭제(위 소유 확정). `TSOCIAL_ACCOUNT`·`TEMAIL_VERIFICATION`은 잔류.
- **SP-DB-15 delta 갱신**: 유지 5 + **참여 7 = 12테이블**. 제거 테이블 16→15(TMEMBER 제외).
- **SP-DB-16 테스트 재명세(→ Tier-0 test respec, conftest.py·test_schema_load.py)**: SC-1 존재 테이블 5→12; SC-2 제거 16→15; **SC-6 재정의** — 참여 테이블의 `FK→TMEMBER`는 이제 **정상**이므로 SC-6은 "프로파일러·소셜·구 제거 테이블 FK 부재"로 좁힌다(TMEMBER 참조 허용); **신규 AU-3**(TMEMBER PII 정확집합 `{LOGIN_EMAIL_NM, NICKNAME_NM}`·비밀번호/실명/전화 컬럼 부재)·**AU-4**(TSESSION·TAUTH_CODE·TEMPLOY_VERIFICATION 원문 토큰/코드/회사이메일 컬럼 0·`*_HASH_VAL`만). conftest `REMOVED_TABLES`에서 TMEMBER 제거 + `TABLE_CREATE_ORDER`에 7종 위상순 추가.
- **DC-6 게이트(test_data_contract)**: `BADGE_CD` 허용집합 2값→3값(`{official, est, verified}`, SP-DB-16.3 DC-6 반영 완료). DEC-2 백필은 `est`→`official`만 승격하고 `verified`는 사용자 기여 경로만 생성.
- **참여 신규 제약·값집합 커버리지(CN/DC 확장, Tier-0 test respec)**: 참조 5테이블만 대상이던 CN-*(제약)·DC-*(값집합 린트)를 참여 테이블로 확장 — 신규 UNIQUE 5종(`TOKEN_HASH_VAL`·`COMP_EMAIL_HASH_VAL`·`LOGIN_EMAIL_NM`·`NICKNAME_NM`·`EMAIL_DOMAIN_NM`) 위반 삽입 IntegrityError, FK CASCADE/SET NULL 거동, 참여 5개 CD 값집합(`STATUS_CD`·`PURPOSE_CD`·`VRF_METHOD_CD`·`EDIT_TYPE_CD`) 린트. SP-DB-17.8이 값집합을 앱/린트 강제로 위임했으므로 이 게이트가 강제 지점.
- **`run_tests.sh` 백업(T-13.2.1 최우선)**: 참여 7테이블을 TCOMPARE_LOG와 함께 dump→restore 대상에 포함(공유 스키마 DROP/CREATE로부터 회원 데이터 보호).
