# 로그인·재직인증·복지 편집 기능 도입 계획

작성일: 2026-07-21 · 상태: **계획 확정(사용자 승인) — 구현 미착수**
목적: "로그인 기능 추가" 요청의 실행 계획 정본. 착수 시 본 문서의 ML-A(문서 개정)부터 순서대로 진행한다.
분석 방식: 3개 병렬 탐사(아키텍처·등록/수정 흐름·docs/TASK 계획) 후 설계 검토. 가드 테스트 원문·nginx location 우선순위·스키마 명명 관례는 코드로 직접 재검증 완료.

## §A 배경과 요구사항

사용자 요청 4건:
1. 로그인 없이도 사이트 열람은 지금처럼 가능. 단 **회사·복지 등록/수정은 로그인 사용자만**.
2. **회원 가입은 최대한 간단히** — 개인정보 보관 최소화.
3. 특정 회사의 복지를 수정·추가하려는 사용자가 **진짜 그 회사 재직자인지 검증** 필요.
4. 참고 사이트: 나무위키.

### 사용자 확정 결정 (2026-07-21, AskUserQuestion)

| # | 결정 항목 | 확정안 |
|---|---|---|
| 1 | 구현 범위 | 로그인 + 재직인증 + **복지 등록·수정 API/UI까지**. 회사 신규 등록은 제외(INV-6 회사 ~96 유지) |
| 2 | 가입/로그인 방식 | **무비밀번호 이메일 6자리 코드 로그인**. 저장 PII = 로그인 이메일 1개 + 닉네임(비식별). 세션 ~30일 |
| 3 | 재직 검증 | **회사 도메인 이메일 코드 인증**(블라인드 방식). 인증 후 회사 이메일 원문 파기, HMAC 해시만 보관. 도메인 미등록 회사는 운영자 수동 승인 폴백 |
| 4 | 편집 이력 | 나무위키식 **이력 테이블**(누가·언제·before→after 스냅샷) |

### B-4. 완전 신규 구축이 필요한 것

- 이메일 발송: SMTP 클라이언트·템플릿·설정 슬롯 전무(금지 주석까지 존재).
- DB 쓰기 계층: `server/database.py`는 계약상 read-only(익명 로그 2연산 제외).
- 세션/쿠키: `Set-Cookie` 코드 0, 세션 저장소 없음.

## §C 핵심 설계 결정 (근거 포함)

| 결정 | 내용 | 근거 |
|---|---|---|
| 세션 구현 위치 | 미들웨어가 아닌 **FastAPI 의존성**(`require_member`) — 참여 라우터에만 부착 | Tier-0 TS-2 어서션(`middleware == ["CORSMiddleware"]`) **원문 그대로 유지** 가능. 익명 GET 경로는 세션 코드 미경유 |
| 신규 의존성 | **0개** — 토큰 `secrets`, 해시 `hmac`/`hashlib`, 메일 stdlib `smtplib`(+`asyncio.to_thread`) | 패스워드리스라 JWT/OAuth/passlib 자체가 불필요 → "인증 라이브러리 금지" 정신 유지 |
| 세션 토큰 | 불투명 랜덤(`secrets.token_urlsafe(32)`), DB에는 SHA-256 해시만. 쿠키 `loupit_sid`: `HttpOnly; Secure; SameSite=Lax; Max-Age=2592000; Path=/api/v1` | DB 유출 내성, 서버측 폐기 가능(로그아웃·탈퇴 즉시 무효화) |
| CSRF | SameSite=Lax + 참여 쓰기 라우트의 `X-Loupit-Client` 헤더 필수(nginx 게이트 + **앱 레벨 이중 검사**) | 커스텀 헤더는 크로스오리진에서 preflight 없이 부착 불가, preflight는 CORS 허용목록+`allow_credentials=False`에서 실패 — 표준 custom-header CSRF 방어와 동일 구조. CORS `allow_credentials=False` **유지**(프로덕션은 nginx 동일 오리진 프록시라 쿠키에 CORS 불관여) |
| TCOMPANY_BENEFIT | **DDL 무변경** | 편집자 기록은 휴면 INS_ID/MOD_ID에 MBR_ID 기입으로 활성화, 상세 이력은 신규 TBENEFIT_EDIT_LOG. VERIFIED_BY_ID 부재 가드 유지 |
| 사용자 편집의 배지 | `BADGE_CD='verified'`(**신규 제3값**, 라벨 "재직자 확인") · `AMT_SOURCE_CD='estimated'`(±20%) · `BADGE_SRC_CD='user_report'` · `EXPIRES_DTM=+18개월`(DG-1 TTL 재사용) | `est` 재사용 시 Tier-0 "백필 후 est=0" 게이트가 깨짐 → 제3값이 게이트 무손상 유일해. 재직자 1인의 금액 주장에 stated(±5%)는 과신 — 보수적 ±20%. calc.js 밴드 로직 무변경(INV-5 금액 신뢰도-배지 디커플), 표시 계층만 확장. official 행을 사용자가 수정하면 verified로 강등 |
| 삭제 | v1은 사용자 DELETE 라우트 없음(운영자 CLI 전용, EDIT_TYPE_CD='delete' 로그) | 소프트삭제를 번들·정적생성·calc 전 계층에 심는 침습 회피 + 반달리즘 표면 축소. 소프트삭제 컬럼은 2차 확장 |
| 동시성 | PUT에 `base_dtm`(읽은 행의 `MOD_DTM ?? INS_DTM`) 필수 → 불일치 시 **409 + 현재 행 반환** | 버전 컬럼 추가 없이 기존 감사 컬럼 재사용. 클라이언트는 diff 표시 후 재시도 |
| 닉네임 | 로그인 시 자동 생성(`직장인-{랜덤 base36 6자}`) → 마이페이지에서 변경 | 코드 1회성과 충돌하는 2단계 가입 흐름 제거(가입 최간소화) |
| 관리자 폴백 | 웹 패널 없이 **운영자 CLI**(`python -m server.ops`) | 기존 시드/운영 스크립트 관례. 관리 UI 구축 비용 회피 |
| 회사 도메인 | 별도 테이블(회사당 복수 도메인). **빈 시드 시작** → 점진 등록, 미등록 회사는 자동으로 수동 승인 폴백 | 그룹사 공유 도메인 오인증 방지(보수적 등록, 애매하면 수동) |
| 이메일 정규화·검증 | stdlib 정규식 + 소문자 정규화(email-validator 의존 회피). 해시는 `HMAC-SHA256(pepper, email)` — pepper는 env 필수값 | 의존성 0 원칙 + 레인보우 테이블 방어 |

## §D DB 설계

### D-1. 신규 7테이블 (schema.sql 정본 + `db/migrations/20260722_participation_tables.sql`)

기존 명명 관례 준수: T-접두, `_ID/_CD/_NM/_DTM/_VAL/_CTNT/_YN/_NO` 접미, 전 컬럼 COMMENT, `uq_`/`idx_` 키, InnoDB utf8mb4, `CREATE TABLE IF NOT EXISTS` 멱등. 생성 순서(FK): TMEMBER → TSESSION·TAUTH_CODE → TCOMPANY_EMAIL_DOMAIN → TEMPLOY_VERIFICATION → TEMPLOY_VRF_REQUEST → TBENEFIT_EDIT_LOG.

| 테이블 | 역할 | 핵심 컬럼 |
|---|---|---|
| `TMEMBER` | 회원(패스워드리스 — 비밀번호 컬럼 영구 부재) | `MBR_ID` PK · `LOGIN_EMAIL_NM` VARCHAR(255) NULL UNIQUE(유일 PII, 탈퇴 시 NULL 파기) · `NICKNAME_NM` VARCHAR(30) NOT NULL UNIQUE · `STATUS_CD`(active/banned/deleted) · `ADMIN_YN` · `LAST_LOGIN_DTM` · 감사 4종 |
| `TSESSION` | 서버측 세션(감사 면제 — append+revoke 전용) | `SESSION_ID` PK · `TOKEN_HASH_VAL` CHAR(64) UNIQUE(SHA-256 hex, 원문 무저장) · `MBR_ID` FK CASCADE · `EXPIRES_DTM`(+30일) · `LAST_SEEN_DTM` · `REVOKED_DTM` |
| `TAUTH_CODE` | 이메일 코드 챌린지(로그인·재직인증 공용, 일 1회 퍼지) | `CODE_ID` PK · `PURPOSE_CD`(login/employ_verify) · `EMAIL_HASH_VAL` CHAR(64)(HMAC, 원문 무저장) · `CODE_HASH_VAL` CHAR(64) · `COMP_ID`/`MBR_ID` NULL(employ_verify만) · `ATTEMPT_CNT`(상한 5) · `EXPIRES_DTM`(+10분) · `CONSUMED_DTM`(1회용) |
| `TCOMPANY_EMAIL_DOMAIN` | 회사별 허용 도메인(빈 시드 시작) | `COMP_DOMAIN_ID` PK · `COMP_ID` FK CASCADE · `EMAIL_DOMAIN_NM` VARCHAR(100)(소문자 정규화) · UNIQUE(COMP_ID, 도메인) · 감사 4종 |
| `TEMPLOY_VERIFICATION` | 재직 인증(회사 이메일 원문 무저장) | `EMP_VRF_ID` PK · `MBR_ID`+`COMP_ID` UNIQUE · `COMP_EMAIL_HASH_VAL` CHAR(64) NULL UNIQUE(중복 인증 차단 키 — NULL 다중 허용=수동 승인 무영향) · `VRF_METHOD_CD`(domain/manual) · `VERIFIED_DTM` · `EXPIRES_DTM`(+365일 재인증) · `REVOKED_DTM` · 감사 4종 |
| `TEMPLOY_VRF_REQUEST` | 수동 승인 큐(운영자 CLI 소비) | `VRF_REQ_ID` PK · `MBR_ID`·`COMP_ID` FK · `EVIDENCE_CTNT` VARCHAR(500)(재직 소명 — 기밀·타인정보 금지 안내) · `STATUS_CD`(pending/approved/rejected) · `DECIDED_BY_ID`·`DECIDED_DTM`·`REJECT_RSN_CTNT` · 감사 4종 |
| `TBENEFIT_EDIT_LOG` | 복지 편집 이력(append-only, 공개 열람) | `EDIT_LOG_ID` BIGINT PK · `COMP_ID` FK · `BENEFIT_ID` FK **SET NULL**(행 삭제 후 이력 보존) · `BENEFIT_CD` 비정규화 · `MBR_ID` FK · `EDIT_TYPE_CD`(create/update/delete) · `BEFORE_VAL`/`AFTER_VAL` JSON 스냅샷(롤백 재료) · `EDIT_NOTE_CTNT` VARCHAR(200)(편집 요약) · `INS_DTM` |

## 부록 — 사전 검증에서 확인한 사실 (계획의 전제)

1. nginx `^~ /api/v1/` 블록(loupit.conf:200)은 매칭 시 정규식 location 평가를 생략 — 경로변수 라우트의 nginx 정규식 리밋 불가(§I 전략의 근거). 고정 경로는 `=` 블록 관례(log·health) 존재.
2. TS-2 어서션은 `middleware_names == ["CORSMiddleware"]` 정확 비교(test_surface.py:64) — 세션을 의존성으로 구현하면 원문 유지 가능.
3. `test_database.py:93`은 database 모듈의 쓰기 심볼을 dir() 스캔으로 정확 열거 — 쓰기 헬퍼 추가 시 재명세 필수.
4. `server/requirements.txt`에 `pydantic-settings` pin 누락(기존 결함) — ML-G에서 정정.
5. 테스트가 서빙 스키마를 DROP/CREATE 재사용(conftest + run_tests.sh) — §L 최상위 리스크의 근거.
6. CORS `allow_credentials=False`(main.py:81)는 유지 — 프로덕션 same-origin 프록시 구조에서 쿠키 세션과 충돌 없음.
