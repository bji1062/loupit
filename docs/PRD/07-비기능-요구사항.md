# 비기능 요구사항 (NFR)

**목적**: loupit(로그인 없는 회사 복지·연봉 비교 도구 + 광고 수익)의 품질 속성(성능·SEO·접근성·개인정보·보안·가용성·자원)을 검증 가능한 기준으로 정의한다.

**참조 브리프 섹션**: §2(핵심 결정 1~9), §3(MVP 범위), §6(읽기전용 API·캐시), §8(기술 스택), §9(문서 규칙·언어)

**상위 추적**: 본 문서는 PRD 하위 항목이며 브리프의 확정 결정에 근거한다. 하위 문서(FRD/SPEC/TASK)는 아래 NFR ID를 인용한다.

> 표기: 측정 대상 기준값은 검증(수용) 조건이다. "Good" CWV 임계값은 Google 공개 표준(모바일 필드값 75 백분위)을 채택한다.

---

## 1. 성능 (Performance)

정적 페이지 우선 원칙(§2-7, §8)에 따라 색인·광고 대상 콘텐츠 페이지는 요청 시 서버 렌더링 없이 사전 생성물을 서빙한다.

| ID | 요구사항 | 측정/수용 기준 | 근거 |
|----|----------|----------------|------|
| NFR1 | 회사 상세 200개 + 인기 비교 조합 페이지는 빌드타임 정적 HTML로 생성해 Nginx가 직접 서빙한다. 요청 시 서버 렌더링·DB 조회 없음. | 정적 경로 응답에 백엔드(Uvicorn) 호출·DB 쿼리 0건. 파일은 파이썬 생성 스크립트 산출물. | §2-7, §8 |
| NFR2 | Core Web Vitals가 "Good" 기준을 충족한다. | LCP ≤ 2.5s, INP ≤ 200ms, CLS ≤ 0.1 (모바일, 필드 75 백분위). Lighthouse/CrUX로 검증. | §8, 작성지시 |
| NFR3 | 프론트는 부팅 시 `GET /api/v1/reference/all` 단일 번들만 로드하고 브라우저·CDN 캐시로 재요청을 줄인다. | 응답 헤더 `Cache-Control: public, max-age=3600`. 1시간 내 재방문 시 참조 데이터 네트워크 재요청 없음. | §6 |
| NFR4 | 프론트는 빌드/번들러 없이 동작하는 경량 바닐라 JS(ES6+)·HTML5·CSS3로 구성한다. | 트랜스파일·번들 산출물 없음. Fetch API·localStorage 사용. 초기 JS 다운로드 최소화. | §8 |
| NFR5 | 폰트(Pretendard) 로딩·광고 슬롯 삽입으로 인한 레이아웃 이동을 방지한다. | 광고/이미지 슬롯은 고정 크기 예약. `font-display` 지정으로 CLS ≤ 0.1 유지. | §8, §2-6 |

## 2. SEO

SEO는 수익의 뼈대(§2-7). 모든 정적 페이지는 개별 메타데이터를 갖는다.

| ID | 요구사항 | 측정/수용 기준 | 근거 |
|----|----------|----------------|------|
| NFR6 | 각 정적 페이지(회사·조합·랜딩·정책)는 고유한 `<title>`과 `meta description`을 갖는다. | 페이지별 중복 없는 title/description. 회사명·조합명 반영. | §2-7, §3(F4/F5) |
| NFR7 | OpenGraph 태그(og:title, og:description, og:type, og:url, og:image)를 포함한다. | 소셜 공유 시 미리보기 정상 렌더. 페이지별 값 일치. | 작성지시 |
| NFR8 | 회사 상세 페이지는 JSON-LD 구조화 데이터(Organization 유형: 기업명·산업 등)를 포함한다. | 구조화 데이터 테스트 통과, 파싱 오류 0건. | 작성지시, §4(TCOMPANY) |
| NFR9 | 빌드타임에 전체 정적 URL을 포함한 `sitemap.xml`을 생성한다. | 생성물 검증 테스트(§10): 회사 200개 + 조합 + 정책·랜딩 URL 누락 0. | §2-7, §8 |
| NFR10 | 사이트 루트에 `robots.txt`를 제공하고 sitemap 위치를 명시한다. | `robots.txt`가 크롤링 정책 + `Sitemap:` 라인 포함. | 작성지시 |
| NFR11 | 각 페이지는 자기 자신을 가리키는 `<link rel="canonical">`을 갖는다. | 조합 페이지의 역순 쌍 등 중복 URL은 단일 canonical로 정규화. | 작성지시, §3(F5) |
| NFR12 | 콘텐츠는 HTTPS(loupit.co)로 서빙되고 JS 없이도 크롤러가 본문을 읽을 수 있다. | Let's Encrypt TLS. 정적 페이지 본문이 초기 HTML에 존재(비-JS 렌더). | §1, §2-9, §8 |

## 3. 접근성 (Accessibility)

## 5. 보안 (Security)

| ID | 요구사항 | 측정/수용 기준 | 근거 |
|----|----------|----------------|------|
| NFR20 | 백엔드는 읽기 전용이다. 쓰기 엔드포인트·인증 미들웨어가 없다. | API 표면은 health/reference/all/companies.search/companies.{id}로 한정, 모두 GET. POST/PUT/DELETE 없음. | §2-4, §6 |
| NFR21 | 렌더링 시 회사·복지 데이터 및 사용자 입력을 HTML 이스케이프하여 XSS를 차단한다. | 복지 NOTE_CTNT/별칭/입력값을 `innerHTML` 직접 삽입 금지, 이스케이프/textContent 사용. XSS 페이로드 주입 테스트 통과. | §4, §7, 작성지시 |
| NFR22 | 비밀정보(개인 액세스 토큰(PAT)·DB 자격증명·배포 토큰)를 클라이언트 코드·정적 산출물·저장소에 노출하지 않는다. | 프론트/HTML에 시크릿 없음. DB 자격증명은 서버측 한정. AdSense client id는 플레이스홀더. 저장소 스캔에서 PAT 검출 0. | §8, 작성지시 |
| NFR23 | 전 구간 HTTPS(TLS)로 서빙한다. | Nginx + Let's Encrypt. HTTP→HTTPS 리다이렉트, 유효 인증서. | §2-9, §8 |

## 6. 오프라인 / 점진적 향상 (Progressive Enhancement)

| ID | 요구사항 | 측정/수용 기준 | 근거 |
|----|----------|----------------|------|
| NFR24 | 정적 콘텐츠 페이지(회사·조합·정책)는 JavaScript 없이도 본문 콘텐츠가 표시된다. | JS 비활성 상태에서 기업정보·복지 본문 가독. 인터랙션(비교 CTA)만 JS 향상. | §2-7, §3(F4/F5), NFR12 |
| NFR25 | localStorage 미지원·차단·용량초과 시에도 비교 기능은 정상 동작한다. | 저장 실패를 try/catch로 흡수, 오류 없이 진행. "최근 비교"만 미저장. | §2-2, §7 |
| NFR26 | `reference/all`·검색 API 실패 시 사용자에게 오류 상태를 알리고 앱이 크래시하지 않는다. | 네트워크 실패 시 재시도/안내 표시. 비교 계산은 클라이언트 사이드라 로드 성공 후 오프라인에서도 계산 가능. | §2-8, §6, §7 |

## 7. 자원 제약 (Resource Constraints)

Oracle Cloud ARM Always Free(춘천) 단일 인스턴스에서 동작해야 한다(§2-9).

| ID | 요구사항 | 측정/수용 기준 | 근거 |
|----|----------|----------------|------|
| NFR27 | 백엔드는 Always Free 자원 한도 내에서 동작하는 슬림 read-only 서비스로 유지한다. | FastAPI async + aiomysql 커넥션 풀. 상시 백그라운드 잡·무거운 처리 없음. 비교 계산은 클라이언트로 오프로드. | §2-4, §2-8, §8 |
| NFR28 | 트래픽·DB 부하를 정적 서빙과 캐시로 흡수한다. | 정적 파일은 Nginx 직접 서빙, `/api/v1/*`만 Uvicorn 프록시. `reference/all` 1시간 캐시로 DB 재조회 최소화. | §2-9, §6 |
