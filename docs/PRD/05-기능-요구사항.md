# 기능 요구사항

**목적**: loupit MVP(1차 릴리스)가 반드시 제공해야 하는 기능(F1~F8)을 확정 범위에 근거하여 정의하고, 하위 문서(USECASE/FRD)가 인용할 안정적 ID를 부여한다.

**참조 브리프 섹션**: §3(MVP 범위, F1~F8 정의) 근거. §2(확정 결정), §4(데이터 모델), §6(API 표면), §7(비교 알고리즘)을 관련 항목에서 참조.

**상위 추적성**: 본 문서는 PRD의 기능 요구사항 항목이며, USECASE→PRD, FRD→USECASE 체인의 뿌리다. 각 기능 ID(F1~F8)는 하위 문서에서 그대로 인용한다.

---

## 요구사항 인덱스

| ID | 기능명 | 유형 | 렌더링 |
|----|--------|------|--------|
| F1 | 회사 검색 + 선택 | 인터랙션 | 클라이언트(SPA) |
| F2 | 비교 입력 | 인터랙션 | 클라이언트(SPA) |
| F3 | 클라이언트 비교 리포트 생성 | 계산/출력 | 클라이언트(SPA) |
| F4 | 회사 상세 페이지 ~96개 | 콘텐츠 | 정적(빌드타임, SEO) |
| F5 | 인기 비교 조합 페이지 | 콘텐츠 | 정적(빌드타임, SEO) |
| F6 | 광고/제휴 슬롯 | 수익화 | 정적+클라이언트 |
| F7 | 정책 페이지 | 규정 준수 | 정적 |
| F8 | 읽기 전용 API | 백엔드 | FastAPI(읽기 전용) |

전역 제약(이 익명 열람·비교 흐름 공통, §2): 익명 흐름은 로그인/인증/회원/서버 사용자저장 없음(무로그인·무저장 유지). 로그인·회원·계정은 복지 등록·수정 기여(SC14, 문서 09/13·SP-AUTH) 한정 In-scope이며 이 익명 흐름/모듈엔 비등장. 사용자 데이터는 브라우저 localStorage에만 저장. 비교 계산은 100% 클라이언트. 백엔드 쓰기는 익명 비교 로그 POST 1종(POST /comparisons/log)에 한정되고, SC14 기여 쓰기는 세션·재직 검증 통과 요청에 한정된다.

**MVP 제외(§3, 참고)**: 프로파일러(영구 제외), 복지 카테고리 가이드 콘텐츠, 회사 랭킹/리스트 페이지(2차 검토), 사용자 계정·저장·신고 기능.

## F1. 회사 검색 + 선택

**설명**: 사용자가 비교 대상 회사를 이름 또는 별칭으로 검색하여 선택한다. 별칭 매칭으로 약칭·영문·통칭 입력을 정식 회사로 연결한다. 현직(a)과 이직처(b) 두 슬롯 각각에 회사를 지정하는 진입점이다.

**주요 입력**
- 검색어 텍스트(회사명/별칭 일부)
- 선택 대상 슬롯(a=현직 / b=이직처)

**주요 출력**
- 매칭 회사 후보 리스트(최대 20건): `{comp_id, comp_nm, comp_tp_cd, industry_nm, logo_nm}`
- 선택된 회사가 해당 슬롯에 반영된 상태

**개괄적 수용기준**
1. 정식명 또는 별칭 일부 입력 시 해당 회사가 후보에 노출된다(§6 `GET /api/v1/companies/search?q=`, 이름/별칭 LIKE, 최대 20건).
2. 후보 선택 시 선택 회사의 유형·산업·복지 데이터가 비교 입력(F2)에 연동된다(§4 TCOMPANY/TCOMPANY_ALIAS/TCOMPANY_BENEFIT 기준).
3. a·b 두 슬롯에 서로 다른 회사를 각각 지정할 수 있다.
4. 검색은 로그인 없이 동작하며 사용자별 서버 저장을 발생시키지 않는다(§2).

**참조**: 검색 API는 §6, 별칭 데이터는 §4 TCOMPANY_ALIAS.

## F2. 비교 입력

**설명**: 두 직장(현직=a / 이직처=b)의 조건을 입력한다. 연봉, 복지 항목 체크/금액, 근무형태, 통근시간, 비교 우선순위, 희생요소를 사용자가 직접 지정한다. 프로파일러(가치관 진단)는 제외이며 우선순위·희생요소는 사용자가 직접 선택한다(§2-3).

**주요 입력**
- 연봉(a/b, 만원 단위 — §4 금액 단위 규칙)
- 복지 항목: 카테고리별 체크 여부 + 금액(정성적 항목은 금액 없음)
- 근무형태: 재택(remote), 유연(flex), 야근(overtime/ot), 임금유형(wage=포괄/비포괄)
- 통근시간
- 우선순위(직접 선택): 연봉 / 워라밸 / 복지 / 브랜드
- 희생요소(curSacrifice, 직접 선택): 무엇을 감수할지 사용자 선택(§7 비교 알고리즘 입력)

**주요 출력**
- a/b 각각에 대한 입력 상태 객체(연봉·복지·근무형태·통근·우선순위·희생요소)
- 실데이터 복지로 채워진 복지 항목 기본값(등록 회사 선택 시) 또는 기업유형 프리셋 템플릿 기본값(등록 회사에 없어 직접 입력 모드로 진입한 경우)

**개괄적 수용기준**
1. 우선순위(연봉/워라밸/복지)를 사용자가 직접 선택할 수 있고 프로파일러 설문은 존재하지 않는다(§2-3).
2. 희생요소(curSacrifice)를 사용자가 직접 선택할 수 있으며, 이 값은 F3 비교 리포트의 우선순위 기반 판정 입력으로 전달된다(§7 비교 알고리즘 입력, F3 주요 입력과 정합).
3. 복지 항목은 카테고리(§4 BENEFIT_CTGR_CD 9종: compensation/flexibility/work_env/time_off/health/family/growth/leisure/perks)로 구성되며 체크/금액을 지정할 수 있다.
4. 근무형태 입력에 재택·유연·야근·임금유형(포괄/비포괄)이 포함된다(§7 입력 정의와 일치).
5. 등록 회사 선택 시 실데이터 복지로 초기값이 채워지고, 등록 회사에 없는 회사는 직접 입력 모드에서 기업유형 프리셋 템플릿으로 초기값이 채워지며(§4 TBENEFIT_PRESET 직접 입력 모드, UC-15/21), 입력 화면 광고는 최소화한다(§2-6).

**참조**: 입력 항목 정의(우선순위·희생요소 포함)는 §7, 복지 카테고리·프리셋은 §4, 광고 최소화 정책은 §2-6.

## F3. 클라이언트 비교 리포트 생성

**설명**: F2의 입력을 받아 100% 클라이언트 사이드에서 비교 리포트를 생성한다(§2-8). 총보상, 시간조정 가치, 워라밸·자율성, 복지 카테고리별 델타, 우선순위 기반 판정을 두 관점(a/b)으로 산출한다.

**주요 입력**
- F2의 a/b 입력 상태(연봉, 복지 금액, 근무형태, 통근시간, 우선순위, 희생요소)
- §6 `reference/all`로 로드된 참조 데이터(기업유형 상승률·안정성, 프리셋)

**주요 출력**
- 총보상(실효연봉=연봉+순복지)
- 시간조정(시간당) 가치(주간근무시간·야근수당 반영)
- 워라밸·자율성 요인(재택절약·유연·무제한휴가)
- 복지 카테고리별 델타 + 복지 불확실성 밴드(배지별 ±)
- 우선순위 기반 "누가 유리/무엇을 택할지" 판정(vdCard)

**개괄적 수용기준**
1. 모든 계산이 클라이언트에서 수행되며 서버 쓰기/판정 호출이 없다(§2-8).
2. 실효연봉·시간당 가치·워라밸·카테고리별 복지·우선순위 판정이 리포트에 포함된다(§7 출력 정의).
3. 복지 배지에 따라 불확실성 밴드가 적용된다(§7: official ±5%, est ±20%, 만료 ±15%).
4. 기존 로그인 게이트·신고버튼·저장호출은 리포트 경로에 존재하지 않는다(§7).

**참조**: 계산·함수(compare/calc/getWSHours/getOTPay 등)와 밴드 규칙은 §7, 참조 데이터 로드는 §6 `reference/all`.

## F4. 회사 상세 페이지 ~96개

**설명**: 복지 데이터가 있는 회사(약 96개)의 상세 페이지를 빌드타임에 정적 생성한다(§2-7, §8). 기업정보·복지·근무형태를 노출하고 "이 회사로 비교하기" CTA로 비교 툴(F2)에 연결한다. SEO·AdSense 승인 콘텐츠의 핵심이다.

**주요 입력**
- 회사 데이터(§4/§5): 영문id, 정식명, 별칭[], 유형, 산업, 로고, WORK_STYLE_VAL, 복지(실데이터 — 등록 회사는 모두 실복지 보유)
- 빌드타임 파이썬 생성 스크립트 입력(§8)

**주요 출력**
- 회사별 정적 HTML 페이지 ~96개
- sitemap.xml 항목
- "이 회사로 비교하기" CTA(해당 회사를 슬롯에 지정한 비교 진입)

**개괄적 수용기준**
1. 복지 데이터가 있는 회사(약 96개) 각각에 대해 정적 HTML 페이지가 생성된다(§3 F4, §5 복지 데이터가 있는 회사).
2. 페이지에 기업정보·복지·근무형태가 표시되며, 등록 회사는 모두 실복지 데이터를 보유하므로 회사페이지 프리셋 폴백은 불필요하다(§4/§5).
3. "이 회사로 비교하기" CTA가 해당 회사를 비교 입력(F2)에 연결한다.
4. 생성물이 sitemap.xml에 포함되어 검색 색인 대상이 된다(§2-7, §8).

**참조**: 데이터는 §4/§5, 생성 파이프라인은 §8, 콘텐츠 페이지 광고 배치는 §2-6.

## F5. 인기 비교 조합 페이지

**설명**: 자주 찾는 회사쌍(예: 삼성전자 vs SK하이닉스)의 비교 조합 페이지를 빌드타임에 정적 사전 생성한다(§3 F5). SEO 유입과 AdSense 콘텐츠 확보가 목적이다.

**주요 입력**
- 사전 정의된 인기 회사쌍 목록
- 각 회사의 참조·복지 데이터(§4/§5)

**주요 출력**
- 회사쌍별 정적 비교 조합 HTML 페이지
- sitemap.xml 항목
- 해당 조합을 비교 툴(F2/F3)에서 여는 진입

**개괄적 수용기준**
1. 정의된 인기 회사쌍에 대해 정적 조합 페이지가 빌드타임에 생성된다(§3 F5).
2. 페이지가 두 회사의 핵심 비교 정보를 노출하고 비교 툴 진입 경로를 제공한다.
3. 생성물이 sitemap.xml에 포함되어 검색 색인 대상이 된다(§2-7).
4. 조합 페이지는 콘텐츠 페이지로서 광고 배치 대상이다(§2-6).

**참조**: 데이터는 §4/§5, 생성 파이프라인은 §8, 광고 배치 정책은 §2-6.

## F6. 광고/제휴 슬롯

**설명**: Google AdSense(자동광고 + 수동 슬롯)와 제휴(affiliate) 링크로 수익화한다(§2-5). 제휴 컴포넌트는 `affiliate.json` 데이터 기반이다. 광고/제휴는 "광고"로 명시하여 공정위 표시광고 규정을 준수한다.

**주요 입력**
- AdSense 스니펫(플레이스홀더 client id, §8)
- 제휴 데이터 `affiliate.json`
- 배치 컨텍스트(페이지 유형: 랜딩/회사/인기조합/리스트/비교 입력/비교 결과)

**주요 출력**
- 페이지 내 렌더링된 광고 슬롯(자동/수동)
- `affiliate.json` 기반 제휴 링크 컴포넌트
- "광고" 표기가 부착된 광고/제휴 영역

**개괄적 수용기준**
1. AdSense 자동광고와 수동 슬롯, `affiliate.json` 기반 제휴 컴포넌트를 모두 지원한다(§2-5, §8).
2. 광고 배치는 콘텐츠 페이지 위주(랜딩·회사·인기조합·리스트)이며 비교 입력 화면은 광고 최소화, 비교 결과 화면은 절제된 슬롯만 허용한다(§2-6).
3. 모든 광고/제휴 영역에 "광고" 표기가 명시된다(§2-5, 공정위 표시광고 규정).
4. 제휴 링크는 `affiliate.json` 데이터로 구성되어 데이터 교체가 코드 수정 없이 가능하다(§2-5).

**참조**: 배치 정책은 §2-6, 표기 규정은 §2-5, 스택은 §8, 고지 페이지는 F7.

## F7. 정책 페이지

**설명**: 서비스 운영에 필요한 정책·고지 페이지를 정적으로 제공한다(§3 F7). 개인정보처리방침, 이용약관, 데이터 정확성 면책조항, 광고/제휴 고지를 포함한다.

**주요 입력**
- 정책 문안(개인정보처리방침/이용약관/면책/광고·제휴 고지)

**주요 출력**
- 정책 정적 페이지 4종(개인정보처리방침, 이용약관, 면책조항, 광고/제휴 고지)

**개괄적 수용기준**
1. 개인정보처리방침·이용약관·면책조항(데이터 정확성)·광고/제휴 고지 페이지가 각각 존재한다(§3 F7).
2. 개인정보처리방침은 서버 사용자저장이 없고 데이터는 localStorage에만 저장된다는 사실(§2-1, §2-2)과 정합한다.
3. 광고/제휴 고지가 F6의 "광고" 표기 및 공정위 표시광고 규정과 정합한다(§2-5).
4. 면책조항이 비교 데이터·복지 정보의 정확성 한계를 명시한다(§3 F7).

**참조**: 데이터 저장 정책은 §2-1/§2-2, 광고/제휴 표기는 §2-5 및 F6.

## F8. 읽기 전용 API

**설명**: 프론트가 사용하는 슬림 읽기 전용 FastAPI 표면(§2-4, §6). 익명 쓰기는 비교 로그 POST 1종(POST /comparisons/log)에 한정되고 SC14 기여 쓰기는 세션·재직 검증 통과 요청에 한정되며, 인증 미들웨어는 없음. 참조 번들과 회사 검색·상세, 인기 비교 조합(트렌딩), 헬스체크를 제공한다.

**주요 입력**
- 검색어 `q`(companies/search)
- 회사 식별자 `comp_id`(companies/{id})

**주요 출력**
- `GET /api/v1/health`: 헬스체크
- `GET /api/v1/reference/all`: 전체 참조 번들(company_types[], benefit_presets{type_cd:[…]}, companies[{…, aliases[], benefits[]}]), 1시간 캐시(`Cache-Control: public, max-age=3600`)
- `GET /api/v1/companies/search?q=`: `{comp_id, comp_nm, comp_tp_cd, industry_nm, logo_nm}` 최대 20건
- `GET /api/v1/companies/{comp_id}`: `{…, work_style_val, aliases[], benefits[]}`(등록 회사는 실복지 보유 → 프리셋 폴백 병합 불필요, 없는 comp_id는 404)
- `GET /api/v1/comparisons/trending`: 인기 비교 조합(트렌딩) 목록(§6)

**개괄적 수용기준**
1. 5개 엔드포인트(health, reference/all, companies/search, companies/{id}, comparisons/trending)가 명세대로 응답한다(§6).
2. 익명 쓰기는 비교 로그 POST 1종에 한정되고 SC14 기여 쓰기는 세션·재직 검증 통과 요청에 한정되며, 인증 미들웨어는 존재하지 않는다(§2-4).
3. `reference/all`이 `Cache-Control: public, max-age=3600`를 반환하고 번들 최상위 키(company_types/benefit_presets/companies)를 포함하며 프로파일러 관련 키(profiles/job_groups/questions)는 없다(§6).
4. `companies/{id}` 응답이 실복지 데이터를 반환하며(등록 회사는 모두 실복지 보유 → 프리셋 폴백 병합 불필요), 없는 comp_id는 404를 반환한다(§4/§6).

**참조**: 엔드포인트·번들 구조는 §6, 데이터 모델은 §4, 스택(FastAPI/aiomysql/원시 SQL)은 §8. 이 API는 F1(검색), F2/F3(참조 데이터), F4(상세 데이터)의 데이터 공급원이다.
