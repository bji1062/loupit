# 제약과 가정

**목적**: loupit MVP 설계·구현이 반드시 준수해야 할 기술·법적/정책 제약과, 제품 성립의 전제가 되는 핵심 가정을 명시하여 이후 문서(FRD/SPEC/TASK)의 판단 근거를 고정한다.

**참조 브리프 섹션**: §2(확정 핵심 결정), §3(MVP 범위), §5(보존 데이터), §8(기술 스택), §9(문서 규칙)

---

## 1. 기술 제약 (Technical Constraints)

빌드·배포·런타임에서 반드시 지켜야 할 확정 제약. 위반 시 설계 자체가 성립하지 않는다.

| ID | 제약 | 근거(브리프) | 함의 / 대응 |
|----|------|------|------|
| **AS1** | 인프라는 Oracle Cloud ARM **Always Free**(춘천 리전) 단일 구성만 사용. 유료 자원·오토스케일·매니지드 DB 없음. | §2-9, §8 | CPU/메모리/네트워크가 무료 한도로 고정. 트래픽 급증 대비는 정적 파일 서빙(Nginx)과 캐시로 흡수. DB는 자체 호스팅 MySQL 8.0. |
| **AS2** | 회사 상세 ~96개(복지 데이터가 있는 회사) + 인기 비교 조합 페이지는 **빌드타임에 파이썬 스크립트로 정적 HTML + sitemap.xml 생성**. 런타임 서버사이드 렌더링 없음. | §2-7, §3(F4·F5), §8 | 데이터가 바뀌면 페이지가 자동 갱신되지 않음 → 재생성·재배포 필요. 정적 생성물은 검증 테스트 대상(§10). |
| **AS3** | 프론트엔드는 **바닐라 JS(ES6+) + HTML5 + CSS3, 빌드 단계 없음**. 번들러·트랜스파일러·npm 빌드 미사용. Fetch API·localStorage·브라우저 네이티브 기능만. | §2-8, §8 | 외부 프레임워크(React 등) 불가. 코드 분할·트리셰이킹 없음 → 파일 구조와 로딩 순서를 수동 관리. 폰트는 Pretendard. |
| **AS4** | 백엔드는 **슬림 읽기 전용 FastAPI + MySQL**. 쓰기 엔드포인트·인증 미들웨어 없음. API 표면은 §6의 4개(`health`, `reference/all`, `companies/search`, `companies/{id}`)로 한정. | §2-4, §2-8, §6 | 데이터 갱신은 seed 재적재 + 재배포로만 수행. 서버는 참조 데이터 제공만 담당, 비즈니스 계산은 하지 않음. |
| **AS5** | **비교 계산 로직은 100% 클라이언트 사이드**. 서버는 참조 데이터만 제공. | §2-8, §7 | 계산 성능·정확성은 브라우저에 의존. 순수함수로 분리하여 단위테스트 가능하게 구성(§10). |
| **AS6** | 사용자 데이터(예: "최근 비교")는 **브라우저 localStorage에만** 저장. 서버 전송·저장 없음, 사용자별 서버 상태 없음. | §2-1, §2-2 | 기기·브라우저 간 동기화 불가. 데이터 유실(캐시 삭제) 허용 범위 내 기능만 설계. 회원·로그인·소셜피드 전면 제외. |

## 2. 법적 / 정책 제약 (Legal & Policy Constraints)

수익화(광고·제휴)와 복지 데이터 노출에 따르는 규정 준수 제약.

| ID | 제약 | 근거(브리프) | 함의 / 대응 |
|----|------|------|------|
| **AS7** | 광고·제휴 링크는 **"광고"로 명시 표기**(공정거래위원회 표시·광고 규정 준수). | §2-5, §3(F6) | AdSense 자동/수동 슬롯과 affiliate 컴포넌트 모두 광고 라벨 노출 필수. 라벨 누락 시 규정 위반. |
| **AS8** | 광고 배치는 **콘텐츠 페이지 위주**(랜딩·회사 페이지·인기 조합·리스트). **비교 툴 입력 화면에서는 광고 최소화**, 결과 화면은 절제된 슬롯만 허용. | §2-6 | 입력 UX 보호가 배치 정책보다 우선. SPEC/WIREFRAME은 이 배치 규칙을 슬롯 단위로 명시해야 함. |
| **AS9** | **개인정보처리방침·이용약관·면책조항·광고/제휴 고지** 정책 페이지를 MVP에 반드시 포함·게시. | §3(F7) | localStorage·광고 쿠키(AdSense) 사용 사실을 개인정보처리방침에 기재. 정책 페이지는 릴리스 게이트 항목. |
| **AS10** | 복지 데이터에는 **출처·정확성 면책**을 명시. 데이터는 배지로 출처·신뢰도를 구분(공식/추정, 출처 URL, 검증·만료일시). | §3(F7), §4(TCOMPANY_BENEFIT), §5 | 추정 데이터는 불확실성 밴드로 처리(§7). 면책조항에 "데이터 정확성 보증 없음" 고지. 원출처 저작권·이용조건을 침해하지 않는 범위에서 노출. |

## 3. 핵심 가정 (Core Assumptions)

제품·수익 모델 성립의 전제. 거짓으로 판명되면 목표(SEO 기반 광고 수익)가 흔들리므로 검증·모니터링 대상이다.

| ID | 가정 | 근거(브리프) | 검증 방법 / 리스크 |
|----|------|------|------|
| **AS11** | 정적 회사/조합 페이지(회사 ~96개 + 조합 페이지)가 **검색 색인되어 유기적 검색 트래픽이 유입**된다. SEO가 수익의 뼈대로 작동한다. | §2-7, §3(F4·F5) | 검증: sitemap 제출 후 색인 수·유입 세션 추적(검색 콘솔). 리스크: 색인 지연·저품질 판정 시 유입 부재 → 광고 수익 미발생. |
| **AS12** | 발행 콘텐츠로 **Google AdSense 승인을 획득**한다(승인 후 자동/수동 슬롯 활성화). | §2-5, §2-7, §3(F6) | 검증: 정적 콘텐츠(회사 상세 ~96개 + 조합) + 정책 페이지 완비 후 승인 신청. 리스크: 콘텐츠·정책 미비로 승인 거절 시 수익화 지연(플레이스홀더 client id는 승인 전 임시). |
| **AS13** | 보존 대상 **큐레이션 데이터(복지 데이터가 있는 회사 ~96개·유형 6종·복지 SQL 96개 파일)가 유효**하며 신규 스키마로 재이식 가능하다. | §5, §4 | 검증: seed 재적재 + `reference/all` 응답 검증 테스트. 리스크: 데이터 노후·별칭 누락 시 검색/비교 품질 저하. 등록 회사는 모두 실복지 보유 → 회사페이지 프리셋 폴백 불필요(프리셋은 비교 툴 직접 입력 모드용, UC-15/21). |
| **AS14** | 제휴(affiliate) 프로그램을 확보하여 `affiliate.json` 기반 컴포넌트로 노출할 링크가 존재한다. | §2-5, §3(F6), §8 | 검증: affiliate.json 실데이터 확보 여부. 리스크: 제휴처 미확보 시 제휴 수익 라인 부재(AdSense 단독으로 축소). |

## 4. 추적성 메모

- **파생 문서 인용 규칙**: FRD/FLOW/SPEC/TASK는 관련 제약·가정을 `AS#`로 인용한다(예: 광고 슬롯 배치 SPEC은 AS7·AS8 인용, 정책 페이지 TASK는 AS9 인용).
- **릴리스 게이트로 승격되는 항목**: AS7(광고 라벨), AS9(정책 페이지), AS12(AdSense 승인)은 1차 릴리스 전 충족 확인 필수.
- **변경 금지**: AS1~AS6은 §2 확정 결정에서 유도된 불변 제약이다. 변경 시 브리프(진실 소스) 개정이 선행되어야 한다.
