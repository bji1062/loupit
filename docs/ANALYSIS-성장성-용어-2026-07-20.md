# 성장성 용어 정리 — 코드·문서·DB 전반 사용처 분석

작성일: 2026-07-20 · 목적: "성장성 관련 표현 정리" 작업의 범위 판단 근거
분석 방식: 7개 서브시스템(계산엔진·렌더·웹카피·테스트·SPEC/FRD·PRD/FLOW/USECASE/TASK·DB/생성기) 병렬 조사 후 통합.
검증 완료 3건: ① 문자열 원천은 `calc.js:383/388` 뿐 ② `report.js:64`는 `p.label`을 그대로 렌더(하드코딩 없음) ③ `generator/pages/company.py:101`은 `growth_rate_val`(수치) 미소비.

## 1. 결론 — 문구 vs 실제 지표

**성장성은 "단순 마케팅 문구"가 아니라 `brandProjection()`이 실제로 산출하는 계산 지표다.**
단, 동일 용어가 표면에 따라 3계층으로 존재한다:

| 계층 | 성격 | 위치 |
|---|---|---|
| ① **계산 지표** | 엔진이 실제 산출 (3년 복리 투영·누적차) | `calc.js` — 비교 리포트 '브랜드' 축 |
| ② **저장 문구 렌더** | 수치 계산 없이 라벨/점수만 표시 | `generator/` 회사·조합 정적 페이지 |
| ③ **마케팅 카피** | 순수 판매 문구 (지표 로직과 분리) | `web/*.html` meta·og·lede 5곳 |

**핵심 산식(정본: SPEC/05 §13.3~13.4, FR-39):**
- `proj[0] = salMid; proj[y] = round(proj[y-1] × (1 + growth_rate_val))` (y=1..3)
- `cumDiff = Σ(projB[y] − projA[y])` → 부호로 **성장성** 승자 판정 (`calc.js:384`)
- **안정성** 승자 = `stability_score_no` 큰 쪽 (`calc.js:389`)
- 폴백: `GROWTH_RATE_FALLBACK = 0.04`, 한쪽 회사 미선택 시 `limited = true`

## 2. ⚠️ 가장 큰 함정 — `성장성` ≠ `growth(성장)`

검색되는 `growth`/`성장` 히트의 **대다수는 복지 9카테고리 중 하나("성장·커리어" 자기계발 복지)**로, 성장성 지표와 완전히 무관하다. 성장성 작업 시 아래를 건드리면 복지표·버터플라이·복지 매트릭스가 깨진다 — **절대 손대지 말 것**:
- `CATEGORY_LABEL.growth = '성장'` (report.js:9, company.js:13)
- `CTGR9` allowlist / `BENEFIT_CATEGORIES` 9종 고정 순서 (inputs.js:13, calc.js:139)
- 시드 복지행 `BENEFIT_CTGR_CD = 'growth'` (약 90개 SQL 파일)

| 구분 | **성장성**(+안정성) — brand축 지표 | **growth(성장)** — 복지 카테고리 |
|---|---|---|
| 정체 | 비교 판정 vdCard의 브랜드 축 관점(p1 성장성 / p2 안정성) | 복지 9분류 코드 중 하나 |
| 데이터 원천 | `TCOMPANY_TYPE.GROWTH_RATE_VAL`·`STABILITY_SCORE_NO` | `TCOMPANY_BENEFIT.BENEFIT_CTGR_CD='growth'` |
| 코드 원천 | `calc.js:383/388` 문자열, `GROWTH_RATE_FALLBACK=0.04`, `PROJECTION_YEARS=3` | `CATEGORY_LABEL.growth`, `CTGR9` |
| 계산 여부 | O — 3년 복리 투영·cumDiff·승자 판정 | X — 그룹핑/정렬용 라벨 |
| 표시 라벨 | "성장성" / "안정성" (엔진 소유) | "성장" |
| 쓰이는 곳 | 비교 리포트 브랜드 카드 + 정적페이지 라벨 | 복지 델타표·버터플라이·매트릭스·회사상세 복지 그룹 |

## 4. 변경 시나리오별 파급범위

### (a) 문구만 정리 — 마케팅 카피 텍스트 변경 · 난이도 낮음
- **건드릴 파일**: `web/index.html:10,16`, `web/compare/index.html:10,17,40` (5곳).
- **테스트 파급**: 없음(이 문안을 검증하는 테스트 미발견). SEO/OG 미리보기 문구만 바뀜.

## 5. 주의점 (함정)

1. **동명이의 최대 함정 — growth ≠ 성장성**: `growth`/`성장` 히트 대다수는 복지 카테고리. `CTGR9`·`CATEGORY_LABEL.growth`·BENEFIT_CATEGORIES 순서·시드 `BENEFIT_CTGR_CD='growth'`는 성장성과 무관 → 건드리면 복지 렌더가 깨진다.
2. **성장성 문자열 원천은 딱 1곳**: `calc.js:383/388`. report.js·ui.js·company.js·inputs.js 어디에도 리터럴 없음. 라벨 변경은 엔진에서 해야 하며 report.js에서 문자열을 찾는 건 헛수고.
3. **growth_rate_val(수치)은 정적 생성기에서 dead**: 정적페이지는 수치 미소비, `growth_label_nm`(문구)만 렌더. "성장성 수치가 정적페이지에 반영된다"는 오판. 수치는 오직 비교 리포트 brand축과 서버 번들에서만 산다.
4. **성장성 vs 안정성은 한 쌍이나 소스가 다름**: 성장성=`GROWTH_RATE_VAL`(계산 입력)+`GROWTH_LABEL_NM`(표시 문구) 2컬럼 분리, 안정성=`STABILITY_SCORE_NO`(입력 겸 표시).
5. **문서 오탐**: `docs/FRD/06-리포트-표현.md:53`의 "안정성"은 렌더 배치 안정성(브랜드 지표 아님). `docs/PRD/02`의 "성장"은 UV/사업 성장(북극성 지표).
6. **테스트가 값 단위로 못박음**: 산식/폴백(0.04)/투영 연차(3년)/반환 스키마 변경 시 T-ENGINE-29/30 즉시 실패. 필드 개명은 4개 파일 픽스처 연쇄 실패. 함수 export 이름 변경은 스위트 전체 로드 실패.
7. **랜딩·SPA 셸 문안 미러링**: 마케팅 문구가 index.html/compare/index.html에 중복 → 한쪽만 고치면 불일치.

## 6. 권고

지난 랜딩 변경(특징 리스트에서 "워라밸·성장성" → "워라밸·문화")의 연장선에서는 **(a) 마케팅 문구 정리**가 위험 없이 가장 자연스럽다. 브랜드 축의 성장성/안정성 비교는 실제 계산·노출되는 기능이므로, 문구 정합만으로 랜딩·SEO에서 성장성을 더 이상 앞세우지 않게 된다. 리포트 표시 라벨 자체를 바꾸려면 (b), 기능을 걷어내려면 (c) — (c)는 4계층+문서 동시 개정이 필요한 큰 작업이다.

> 참고 근거 데이터: 이 분석은 세션 워크플로 `growth-term-audit`(8 에이전트, 68 tool_uses)의 통합 결과이며, 위 3건의 핵심 주장은 원문 재확인으로 검증됨.
