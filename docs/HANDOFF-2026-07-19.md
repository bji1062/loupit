## C. 반드시 알아야 할 함정

### C-1. 테스트는 `run_tests.sh`로만 (기존 규칙, 여전히 유효)

release 게이트 경유로만. generator·web 테스트는 DB 무관이라 직접 실행 안전:
```bash
python3 -m pytest -q generator/tests/     # DB 무접촉
node --test 'web/**/*.test.js'            # ⚠ node v24는 `node --test web/`를 모듈로 오해 — glob 필수
```

### C-2. 배포 = `RELEASE_CONFIRM=1 bash infra/deploy/release.sh`
게이트가 최선두라 실패 시 서빙 상태가 바뀌지 않는다. TCOMPARE_LOG는 백업→재시드→재주입으로 보존된다.

### C-3. ⚠️ **nginx conf는 리포→`/etc` 자동 배포가 없다**
`release.sh`는 `reload`만 하고 **conf를 복사하지 않는다**. 프로덕션 conf를 바꿨다면 수동 배치 필요:
```bash
sudo cp /etc/nginx/sites-available/loupit.conf /root/nginx-loupit.conf.bak-$(date +%Y%m%d-%H%M%S)
sudo cp infra/nginx/loupit.conf /etc/nginx/sites-available/loupit.conf
sudo nginx -t && sudo systemctl reload nginx
```
- **순서 주의**: 작업트리가 곧 라이브 문서루트라, 새 경로를 참조하는 HTML을 먼저 커밋하면
  conf 적용 전까지 그 경로가 전부 404가 된다. **conf 먼저 → 참조 변경 나중.**
- `nginx -t`는 `alias`/`root` 대상 디렉터리 존재를 검사하지 않는다(오타는 런타임 404).
  스모크 SM-15a~e가 이걸 잡는 유일한 선이다.
- **beta conf는 리포본을 복사하지 말 것** — 리포는 `beta.jobcho.wiki` 인증서를 참조하는데 미발급이라
  nginx 기동이 실패한다. 라이브 beta는 `beta.loupit.co` 변형에 필요한 블록만 제자리 패치해 둔 상태.

### C-3b. ⚠️ 큰 프록시 응답은 nginx 임시파일을 타고, 실패해도 5xx가 아니라 **조용히 잘린다**
2026-07-20 실장애. `reference/all`이 약 600KB인데 `proxy_buffers` 기본값(8×4k=32KB)으로는
메모리에 못 담아 nginx가 `/var/lib/nginx/proxy/`로 흘리는데, 그 디렉터리가 `nobody:root 0700`이라
워커(`www-data`)가 `open()`에 EACCES로 실패했다. 그러면 **응답이 그 지점에서 끊긴다** —
`Content-Length`는 이미 전체 길이(606531)로 나간 뒤라 클라이언트는 200을 받고 JSON 파싱만 실패한다.
- **증상**: F5 새로고침 시 "비교 도구를 불러오지 못했습니다. 네트워크 상태를 확인해주세요."
  절단률 약 40%라 "다시 시도"를 눌러도 자주 또 실패해 **버튼이 고장난 것처럼 보였다**
  (버튼 자체는 정상 — 실패 주입 후 클릭하면 정상 복구됨을 실증).
- **진단 지름길**: `sudo tail /var/log/nginx/error.log | grep 'Permission denied'`.
  업스트림 직접 호출(`curl http://127.0.0.1:8000/...`)은 항상 정상이므로 **API를 의심하면 헛짚는다**.
  같은 URL을 여러 번 받아 `size_download`가 들쭉날쭉하면 이 문제다.
- **조치(둘 다 적용됨)**: `loupit.conf`의 API 블록에 `proxy_buffers 16 64k` 등으로 전량 메모리 수용
  (디스크 경로를 아예 안 탐) + `provision.sh`가 `/var/lib/nginx/proxy` 소유권 보장(이중 방어).
- **회귀 가드**: `SM-16 ref body intact(5x)` 신설. 기존 `SM-4`는 **HEAD로 헤더만** 봐서 본문이
  잘려도 통과했다 — 이것이 실장애를 놓친 사각지대였다. 간헐 실패라 5회 반복 검사한다.

### C-4. 자산은 `/assets/v2/` 세대 + no-cache
- **`web/assets`에 `.gz` 사전압축 금지**(불변식). gzip_static이 원본과 신선도를 비교하지 않아
  no-cache 재검증을 조용히 무력화한다. v2 location은 `gzip_static off`.

### G-4. 정정 — 출처 URL은 애드센스 요소가 **아니다**

미복제·사실만 재서술"에서 나오고 사실 자체는 저작권 대상이 아니다.
→ 출처 URL의 근거는 **검증 가능성과 재검증 가능성(G-3)**이지 애드센스가 아니다.

## L. 스크래핑 방어 3층 — 완료(2026-07-21, 사용자 요청)

**완전 차단은 불가능하다.** 같은 복지 데이터가 95개 SEO 정적 페이지에도 있고 Googlebot이
크롤해야 애드센스 수익이 난다. 목표는 "벌크·게으른 벡터 차단 + 비용 상승"이지 "불가능하게"가
아니다. A·B·D 3층을 넣었다(C 정적 throttle은 정상 사용자 429 위험으로 사용자가 제외).

### L-1. Layer A — 데이터 GET 출처 제한(가장 큰 이득)
`reference/all`은 **인증 없는 GET 1회 = 600KB 전체 덤프**였다. rate limit(300r/m)은 1회 표적에
무력하다. → `apiFetch`(api.js)와 `trending.js`가 `X-Loupit-Client: web` 헤더를 보내고,
nginx `^~ /api/v1/` 블록이 `if ($http_x_loupit_client = "") { return 403; }`로 요구한다.
- 맨 curl·봇은 헤더 없어 403. 예외: `health`는 `= /api/v1/health` 별도 블록(모니터링),
  `comparisons/log`는 별도 블록(sendBeacon은 커스텀 헤더 불가).
- **정직한 한계**: 헤더값은 공개 JS에 노출돼 비밀이 아니다. JS를 읽는 스크래퍼는 흉내낼 수
  있다. 이 층은 **게으른/드라이브바이 벌크 스크래퍼**를 막지 완전 차단이 아니다.

### L-2. Layer B — 악성 봇 UA 차단
`infra/nginx/loupit-botdefense.conf`의 `$loupit_bad_bot` map + 서버 레벨 `if→403`.
- 차단: python-requests/scrapy/go-http/java/okhttp/headless/puppeteer/playwright/셀레늄 +
  공격적 SEO봇(Ahrefs/Semrush/MJ12/Petal 등).
- ★ **화이트리스트 우선**(map은 첫 일치 우선): Googlebot·Bingbot·네이버·다음·**AdSense
  (Mediapartners-Google·AdsBot-Google)**·소셜 크롤러는 절대 차단 금지. 실수로 막으면 SEO·수익
  사망. `curl`·`wget`은 **의도적 제외**(smoke·모니터링 보호 + Layer A가 이미 데이터 API에서 curl 차단).
- UA는 위조 가능 → Layer A·D와 다층.

### L-3. Layer D — robots.txt 강화
`generator/pages/sitemap.py`가 공격 스크래퍼 전면 Disallow + `Crawl-delay: 2`. honor-system
보조(정직한 봇만 따름). 검색·AdSense는 규칙 미적용이라 무영향.

### L-4. ★ 함정 (다음에 API 게이트 건드릴 때 반드시 볼 것)
1. **raw fetch 누락**: `apiFetch`에 헤더를 넣어도 **`trending.js`는 raw fetch**라 헤더가 빠져
   실시간 위젯이 403으로 조용히 사라졌다(부팅·검색은 정상이라 안 보였다). **데이터를 부르는
   모든 fetch 경로**(apiFetch 외 trending)를 다 찾을 것. playwright 실브라우저 검증이 아니었으면
   놓쳤다 — curl 검증만으론 위젯 파손을 못 본다.
2. **배포 순서**: 클라이언트가 헤더를 **먼저** 보내게(무해) 배포한 뒤 nginx가 요구해야 SPA
   무중단. 반대로 하면 배포~클라 반영 사이 전 사용자가 403.
3. **테스트 하네스 == 프로덕션 배선**(또 나옴): 게이트의 `loupit.test.conf`에 botdefense map
   include를 안 넣으면 `unknown "loupit_bad_bot" variable`로 게이트가 실패한다. conf.d map을
   추가하면 test.conf에도 include할 것(limits.conf와 동일).
4. **playwright는 HeadlessChrome UA라 Layer B에 막힌다** — 검증 스크립트는 `newContext({
   userAgent: 실브라우저 })`로 실제 UA를 줘야 한다.

### L-5. 배포·회귀 가드
- **conf.d 수동 배치**(§C-3 절차): `botdefense.conf` → `/etc/nginx/conf.d/`, `loupit.conf`
  → sites-available, `nginx -t`, reload. ⚠ conf.d 파일(limits·botdefense)은 provision.sh가
  복사하지 않는다 — **호스트 재구축 시 둘 다 수동 복사**(각 파일 헤더에 설치법 있음).
- 스모크: **SM-17**(무헤더 ref 403) · **SM-18**(봇UA 403) · **SM-19/19b**(Googlebot·AdSense 허용)
  · **SM-20**(health 무헤더 200). SM-4·16은 헤더 추가.
- 단위: api.test·trending.test(헤더 전송 단언 — 빠지면 앱/위젯 사망), test_sitemap_robots
  (공격봇 Disallow + 검색/AdSense 비차단 단언).

### L-6. 남는 잔여 벡터(정직한 고지)
- **정적 HTML 95개를 curl 루프로 긁는 것**은 여전히 가능하다(Layer A는 API만, B는 curl 제외,
  C는 사용자가 제외). 그건 애초에 공개된 SEO 산출물이라 수용한다.
- 헤더값·UA는 위조 가능. 결정적 스크래퍼는 못 막는다. **비용을 올렸을 뿐이다.**
- 더 강하게 가려면: 정적 페이지 per-IP throttle(C, 정상 사용자 429 위험) 또는
  Cloudflare 등 앞단 WAF(별도 인프라).
