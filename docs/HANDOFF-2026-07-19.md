## C. 반드시 알아야 할 함정

### C-1. 테스트는 `run_tests.sh`로만 (기존 규칙, 여전히 유효)
서빙 스키마 `LOUPIT`을 테스트에도 재사용한다. 맨 `pytest server/tests`는 conftest 가드가
`[C-1 안전장치]`로 차단한다(정상 동작). 백엔드 검증은 **`bash infra/deploy/run_tests.sh`** 또는
release 게이트 경유로만. generator·web 테스트는 DB 무관이라 직접 실행 안전:
```bash
python3 -m pytest -q generator/tests/     # DB 무접촉
node --test 'web/**/*.test.js'            # ⚠ node v24는 `node --test web/`를 모듈로 오해 — glob 필수
```

### C-2. 배포 = `RELEASE_CONFIRM=1 bash infra/deploy/release.sh`
게이트가 최선두라 실패 시 서빙 상태가 바뀌지 않는다. TCOMPARE_LOG는 백업→재시드→재주입으로 보존된다.

### C-3. ⚠️ **nginx conf는 리포→`/etc` 자동 배포가 없다**
`release.sh`는 `reload`만 하고 **conf를 복사하지 않는다**. 프로덕션 conf를 바꿨다면 수동 배치 필요:
```bash
sudo cp /etc/nginx/sites-available/loupit.conf /root/nginx-loupit.conf.bak-$(date +%Y%m%d-%H%M%S)
sudo cp infra/nginx/loupit.conf /etc/nginx/sites-available/loupit.conf
sudo nginx -t && sudo systemctl reload nginx
```
- **순서 주의**: 작업트리가 곧 라이브 문서루트라, 새 경로를 참조하는 HTML을 먼저 커밋하면
  conf 적용 전까지 그 경로가 전부 404가 된다. **conf 먼저 → 참조 변경 나중.**
- `nginx -t`는 `alias`/`root` 대상 디렉터리 존재를 검사하지 않는다(오타는 런타임 404).
  스모크 SM-15a~e가 이걸 잡는 유일한 선이다.
- **beta conf는 리포본을 복사하지 말 것** — 리포는 `beta.jobcho.wiki` 인증서를 참조하는데 미발급이라
  nginx 기동이 실패한다. 라이브 beta는 `beta.loupit.co` 변형에 필요한 블록만 제자리 패치해 둔 상태.

### C-4. 자산은 `/assets/v2/` 세대 + no-cache
- **`web/assets`에 `.gz` 사전압축 금지**(불변식). gzip_static이 원본과 신선도를 비교하지 않아
  no-cache 재검증을 조용히 무력화한다. v2 location은 `gzip_static off`.
- 새 자산 참조는 반드시 `/assets/v2/…`. 구 `/assets/` 경로는 하위호환용 no-cache 서빙만 유지.

### C-5. 백업은 살아 있다
`loupit-backup.timer` enabled+active(매일 03:00), `/var/backups/loupit/`에 축적 중,
6테이블 전량 + TCOMPARE_LOG 데이터 포함, `Dump completed` 트레일러 정상. 다만 §B-4의 갭 참고.

### C-6. 서버 접촉 작업은 이 호스트에서만

## F. B-1 해시 딥링크 결함 — 처리 완료(2026-07-20)

### F-1. 실제 결함면은 문서보다 넓었다
라이브 크롤(playwright, 빈 localStorage)로 실증한 결과 **`#input`도 깨져 있었고**, 무엇보다
**랜딩 `/`가 `/compare`와 동일 셸·동일 `app.js`를 공유**해 `https://jobcho.wiki/#report` 등도
전부 같은 증상이었다. `/`는 canonical·sitemap 등재 URL이라 노출면이 `/compare`보다 넓다.

| 해시 | 수정 전(양 호스트) | 수정 후 |
|---|---|---|
| `#report` | `#report-body` 자식 0 — "비교 리포트 입력 수정 새 비교"만 | 레코드 있으면 **복원**(자식 7), 없으면 검색 뷰 |
| `#input` | "비교 입력 비교하기"(10자) — 빈 슬롯 | 검색 뷰 |
| `#company` | sr-only h2뿐인 **백지**(8자) | 검색 뷰 |

핵심 실증: 비교 완료 → 새로고침 시 localStorage에 **레코드가 그대로 1건 남아 있는데도**
빈 리포트가 떴다. 복원 재료가 있는데 부팅 경로가 쳐다보지 않은 것이 결함의 전부였다.

### F-2. 계약상 위치 — 신규 계약이 아니라 **기존 조항 위반**
`docs/SPEC/06:152` SP-FE-3.3 규칙 (3)은 최초판부터 "부팅 시 … **상태가 없으면 `search`**"를
명령하고 있었다. 그런데 같은 문서의 규범 의사코드(SP-FE-3.2·SP-FE-5.1)와 `TASK/06`의 T-06.5.2가
조건을 누락한 `go(parseHash()||'search')`를 박아 두어, 구현자가 산문 대신 의사코드를 따라갔다.
→ **계약을 바꾸면 산문·의사코드·TASK 리프를 같은 커밋에서 함께 고쳐야 한다**(세션 교훈 #5의 재확인).

### F-3. 적대적 검증이 잡아낸 파생 결함 3건 (전부 수정·라이브 확인)
최초 수정만으로는 부족했다. 5개 렌즈 병렬 리뷰 + 라이브 재현으로 확정한 것:

1. **복원 후 "입력 수정" → 빈 입력 뷰** (치명적, B-1이 한 클릭 거리에 재발)
   `mountUI`는 마운트 시점에 슬롯이 없으면 입력 뷰를 렌더하지 않는데(`ui.js:337`), 부팅 복원은
   그 **이후**에 상태를 채운다. → `restoreComparison`이 `renderInputView`를 호출하도록 수정.
   **"불러오기" 버튼 경로에도 이미 있던 선행 결함**이며 함께 해소됐다.
2. **새로고침마다 레코드 재저장** — `runReport`가 무조건 `saveRecentComparison`을 부른다.
   자동 복원이 이를 타면 `id`·`savedAt`이 새로 발급돼 "최근 비교" 순서·식별자가 요동친다.
   → `runReport({save:false})` 옵션 신설.
3. **무상태 `popstate` 폴백 회귀(내가 넣은 것)** — 해시 없는 히스토리 항목에서 폴백이
   `search`→`input`으로 바뀌어 뒤로가기가 입력 뷰에 눌러앉았다. → `popstate`는 해시가 없으면
   기존대로 `search`. **부팅 폴백 규칙을 `popstate`에 그대로 쓰면 안 된다.**

기각: report_bottom 광고 슬롯 미마운트 의혹 → `AD_CLIENT`가 아직 플레이스홀더라 승인 전 정상
상태이고, `mountAds`는 뷰 가시성과 무관하게 문서 전체를 스캔하므로 순서 문제도 없다.

### F-4. 테스트에서 배운 것 — **조용히 무효한 테스트 3종**
신규 테스트를 쓰는 과정에서 셋 다 "green인데 아무것도 검증하지 않는" 상태를 거쳤다.
같은 함정이 반복되므로 기록한다.

1. **필터가 URL과 안 맞았다** — `u.includes('compare')`로 비교 로그를 걸렀는데 실제 경로는
   `/api/v1/comparisons/log`다. `'comparisons'`에 `'compare'`는 없다. → 상수를 import해 비교.
2. **픽스처가 검증 대상 분기를 건너뛰게 만들었다** — jsdom 시드가 `#report-body`를 미리 렌더해
   `hasRenderedReport()`가 true가 되고 **복원 경로가 통째로 실행되지 않았다**. 새 페이지 로드는
   `#report-body`가 비어 있다. → 시드는 분리 노드에 렌더.
3. **모듈 전역 상태가 테스트 간에 샜다** — `ui.test.js`가 `App.state`를 리셋하지 않아 앞 테스트가
   채운 `matched`로 `mountUI`가 입력 뷰를 렌더해 버렸다. → `loadShell` 시 `App.state` 리셋.

공통 교훈: **테스트가 통과하면 왜 통과하는지 확인하라.** 뮤테이션을 실제로 주입해
빨간불이 켜지는지 본 것이 세 번 다 결정적이었다.

### F-6. ⚠️ 배포에 관해 반드시 알 것
`infra/nginx/loupit.conf:96` — `alias /home/ubuntu/loupit/web/assets/`. **작업트리가 곧 라이브
문서루트이고 빌드 단계가 없다.** 즉 `web/assets/js/*.js`를 저장하는 **순간 프로덕션에 반영된다**
(no-cache라 재검증도 즉시). `release.sh` 게이트를 거치지 않는다. JS를 편집할 때는 이 점을
전제하고, 테스트 green을 **저장 전에** 확보할 것.

### F-7. ⚠️ 이번 세션에 실제로 겪은 사고 — 뮤테이션 테스트가 프로덕션을 깼다
테스트 실효성을 확인하려고 `web/assets/js/app.js`에 의도적 뮤테이션을 주입했는데, **그 파일이
라이브 문서루트**라 주입된 몇 초~수 분 동안 프로덕션이 실제로 깨진 상태로 서빙됐다.
(병렬 검증 에이전트가 같은 기법을 쓰면서 `hasRenderedReport()`를 `return true`로 바꿔 둔 것을
포착 → 즉시 중단·원복·424 green·라이브 재확인으로 수습.)

**규칙**: `web/assets/**` 파일에는 뮤테이션을 주입하지 말 것. 필요하면
`cp -r web/assets /tmp/mutate/` 로 **복제본에서** 하거나, 최소한 주입~원복을 한 명령 안에서
끝내고 직후 라이브를 확인할 것. 자동화 에이전트에 이 디렉터리 편집 권한을 주는 것도 금물이다.
