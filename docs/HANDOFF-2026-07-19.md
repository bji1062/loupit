## C. 반드시 알아야 할 함정

### C-1. 테스트는 `run_tests.sh`로만 (기존 규칙, 여전히 유효)
서빙 스키마 `LOUPIT`을 테스트에도 재사용한다. 맨 `pytest server/tests`는 conftest 가드가
`[C-1 안전장치]`로 차단한다(정상 동작). 백엔드 검증은 **`bash infra/deploy/run_tests.sh`** 또는
release 게이트 경유로만. generator·web 테스트는 DB 무관이라 직접 실행 안전:
```bash
python3 -m pytest -q generator/tests/     # DB 무접촉
node --test 'web/**/*.test.js'            # ⚠ node v24는 `node --test web/`를 모듈로 오해 — glob 필수
```

### C-2. 배포 = `RELEASE_CONFIRM=1 bash infra/deploy/release.sh`
게이트가 최선두라 실패 시 서빙 상태가 바뀌지 않는다. TCOMPARE_LOG는 백업→재시드→재주입으로 보존된다.

### C-3. ⚠️ **nginx conf는 리포→`/etc` 자동 배포가 없다**
`release.sh`는 `reload`만 하고 **conf를 복사하지 않는다**. 프로덕션 conf를 바꿨다면 수동 배치 필요:
```bash
sudo cp /etc/nginx/sites-available/loupit.conf /root/nginx-loupit.conf.bak-$(date +%Y%m%d-%H%M%S)
sudo cp infra/nginx/loupit.conf /etc/nginx/sites-available/loupit.conf
sudo nginx -t && sudo systemctl reload nginx
```
- **순서 주의**: 작업트리가 곧 라이브 문서루트라, 새 경로를 참조하는 HTML을 먼저 커밋하면
  conf 적용 전까지 그 경로가 전부 404가 된다. **conf 먼저 → 참조 변경 나중.**
- `nginx -t`는 `alias`/`root` 대상 디렉터리 존재를 검사하지 않는다(오타는 런타임 404).
  스모크 SM-15a~e가 이걸 잡는 유일한 선이다.
- **beta conf는 리포본을 복사하지 말 것** — 리포는 `beta.jobcho.wiki` 인증서를 참조하는데 미발급이라
  nginx 기동이 실패한다. 라이브 beta는 `beta.loupit.co` 변형에 필요한 블록만 제자리 패치해 둔 상태.

### C-3b. ⚠️ 큰 프록시 응답은 nginx 임시파일을 타고, 실패해도 5xx가 아니라 **조용히 잘린다**
2026-07-20 실장애. `reference/all`이 약 600KB인데 `proxy_buffers` 기본값(8×4k=32KB)으로는
메모리에 못 담아 nginx가 `/var/lib/nginx/proxy/`로 흘리는데, 그 디렉터리가 `nobody:root 0700`이라
워커(`www-data`)가 `open()`에 EACCES로 실패했다. 그러면 **응답이 그 지점에서 끊긴다** —
`Content-Length`는 이미 전체 길이(606531)로 나간 뒤라 클라이언트는 200을 받고 JSON 파싱만 실패한다.
- **증상**: F5 새로고침 시 "비교 도구를 불러오지 못했습니다. 네트워크 상태를 확인해주세요."
  절단률 약 40%라 "다시 시도"를 눌러도 자주 또 실패해 **버튼이 고장난 것처럼 보였다**
  (버튼 자체는 정상 — 실패 주입 후 클릭하면 정상 복구됨을 실증).
- **진단 지름길**: `sudo tail /var/log/nginx/error.log | grep 'Permission denied'`.
  업스트림 직접 호출(`curl http://127.0.0.1:8000/...`)은 항상 정상이므로 **API를 의심하면 헛짚는다**.
  같은 URL을 여러 번 받아 `size_download`가 들쭉날쭉하면 이 문제다.
- **조치(둘 다 적용됨)**: `loupit.conf`의 API 블록에 `proxy_buffers 16 64k` 등으로 전량 메모리 수용
  (디스크 경로를 아예 안 탐) + `provision.sh`가 `/var/lib/nginx/proxy` 소유권 보장(이중 방어).
- **회귀 가드**: `SM-16 ref body intact(5x)` 신설. 기존 `SM-4`는 **HEAD로 헤더만** 봐서 본문이
  잘려도 통과했다 — 이것이 실장애를 놓친 사각지대였다. 간헐 실패라 5회 반복 검사한다.

## F. B-1 해시 딥링크 결함 — 처리 완료(2026-07-20)

### F-1. 실제 결함면은 문서보다 넓었다
라이브 크롤(playwright, 빈 localStorage)로 실증한 결과 **`#input`도 깨져 있었고**, 무엇보다
**랜딩 `/`가 `/compare`와 동일 셸·동일 `app.js`를 공유**해 `https://jobcho.wiki/#report` 등도
전부 같은 증상이었다. `/`는 canonical·sitemap 등재 URL이라 노출면이 `/compare`보다 넓다.

| 해시 | 수정 전(양 호스트) | 수정 후 |
|---|---|---|
| `#report` | `#report-body` 자식 0 — "비교 리포트 입력 수정 새 비교"만 | 레코드 있으면 **복원**(자식 7), 없으면 검색 뷰 |
| `#input` | "비교 입력 비교하기"(10자) — 빈 슬롯 | 검색 뷰 |
| `#company` | sr-only h2뿐인 **백지**(8자) | 검색 뷰 |

핵심 실증: 비교 완료 → 새로고침 시 localStorage에 **레코드가 그대로 1건 남아 있는데도**
빈 리포트가 떴다. 복원 재료가 있는데 부팅 경로가 쳐다보지 않은 것이 결함의 전부였다.

### F-2. 계약상 위치 — 신규 계약이 아니라 **기존 조항 위반**
`docs/SPEC/06:152` SP-FE-3.3 규칙 (3)은 최초판부터 "부팅 시 … **상태가 없으면 `search`**"를
명령하고 있었다. 그런데 같은 문서의 규범 의사코드(SP-FE-3.2·SP-FE-5.1)와 `TASK/06`의 T-06.5.2가
조건을 누락한 `go(parseHash()||'search')`를 박아 두어, 구현자가 산문 대신 의사코드를 따라갔다.
→ **계약을 바꾸면 산문·의사코드·TASK 리프를 같은 커밋에서 함께 고쳐야 한다**(세션 교훈 #5의 재확인).

### F-3. 적대적 검증이 잡아낸 파생 결함 3건 (전부 수정·라이브 확인)
최초 수정만으로는 부족했다. 5개 렌즈 병렬 리뷰 + 라이브 재현으로 확정한 것:

1. **복원 후 "입력 수정" → 빈 입력 뷰** (치명적, B-1이 한 클릭 거리에 재발)
   `mountUI`는 마운트 시점에 슬롯이 없으면 입력 뷰를 렌더하지 않는데(`ui.js:337`), 부팅 복원은

### J-3. ★ 두 가지 함정 (다음 배치에서도 그대로 적용된다)

**(a) 실패가 곧 결함은 아니다 — 배선을 먼저 의심하라.**
MB-4가 처음 실패했는데 결함이 아니라 내 테스트가 틀린 것이었다. 서버 검색 실패 시
**REF 번들 폴백 매칭을 먼저 시도**하므로(FR-E2, `search.js:78`) 폴백이 회사를 찾으면 후보가
렌더되고 오류가 아니다 — 그것이 설계된 동작이다. `error` 상태를 보려면 **폴백도 못 찾는
질의**를 써야 한다(`search.js:117`). 결함으로 보고하기 전에 코드 경로를 확인할 것.

**(b) 뮤테이션은 디스크가 아니라 브라우저에서 하라.**
`web/assets`는 라이브 문서루트라 파일을 변조하면 즉시 프로덕션이 깨진다(§F-7 실사고).
대신 **playwright `route` 가로채기로 응답만 바꾼다** — 디스크 무접촉:
```js
const ORIG = readFileSync('app-orig.js', 'utf8');       // 미리 받아둔 원본
await page.route('**/assets/v2/js/app.js', (route) =>
  route.fulfill({ status: 200,
    headers: { 'content-type': 'text/javascript; charset=utf-8' },
    body: ORIG.replace('...원본...', '...뮤테이션...') }));
```
이 방식으로 REF 2회 로드 / 오류박스 은폐 / 금지 POST 주입 3종을 넣어 하네스가 전부
검출함을 확인했다(= 공허한 통과가 아님). ⚠ 뮤테이션 블록에서 `waitUntil:'networkidle'`은
주입한 요청 때문에 안 끝날 수 있다 — `domcontentloaded`를 쓸 것.
