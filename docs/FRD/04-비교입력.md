# 비교 입력 (FRD)

**목적**: 비교 툴에서 두 직장(현직 a / 이직처 b)의 조건을 사용자가 직접 입력하는 기능 F2를 필드 단위로 정밀 명세한다. 연봉·연봉범위·상승률, 회사 복지 프리셋 자동 채움, 복지 체크·금액 조정, 근무형태(재택/유연/야근/임금유형), 통근시간, 우선순위·희생요소 직접 선택 각각에 대해 **입력 필드(타입·범위·기본값·검증)**, **출력·상태 변수**, **비즈니스 규칙**을 확정한다. "무엇을 어떤 규칙으로" 입력받는지만 규정하며, 리포트 계산식(F3)·회사 검색(F1)·구현 배치는 각 담당 문서로 넘긴다.

**상위 추적**: FRD → USECASE → PRD → 브리프. 상위 근거 = USECASE `03-비교입력.md`(UC-20~UC-25), PRD `05-기능-요구사항.md`(F2), 브리프 §7(비교 알고리즘 입력·함수)·§4(데이터 모델·만원 단위)·§2-3(프로파일러 제외·직접 선택)·§2-6(입력 화면 광고 최소화). 본 문서의 전역 규약·용어·상태 모델은 FRD `01-개요와-FR규칙.md`(FR-01~FR-08, §4 용어, §5 상태 모델), 데이터 필드 계약은 FRD `02-데이터-계약.md`(FR-D3·FR-D4·FR-D5·FR-D8)를 인용하며 재정의하지 않는다.

**범위 경계**: 본 문서는 F2(비교 조건 입력)만 다룬다. 회사를 찾아 슬롯에 지정하는 상호작용은 FR-1x(문서 02, UC-10~15), 입력을 받아 실효연봉·시간가치·워라밸·판정(vdCard)을 산출하는 계산은 FR-3x(문서 04 비교 리포트, UC-30~36)가 소유한다. 본 문서의 FR은 각 입력을 **F3 계산의 입력으로 전달**하는 지점까지만 명세한다. 로그인/회원/프로파일러/서버측 사용자 쓰기(영구 제외, §2-1·§2-3·§2-4)는 어떤 입력에도 등장하지 않는다. 우선순위는 프로파일러 산출이 아니라 사용자 직접 선택이다(§2-3, FR-03).

**FR-ID 대역**: 본 문서는 **FR-2x 대역**(FR-20~FR-25)을 소유한다(FRD 01 §2.3). 부여한 ID는 재사용하지 않는다(안정 ID). 각 FR은 UC-20~UC-25와 1:1 대응하며, 하위 문서(FLOW/WIREFRAME/SPEC/TASK)가 이 ID를 인용한다.

## 공통 규약 (전 FR 적용)

- **단위**: 연봉·복지 금액은 **만원 단위 정수**다(§4, FR-04). 통근시간만 **분(minute) 단위 정수**다(본 문서 FR-24). 금액화 불가 항목은 `null`.
- **슬롯별 입력**: 모든 입력은 슬롯 a(현직)·b(이직처)에 대해 독립적으로 존재한다(FR-01 §4 용어). 아래 상태 변수는 슬롯 접미 `.a`/`.b`로 쓴다.
- **F2 소유 입력 상태**: FRD 01 §5의 공용 상태(`matched`/`benS`/`wsState`/`curPri`/`curSacrifice`/`selectedRate`) 중 **F2가 채우는** 상태(`benS`·`wsState`·`curPri`·`curSacrifice`·`selectedRate`)와, §5가 형태만 확정하고 본 문서가 상세화하는 **슬롯별 스칼라 입력** `salS`(연봉)·`cmtS`(통근시간)를 F2가 소유한다. `salS`·`cmtS`는 §7이 입력으로 명시한 "연봉 범위"·"통근시간"을 담는 F2 전용 입력 상태로, §5 공용 상태를 보완한다(신규 상태 도입이 아니라 F2 입력 슬롯 스칼라의 명시화).
- **초기값 소스**: 슬롯에 회사가 지정되면 복지·근무형태 초기값을 회사 실데이터 또는 기업유형 프리셋 폴백으로 채운다(FR-06, FR-D3·FR-D8). 미지정(직접 입력, UC-15) 슬롯은 사용자가 직접 입력한다.
- **실시간 반영**: 입력 변경은 즉시 F3의 실시간 요약(`calc`)에 반영되어 대략 비교가 갱신되며, 확정 리포트(`compare`) 생성은 F3가 수행한다(§7). 본 문서는 값 지정까지만 담당한다.
- **지속성·미전송**: 입력 상태는 클라이언트 메모리와 localStorage에만 존재하고 서버로 전송·저장되지 않는다(NFR16·NFR17, FR-07). 저장 실패는 흡수하고 비교는 정상 동작한다(NFR25).
- **광고 최소화**: 비교 입력 화면은 광고를 최소화한다(§2-6, F2 수용기준 5). 본 대역의 어떤 FR도 입력 화면에 광고 슬롯을 요구하지 않는다.
- **렌더 안전**: 회사·복지 문자열과 사용자 입력값은 렌더 시 이스케이프/`textContent`로 삽입한다. `innerHTML` 직접 삽입 금지(NFR21).
- **amt_source 전이**: 사용자가 자동 채움 값(금액/체크)을 수정하면 해당 복지 항목의 `amt_source`가 `user`로 전이하고 배지 근거가 사용자 입력으로 대체된다(FR-04, FR-D9).

## [FR-20] 연봉·연봉범위·상승률 입력

**설명**: 슬롯 a/b 각각의 연봉을 **단일 금액** 또는 **범위(하한~상한)**로 입력받고, 이직처(b)는 오퍼 범위 직접 입력 또는 현직(a) 대비 **상승률(selectedRate)** 기반 파생 입력을 허용한다(§7 입력: 연봉 범위, 함수 `getSalRange`/`getOfferRange`). 입력값은 F3 실효연봉(연봉+순복지) 계산의 기준값이 된다.

**상세 동작**
- 정상
  1. 사용자가 슬롯 a·b 각각에 연봉을 만원 단위로 입력한다. 입력 모드는 단일/범위 중 선택한다. 단일 입력 시 내부적으로 `low = high`로 취급한다.
  2. 범위 입력 시 하한·상한을 입력한다(a=`getSalRange`, b 오퍼=`getOfferRange`).
  3. 이직처(b)는 대안으로 상승률(`selectedRate`, %)을 입력해 `b연봉 = a연봉 × (1 + rate/100)`로 파생할 수 있다(FRD 01 §5). 상승률 모드와 직접 입력 모드는 슬롯 b에 대해 배타적이다.
  4. 입력 변경은 즉시 F3 실시간 요약(`calc`)에 반영된다(§7).
- 대안·예외

  | 분기 | 조건 | 처리 |
  |------|------|------|
  | 2b | 범위 하한 > 상한 | 하한 ≤ 상한이 되도록 정정을 안내하고 계산에 반영하지 않는다(UC-20 2b) |
  | 1a | 연봉 미입력(빈 값) | 해당 슬롯 연봉을 `null`로 두고 실효연봉 계산을 입력 시점까지 보류한다(UC-20 1a) |
  | 1b | 음수·비수치 입력 | 0 이상의 유효 금액만 허용하고 그 외 입력은 거부한다(UC-20 1b, FR-04) |
  | 3a | 상승률 모드에서 a 연봉 미입력 | a 연봉 확정 전까지 b 파생값을 산출하지 않는다 |

**입력**

| 필드 | 타입 | 범위·제약 | 기본값 | 상태변수 |
|------|------|-----------|--------|----------|
| 입력 모드 | enum {`single`, `range`} | 슬롯별 | `single` | (UI 모드) |
| a 연봉 하한 | int(만원) | ≥ 0 | `null` | `salS.a.low` |
| a 연봉 상한 | int(만원) | ≥ 하한 (단일 시 = 하한) | `null` | `salS.a.high` |
| b 연봉 하한 | int(만원) | ≥ 0 | `null` | `salS.b.low` |
| b 연봉 상한 | int(만원) | ≥ 하한 (단일 시 = 하한) | `null` | `salS.b.high` |
| b 파생 상승률 | number(%) | 실수 | `null` | `selectedRate` |

**출력·상태**
- `salS.a`·`salS.b`(및 필요 시 `selectedRate`)가 입력 상태에 반영된다.
- 확정 연봉/범위는 F3 실효연봉·시간조정 가치 계산의 입력으로 전달된다(§7). 서버 전송 없음(NFR16).

**검증·비즈니스 규칙**
- 금액은 **만원 단위 정수**이며 음수·비수치는 거부한다(FR-04). 단위는 데이터 모델 금액 컬럼과 동일(§4).
- 범위는 항상 `low ≤ high`. 위반 시 정정 안내(계산 보류).
- 미입력(`null`) 슬롯은 실효연봉 산출에서 보류되며 크래시하지 않는다.
- `selectedRate` 사용 시 b 연봉은 a 연봉에서 파생되며, 파생값도 만원 정수·`low ≤ high` 규칙을 만족한다.

**추적**: UC-20 / F2(→F3 실효연봉) / FR-03·FR-04(FRD 01) / NFR16 / PRD-SEG-1·SEG-2·PER-1 / §4(만원 단위)·§7(연봉 범위·`getSalRange`/`getOfferRange`·실효연봉).

## [FR-21] 복지 프리셋 자동 채움·초기화

**설명**: 슬롯에 회사가 지정되면(UC-14) 해당 회사의 복지 데이터로 `benS[s]`를 자동 채우고, 실데이터가 없으면 기업유형 프리셋으로 폴백하여 각 항목의 초기 체크 상태를 `default_checked_yn`으로 설정한다(§4, FR-06, FR-D3·FR-D8). 직접 입력 모드(UC-15)는 유형 선택 시 프리셋을 복사한다.

**상세 동작**
- 정상
  1. 회사 지정 시 클라이언트가 회사 객체(`companies[]` 인라인 또는 `companies/{id}`, FR-D4)를 `normalizeCompany`로 정규화하여 `benefits`를 `benS[s]`의 초기 소스로 사용한다(FR-D8).
  2. 서버는 실데이터 우선·프리셋 폴백을 이미 병합해 전달하므로(`benefits`는 항상 비어 있지 않음, FR-D4·FR-D7), 실데이터 항목은 `amt_source=real`, 프리셋 폴백 항목은 `amt_source=preset`으로 표기된다(FR-04·FR-D9).
  3. 프리셋 유래 항목의 초기 체크는 `default_checked_yn`을, 회사 실데이터 항목의 초기 체크는 `true`를 적용한다(FR-D8.1·FR-D8.2).
  4. 각 항목은 배지(`badge_cd`: 공식 official / 추정 est, 만료 파생 포함)로 출처 신뢰도를 표시한다(FR-D9).
  5. 사용자는 자동값을 그대로 두거나 개별 항목을 수정한다 → 상세는 FR-22로 연결된다.
- 대안·예외

  | 분기 | 조건 | 처리 |
  |------|------|------|
  | 2a | 실데이터가 일부 항목만 존재 | 서버가 실데이터∪프리셋으로 병합해 전달하며, 클라이언트는 병합된 `benefits`를 그대로 채운다(FR-06·FR-D4) |
  | 1a | 직접 입력 모드 슬롯(UC-15) | `matched[s]=null` 유지. 사용자가 유형(`comp_tp_cd`)을 고르면 `benefit_presets[comp_tp_cd]`를 정규화해 `benS[s]`로 복사한다(FR-D3·FR-D8) |
  | 1b | 슬롯 회사 교체(UC-14 2a) | 새 회사의 실데이터/프리셋으로 `benS[s]`를 재설정하고, 기존 사용자 수정값이 대체됨을 안내한다 |
  | 5a | 사용자가 자동값 수정 | 해당 항목 `amt_source=user`로 전이(FR-22, FR-04). 만료 항목도 사용자가 수정 가능 |

**의사코드(초기 채움)**
```
fillBenefits(slot):
  if matched[slot] != null:                     # 회사 지정(UC-14)
    src = matched[slot].benefits                 # 서버 병합(실데이터∪프리셋)
    benS[slot] = src.map(b => normalize(b, checked = (b.amt_source=='preset'
                                                       ? b.default_checked_yn : true)))
  elif chosenType[slot] != null:                 # 직접 입력 + 유형 선택(UC-15)
    src = REF.benefit_presets[chosenType[slot]]
    benS[slot] = src.map(b => normalize(b, checked = b.default_checked_yn))
  else:
    benS[slot] = []                              # 빈 상태에서 사용자 직접 입력
```

**입력**
| 필드 | 타입 | 제약 | 상태변수 |
|------|------|------|----------|
| 지정 회사 | 회사 객체(FR-D4) \| null | UC-14 지정 또는 null(UC-15) | `matched[s]` |
| 직접 입력 유형 | ∈ 6종 `comp_tp_cd` \| null | 직접 입력 모드에서만 | (UI 선택) |
| (자동) 복지 소스 | `benefits[]` 또는 `benefit_presets[cd]` | 비어 있지 않음(FR-D4) | — |

**출력·상태**
- `benS[s]` = 실데이터/프리셋으로 초기화된 복지 항목 배열(각 항목에 `checked`·`amt_source` 부여).
- 확정 복지는 F3 순복지·카테고리 델타 계산 입력이 된다(§7).

**검증·비즈니스 규칙**
- `checked` 초기값 = 프리셋 항목이면 `default_checked_yn`, 회사 실데이터 항목이면 `true`(FR-D8.1).
- `normalizeCompany` 폴백 적용: `benefit_ctgr_cd` 미상 → `perks`, `badge_cd` 미상 → `est`, `qual_yn`은 불리언 강제(FR-D8.2).
- 폴백은 F2 입력 초기값·F4/F5 정적 페이지·F8 응답에서 동일 규약으로 적용된다(FR-06).
- 정성 복지(`qual_yn=true`)는 금액 없이(`benefit_amt=null`) 채워진다(FR-D5).

**추적**: UC-21 / F2(F1 회사 지정·F3 복지 계산) / FR-06·FR-D3·FR-D4·FR-D8·FR-D9 / PRD-SEG-1·SEG-2·SEG-3 / §4(`TCOMPANY_BENEFIT`/`TBENEFIT_PRESET`/`DEFAULT_CHECKED_YN`)·§5(프리셋 폴백)·§6.

## [FR-22] 복지 항목 체크·금액 조정

**설명**: 복지 항목을 9개 카테고리로 표시하고, 각 항목의 비교 포함 여부(`checked`)와 정량 항목의 금액(만원)을 사용자가 조정한다(§4 `BENEFIT_CTGR_CD` 9종, F2 수용기준 3). 정성 항목(`qual_yn=true`)은 금액 없이 체크·서술만 다룬다.

**상세 동작**
- 정상
  1. 복지 항목이 9개 카테고리로 그룹화되어 표시된다: `compensation`/`flexibility`/`work_env`/`time_off`/`health`/`family`/`growth`/`leisure`/`perks`(§4·FR-D5).
  2. 사용자가 각 항목의 `checked`(비교 포함/제외)를 토글한다.
  3. 정량 항목은 금액(만원)을 입력·수정한다. 사용자 수정 시 `amt_source=user`로 전이한다(FR-04·FR-D9).
  4. 정성 항목(`qual_yn=true`)은 금액 없이 체크만 하며 `qual_desc_ctnt`로 표시된다(§4: 정성이면 `benefit_amt=null`, FR-D5).
  5. 체크/금액 변경은 F3의 카테고리별 복지 합산·순복지 계산 입력으로 반영된다(§7 `getBenTotal`·`renderBenCatCompare`·`renderQualCompare`).
- 대안·예외

  | 분기 | 조건 | 처리 |
  |------|------|------|
  | 2a | 항목 체크 해제 | 순복지·카테고리 합산에서 제외한다(F3) |
  | 3a | 정량 항목 금액 미입력 | 계산상 0으로 취급한다(`Number(v)||0`, FR-D8.2). 입력 시 합산 반영 |
  | 3b | 프리셋 자동값 수정 | FR-21 5a와 동일: 배지 근거가 사용자 입력으로 전환(`amt_source=user`) |
  | 4a | 정성 항목에 금액 입력 시도 | 정성 항목은 금액 필드 없음(`benefit_amt=null` 유지) |

**입력**
| 필드 | 타입 | 범위·제약 | 기본값 | 상태변수 |
|------|------|-----------|--------|----------|
| 체크 여부 | bool | — | FR-21 초기값 | `benS[s][i].checked` |
| 금액 | int(만원) | ≥ 0. `qual_yn=true`면 편집 불가(`null`) | FR-21 초기값 | `benS[s][i].benefit_amt` |

**출력·상태**
- `benS[s]`의 각 항목이 9개 카테고리별로 체크/금액 확정 상태가 된다.
- 확정 값은 F3 순복지·카테고리 델타 계산 입력으로 전달된다(§7).

**검증·비즈니스 규칙**
- 금액은 만원 단위 정수 ≥ 0. 음수·비수치 거부(FR-04).
- `qual_yn=true` ⇒ `benefit_amt=null`, `amt_source=none`(정성복지 금액 없음, FR-D5).
- 체크 해제 항목은 순복지·불확실성 밴드 합산에서 제외된다(FR-05).
- 카테고리는 9종 집합을 벗어나지 않으며 미상은 `perks`로 폴백된다(FR-D8.2).
- 사용자 입력·복지 문자열은 렌더 시 이스케이프한다(NFR21).

**추적**: UC-22 / F2(→F3 카테고리 델타·순복지) / FR-04·FR-05·FR-D5·FR-D8 / NFR21 / PRD-SEG-1·SEG-2 / §4(`BENEFIT_CTGR_CD` 9종·`QUAL_YN`·`QUAL_DESC_CTNT`)·§7(`getBenTotal`·`renderBenCatCompare`·`renderQualCompare`).

## [FR-23] 근무형태 입력 (재택/유연/야근/임금유형)

**설명**: 슬롯 a/b 각각의 근무형태 4축(`wsState = {ot, wage, remote, flex}`)을 입력받는다(FRD 01 §5). 회사 지정 슬롯은 `work_style_val`(§4 JSON)에서 초기값을 제안하고 사용자가 수정한다. 입력은 F3의 주간근무시간(`getWSHours`)·야근수당(`getOTPay` 포괄/비포괄 분기)·워라밸·자율성 계산 입력이 된다(§7).

**상세 동작**
- 정상
  1. 사용자가 4축을 선택한다: 야근 수준(`ot`), 임금유형(`wage`), 재택(`remote`), 유연(`flex`).
  2. 재택(`remote`)은 F3 재택절약(`REMOTE_SAVE`) 워라밸 요인 입력, 유연(`flex`)은 자율성 요인 입력이 된다(§7).
  3. 야근 수준(`ot`)은 F3 야근수당(`getOTPay`)·주간근무시간(`getWSHours`) 산정 입력이 된다(§7).
  4. 임금유형(`wage`)은 포괄/비포괄로 `getOTPay`의 분기를 결정한다(§7 wage=포괄/비포괄).
- 대안·예외

  | 분기 | 조건 | 처리 |
  |------|------|------|
  | 1a | 슬롯에 회사 지정됨 | `matched[s].work_style_val`(remote/flex/overtime)로 초기값을 제안하고 사용자가 수정 가능(UC-23 1a, FR-D4.1) |
  | 5a | 임금유형(`wage`) 미지정 | `getOTPay`가 기본 분기로 처리(야근수당 산정 보류/기본, UC-23 5a). 본 FR은 값 지정까지만 담당 |
  | 2a | `unlimitedPTO`·`refreshLeave` 등 `work_style_val` 항목 | 편집 4축(`wsState`)에 포함되지 않으며 `matched[s].work_style_val`로 보존되어 F3 워라밸 요인 입력에 반영된다(§7 무제한휴가 요인) |

**의사코드(회사 지정 시 초기 제안)**
```
initWsState(slot):
  ws = matched[slot]?.work_style_val
  wsState[slot] = {
    ot:     null,                                # 야근 수준은 사용자 입력(기본 미선택)
    wage:   null,                                # 임금유형 사용자 입력(미지정 시 getOTPay 기본)
    remote: ws?.remote ? proposeRemote(ws.remote) : null,
    flex:   ws?.flex   ? proposeFlex(ws.flex)     : null
  }                                              # 제안값은 사용자가 자유 수정
```

**입력**
| 축 | 타입 | 값 영역 | 기본값 | 상태변수 |
|----|------|---------|--------|----------|
| 야근 수준 | enum | {`low`, `mid`, `high`} | `null`(미선택) | `wsState[s].ot` |
| 임금유형 | enum | {`inclusive`(포괄), `separate`(비포괄)} | `null` | `wsState[s].wage` |
| 재택 | enum | {`none`, `partial`, `hybrid`, `free`} | `null`(회사 지정 시 `work_style_val` 제안) | `wsState[s].remote` |
| 유연 | enum | {`none`, `stagger`, `flexible`} | `null`(회사 지정 시 제안) | `wsState[s].flex` |

**출력·상태**
- `wsState[s] = {ot, wage, remote, flex}`가 입력 상태에 반영된다.
- 반영 값은 F3 시간조정 가치·워라밸·자율성 계산 입력으로 전달된다(§7). 무제한휴가·리프레시휴가는 `matched[s].work_style_val` 경로로 F3에 전달된다.

**검증·비즈니스 규칙**
- 각 축은 정의된 값 영역 밖 값을 갖지 않는다(FRD 01 §5, FR-D4.1). 미선택은 `null`.
- `wage` 미지정 시 F3 `getOTPay`가 기본 분기로 처리하며 크래시하지 않는다.
- `unlimitedPTO`·`refreshLeave`는 `wsState` 편집 대상이 아니라 회사 `work_style_val`에서 읽는 파생 입력이다(§7).

**추적**: UC-23 / F2(→F3 시간가치·워라밸) / FR-03·FR-D4·FR-D8 / PRD-SEG-1·SEG-2·PER-1 / §4(`WORK_STYLE_VAL`)·§7(`getWSHours`·`getOTPay` 포괄/비포괄·`REMOTE_SAVE`).

## [FR-24] 통근시간 입력

**설명**: 슬롯 a/b 각각의 통근시간을 **분 단위 정수**로 입력받는다(§7 입력: 통근시간). 입력은 F3의 시간조정(시간당) 가치·워라밸 요인 계산 입력이 된다(§7).

**상세 동작**
- 정상
  1. 사용자가 슬롯 a·b 각각의 통근시간(분)을 입력한다.
  2. 통근시간은 F3 시간조정 가치·워라밸 요인 계산 입력으로 전달된다(§7 출력: 시간조정 가치·워라밸).
  3. 입력 변경은 즉시 F3 실시간 요약(`calc`)에 반영된다(§7).
- 대안·예외

  | 분기 | 조건 | 처리 |
  |------|------|------|
  | 1a | 통근시간 미입력 | `null`로 두고 미반영 상태 유지, 입력 시점에 계산 반영(UC-24 1a) |
  | 1b | 음수·비수치 입력 | 0 이상의 정수만 허용, 그 외 거부 |
  | 1c | 재택(FR-23)과의 관계 | 재택 비중이 통근 부담을 낮추는 반영은 F3 계산에서 처리. 본 FR은 통근시간 값 지정까지만 담당(UC-24 1b) |

**입력**
| 필드 | 타입 | 범위·제약 | 기본값 | 상태변수 |
|------|------|-----------|--------|----------|
| 통근시간(편도) | int(분) | ≥ 0 | `null` | `cmtS[s]` |

**출력·상태**
- `cmtS.a`·`cmtS.b`(분)가 입력 상태에 반영되어 F3 시간가치·워라밸 계산 입력으로 전달된다.
- 서버 전송 없음(NFR16).

**검증·비즈니스 규칙**
- 값은 **분 단위 정수** ≥ 0. 음수·비수치 거부(FR-04 금액 규약을 분 단위에 준용).
- 미입력(`null`)은 계산에서 보류되며 크래시하지 않는다.

**추적**: UC-24 / F2(→F3 시간가치·워라밸) / FR-03 / PRD-SEG-1·SEG-2 / §7(통근시간·시간조정 가치).

## [FR-25] 우선순위·희생요소 직접 선택

**설명**: 비교 판정 축인 우선순위(`curPri`)와 희생요소(`curSacrifice`)를 사용자가 **직접 선택**한다. 프로파일러(가치관 진단 설문)는 존재하지 않으며 자동 추천도 없다(§2-3, F2 수용기준 1, SC9 영구 제외). 선택 값은 F3의 우선순위 기반 판정(`vdCard`) 입력이 된다(§7).

**상세 동작**
- 정상
  1. 사용자가 우선순위를 연봉 / 워라밸 / 복지 / 브랜드 중 **하나** 직접 선택한다(`curPri`, 단일 선택, §7). 기본값 `워라밸`이 사전 선택되어 있으며 사용자가 명시적으로 변경할 수 있다(FRD 01 §5).
  2. 사용자가 희생요소(`curSacrifice`)를 직접 선택한다 — 무엇을 감수할지 지정하며, 선택 안 함(`null`)도 허용된다(§7, F2 수용기준 2).
  3. 선택한 `curPri`·`curSacrifice`는 F3 우선순위 기반 "누가 유리/무엇을 택할지" 판정(`vdCard`) 입력으로 전달된다(§7 출력).
- 대안·예외

  | 분기 | 조건 | 처리 |
  |------|------|------|
  | 1a | 우선순위 미명시 | 기본값(`워라밸`)이 적용되나, UI는 판정 전 명시적 선택을 유도한다(UC-25 1a, §2-3) |
  | 1b | 우선순위·희생요소 변경 | 변경 시 F3 판정(`vdCard`)이 재계산 입력으로 반영된다(UC-25 1b) |
  | 1c | 프로파일러(자동 추천) 기대 | 해당 기능은 영구 제외(SC9). 직접 선택으로 대체됨을 안내한다(§2-3, FR-01) |

**입력**
| 필드 | 타입 | 값 영역 | 기본값 | 상태변수 |
|------|------|---------|--------|----------|
| 우선순위 | enum(단일) | {`연봉`, `워라밸`, `복지`, `브랜드`} | `워라밸` | `curPri` |
| 희생요소 | 문자열 \| null | `curPri`와 상이한 값, 또는 `null`(선택 안 함) | `null` | `curSacrifice` |

**출력·상태**
- `curPri`·`curSacrifice`가 확정되어 F3 판정(`vdCard`) 입력으로 전달된다.
- 서버 전송·서버 저장 없음(NFR16). 프로파일러 상태(`pfResult` 등) 미사용(FRD 01 §5.1, FR-D10).

**검증·비즈니스 규칙**
- `curPri`는 4종 집합의 단일 값이며, 프로파일러 산출이 아니라 사용자 직접 선택으로만 설정된다(§2-3, FR-03).
- `curSacrifice`는 `curPri`와 다른 값이거나 `null`이다(FR-03 검증 규칙 (2)와 정합).
- 우선순위·희생요소는 F3 `vdCard` 판정의 유일한 판정 축 입력이며, 페르소나·질문 응답을 입력으로 갖지 않는다(FR-D10).

**추적**: UC-25 / F2(→F3 `vdCard` 판정) / FR-03·FR-D10 / PRD-SEG-1·SEG-2·PER-2 / §2-3(프로파일러 제외·직접 선택)·§7(`curPri`·`curSacrifice`·우선순위 판정).
