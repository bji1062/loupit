# 인기 비교 조합 정적 생성 (FRD)

**목적**: loupit의 인기 비교 조합 페이지(F5, 예: "삼성전자 vs SK하이닉스")를 **빌드타임 정적 생성**으로 구현하기 위한 기능 요구사항을 구현 이전 수준에서 정밀 명세한다. 구체적으로 (1) 조합 선정 기준과 조합 페이지 URL 규칙(`/vs/{a}-{b}`)·정적 산출 규약, (2) 방문자 개인 입력 없는 **사전 계산 비교 요약**의 콘텐츠·기본 가정, (3) "비교 툴에서 열기"를 통한 **양사(a/b) 프리필** 진입, (4) 관련 조합·회사 상세로의 **내부 링크**, (5) 조합 페이지 **SEO 태그·canonical·sitemap 기여**, (6) 조합 페이지 **광고·제휴 슬롯**을 FR-60~FR-65로 확정한다. "어떻게 코드·파일로 배치하는가"(생성 스크립트 구조·템플릿 파일)는 SPEC/TASK로 미룬다.

**참조 상위문서**: USECASE `06-인기비교조합.md`(UC-50~UC-54, 공통 사전조건·사후조건 제약), PRD `07-비기능-요구사항.md`(NFR1·NFR2·NFR5·NFR6~12·NFR16·NFR19·NFR24·NFR29). 전역 규약·용어·상태 모델은 FRD `01-개요와-FR규칙.md`(FR-01·FR-02·FR-03·FR-05·FR-06·FR-07, §4 용어, §5 상태 모델)를 인용한다. 데이터 필드 계약은 FRD `02-데이터-계약.md`(FR-D3 프리셋·FR-D4 회사 객체·FR-D6/FR-D7 응답), 비교 툴 진입(프리필 소비)은 FRD `03-검색-회사선택.md`(FR-16 CTA 프리필 진입·FR-14 회사 반영), 계산·표현은 FRD `05-비교엔진.md`(FR-3x)·`06-리포트-표현.md`(FR-4x)를 인용하고 재정의하지 않는다. 근거 브리프 섹션: §2-5·§2-6(수익화·광고 배치), §2-7(SEO), §2-8(클라이언트 계산), §3(F5), §4/§5(데이터 모델·보존 데이터), §8(빌드타임 정적 생성), §9(문서 규칙).

**상위 추적**: FRD → USECASE(UC-50~UC-54) → PRD(F5) → 브리프. 본 문서(08)는 F5(인기 비교 조합 페이지)에 **FR-6x 대역**을 소유한다(작성 지시). 각 FR은 충족 UC-5x·상위 F5(전환·연동 지점의 F1·F2·F3·F4·F6)·관련 NFR을 인용한다.

**범위 경계**: 본 문서는 **F5의 조합 페이지 생성·요약 렌더·비교 툴 진입·내부 링크·SEO·광고**만 다룬다. 조합 페이지의 빌드타임 정적 생성(FR-60)과 그 조합 URL의 sitemap 기여(FR-64)는 본 문서가 **UC-A4(빌드타임 정적 페이지·사이트맵 생성·배포)의 F5 슬라이스로서 충족**한다. 그러나 **전 생성기 실행 오케스트레이션 + 사이트 전역 `sitemap.xml` 병합 + `robots.txt` + Nginx 배포**(UC-A4의 사이트 전역 부분)의 **primary 소유**는 공통 빌드/배포 FR — 회사 상세 정적 생성(FRD 07)의 빌드 파이프라인 FR-50·사이트맵 등록 FR-56 계열이 사이트 전역 병합·배포까지 확장 소유하며 FR-0x 전역 규약과 정합 — 에 있고, 본 문서는 조합 페이지가 그 사이트 전역 빌드에 제공하는 **산출물·조합 URL 규약**을 규정한다. 크롤러 색인은 UC-A3이 다룬다. 회사 상세 페이지(F4)의 정적 생성·기업정보·복지 렌더·"이 회사로 비교하기" 단일 슬롯 CTA는 F4(FRD 07, FR-5x, UC-40~UC-44)가 소유하며, 본 문서는 내부 링크 대상(FR-63)·양사 프리필(FR-62)로만 참조한다. 비교 조건 입력(연봉·복지·근무형태·통근·우선순위)은 F2(FR-2x, FRD 04), 리포트 산출식·재계산은 F3(FR-3x, FRD 05 / FR-4x, FRD 06)이 소유하고, 본 문서는 조합 페이지에서의 **진입 트리거**로만 인용한다. AdSense·affiliate 슬롯의 렌더링·클릭·데이터 컴포넌트 상세는 F6(광고·제휴 FRD)이 소유하며, 본 문서는 조합 페이지가 **광고 배치 대상임과 슬롯 위치 예약**만 규정한다. 프로파일러(가치관 진단 설문)와 익명 경로 서버측 사용자 데이터 쓰기 저장은 제품 범위에서 영구 제외이며, 로그인·회원·계정은 이 익명 조합 생성 흐름에 등장하지 않는다(로그인·재직 인증은 복지 등록·수정 기여 SC14 한정 — FRD 13; §2-1·§2-3·§2-4, FR-01, NFR16·NFR20). 조합 페이지 열람·요약 확인·비교 툴 진입 전 과정에서 사용자 데이터의 서버 전송·저장이 발생하지 않으며(§2-8, NFR16), 방문자 본인 조건 반영 계산은 비교 툴 진입 후 100% 클라이언트에서 수행된다(§2-8, FR-01).

**FR-ID 대역**: 본 문서는 **FR-6x 대역**(FR-60~FR-65)을 소유한다(작성 지시). 부여한 ID는 재사용하지 않는다(안정 ID, 브리프 §9, 01 §2.2). 본 문서의 헤드라인 커버 UC는 **UC-50~UC-54**이며, 추가로 FR-60·FR-64가 횡단 유즈케이스 **UC-A4(빌드타임 정적 페이지·사이트맵 생성·배포)의 F5 슬라이스**(조합 페이지 생성·조합 URL sitemap 기여)를 **충족**한다(UC-A4 사이트 전역 오케스트레이션의 primary 소유는 공통 빌드/배포 FR — 상세는 문서 말미 "횡단 UC-A4 커버리지" 참조). 하위 문서(FLOW/WIREFRAME/SPEC/TASK)가 이 FR-ID를 인용한다.

## 공통 규약 (전 FR 적용)

아래 전제는 FR-60~FR-65 모두에 적용되며 각 FR에서 재서술하지 않는다(USECASE 06 공통 사전·사후조건, FR-01, 브리프 §8).

1. **정적·읽기 전용 서빙**: 조합 페이지는 빌드타임에 완성된 정적 HTML로 생성되어 Nginx가 직접 서빙한다. 요청 시 서버(Uvicorn) 렌더링·DB 조회가 없다(NFR1). 존재하는 조합만 파일이 존재하며, 미존재 조합 경로는 404를 반환한다(UC-50 분기 3a, NFR/색인 제외 UC-A3).
2. **데이터 출처**: 조합 페이지의 모든 값은 빌드타임 생성 스크립트가 두 회사의 참조·복지 데이터(§4/§5: `TCOMPANY`·`TCOMPANY_BENEFIT`; 조합 페이지는 등록 회사(모두 실복지 보유)만 대상이므로 회사페이지용 프리셋 폴백은 불필요)에서 산출한 정적 값이며, 방문자별 개인 입력(연봉·우선순위 등)을 포함하지 않는다(UC-51 사전조건, §2-8).
3. **비-JS 가독·인코딩**: 조합 페이지 본문(비교 요약·내부 링크)은 JavaScript 없이도 초기 HTML에 존재해 크롤러·비활성 환경에서 읽을 수 있어야 한다(NFR12·NFR24). `<html lang="ko">`·`charset=utf-8`로 인코딩한다(NFR29). 인터랙션("비교 툴에서 열기" 등)만 JS로 향상한다.
4. **무서버쓰기**: 열람·요약 확인·비교 툴 진입 전 과정에서 서버로의 사용자 데이터 쓰기가 발생하지 않는다(§2-1·§2-4, NFR16·NFR20, FR-01·FR-07).

## [FR-60] 인기 조합 선정·`/vs/{a}-{b}` URL 규칙·빌드타임 정적 생성

**설명**: 자주 찾는 회사쌍("A vs B")의 비교 페이지를 **사전 정의된 인기 조합 목록**에 근거하여 빌드타임에 정적 HTML로 생성하고, 각 조합에 정규화된 `/vs/{a}-{b}` URL을 부여한다. 이 FR은 UC-50(검색엔진 유입·열람)의 페이지 존재·URL·서빙 근거를 소유하며, 동시에 **UC-A4(빌드타임 정적 페이지·사이트맵 생성·배포)의 F5 조합 페이지 생성 슬라이스를 충족**한다(UC-A4 기본 흐름 2의 "인기 조합 페이지(F5)를 정적 HTML로 생성" 단계). 전 생성기 실행·사이트 전역 sitemap 병합·배포 오케스트레이션은 UC-A4의 primary 소유 FR(공통 빌드/배포 FR, 아래 R5)이 담당한다.

**상세 동작**

- 정상(선정): 빌드 입력으로 **사전 정의 조합 목록**(큐레이션된 인기 회사쌍 데이터, 예: `combinations.json` 유형의 빌드타임 데이터 파일 — 코드 수정 없이 목록 교체 가능, `affiliate.json` 규약과 동종)을 받는다. 각 항목은 두 회사를 `COMP_ENG_NM`(또는 `COMP_ID`)로 지시한다. 목록은 전수 조합(등록 회사(복지 데이터가 있는 약 96개)의 모든 쌍)이 아니라 **대표성 있는 인기 회사쌍**만 담는다(선정 가이드: 동일/인접 산업 `INDUSTRY_NM`의 경쟁사, 동일 기업유형 대표사, 검색 수요가 높은 대표 회사쌍). 목록에 없는 임의 조합의 페이지는 생성하지 않는다(UC-50 분기 3a, UC-53 분기 4a).
- 정상(URL 정규화): 각 조합의 두 회사 식별자를 URL-safe **slug**로 변환하고(소문자·URL 안전 문자), 두 slug를 **사전식(정렬) 순서**로 배치해 단일 정규 경로 `/vs/{first}-{second}`를 만든다. 정렬 순서를 canonical로 고정하므로 역순 쌍(B vs A)은 별도 페이지를 만들지 않고 동일 canonical URL로 귀속된다(NFR11).
- 정상(생성·서빙): 빌드타임 생성 스크립트(파이썬, §8)가 조합별로 완성된 정적 HTML 파일을 산출하고(요약 FR-61·프리필 진입 FR-62·내부 링크 FR-63·SEO 태그 FR-64·광고 슬롯 예약 FR-65 포함), Nginx가 요청 시 서버 렌더링·DB 조회 없이 직접 서빙한다(NFR1, 공통 규약 1). 조합 URL은 sitemap에 기여된다(FR-64).
- 대안(역순·미정규 요청): 역순·대소문자 변형 등 비정규 경로 요청은 canonical 경로로 정규화된다(canonical 링크 + 서버 리다이렉트/단일화, 상세 서빙 규칙은 SPEC). 색인 중복은 canonical로 흡수된다(NFR11).
- 대안(미존재 조합): 사전 정의 목록에 없어 페이지가 생성되지 않은 경로는 404를 반환하며 색인 대상에서 제외된다(UC-50 분기 3a, UC-A3).
- 대안(구성원 회사 부재·비활성): 목록 항목의 한 회사가 참조 데이터에 없으면(식별자 불일치) 해당 조합 페이지를 생성하지 않고 빌드 경고로 남긴다(무효 조합 스킵).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 사전 정의 조합 목록 | 데이터 파일(항목 배열) | 각 항목 = 두 회사 식별자(`COMP_ENG_NM` 또는 `COMP_ID`). 빌드타임 입력. 코드 무수정 교체 가능 |
| 회사 식별자 | `COMP_ENG_NM`(string, unique) / `COMP_ID`(int) | §4 `TCOMPANY`. slug 원본. 두 식별자는 참조 데이터에 존재해야 함 |
| 참조·복지 데이터 | 빌드타임 참조 번들/DB 스냅샷 | §4/§5. 요약(FR-61)·프리필(FR-62) 값의 출처 |

**출력·상태**

- 조합별 정적 HTML 파일 1개(canonical 경로 `/vs/{first}-{second}`).
- 정규 URL 집합(sitemap 기여 소스, FR-64). 역순 쌍은 별도 URL 없이 canonical로 단일화.
- 서버 사용자 상태 0건(정적 산출물, FR-01).

**검증·비즈니스 규칙**

- (R1) **존재하는 조합만 페이지가 존재**한다. 페이지 집합 = 사전 정의 목록에서 유효(양 회사 참조 데이터 존재)한 항목 집합. 목록 외 경로는 404(UC-50 3a).
- (R2) URL은 두 회사 slug의 **사전식 순서**로 정규화하며, 한 조합에 canonical URL이 정확히 1개다(NFR11). 역순·변형 경로는 canonical로 귀속된다.
- (R3) slug는 URL-safe이며, `/vs/{a}-{b}`의 쌍 구분자(`-`)와 slug 내부 문자가 충돌하지 않도록 slug를 정규화한다(예: 내부 구분 문자 정리). 두 세그먼트 매핑의 모호성은 **알려진 회사 slug 사전**(빌드타임 확정된 회사 집합)에 대한 매칭으로 해결한다. 단, 비교 툴 진입의 회사 지시는 경로 파싱이 아니라 명시적 파라미터로 전달한다(FR-62 R2).
- (R4) 요청 시 백엔드(Uvicorn) 호출·DB 쿼리 0건(NFR1). 정적 파일은 파이썬 생성 스크립트 산출물이며 빌드 생성물 검증 테스트(브리프 §10) 대상이다.
- (R5) 본 FR은 UC-A4의 **F5 조합 페이지 생성 슬라이스를 충족**한다(조합별 정적 HTML 산출·URL 규약). 이와 별개로 **전 생성기 실행 오케스트레이션(F4 회사 페이지 생성기 + F5 조합 생성기 실행)·사이트 전역 `sitemap.xml` 병합·Nginx 배포**는 UC-A4의 **primary 소유 FR = 공통 빌드/배포 FR**(회사 상세 정적 생성 FRD 07의 빌드 파이프라인 FR-50·사이트맵 등록 FR-56 계열이 사이트 전역 병합·배포까지 확장 소유; FR-0x 전역 규약과 정합)이 충족하며, 본 FR은 조합 페이지가 그 사이트 전역 빌드에 제공하는 산출물·URL 규약을 규정한다(범위 경계).

**추적**: UC-50(기본 흐름 1~3, 분기 3a)·UC-A4(F5 조합 생성 슬라이스 충족, 기본 흐름 2) / F5 (UC-A4 사이트 전역 primary=공통 빌드/배포 FR[07 FR-50·FR-56 계열]·UC-A3 색인 연계) / FR-D4(회사 객체) / NFR1·NFR11·NFR12·NFR29 / 브리프 §3(F5)·§4·§8·§2-7.

## [FR-61] 사전 계산 비교 요약 렌더(기본 가정 하)

**설명**: 조합 페이지에 두 회사(A/B)의 핵심 비교 요약을 **빌드타임에 사전 계산**해 렌더한다. 요약은 방문자 개인 입력이 없는 **기본 가정** 하의 데이터 대조이며, 개인 조건을 반영한 실효연봉·시간가치·우선순위 판정은 비교 툴 진입(FR-62)으로 안내한다. UC-51을 소유한다.

**상세 동작**

- 정상(요약 콘텐츠): 각 회사의 참조·복지 데이터에서 아래 대조 블록을 산출·렌더한다.
  - **기업정보 대조**: 정식명·기업유형(`COMP_TP_NM`)·산업(`INDUSTRY_NM`)·로고(`LOGO_NM`)를 양사 나란히 대조한다(§4 `TCOMPANY`·`TCOMPANY_TYPE`).
  - **근무형태 대조**: `WORK_STYLE_VAL`(JSON: remote/flex/unlimitedPTO/refreshLeave/overtime) 5축을 양사 나란히 대조한다(§4 `TCOMPANY`).
  - **복지 대조(카테고리별)**: 9종 카테고리(`compensation`/`flexibility`/`work_env`/`time_off`/`health`/`family`/`growth`/`leisure`/`perks`)별로 항목 수·대표 복지·정량 복지 금액(만원, `BENEFIT_AMT`)을 대조한다. 금액은 각 항목의 배지 상태(공식/추정/만료)와 함께 표기한다(FR-05 인용).
- 정상(기본 가정): 요약은 아래 **기본 가정**으로 산출된다 — (a) 방문자 개인 변수(개인 연봉·통근시간·우선순위·희생요소·야근 강도)를 **입력받지 않으며 반영하지 않는다**, (b) 복지 체크 상태는 회사 실데이터(등록 회사)의 복지 항목을 기본 체크로 간주한 데이터 수준 요약이다, (c) 야근수당·시간조정 가치 등 개인 근무 변수 의존 계산은 산출하지 않고 **데이터 대조(정성·정량)** 만 제시한다. 개인화 판정(vdCard)은 조합 페이지에 표시하지 않는다.
- 정상(안내): 요약이 개인 입력 없는 **사전값 기준**임을 페이지에 명시하고, 개인 조건 반영을 원하면 "비교 툴에서 열기"(FR-62)로 안내한다(UC-51 기본 흐름 4).
- 대안(실데이터 없음 — 해당 없음): 조합 페이지는 등록 회사(모두 실복지 보유)만 대상이므로 요약은 항상 실데이터로 산출되며, 실데이터 부재에 따른 회사페이지용 프리셋 폴백은 발생하지 않는다(등록 정책 변경, UC-51 분기 2a 미발생).
- 대안(정성 복지): 정성 복지(`QUAL_YN=Y`, 금액 없음)는 금액 대조·합산에서 제외하고 정성 항목(`QUAL_DESC_CTNT`)으로 별도 표기한다(FR-05 대안, UC-51 분기 2b).
- 대안(면책 정합): 요약을 방문자 자신의 상황으로 오해할 여지에 대해, 값이 사전값 기준임을 명시하고 복지 데이터 정확성 한계는 면책조항(F7)과 정합하게 고지한다(UC-51 분기 3a).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 회사 A/B 참조 객체 | 회사 객체(`FR-D4`) | 기업정보·`WORK_STYLE_VAL`·`benefits[]`(등록 회사는 실복지 보유 → 실데이터) |
| 기업유형 | `TCOMPANY_TYPE` 필드 | `COMP_TP_CD`·`COMP_TP_NM`(유형 명칭 대조용) |
| 복지 항목 | 복지 객체 | `BENEFIT_CTGR_CD`(9종)·`BENEFIT_AMT`(만원\|null)·`BADGE_CD`·`EXPIRES_DTM`·`QUAL_YN`·`QUAL_DESC_CTNT` |

**출력·상태**

- 정적 HTML 내 사전 렌더된 비교 요약(기업정보·근무형태·복지 카테고리 대조). 초기 HTML에 존재(비-JS 가독, NFR12·NFR24, 공통 규약 3).
- 배지 라벨(공식/추정/만료)이 부착된 정량 복지 금액(만원). 정성 복지는 정성 표기.
- "사전값 기준" 고지 + 비교 툴 진입 안내. 서버 런타임 계산·사용자 저장 없음(§8, 공통 규약 4).

**검증·비즈니스 규칙**

- (R1) 요약은 **개인 입력을 반영하지 않는다**(개인 연봉·우선순위·통근·야근 부재). 개인화 지표(개인 실효연봉·시간조정 가치·vdCard 판정)는 조합 페이지에 렌더하지 않는다(UC-51 기본 흐름 2, §2-8).
- (R2) 배지·불확실성 밴드 규약은 FR-05(전역)를 그대로 따르며 재정의하지 않는다(공식 ±5%/추정 ±20%/만료 ±15% 계수, 배지 상태 산출). 밴드를 요약에 표시할 경우 F7 면책과 정합해야 한다(UC-51 3a).
- (R3) 조합 페이지는 등록 회사(모두 실복지 보유)만 대상이므로 실데이터 부재에 따른 회사페이지용 프리셋 폴백은 발생하지 않는다(amt_source=`real`, FR-06·FR-D3 참조).
- (R4) 정성 복지(`QUAL_YN=Y`)는 금액 합산에서 제외하고 정성 항목으로만 표기한다(FR-04·FR-05).
- (R5) 요약 렌더 시 회사·복지 텍스트(별칭·`NOTE_CTNT`·`QUAL_DESC_CTNT` 등)는 HTML 이스케이프하여 정적 산출물에 안전하게 삽입한다(NFR21). 정적 생성이므로 방문자 입력 경로는 없다.

**추적**: UC-51(기본 흐름 1~4, 분기 2a·2b·3a) / F5 (F7 면책 정합) / FR-05·FR-06·FR-D3·FR-D4 / NFR5·NFR12·NFR24·NFR21 / 브리프 §8·§4/§5·§2-8.

## [FR-62] 조합 페이지 → 비교 툴 진입(양사 a/b 프리필)

**설명**: 조합 페이지의 "비교 툴에서 열기" 경로는 해당 조합의 **두 회사를 슬롯 a(현직)·b(이직처)에 동시에 사전 지정**한 상태로 비교 툴(SPA)에 진입시킨다. 이는 회사 상세의 단일 슬롯 CTA(FR-16)를 **양사 프리필**로 확장한 것이며, 진입 지시자를 소비해 두 슬롯을 채우는 클라이언트 동작을 규정한다. UC-52를 소유한다.

**상세 동작**

- 정상(진입 경로 구성): 조합 페이지가 "비교 툴에서 열기" 링크에 **두 회사 식별자와 슬롯 배정**을 담아 비교 툴 진입점으로 연결한다(예: URL 쿼리 파라미터 `a=<식별자>&b=<식별자>`). 슬롯 기본 배정은 URL 조합 순서(정규 첫 회사→슬롯 a, 둘째 회사→슬롯 b)를 따른다. 이 진입 경로(정적 페이지의 링크·파라미터) **생성**은 F5(본 FR)가 소유한다.
- 정상(부팅·소비): 비교 툴이 로드되며 참조 번들(`reference/all`)을 1회 로드한다(FR-02, UC-A1). 클라이언트가 두 지시자를 읽어 각 대상 슬롯의 회사 객체를 REF 인라인(또는 `companies/{id}`, FR-D6/FR-D7)에서 확보하고, FR-16/FR-14의 정규화·복지 초기화 흐름에 **양 슬롯 모두** 합류시킨다. 사용자는 검색 단계를 건너뛰고 두 회사가 a·b에 지정된 상태로 시작한다(UC-52 기본 흐름 2~3, UC-14 연동).
- 정상(본인 조건 재계산): 사용자가 본인 조건(연봉, 복지 체크/금액, 근무형태 재택·유연·야근·임금유형, 통근시간, 우선순위 직접 선택)을 입력하면(F2, UC-20~UC-25), 클라이언트가 본인 조건으로 리포트를 재계산한다(F3 `compare()`, UC-30). 서버 호출·저장은 발생하지 않는다(§2-8, NFR16, FR-01).
- 대안(슬롯 교체): 사용자가 슬롯 회사를 교체하려 하면 검색(FR-10~FR-14)·슬롯 재지정(UC-14)으로 변경 후 재계산한다(UC-52 분기 3a).
- 대안(본인 조건 미입력 진행): 조건 입력 없이 진행하면 조합 기본값(사전값, 등록 회사의 실복지)으로 리포트를 산출하고, 이후 UC-35 재계산으로 개인 조건 반영을 안내한다(UC-52 분기 4a).
- 대안(비등록 회사 직접 입력): 프리필된 두 회사는 등록 회사로 실복지가 채워진다. 사용자가 슬롯을 비등록 회사로 교체하면(직접 입력 모드) 기업유형 프리셋 초기값으로 복지가 채워지고(FR-06·FR-D3), 사용자가 수정 가능하다(UC-52 분기 4b, UC-21/UC-22).
- 대안(무효 지시자): 존재하지 않거나 해석 불가한 회사 지시자는 해당 슬롯을 미선택으로 두고 정상 검색 진입으로 폴백한다(FR-16 대안·FR-D11·NFR26). 둘 중 한 슬롯만 유효하면 유효 슬롯만 프리필된다.
- 대안(로컬 보관): 사용자가 결과를 기기 내 보관하려 하면 최근 비교를 브라우저 localStorage에만 저장한다(FR-07, UC-36/UC-A2, 서버 저장 없음).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 회사 A 지시자 | `COMP_ID`(int) 또는 `COMP_ENG_NM`/회사명(string) | 조합 페이지가 슬롯 a용으로 부여. 진입점에서 소비 |
| 회사 B 지시자 | `COMP_ID`(int) 또는 `COMP_ENG_NM`/회사명(string) | 조합 페이지가 슬롯 b용으로 부여. 진입점에서 소비 |
| 슬롯 배정 | (a, b) 매핑 | 기본 = 정규 조합 순서(첫→a, 둘→b). 사용자가 교체 가능(UC-52 3a) |

**출력·상태**

- 진입 후 상태: `matched.a` = 회사 A 객체, `matched.b` = 회사 B 객체(FR-14 사후조건과 동일, §01-5 상태 모델). `benS.a`/`benS.b`는 각 회사(등록 회사) 실데이터로 초기화(FR-06).
- 이후 F2 입력·F3 재계산은 클라이언트에서 수행되며 서버 전송·저장 없음(§2-8, NFR16, FR-01).
- PRD-SEG-3 → PRD-SEG-1/PRD-SEG-2 전환 성립.

**검증·비즈니스 규칙**

- (R1) 본 FR은 **양사 프리필**로 두 슬롯을 채운다(FR-16의 단일 슬롯 확장). 진입 지시자 **소비**·슬롯 반영의 클라이언트 동작을 규정하고, 부팅·정규화·복지 초기화의 세부는 FR-02·FR-16·FR-14·FR-06을 인용해 재정의하지 않는다.
- (R2) 회사 지시는 **명시적 파라미터**로 전달하며(`/vs/` 경로 세그먼트 파싱에 의존하지 않음), 이로써 slug 구분자 모호성(FR-60 R3)과 무관하게 대상 회사를 확정한다. 지시자가 회사명 기반이면 검색 API로 1건 해석 후 합류할 수 있고(FR-16 대안), 실패 시 검색 진입으로 폴백한다.
- (R3) 프리필된 두 회사(등록 회사)도 FR-14의 정규화·복지 초기화 규약을 동일 적용한다(등록 회사는 실복지 보유 → amt_source=`real`).
- (R4) 진입·입력·재계산 전 과정에서 사용자 데이터의 서버 전송·저장이 없다(§2-8, NFR16, FR-01). 로컬 저장은 localStorage 한정이며 미지원·차단 시에도 비교는 정상 동작한다(FR-07, NFR25).
- (R5) 무효/부분 지시자는 크래시 없이 미선택·검색 진입으로 폴백한다(FR-D11·NFR26).

**추적**: UC-52(기본 흐름 1~6, 분기 3a·4a·4b·6a) / F5 → F2·F3 (F1 슬롯 연동) / FR-02·FR-06·FR-07·FR-14·FR-16·FR-D6·FR-D7·FR-D11 / NFR16·NFR25·NFR26 / 브리프 §7·§2-8, PRD-SEG-3.

## [FR-63] 관련 조합·회사 상세 내부 링크

**설명**: 조합 페이지에 **관련 인기 조합**(다른 회사쌍)과 **개별 회사 상세 페이지**(F4)로 이어지는 내부 링크를 제공해 콘텐츠 순회와 검색 색인·유입 확대(§2-7)에 기여한다. UC-53을 소유한다.

**상세 동작**

- 정상(회사 상세 링크): 조합의 두 회사 각각에 대해 해당 회사 상세 페이지(F4, FRD 07)로 이어지는 내부 링크를 렌더한다. 회사 상세로 이동하면 기업정보·복지·근무형태 열람과 "이 회사로 비교하기" 단일 슬롯 CTA(F4 소유)로 비교 진입할 수 있다(UC-53 기본 흐름 3).
- 정상(관련 조합 링크): 사전 정의 목록(FR-60)에서 **관련 조합**을 선정해 링크한다. 관련성 기준: 두 회사 중 하나를 **공유**하는 다른 조합, 또는 동일/인접 산업(`INDUSTRY_NM`)·동일 기업유형의 조합. 다른 인기 조합으로 이동하면 그 조합에 대해 UC-50/UC-51 흐름을 반복한다(UC-53 기본 흐름 4).
- 대안(대상 미존재): 대상 조합 페이지가 사전 정의 목록에 없어 생성되지 않았으면 링크를 노출하지 않는다(존재하는 조합만 내부 링크 대상, FR-60 R1, UC-53 분기 4a). 회사 상세 링크도 생성된 회사 페이지(F4)만 대상으로 한다.
- 대안(관련 없음): 관련 조합이 없으면 관련 조합 링크 영역을 표시하지 않는다(UC-53 분기 1a). 회사 상세 링크(두 구성 회사)는 항상 존재한다.

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 현재 조합 | 회사쌍(A, B) | FR-60 정규 조합 |
| 사전 정의 조합 목록 | 데이터 파일 | 관련 조합 후보 소스(FR-60 입력) |
| 회사 상세 URL 규칙 | F4(FRD 07) 소유 | 두 구성 회사 상세 페이지 경로 |

**출력·상태**

- 정적 HTML 내 내부 링크 블록: (a) 두 구성 회사의 상세 페이지 링크, (b) 관련 인기 조합 링크(존재 시). 초기 HTML에 존재(비-JS 가독, NFR12·NFR24).
- 내부 링크는 sitemap·색인 커버리지(FR-64)와 함께 유입 순회에 기여. 서버 사용자 저장 없음.

**검증·비즈니스 규칙**

- (R1) 내부 링크 대상은 **실제 생성된 페이지**(조합: FR-60 유효 목록 / 회사 상세: F4 생성 페이지)로 한정한다. 미생성 대상 링크를 노출하지 않아 404 유입을 방지한다(FR-60 R1, NFR/색인 정합).
- (R2) 회사 상세 페이지의 렌더·CTA·정적 생성은 F4(FRD 07, FR-5x)가 소유하며, 본 FR은 F4 페이지로의 **링크 제공**만 규정한다(범위 경계).
- (R3) 관련 조합 선정 로직은 빌드타임에 사전 정의 목록에 대해 계산되며(공유 회사·동일 산업/유형), 런타임 서버 조회 없이 정적으로 렌더된다(NFR1).

**추적**: UC-53(기본 흐름 1~4, 분기 3a·4a·1a) / F5 (F4 연동) / FR-60 / NFR6·NFR9·NFR12·NFR24 / 브리프 §2-7, PRD-SEG-3.

## [FR-64] 조합 페이지 SEO 태그·canonical·sitemap 기여

**설명**: 각 조합 페이지가 고유한 SEO 메타데이터·canonical 링크를 갖고 sitemap·색인에 정상 기여하도록 규정한다. SEO는 수익의 뼈대(§2-7)이며, 본 FR은 조합 페이지에 대해 관련 NFR(NFR6~NFR12)의 검증 조건을 상속한다(재정의하지 않고 인용). 본 FR은 조합 페이지 태그·canonical과 정규 조합 URL의 sitemap 기여로 **UC-A4(빌드타임 정적 페이지·사이트맵 생성·배포)의 F5 조합 URL 기여 슬라이스를 충족**한다(UC-A4 기본 흐름 4의 sitemap 생성에 조합 URL 제공). UC-50 유입과 UC-A3(색인)을 지지하며, 사이트 전역 sitemap 병합·배포(UC-A4 primary)는 공통 빌드/배포 FR(아래 R5)이 충족한다.

**상세 동작**

- 정상(고유 메타): 각 조합 페이지는 조합명("A vs B")·두 회사명을 반영한 **고유 `<title>`과 `meta description`**을 가진다(NFR6). 예: `<title>삼성전자 vs SK하이닉스 복지·연봉 비교 | jobcho.wiki</title>`, description은 두 회사의 처우·복지·근무형태 비교 요지. 페이지별 중복 없음.
- 정상(OpenGraph): `og:title`·`og:description`·`og:type`·`og:url`·`og:image`를 포함해 소셜 공유 미리보기가 정상 렌더되게 한다(NFR7). `og:url`은 canonical 정규 URL과 일치.
- 정상(canonical): 각 조합 페이지는 자기 자신(정규 `/vs/{first}-{second}`)을 가리키는 `<link rel="canonical">`을 가진다. **역순 쌍(B vs A)·변형 경로는 동일 canonical로 정규화**되어 중복 색인을 방지한다(NFR11, FR-60 R2).
- 정상(sitemap 기여): 조합 페이지의 정규 URL을 빌드타임 `sitemap.xml`에 기여한다(회사 상세(~96개)·정책·랜딩과 함께 병합, NFR9). 사이트 전역 sitemap 병합·`robots.txt`(`Sitemap:` 라인 포함, NFR10)는 UC-A4 primary 소유 FR(사이트 수준 공통 빌드/배포 FR — 07 FR-56 계열이 사이트 전역 병합·배포까지 확장 소유)이 충족하고, 본 FR은 조합 URL 집합을 그 입력으로 제공한다(UC-A4 F5 URL 기여 슬라이스).
- 정상(크롤러 가독·인코딩): 본문(요약·내부 링크)이 초기 HTML에 존재해 JS 없이도 크롤러가 읽는다(NFR12·NFR24). `<html lang="ko">`·`charset=utf-8`(NFR29). HTTPS(jobcho.wiki)로 서빙(NFR12·NFR23).
- 대안(구조화 데이터): 회사 상세의 JSON-LD Organization 구조화 데이터 요구(NFR8)는 F4(회사 상세)가 소유한다. 조합 페이지는 필수 요건으로 두지 않으며, 필요 시 두 회사 Organization을 참조하는 구조화 데이터를 추가할 수 있다(선택, F4 규약과 정합).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 조합명·회사명 | string | 두 회사 `COMP_NM` + "vs". title/description/OG 소스 |
| canonical URL | string | FR-60 정규 `/vs/{first}-{second}` |
| 대표 이미지 | 자산 경로 | `og:image` 소스(로고/대표 이미지, `LOGO_NM` 활용) |

**출력·상태**

- 페이지별 고유 `<title>`·`meta description`·OpenGraph 태그·자기참조 canonical.
- sitemap 기여용 정규 URL 집합(UC-A4 primary 소유 공통 빌드/배포 FR로 전달, F5 URL 기여 슬라이스). 역순 쌍 중복 URL 0(단일 canonical).

**검증·비즈니스 규칙**

- (R1) 조합 페이지 간, 그리고 회사·정책·랜딩 페이지와 title/description **중복 0**(NFR6). 조합명·회사명이 반영되어야 한다.
- (R2) 한 조합에 canonical URL이 정확히 1개이며 역순·변형 경로는 그 canonical로 정규화된다(NFR11, FR-60 R2). `og:url` = canonical.
- (R3) 조합 URL은 sitemap 생성물 검증 테스트(브리프 §10, NFR9)에서 누락 0이어야 한다. `robots.txt`는 `Sitemap:` 라인을 포함한다(NFR10, 사이트 수준).
- (R4) 본문은 비-JS 렌더로 초기 HTML에 존재한다(NFR12·NFR24). 인코딩·언어 태그는 NFR29를 따른다.
- (R5) 본 FR은 조합 페이지의 태그·canonical과 정규 조합 URL 기여로 **UC-A4의 F5 조합 URL 기여 슬라이스를 충족**한다. 사이트 전역 `sitemap.xml`·`robots.txt` 파일 자체의 병합·생성·배포는 UC-A4의 **primary 소유 FR = 공통 빌드/배포 FR**(회사 상세 정적 생성 FRD 07의 사이트맵 등록 FR-56·빌드 파이프라인 FR-50 계열이 사이트 전역 병합·배포까지 확장 소유; FR-0x 전역 규약과 정합)이 충족한다(범위 경계).

**추적**: UC-50(유입)·UC-A4(F5 조합 URL 기여 슬라이스 충족, 기본 흐름 4) / F5 (UC-A3 색인·UC-A4 사이트 전역 primary=공통 빌드/배포 FR[07 FR-56·FR-50 계열] 연계) / FR-60 / NFR6·NFR7·NFR9·NFR10·NFR11·NFR12·NFR29 / 브리프 §2-7·§8.

## [FR-65] 조합 페이지 광고·제휴 슬롯("광고" 표기)

**설명**: 조합 페이지는 콘텐츠 페이지로서 광고 배치 대상이다(§2-6). AdSense 자동/수동 슬롯과 `affiliate.json` 기반 제휴 컴포넌트를 노출하되(§2-5, F6), 모든 광고·제휴 영역에 "광고" 표기를 부착하고 콘텐츠(비교 요약) 열람을 해치지 않는 범위에서 배치한다. UC-54를 소유한다.

**상세 동작**

- 정상(배치·노출): 조합 페이지 렌더 시 AdSense 자동광고·수동 슬롯과 제휴 컴포넌트가 노출된다(§2-5). 조합 페이지는 회사 상세·리스트·랜딩과 함께 광고 배치 대상이다(§2-6). 광고 슬롯은 요약·내부 링크 콘텐츠를 밀어내지 않도록 절제해 배치한다.
- 정상("광고" 표기): 모든 광고/제휴 영역에 "광고" 라벨을 부착한다(공정위 표시광고 규정, NFR19, FR-01 §4 용어). 표기·고지는 광고/제휴 고지 정책 페이지(F7)와 정합한다.
- 정상(레이아웃 안정): 광고/이미지 슬롯은 고정 크기를 예약해 폰트·광고 삽입으로 인한 레이아웃 이동을 방지한다(CLS ≤ 0.1, NFR5).
- 대안(제휴 데이터 교체): 제휴 링크는 `affiliate.json` 데이터로 구성되어 코드 수정 없이 교체 가능하다(§2-5, UC-54 분기 1a).
- 대안(슬롯 렌더·클릭 상세): AdSense 슬롯·제휴 컴포넌트의 렌더링·클릭·개인화 동의(광고 쿠키 consent) 상세 동작은 F6(광고·제휴 FRD)이 소유하며, 본 FR은 조합 페이지가 그 대상 페이지임과 슬롯 위치 예약만 규정한다(UC-54 분기 3a, 범위 경계).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 광고 슬롯 위치 | 페이지 템플릿 영역 | 콘텐츠(요약)를 해치지 않는 위치. 고정 크기 예약(NFR5) |
| 제휴 데이터 | `affiliate.json` | 코드 무수정 교체 가능(§2-5). 컴포넌트 상세는 F6 |
| AdSense client id | 플레이스홀더 | 정적 산출물·저장소에 실제 시크릿 노출 금지(NFR22) |

**출력·상태**

- 조합 페이지에 "광고" 표기가 부착된 AdSense/제휴 슬롯 노출(콘텐츠 열람을 해치지 않는 배치).
- 슬롯은 고정 크기 예약으로 CLS 안정(NFR5). 서버 사용자 저장 없음.

**검증·비즈니스 규칙**

- (R1) 모든 광고·제휴 영역에 "광고" 표기가 노출된다(NFR19, §2-5). 표기·고지는 F7 정책과 정합한다(UC-54 분기 2a).
- (R2) 조합 페이지는 콘텐츠 페이지 광고 배치 정책(§2-6)을 따른다. 광고는 비교 요약 열람을 해치지 않는 범위로 배치한다(비교 툴 입력 화면의 광고 최소화 정책과 구분 — 조합 페이지는 배치 대상).
- (R3) 슬롯은 고정 크기를 예약해 레이아웃 이동을 방지한다(NFR5).
- (R4) AdSense client id는 플레이스홀더이며 실제 시크릿을 정적 산출물·저장소에 노출하지 않는다(NFR22).
- (R5) 슬롯 렌더링·클릭·개인화 동의 등 광고/제휴 슬롯 상세는 F6(광고·제휴 FRD)이 소유하고, 본 FR은 조합 페이지의 배치 대상성·위치 예약·"광고" 표기 부착만 규정한다(범위 경계).

**추적**: UC-54(기본 흐름 1~3, 분기 1a·2a·3a) / F5 → F6 (F7 고지 정합) / NFR5·NFR19·NFR22 / 브리프 §2-5·§2-6, A4(광고·제휴 시스템).

UC-A4의 사이트 전역 오케스트레이션(전 생성기 실행 + 사이트 전역 `sitemap.xml` 병합 + `robots.txt` + Nginx 배포)의 **primary 소유**는 공통 빌드/배포 FR(회사 상세 정적 생성 FRD 07의 빌드 파이프라인 FR-50·사이트맵 등록 FR-56 계열이 사이트 전역 병합·배포까지 확장 소유; FR-0x 전역 규약과 정합)이며, 본 문서의 FR-60·FR-64는 그 사이트 전역 빌드에 산출물·조합 URL을 제공한다. 이로써 UC-A4가 ≥1 FR로 커버된다(01 §3.2-3; UC-A4 primary는 01 §7 인덱스가 배정하는 공통 빌드/배포 FR과 정합).
