# 검색·회사 선택 (FRD)

**목적**: PRD 기능 F1(회사 검색 + 선택)이 USECASE UC-10~UC-15로 정의한 사용자 상호작용을, 클라이언트(바닐라 JS SPA)가 제공해야 하는 **기능 요구사항(FR-1x)**으로 구현 이전 수준에서 정밀하게 명세한다. 검색어 입력·디바운스, 이름·별칭·영문 매칭, 자동완성·부분일치 후보 리스트, 무결과·오류 UX, 두 슬롯(현직 a / 이직처 b)으로의 회사 선택·교체·해제, 회사 미선택 직접 입력 모드를 다룬다.

**상위 추적**: 본 문서는 FRD → USECASE → PRD → 브리프 체인에 속한다. 상위 근거 = USECASE `02-검색과-회사선택.md`(UC-10~UC-15), PRD `05-기능-요구사항.md`(F1), 브리프 §2(핵심 결정)·§4(데이터 모델)·§6(읽기 전용 API)·§7(비교 알고리즘). 전역 규칙·용어·상태 모델은 FRD `01-개요와-FR규칙.md`(FR-01~FR-08, §4 용어, §5 상태 모델), 데이터 계약은 FRD `02-데이터-계약.md`(FR-D3·FR-D4·FR-D6·FR-D7·FR-D8·FR-D11)를 인용한다. 각 FR은 관련 NFR(NFR2·NFR3·NFR14·NFR16·NFR17·NFR20·NFR21·NFR25·NFR26·NFR29, PRD `07-비기능-요구사항.md`)을 참조한다.

**범위 경계**: 본 문서는 **F1의 검색·선택 상호작용**만 명세한다. 슬롯에 회사가 반영된 이후의 조건 입력 상세(연봉·복지 체크/금액·근무형태·통근·우선순위·희생요소)는 F2(FRD 04 비교 입력), 리포트 산출은 F3(FRD 05 비교 리포트)가 소유한다. 검색·상세 API의 응답 스키마·상한·오류 형태는 데이터 계약(FR-D6·FR-D7·FR-D11)이 확정하며, 본 문서는 그 계약을 **소비하는 클라이언트 동작**을 명세한다. 로그인/회원/프로파일러/서버측 사용자 쓰기는 제품 범위에서 영구 제외이므로(브리프 §2-1·§2-3·§2-4, FR-01) 어떤 검색·선택 흐름에도 등장하지 않는다. 선택·직접 입력 상태는 클라이언트 메모리와 localStorage에만 존재하며 서버로 전송·저장되지 않는다(NFR16·NFR17).

**FR-ID 대역**: 본 문서는 **FR-1x 대역**(FR-10~FR-17)을 소유한다(FRD 01 §2.3). 부여한 ID는 재사용하지 않는다(안정 ID). 하위 문서(FLOW/WIREFRAME/SPEC/TASK)가 이 ID를 인용한다.

## 공통 규약 (전 FR 적용)

- **슬롯 표기**: `s ∈ {a, b}`. `a`=현직(비교 기준), `b`=이직처(비교 대상). 두 슬롯은 완전히 독립적으로 검색·선택·직접 입력을 수행한다(FRD 01 §4 용어).
- **참조원**: 검색은 서버 검색 API(`GET /api/v1/companies/search?q=`, FR-D6)를 1차로 사용한다. 부팅 시 로드한 참조 번들(REF, FR-02·FR-D1)이 존재하므로, 참조 데이터(유형·프리셋·회사 인라인 복지)는 REF에서 조회하며 회사 선택 시 상세를 별도 호출하지 않을 수 있다(FR-D7 대안).
- **상태 모델**: 본 문서가 채우는 공용 상태는 `matched.{a,b}`(FRD 01 §5, FR-D8)이다. 회사 선택 시 파생되는 `benS.{a,b}`·`wsState.{a,b}` 초기화는 F2(FRD 04)가 소유하며, 본 문서는 그 초기화를 **트리거**하는 지점까지 명세한다.
- **네트워크 제약**: 모든 검색·상세 요청은 GET·읽기 전용이다(NFR20). 검색어·선택 결과를 포함한 어떤 사용자 입력도 서버로 쓰기 전송하지 않는다(NFR16).
- **렌더 안전**: 후보 리스트·회사명·별칭·산업명은 `textContent`/이스케이프로 삽입하며 `innerHTML` 직접 삽입을 금지한다(NFR21).
- **디바운스 기본값**: 본 문서가 명세하는 검색 입력 디바운스 지연의 기본값은 **300ms**다(레거시 `job_change/app.js` `doSearch` 이식값). 정확한 수치는 SPEC에서 조정 가능하나, "입력 갱신마다 즉시 호출하지 않고 지연·병합한다"는 규칙 자체는 불변이다.

## [FR-10] 검색어 입력·디바운스·요청 트리거

**설명**: 특정 슬롯(a/b)의 검색 입력란에 사용자가 입력한 검색어로 서버 검색을 트리거한다. 입력 갱신마다 즉시 호출하지 않고 디바운스로 병합하며, 빈 입력·최소 길이·응답 경합을 규정한다. UC-10(회사명 검색)·UC-12(자동완성·부분일치)의 입력측 동작을 담당한다.

**상세 동작**
- 정상: 사용자가 슬롯 `s`의 입력란에 글자를 입력 → 입력 이벤트마다 검색어 `q = 입력값.trim()`을 계산 → `q.length ≥ 1`이면 디바운스 타이머(기본 300ms)를 리셋하고, 만료 시 `GET /api/v1/companies/search?q={encodeURIComponent(q)}`를 1회 호출한다(FR-D6). 디바운스 대기·요청 중에는 해당 슬롯의 후보 패널을 "검색 중…" 로딩 상태로 표시한다(레거시 `doSearch` 이식).
- 대안(빈/공백 입력): `q.length < 1`(빈 문자열 또는 공백만)이면 검색을 호출하지 않고 해당 슬롯의 후보 패널을 닫아 초기화한다(UC-10 분기 3a, UC-12 분기 2a). 대기 중이던 디바운스 타이머가 있으면 취소한다.
- 대안(연속 입력 경합): 입력이 계속 변경되면 후보 리스트는 항상 **최신 검색어 기준** 결과만 반영한다. 이전 검색어에 대한 응답이 늦게 도착해도 현재 입력어와 불일치하면 폐기한다(UC-12 분기 3a). 슬롯별 디바운스 타이머는 독립(`_searchTimers[s]`)이다.
- 대안(요청 실패): 검색 요청이 네트워크·5xx로 실패하면 무결과와 구분되는 오류 상태로 전이한다(FR-13, UC-10 분기 3b).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 검색어 `q` | string | 입력란 원문을 `trim()`한 값. 최소 유효 길이 1자(공백 제외). URL 인코딩 후 쿼리로 전달 |
| 대상 슬롯 `s` | enum | `a` 또는 `b`. 슬롯별 입력란·후보 패널·디바운스 타이머가 1:1 대응 |

**출력·상태**
- 검색 트리거(디바운스 만료 시 GET 1회) 또는 트리거 억제(빈/공백 입력).
- 슬롯 `s`의 후보 패널 표시 상태: `닫힘`(빈 입력) / `로딩`("검색 중…") / `결과`(FR-12) / `무결과·오류`(FR-13).
- 서버·localStorage에 어떤 상태도 기록하지 않는다(입력 단계, NFR16).

**검증·비즈니스 규칙**
- (R1) `q.length ≥ 1`일 때만 검색을 호출한다(레거시 및 FR-D6 `min_length=1`과 정합). 빈/공백 검색어로는 요청을 발생시키지 않아 네트워크 낭비를 방지한다(FR-D11).
- (R2) 입력 이벤트당 즉시 호출을 금지하고 디바운스로 병합한다. 이는 모바일 우선 타이핑 부담·요청 빈도를 낮춰 응답성(NFR2 INP)과 자원(NFR3)을 보호한다(UC-12, PRD-CTX-1).
- (R3) 최신 검색어 원칙: 후보 리스트에는 현재 입력어에 대응하는 응답만 반영한다(경합 응답 폐기).
- (R4) 검색어·입력값을 서버에 쓰기 전송하지 않는다(NFR16, NFR20).

**추적**: UC-10·UC-12 / F1 → FR-D6·FR-D11 / NFR2·NFR3·NFR16·NFR20 / 브리프 §6, PRD-CTX-1.

## [FR-11] 이름·별칭·영문 매칭과 결과 상한(최대 20)

**설명**: 검색어를 회사 정식명과 별칭(약칭·영문·그룹 통칭)에 부분일치시켜, 입력한 표현이 정식명이든 별칭이든 동일 정식 회사로 수렴하도록 한다. 결과는 최대 20건으로 상한한다. UC-10(회사명 검색)·UC-11(별칭·영문 통칭 검색)을 담당한다.

**상세 동작**
- 정상: 서버 검색 API가 검색어 `q`를 `%q%` 부분일치로 회사 정식명(`COMP_NM`)과 별칭(`TCOMPANY_ALIAS.ALIAS_NM`) 양쪽에 매칭하여 후보를 반환한다(브리프 §6, FR-D6, 레거시 `WHERE c.COMP_NM LIKE %s OR ca.ALIAS_NM LIKE %s`). 정식명 일부 입력(UC-10)과 별칭·영문·통칭 입력(UC-11)이 **동일 API·동일 경로**로 처리된다.
- 정상(별칭 → 정식명 표시): 별칭으로 매칭된 후보도 후보 리스트에는 **정식명(`comp_nm`)**으로 표시되어, 사용자가 약칭·영문 입력만으로도 정식 회사를 식별한다(UC-11 기본 흐름 5).
- 대안(중복 수렴): 서로 다른 별칭이 동일 회사로 수렴하거나 정식명·별칭이 동시에 매칭되어도, 후보에는 회사당 1건(정식명 기준)만 나타난다. 서버 응답은 `comp_id` 기준 중복 없이 반환되며(레거시 `SELECT DISTINCT`), 클라이언트는 `comp_id` 기준으로 재중복 제거를 방어적으로 적용한다(UC-11 분기 4a; §4 `unique(COMP_ID, ALIAS_NM)`로 회사 내 별칭 중복은 없음).
- 대안(20건 초과): 매칭이 20건을 초과하면 상위 20건만 반환·표시하고(§6·FR-D6 상한), 사용자에게 "글자를 더 입력해 좁히세요"를 안내한다(UC-10 분기 4a, UC-12 분기 4a).
- 대안(0건): 이름·별칭 매칭이 0건이면 무결과 처리로 전이한다(FR-13, UC-10 분기 4b / UC-11 분기 4b).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 검색어 `q` | string | FR-10에서 전달된 유효 검색어(≥1자). 정식명·별칭·영문·통칭 어느 표현이든 허용 |

**출력·상태**: 매칭 후보 배열(최대 20건). 각 항목은 검색 응답 축소 투영 `{comp_id:int, comp_nm:string, comp_tp_cd:string(6종), industry_nm:string|null, logo_nm:string|null}`이다(FR-D6). 이 항목이 FR-12의 후보 리스트 렌더 입력이 된다.

**검증·비즈니스 규칙**
- (R1) 매칭 축은 **정식명 + 별칭** 두 컬럼 부분일치로 고정한다(§6, FR-D6). 별칭 테이블이 영문 표기·약칭·그룹 통칭을 정식 회사로 잇는 유일한 연결 데이터다(§4 TCOMPANY_ALIAS, 브리프 §5 회사 200개 각 별칭[]).
- (R2) 결과 상한 20건. 20건 초과 반환·표시를 금지한다(§6, FR-D6).
- (R3) 후보는 `comp_id` 기준 유일하다(중복 회사 미표기). 서버 DISTINCT + 클라이언트 방어적 dedup.
- (R4) 후보 항목의 `comp_tp_cd`는 6종(`large`/`startup`/`mid`/`foreign`/`public`/`freelance`) 밖의 값을 갖지 않는다(FR-D2·FR-D6).

**추적**: UC-10·UC-11 / F1 → FR-D6 / 브리프 §4(TCOMPANY_ALIAS)·§5·§6.

## [FR-12] 자동완성·부분일치 후보 리스트·정렬

**설명**: FR-11이 반환한 후보를 슬롯별 자동완성 리스트로 렌더링한다. 입력 글자가 늘어날수록 후보가 좁혀지고, 각 후보는 정식명·기업유형·산업·로고와 함께 안전하게 표시되며 결정적 순서로 정렬된다. UC-12(자동완성·부분일치)·UC-11(별칭 후보 표시)을 담당한다.

**상세 동작**
- 정상: FR-10 디바운스 만료 후 도착한 후보 배열을 슬롯 `s`의 후보 패널에 리스트로 렌더한다. 각 후보 행은 `comp_nm`(정식명)을 주 라벨로, `comp_tp_cd`의 표시 라벨·`industry_nm`·`logo_nm`을 보조 정보로 노출한다(UC-10 기본 흐름 5). 로고는 `logo_nm` 약어로 매핑하되 의미 이미지에는 대체 텍스트를 부여한다(NFR15).
- 정상(부분일치 좁힘): 사용자가 글자를 더 입력하면 FR-10이 새 검색을 트리거하고, 새 후보로 리스트를 교체한다. 입력이 늘수록 후보 수가 줄어드는 좁힘 UX를 제공한다(UC-12 기본 흐름 5).
- 정상(정렬): 후보 표시 순서는 클라이언트에서 결정적으로 정렬한다 — (1) 검색어와 **정식명 완전일치** → (2) 정식명 **접두 일치**(startsWith) → (3) 별칭 접두 일치 → (4) 정식명 길이 오름차순 → (5) 정식명 가나다(로케일 `ko`) 순. 서버 응답 순서(브리프 §6·FR-D6는 결과 순서를 규정하지 않음)에 의존하지 않고 최상위에 가장 근접한 후보가 오도록 보정한다.
- 대안(로딩·전이): FR-10의 "검색 중…" 로딩 상태에서 응답 도착 시 결과 리스트로, 0건·오류 시 FR-13 빈 상태로 전이한다.
- 대안(선택): 사용자가 후보 1건을 클릭·키보드 선택하면 FR-14(회사 선택)로 연결한다.

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 후보 배열 | `{comp_id, comp_nm, comp_tp_cd, industry_nm, logo_nm}[]` | FR-11 출력. 길이 0~20 |
| 현재 검색어 `q` | string | 정렬의 완전/접두 일치 판정 기준 |

**출력·상태**: 슬롯 `s`의 후보 리스트 UI(정렬·렌더된 상태). 선택 대기 상태이며 아직 `matched[s]`를 변경하지 않는다.

**검증·비즈니스 규칙**
- (R1) 각 후보 행의 텍스트(정식명·산업·유형 라벨)는 `textContent`/이스케이프로 삽입한다. `innerHTML` 직접 삽입 금지(NFR21, XSS 차단).
- (R2) 후보 리스트는 키보드만으로 이동·선택 가능하고 각 항목이 라벨을 가진다(NFR14 접근성; 목록 시맨틱·포커스 표시).
- (R3) 정렬은 결정적(동일 입력→동일 순서)이며 서버 반환 순서에 의존하지 않는다.
- (R4) 표시 문자열은 한국어이며 UTF-8로 렌더한다(NFR29).
- (R5) 최대 20건만 렌더한다(FR-11 R2와 정합). 20건 초과 안내 문구를 리스트 하단에 표시할 수 있다.

**추적**: UC-11·UC-12 / F1 → FR-D6 / NFR14·NFR15·NFR21·NFR29 / 브리프 §6.

## [FR-13] 무결과·검색 오류 빈 상태 UX

**설명**: 검색 결과가 0건이거나 검색 API가 실패했을 때, 후보 리스트 대신 상태에 맞는 빈 상태를 표시하고 다음 행동을 안내한다. **무결과(정상 0건)**와 **오류(네트워크·5xx)**를 구분해, 오류를 무결과로 오인해 직접 입력으로 넘기지 않는다. UC-13(검색결과 없음 처리)을 담당한다.

**상세 동작**
- 정상(무결과): 이름·별칭 매칭이 0건(HTTP 200, 빈 배열 `[]`, FR-D6·FR-D11)이면 "검색 결과 없음" 빈 상태를 표시하고 두 대안을 안내한다 — (a) 다른 검색어·별칭으로 재검색, (b) 직접 입력 모드로 진행(FR-17, UC-15). 무결과는 오류로 취급하지 않는다.
- 대안(재검색): 사용자가 철자·약칭을 수정하면 FR-10~FR-12로 재진입한다(UC-13 분기 3a).
- 대안(직접 입력 전환): 사용자가 "직접 입력"을 선택하면 FR-17로 전이한다(UC-13 분기 3b).
- 대안(검색 오류): 검색 요청이 네트워크 오류·타임아웃·5xx로 실패하면 무결과와 **구분되는** 오류 상태("검색을 불러오지 못했습니다")를 표시하고 재시도 경로를 우선 안내한다. 직접 입력(FR-17)도 대안으로 제시하되, 0건 무결과 화면과 문구·행동을 다르게 한다(UC-10 분기 3b, UC-13 분기 1a).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 검색 결과 | `[]` (빈 배열) 또는 오류 신호 | 빈 배열=무결과, 예외/비정상 응답=오류 |
| 대상 슬롯 `s` | enum | 빈 상태를 표시할 슬롯 |

**출력·상태**
- 슬롯 `s`의 후보 패널: `무결과` 빈 상태(재검색·직접입력 안내) 또는 `오류` 상태(재시도 안내).
- 대상 슬롯의 `matched[s]`는 미선택(`null`)로 유지된다(회사 미반영).
- 앱은 크래시하지 않으며 다른 슬롯·기능은 정상 동작한다(NFR26).

**검증·비즈니스 규칙**
- (R1) 무결과(200 `[]`)와 오류(예외·비정상 상태 코드)를 서로 다른 상태·문구로 구분한다. 오류를 무결과로 오인하지 않는다(UC-13 분기 1a, NFR26).
- (R2) 무결과 화면은 재검색과 직접 입력(FR-17) 두 경로를 모두 제시한다(UC-13 기본 흐름 3).
- (R3) 검색 실패가 앱 크래시를 유발하지 않는다(방어적 처리, FR-D11·NFR26). 비교 계산은 클라이언트 사이드이므로 REF 로드가 성공했다면 오프라인에서도 이후 흐름이 진행된다(NFR26).

**추적**: UC-13(및 UC-10 분기 3b) / F1 → FR-D11 / NFR26 / 브리프 §6.

## [FR-14] 회사 선택과 슬롯(a/b) 반영·비교 입력 연동

**설명**: 후보 리스트에서 회사 1건을 선택하면 그 회사를 지정 슬롯(a/b)에 반영하고, 회사의 유형·산업·복지 데이터를 비교 입력(F2)에 연동한다. UC-14(비교 대상 회사 선택)의 기본 흐름을 담당한다.

**상세 동작**
- 정상: 사용자가 후보를 선택 → 클라이언트가 해당 `comp_id`의 완전 회사 객체를 확보한다. 참조 소스는 두 가지이며 동일 스키마다 — (1) 부팅 번들 REF의 `companies[]` 인라인 항목(FR-02·FR-D4, 상세 호출 불필요) 또는 (2) `GET /api/v1/companies/{comp_id}`(FR-D7). 상세 호출 방식일 때 대기 중 "회사 정보 불러오는 중…" 상태를 표시한다(레거시 `selComp` 이식).
- 정상(정규화·반영): 확보한 회사 객체를 `normalizeCompany`(FR-D8.2)로 정규화하여 `matched[s]`에 담고, 입력란 표시값을 정식명으로 채운다. 슬롯의 기업유형 선택값을 회사의 `comp_tp_cd`로 설정한다.
- 정상(비교 입력 연동): 회사의 `benefits[]`(실데이터 또는 유형 프리셋 폴백 병합, FR-06·FR-D4)를 각 항목 `checked=true`로 하여 `benS[s]`의 초기값으로 채우고, `work_style_val`(FR-D4.1)에서 `wsState[s]`를 초기화한다. "회사 데이터로 복지 자동 채움" 취지를 표시한다(UC-14 기본 흐름 3; 세부 편집·검증은 F2/FRD 04 소유).
- 대안(실데이터 없음): 선택 회사에 실데이터 복지가 없으면 서버·번들에서 이미 기업유형 프리셋으로 폴백 병합된 `benefits[]`가 반영된다(FR-06·FR-D3, amt_source=`preset`). 클라이언트는 빈 복지에도 크래시하지 않는다(UC-14 분기 3a).
- 대안(상세 로드 실패): 상세 호출 방식에서 미존재 ID·오류 응답이면 "회사 정보를 불러올 수 없습니다"를 표시하고 슬롯을 미선택 상태로 유지한다(FR-D11).
- 반복: 나머지 슬롯에 대해 FR-10~FR-12를 반복해 서로 다른 회사를 지정한다(FR-15와 연계).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 선택 회사 식별자 `comp_id` | int | FR-12 후보 항목의 `comp_id` |
| 대상 슬롯 `s` | enum | `a` 또는 `b` |
| 회사 객체 소스 | REF 인라인(FR-D4) 또는 `companies/{id}`(FR-D7) | 동일 스키마. `work_style_val`·`aliases[]`·`benefits[]` 포함 |

**출력·상태**
- `matched[s]` = 정규화된 회사 객체(`{comp_id, comp_nm, comp_tp_cd, industry_nm, logo_nm, work_style_val, aliases[], benefits[], …}`, FR-D8).
- `benS[s]` = 회사 복지 초기값(각 `checked=true`), `wsState[s]` = `work_style_val` 기반 초기값(F2 소유 필드의 초기화 트리거).
- 슬롯 입력란 표시값 = 정식명. 후보 패널은 닫힘.
- 선택 상태는 클라이언트 메모리에만 존재하고, 영구 저장은 localStorage에만 발생한다(요약 id·명 수준). 서버로의 사용자 데이터 저장·전송은 없다(NFR16·NFR17, FR-01·FR-07, F1 수용기준 4).

**검증·비즈니스 규칙**
- (R1) 선택 회사 객체는 정규화(FR-D8.2)를 거쳐 `matched[s]`에 반영된다(카테고리 폴백 `perks`·배지 폴백 `est`·불리언 강제 포함).
- (R2) 복지 초기값은 실데이터 우선·프리셋 폴백 규약을 따른다(FR-06·FR-D3, UC-93). 세부 체크/금액/카테고리 검증은 F2가 소유한다.
- (R3) 선택 결과의 서버 쓰기 전송 0(NFR16, NFR20). 영구 저장소는 localStorage 한정(NFR17).
- (R4) 렌더되는 회사명·별칭·복지 문자열은 이스케이프한다(NFR21).

**추적**: UC-14 / F1 → F2 / FR-D4·FR-D7·FR-D8·FR-06 / NFR16·NFR17·NFR21 / 브리프 §4·§6·§7.

## [FR-15] 슬롯 재선택·교체·동일 회사 처리·해제

**설명**: 이미 회사가 지정된 슬롯의 재선택(교체), 두 슬롯에 동일 회사를 지정하려는 경우, 슬롯 해제(clear)의 상태 전이 규칙을 규정한다. UC-14의 대안 흐름(2a 교체 / 4a 동일 회사)과 슬롯 초기화를 담당한다.

**상세 동작**
- 정상(교체): 이미 회사가 반영된 슬롯 `s`에서 사용자가 새 회사를 선택하면 기존 선택을 새 선택으로 교체한다. `matched[s]`·`benS[s]`·`wsState[s]`가 새 회사 기준으로 갱신된다(UC-14 분기 2a; 갱신 절차는 FR-14와 동일).
- 대안(동일 회사): `a`와 `b`에 동일 `comp_id`를 지정하려 하면, 비교는 서로 다른 회사를 전제하므로(F1 수용기준 3) 다른 회사 선택을 안내한다. 두 슬롯에 서로 다른 회사를 지정하는 것을 기본 규약으로 유지한다(UC-14 분기 4a).
- 대안(해제): 사용자가 슬롯을 해제하면 `matched[s]=null`, `benS[s]=[]`, `wsState[s]={ot:null, wage:null, remote:null, flex:null}`로 초기화하고 입력란·후보 패널을 비운다(레거시 `clearM` 이식). 해제된 슬롯은 재검색(FR-10) 또는 직접 입력(FR-17)으로 다시 구성 가능하다.

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 대상 슬롯 `s` | enum | `a` 또는 `b` |
| 새 선택 `comp_id` | int \| 없음 | 교체 시 새 회사 id. 해제 시 없음 |
| 반대 슬롯 선택 `comp_id` | int \| null | 동일 회사 판정 기준 |

**출력·상태**
- 교체: 슬롯 `s` 상태가 새 회사로 전면 갱신.
- 동일 회사: 반영 보류 + 다른 회사 선택 안내(상태 미변경).
- 해제: 슬롯 `s`의 `matched`/`benS`/`wsState` 초기화.
- 모든 전이는 클라이언트 상태에만 반영(서버 쓰기 없음, NFR16·NFR17).

**검증·비즈니스 규칙**
- (R1) 교체는 이전 슬롯 상태를 완전히 대체한다(잔여 복지·근무형태 상태 없음).
- (R2) `matched.a?.comp_id === matched.b?.comp_id`(양쪽 지정·동일)를 금지 규약으로 안내한다. 비교는 서로 다른 회사를 전제한다(F1 수용기준 3, UC-14 분기 4a).
- (R3) 해제 후 상태는 신규 슬롯과 동일한 초기값(빈 복지·미선택 근무형태)이다.

**추적**: UC-14(분기 2a·4a) / F1 / FR-D8 / NFR16·NFR17 / 브리프 §7.

## [FR-16] 회사 상세 CTA 프리필 진입 (F4 연계)

**설명**: 회사 상세 페이지(F4)의 "이 회사로 비교하기" CTA로 비교 툴에 진입할 때, 검색 없이 해당 회사를 슬롯에 사전 지정한 상태로 시작한다. UC-14의 대안 흐름 1a(주로 PRD-SEG-3의 전환 경로)를 담당한다.

**상세 동작**
- 정상: F4 CTA가 대상 회사 식별자를 담아 비교 툴 진입점으로 연결한다(예: URL 쿼리 파라미터로 회사 지시). 비교 툴 부팅 후 클라이언트가 이 지시자를 읽어 대상 슬롯의 회사 객체를 확보하고(REF 인라인 또는 `companies/{id}`, FR-D6/FR-D7), FR-14의 기본 흐름 2단계(정규화·반영)에 합류시킨다. 사용자는 검색 단계를 건너뛰고 해당 회사가 슬롯에 지정된 상태로 시작한다(UC-14 분기 1a).
- 대안(식별자 매칭): 진입 지시자가 회사명 기반이면 클라이언트가 검색 API로 1건 해석 후 선택 흐름에 합류할 수 있다(레거시 URL 파라미터 → 검색 → 자동 선택 이식). 해석 실패 시 일반 검색 진입으로 폴백한다.
- 대안(무효 지시자): 존재하지 않거나 해석 불가한 회사 지시자는 슬롯을 미선택으로 두고 정상 검색 진입(FR-10)으로 폴백한다(FR-D11).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 회사 지시자 | `comp_id`(int) 또는 회사명(string) | F4 CTA가 부여. 진입점에서 읽음 |
| 대상 슬롯 | enum(기본 `a`) | CTA가 지정하거나 기본 현직 슬롯 |

**출력·상태**: 지정 슬롯에 회사가 사전 반영된 상태로 비교 툴이 시작(FR-14 사후조건과 동일). 서버 쓰기 없음(NFR16).

**검증·비즈니스 규칙**
- (R1) CTA **생성**(정적 페이지의 링크·파라미터 구성)은 F4(FRD 05, FR-4x)가 소유한다. 본 FR은 진입점에서 지시자를 **소비**해 슬롯을 프리필하는 클라이언트 동작만 규정한다(경계 명시).
- (R2) 프리필된 회사도 FR-14의 정규화·복지 초기화 규약을 동일하게 적용한다.
- (R3) 무효 지시자는 크래시 없이 검색 진입으로 폴백한다(FR-D11·NFR26).

**추적**: UC-14(분기 1a) / F1·F4 → F2 / FR-D6·FR-D7·FR-D11 / 브리프 §3(F4)·§6, PRD-SEG-3.

## [FR-17] 회사 미선택 직접 입력 모드

**설명**: 회사를 선택하지 않고 슬롯을 구성하는 경로를 제공한다. 검색 결과가 없거나(FR-13/UC-13) 사용자가 선택을 건너뛴 경우, 회사 연동 없이 조건만 직접 입력해 비교 입력(F2)으로 진입한다. UC-15(회사 미선택 시 직접 입력 모드)를 담당한다.

**상세 동작**
- 정상: 사용자가 "직접 입력"을 선택하면 슬롯 `s`는 `matched[s]=null`을 유지한 채 비교 입력(F2)으로 진입한다. 회사 미지정이므로 실데이터 자동 초기값 없이, 사용자가 기업유형·연봉·복지 체크/금액·근무형태·통근시간 등 조건을 직접 입력한다(UC-15 기본 흐름 3; 입력 항목 상세는 F2/FRD 04 소유).
- 정상(유형 선택 시 프리셋 채움): 사용자가 회사 대신 **기업유형만** 선택하면, 클라이언트가 해당 `comp_tp_cd`의 프리셋 배열(`benefit_presets`, FR-D3)을 `benS[s]`의 초기값으로 복사한다(각 항목 `checked` = `default_checked_yn`, amt_source=`preset`). 이는 회사 선택과 달리 `matched[s]=null`을 유지한다(레거시 `onTypeChange` 이식, UC-15 · FR-06 폴백).
- 대안(프리셋 없는 유형): 프리셋이 비어있는 유형(예: `freelance`)은 복지 목록이 빌 수 있으며, 사용자가 항목을 직접 추가·입력한다(FR-06 대안, UC-15).
- 대안(슬롯별 혼합): 한 슬롯은 회사 선택(FR-14), 다른 슬롯은 직접 입력을 허용한다. 두 슬롯은 독립 구성된다(UC-15 분기 1a).
- 대안(사후 회사 선택 전환): 직접 입력 중 사용자가 회사를 검색·선택하기로 바꾸면 FR-10~FR-14로 전이한다. 회사 선택 시 복지 초기값이 채워져 직접 입력값을 대체할 수 있음을 안내한다(UC-15 분기 2a).
- 입력된 조건은 F3 비교 리포트 생성의 입력으로 전달된다(UC-15 기본 흐름 4; 산출은 F3 소유).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 진입 트리거 | 사용자 액션 | "직접 입력" 선택 또는 무결과(FR-13)에서 전환 |
| 기업유형(선택적) | enum(6종) \| 없음 | 선택 시 프리셋 자동 채움 소스(FR-D3) |
| 대상 슬롯 `s` | enum | `a` 또는 `b` |

**출력·상태**
- `matched[s] = null` 유지(회사 미지정).
- `benS[s]` = 유형 프리셋 복사값(유형 선택 시) 또는 빈 배열/사용자 직접 추가값. 이후 편집·검증은 F2 소유.
- 서버 쓰기·전송 없음. 저장은 localStorage 한정이며 미지원·차단 시에도 비교는 정상 동작한다(NFR16·NFR17·NFR25, FR-07).

**검증·비즈니스 규칙**
- (R1) 직접 입력 모드는 `matched[s]=null`을 불변으로 유지한다(회사 객체가 없음). 회사 선택(FR-14)과 상태상 구분된다.
- (R2) 유형 프리셋 채움은 FR-06·FR-D3 폴백 규약을 따른다(amt_source=`preset`, 배지 주로 `est`).
- (R3) 슬롯별 독립 구성(한쪽 회사 선택 + 한쪽 직접 입력 혼합)을 허용한다(UC-15 분기 1a).
- (R4) 직접 입력 → 회사 선택 전환 시 초기값 대체 가능성을 사용자에게 안내한다(UC-15 분기 2a).

**추적**: UC-15 / F1 → F2 / FR-D3·FR-D8·FR-06 / NFR16·NFR17·NFR25 / 브리프 §2·§4·§7.
