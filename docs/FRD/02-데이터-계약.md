# 데이터 계약 (FRD)

> **이력**: 브랜드 축(성장성·안정성)은 2026-07-20 제거됐다. 우선순위는 연봉·워라밸·복지 3종.
> 경위·파급범위: `docs/ANALYSIS-성장성-용어-2026-07-20.md` §6-1 · `docs/HANDOFF-2026-07-19.md` §K.


**목적**: loupit 프론트(바닐라 JS SPA·정적 페이지)와 읽기 전용 API(FastAPI) 사이의 **데이터 계약**을 필드 단위로 확정한다. `reference/all` 부팅 번들과 `companies/search`·`companies/{id}` 응답의 JSON 스키마(키·타입·널·제약), 각 복지(benefit) 객체의 필드, 클라이언트 런타임 데이터 모델(`matched.a/b`, `benS.a/b`), 그리고 프로파일러 계열 키·모델의 제거를 명세한다.

**상위 추적**: 본 문서는 FRD→USECASE→PRD 체인에 속한다. 상위 근거 = PRD `08-데이터-요구사항` D1~D6, 브리프 §6(읽기 전용 API 표면)·§4(데이터 모델)·§7(클라이언트 비교)·§2(핵심 결정), RESEARCH `benefit-scraping` §4(신뢰도 디커플링, DEC-2). 각 FR은 관련 UC(UC-10~15 검색·선택 / UC-30~36 비교 리포트 / UC-40~44 회사 상세)·기능(F1·F3·F4·F8)·NFR(NFR3·NFR16·NFR17·NFR20·NFR21)을 인용한다.

**범위 경계**: 본 문서는 **클라이언트↔API로 오가는 데이터의 형태**만 규정한다. API 라우팅·SQL·캐시 구현은 SPEC 단계, 비교 계산식은 FRD 비교 리포트 항목, DB 스키마 원문은 PRD D1이 규정한다. 로그인/회원/프로파일러/서버측 사용자 쓰기(영구 제외, 브리프 §2-1·§2-3·§2-4)는 어떤 계약에도 등장하지 않는다. 사용자 상태의 유일한 영구 저장소는 브라우저 localStorage다(§2-2, NFR17). 모든 응답은 GET·읽기 전용이며 쓰기 계약은 없다(NFR20).

**FR-ID 대역**: 본 문서는 **FR-D** 대역(FR-D1~FR-D11)을 사용한다. 부여한 ID는 재사용하지 않는다(안정 ID). 하위 문서(FLOW/SPEC/TASK)가 이 ID를 인용한다.

## 공통 규약 (전 FR 적용)

- **명명**: JSON 키는 소문자 snake_case이며 DB 컬럼(대문자·`T` 접두)에서 접두·대문자를 벗긴 형태와 1:1 대응한다(예: `TCOMPANY_BENEFIT.BENEFIT_AMT` → `benefit_amt`). 코드 식별자는 영어(NFR29).
- **금액 단위**: 모든 복지·보상 금액은 **연간 만원 단위 정수**다(§4). 금액화 불가 항목은 `null`.
- **일시**: `*_dtm` 필드는 ISO 8601 문자열 또는 `null`. 클라이언트는 표시 시 앞 10자(`YYYY-MM-DD`)만 사용한다.
- **불리언**: DB의 `*_YN`(VARCHAR) 컬럼은 API 직렬화 시 JSON `boolean`으로 노출한다(예: `QUAL_YN` → `qual_yn`).
- **미노출 컬럼**: 감사 4종(`INS_ID`/`INS_DTM`/`MOD_ID`/`MOD_DTM`)과 내부 PK(`ALIAS_ID`, `PRESET_ID`, `BENEFIT_ID`)는 API 응답에 포함하지 않는다.
- **이스케이프**: 문자열 필드(복지명·별칭·`note_ctnt`·`qual_desc_ctnt` 등)는 렌더 시 `textContent`/이스케이프로 삽입한다. `innerHTML` 직접 삽입 금지(NFR21, XSS 차단).
- **표기값의 열거 제약**: `comp_tp_cd`·`benefit_ctgr_cd`·`badge_cd`·`badge_src_cd`·`amt_source`는 아래 각 FR이 정의하는 유한 집합 밖의 값을 갖지 않는다(DB는 ENUM 금지·VARCHAR+COMMENT이나 계약상 값 집합은 고정, §4). 클라이언트 런타임 전용 프로버넌스 필드 `value_source`(∈{`real`,`preset`,`user`}, FR-D8)도 고정 집합을 가지나 API 응답·DB 컬럼에는 포함되지 않는다.
- **두 축의 분리(중요)**: **금액 신뢰도** `amt_source`(∈{`stated`,`estimated`,`none`}, DEC-2·FR-D5·FR-D9)와 **값 출처(프로버넌스)** `value_source`(∈{`real`,`preset`,`user`}, FR-D8)는 **이름·의미·값 집합이 다른 독립 필드**다. 한 필드로 합치지 않는다. 밴드 계산은 `amt_source`를, 값 출처 표기·전이는 `value_source`를 읽는다.

## [FR-D1] `reference/all` 부팅 번들 계약

**설명**: 프론트가 부팅 시 1회 로드하는 단일 참조 소스(§6, NFR3). 최상위는 정확히 3개 키만 가진다. 그 외 키(특히 프로파일러 계열)는 추가하지 않는다(FR-D10).

**상세 동작**
- 정상: 클라이언트가 부팅 시 `GET /api/v1/reference/all`로 전체 번들을 받아 `company_types`·`benefit_presets`·`companies`를 메모리에 적재한다(UC-A1). 이후 회사 선택·유형 프리셋·비교 계산은 이 번들을 참조한다.
- 대안: 번들 로드 실패 시 FR-D11에 따라 오류 상태를 표시하고 크래시하지 않는다(NFR26).
- 대안: `companies[].benefits`는 서버에서 이미 실데이터로 인라인 포함되어 전달되므로(등록 회사는 모두 실복지 보유), 클라이언트는 회사 선택 시 별도 상세 호출 없이도 인라인 복지를 사용할 수 있다.

**입력**: 없음(경로 파라미터·쿼리 없음). GET.

**출력·상태**

| 키 | 타입 | 널 | 내용 |
|----|------|----|------|
| `company_types` | benefit_type 객체 배열 | N(길이 6) | 기업유형 6종(FR-D2) |
| `benefit_presets` | 객체 `{comp_tp_cd: benefit[]}` | N | 유형별 기본복지 목록(FR-D3) |
| `companies` | 회사 객체 배열 | N(길이 ~96) | 복지 데이터가 있는 회사 ~96개 + `aliases[]` + `benefits[]` 인라인(FR-D4) |

- HTTP 200, `Content-Type: application/json`.
- 응답 헤더 `Cache-Control: public, max-age=3600`(§6, NFR3). 1시간 내 재방문 시 참조 데이터 재요청 없음.

**검증·비즈니스 규칙**
- 최상위 키는 위 3개로 한정. `profiles`/`questions`/`job_groups` 등 프로파일러 키 부재를 계약으로 강제한다(FR-D10, D5.4).
- `company_types` 길이 = 6, `companies` 길이 = ~96(복지 데이터가 있는 회사만 등록, D2.6). `companies[]`의 모든 항목은 `benefits` 배열이 비어 있지 않다(등록 회사는 모두 실복지 보유, D2.6).
- 본 번들은 읽기 전용 GET 단일 엔드포인트다(NFR20).

**추적**: F8 / UC-A1(부팅 로드) / NFR3(단일 번들·1시간 캐시) / D6.1~D6.3, D5.4 / §6.

## [FR-D2] `company_types[]` 항목 스키마

**설명**: 기업유형 6종 마스터(`TCOMPANY_TYPE`, D1.1). 유형 라벨의 소스.

**상세 동작**
- 정상: 클라이언트가 회사의 `comp_tp_cd`로 유형 항목을 조회해 라벨(`comp_tp_nm`)을 표시에 사용한다(UC-33).
- 대안: 회사 미선택 직접 입력(UC-15)에서도 사용자가 고른 유형 코드로 동일 항목을 참조한다.

**입력**: 없음(번들 내 배열).

**출력·상태 (항목 필드)**

| 필드 | 타입 | 널 | 제약/값 | DB 컬럼 |
|------|------|----|---------|---------|
| `comp_tp_id` | int | N | PK | `COMP_TP_ID` |
| `comp_tp_cd` | string | N | ∈ {`large`,`startup`,`mid`,`foreign`,`public`,`freelance`} | `COMP_TP_CD` |
| `comp_tp_nm` | string | N | 표시 라벨 | `COMP_TP_NM` |

**검증·비즈니스 규칙**
- 항목은 위 3필드만 갖는다(감사 4종은 미노출, 공통 규약).
- `comp_tp_cd`는 6종 집합 밖의 값을 갖지 않으며 배열 내 유일하다.

**추적**: F8 / UC-33(유형 라벨) / D1.1, D6.1 / §4·§6·§7.

## [FR-D3] `benefit_presets{}` 구조·프리셋 항목

**설명**: 유형별 기본복지(`TBENEFIT_PRESET`, D1.5). 비교 툴 **직접 입력 모드**에서 사용자가 유형만 선택하면 자동 채움되는 소스(UC-15). 회사페이지는 복지 실데이터가 있는 회사만 생성하므로 회사페이지 프리셋 폴백은 불필요하다.

**상세 동작**
- 정상: `benefit_presets`는 `comp_tp_cd`를 키로, benefit 객체 배열을 값으로 갖는 객체다. 예: `{"large":[…], "startup":[…], …}`. 각 항목은 FR-D5 benefit 객체에 프리셋 전용 필드 `default_checked_yn`을 더한 형태다.
- 참고: 등록 회사는 모두 실복지 데이터를 보유하므로(회사페이지는 복지 있는 회사만 생성) 회사페이지용 프리셋 폴백 병합은 하지 않는다. 프리셋은 아래 직접 입력에서만 쓰인다.
- 대안(직접 입력): 사용자가 유형만 선택하면 클라이언트는 해당 키의 프리셋 배열을 `benS[s]`의 초기값으로 복사한다(UC-15, FR-D8).

**입력**: 없음(번들 내 객체).

**출력·상태 (프리셋 항목 필드)**: FR-D5의 benefit 객체 필드 + 아래 프리셋 전용 필드.

| 필드 | 타입 | 널 | 제약/값 | DB 컬럼 |
|------|------|----|---------|---------|
| `default_checked_yn` | bool | N | 프리셋 자동 채움 시 초기 체크 여부. 클라이언트 `checked` 초기값의 소스(FR-D8) | `DEFAULT_CHECKED_YN` |
| `sort_order_no` | int | Y | 표시 정렬 순서 | `SORT_ORDER_NO` |

**검증·비즈니스 규칙**
- `benefit_presets`의 키 집합 ⊆ 6종 `comp_tp_cd`.
- 프리셋 복지의 `badge_cd`는 일반적으로 `est`이며, 직접 입력 모드에서 자동 채움될 때 "회사 공식 확인이 아님"을 배지로 드러낸다(D3.10).
- 프리셋 항목도 FR-D5 필드 규약(금액 만원 int|null, `qual_yn` 시 `benefit_amt=null`)을 따른다.

**추적**: F8 / UC-15(유형 자동 채움)·UC-32(카테고리 델타) / D1.5, D2.5, D3.10, D6.2 / §4·§6.

## [FR-D4] 회사 객체 스키마 + `aliases[]`·`benefits[]` 인라인

**설명**: `reference/all`의 `companies[]` 항목과 `companies/{id}` 응답이 공유하는 **회사 객체**의 필드 계약. 회사 마스터(`TCOMPANY`, D1.2) + 별칭 배열(`TCOMPANY_ALIAS`) + 복지 배열(실데이터)을 인라인으로 포함한다.

**상세 동작**
- 정상: 클라이언트가 회사 선택 시(UC-14) 이 객체를 정규화(`normalizeCompany`, FR-D8)하여 `matched[s]`에 담고, `benefits`를 `benS[s]`의 초기 소스로 사용한다. 회사 상세 페이지(F4)는 동일 스키마로 기업정보·복지·근무형태를 렌더한다(UC-41).
- 참고: 등록 회사는 모두 실복지 데이터를 보유하므로 `benefits`는 실데이터로 채워지며, 회사페이지용 프리셋 폴백 병합은 발생하지 않는다.

**입력**: 없음(번들 내 항목) / `companies/{id}`는 경로 파라미터 `comp_id`(FR-D7).

**출력·상태 (회사 객체 필드)**

| 필드 | 타입 | 널 | 제약/값 | DB 컬럼 |
|------|------|----|---------|---------|
| `comp_id` | int | N | PK, 배열 내 유일 | `COMP_ID` |
| `comp_eng_nm` | string | N | 영문 식별자(unique). URL·로고 매핑용 | `COMP_ENG_NM` |
| `comp_nm` | string | N | 정식명(unique) | `COMP_NM` |
| `comp_tp_cd` | string | N | ∈ 6종. `COMP_TP_ID` 조인으로 파생 | `TCOMPANY_TYPE.COMP_TP_CD` |
| `industry_nm` | string | Y | 산업 | `INDUSTRY_NM` |
| `logo_nm` | string | Y | 로고 약어 | `LOGO_NM` |
| `work_style_val` | object | Y | 근무형태 JSON(FR-D4 하위 키) | `WORK_STYLE_VAL` |
| `careers_benefit_url` | string(URL) | Y | 공식 채용/복지 페이지(출처 아웃링크) | `CAREERS_BENEFIT_URL` |
| `aliases` | string[] | N | 별칭 문자열 배열(회사당 1건 이상, D2.6) | `TCOMPANY_ALIAS.ALIAS_NM` |
| `benefits` | benefit[] | N | 복지 객체 배열(FR-D5). 비어 있지 않음(실데이터) | `TCOMPANY_BENEFIT` |

**FR-D4.1 — `work_style_val`(JSON) 키**(D1.6):

| 키 | 타입 | 의미 |
|----|------|------|
| `remote` | bool/문자열 코드 | 재택 |
| `flex` | bool/문자열 코드 | 유연근무 |
| `unlimitedPTO` | bool | 무제한 휴가 |
| `refreshLeave` | bool | 리프레시 휴가 |
| `overtime` | bool/문자열 코드 | 야근 성향 |

**검증·비즈니스 규칙**
- `reference/all`의 `companies[]` 항목은 위 필드 전부를, `companies/{id}` 응답은 최소 `work_style_val`·`aliases[]`·`benefits[]`를 포함하는 동일 회사 객체다(§6, D6.4). `companies/search`는 이 객체의 축소 투영이다(FR-D6).
- `benefits`는 절대 빈 배열이 아니다(D2.6). `aliases`는 회사당 1건 이상(D2.6).
- `comp_eng_nm`·`comp_nm`은 배열 전역에서 유일(D1.2).

## [FR-D5] benefit 객체 필드 계약 (공통)

**설명**: 번들 `companies[].benefits`, `benefit_presets{}`의 항목, `companies/{id}.benefits`가 공유하는 **복지 항목 객체**의 필드. 데이터 신뢰도 표기(배지·출처·검증/만료)와 금액 신뢰도(DEC-2 디커플링)를 포함한다.

**상세 동작**
- 정상: 각 복지 항목은 코드·표시명·카테고리·금액과 신뢰도 메타를 가진다. 클라이언트는 이를 카테고리별 합산(UC-32), 배지·밴드 표시(UC-34), 상세 페이지 출처 표기(UC-42)에 사용한다.
- 대안(정성 복지): `qual_yn=true`면 `benefit_amt=null`이며, 금액 합산·불확실성 밴드에서 제외되고 `qual_desc_ctnt`로 서술 비교된다(UC-32의 정성 비교).
- 대안(프리셋 항목): 직접 입력 모드의 프리셋 유래 항목은 `default_checked_yn`을 추가로 가진다(FR-D3).

**입력·출력 (필드 정의)**

| 필드(별칭) | 타입 | 널 | 제약/값 | DB 컬럼 |
|------------|------|----|---------|---------|
| `benefit_cd` | string | N | 회사 내 유니크 코드(예: `meal`,`transport`,`welfare`,`bonus`,`health`,`housing`,`child_edu`,`event`,`stock`,`edu`). `unique(comp_id,benefit_cd)` | `BENEFIT_CD` |
| `benefit_nm` (nm) | string | N | 표시명 | `BENEFIT_NM` |
| `benefit_amt` (amt) | int | Y | 연간 금액(만원). 음수 불가. `qual_yn=true`면 `null` | `BENEFIT_AMT` |
| `benefit_ctgr_cd` (ctgr_cd) | string | N | ∈ 9종 {`compensation`,`flexibility`,`work_env`,`time_off`,`health`,`family`,`growth`,`leisure`,`perks`} | `BENEFIT_CTGR_CD` |
| `badge_cd` | string | N | 출처 신뢰도 배지 ∈ {`official`,`est`}. 복지 "존재"의 확실성 | `BADGE_CD` |
| `amt_source` | string | N | 금액 신뢰도 ∈ {`stated`,`estimated`,`none`}. DEC-2 디커플링(FR-D9). 값 출처(프로버넌스)가 아님 — 프로버넌스는 FR-D8 `value_source` | (신규·백필) |
| `qual_yn` | bool | N | 정성복지 여부. true면 `benefit_amt=null` | `QUAL_YN` |
| `qual_desc_ctnt` | string | Y | 정성 서술(짧은 발췌만, 원문 복제 금지) | `QUAL_DESC_CTNT` |
| `note_ctnt` (note) | string | Y | 비고(예: "성과에 따라 편차 큼") | `NOTE_CTNT` |
| `verified_dtm` | string(ISO8601) | Y | 마지막 검증 시점. 신선도 표기 근거 | `VERIFIED_DTM` |
| `expires_dtm` | string(ISO8601) | Y | 만료 시점. 경과 시 "만료" 상태(FR-D9) | `EXPIRES_DTM` |
| `badge_src_cd` | string | Y | 출처 유형 ∈ {`scrape_official`,`scrape_fallback`,`ai_parse`,`manual`,`user_report`} | `BADGE_SRC_CD` |
| `badge_src_url_ctnt` (src_url) | string(URL) | Y | 근거 URL. 출처표기 의무(R3, D3.10) | `BADGE_SRC_URL_CTNT` |
| `sort_order_no` | int | Y | 표시 정렬 순서 | `SORT_ORDER_NO` |

**검증·비즈니스 규칙**
- `benefit_ctgr_cd`는 9종 집합 밖의 값을 갖지 않는다. 알 수 없으면 클라이언트가 `perks`로 폴백한다(FR-D8, `normalizeCompany`).
- `qual_yn=true` ⇒ `benefit_amt=null` 및 `amt_source=none`(정성복지는 금액 없음, §4·RESEARCH §4).
- `badge_cd=official`이라도 `amt_source=estimated`일 수 있다(출처 공식·금액 추정, DEC-2). 두 축은 독립이다(FR-D9).
- `amt_source`는 **금액 신뢰도**({`stated`,`estimated`,`none`})만 나타낸다. 이 복지 값이 실회사 데이터/유형 프리셋/사용자 편집 중 어디서 왔는지(**값 출처=프로버넌스**)는 별도 클라이언트 필드 `value_source`(∈{`real`,`preset`,`user`}, FR-D8)로 다루며 `amt_source`와 혼동하지 않는다. (`badge_src_cd=user_report`는 큐레이션 시점 제보 출처로, 런타임 사용자 편집을 뜻하는 `value_source=user`와 무관하다.)
- `user_report`는 `badge_src_cd` 값으로 존재하나 MVP에 런타임 신고 수집 기능은 없다. 수작업 큐레이션 시점에만 부여(D4.6).
- `badge_src_url_ctnt`는 출처표기 의무의 핵심 필드로, `badge_cd=official` 항목에는 가능한 채워진다(법적 리스크 R3 완화, RESEARCH §5).

**추적**: F8 / UC-32(카테고리)·UC-34(배지·밴드)·UC-42(출처) / D3.1~D3.8, D4.6, D6.3 / §4·§6·§7, RESEARCH §4(DEC-2).

## [FR-D7] `companies/{id}` 응답 스키마

**설명**: 단일 회사 상세(F1 선택 시·F4 상세 페이지 데이터원). 완전 회사 객체(FR-D4)를 반환한다(§6, D6.4). 등록 회사는 모두 실복지를 보유하므로 회사페이지용 프리셋 폴백 병합은 없으며, 없는 `comp_id`는 404다(FR-D11).

**상세 동작**
- 정상: `GET /api/v1/companies/{comp_id}`가 회사 객체를 반환한다. 최소 `work_style_val`·`aliases[]`·`benefits[]`를 포함하며 회사 마스터 필드를 함께 제공한다(FR-D4). 클라이언트는 회사 선택 시 이를 정규화해 `matched[s]`·`benS[s]`를 구성한다(UC-14).
- 대안(번들 사용): `reference/all`에 이미 동일 회사 객체가 인라인되어 있으므로, 클라이언트는 상세 호출 대신 번들 항목을 사용할 수 있다(FR-D1). 두 소스의 회사 객체 스키마는 동일하다.
- 대안(미존재 ID): 존재하지 않는 `comp_id`는 FR-D11에 따라 오류 응답으로 처리한다.

**입력**

| 파라미터 | 위치 | 타입 | 널 | 제약 |
|----------|------|------|----|------|
| `comp_id` | path | int | N | 회사 PK. 정수 아님·범위 밖 시 FR-D11 |

**출력·상태**: FR-D4 회사 객체(`benefits`는 실데이터, `aliases`·`work_style_val` 포함). HTTP 200/JSON 객체.

**검증·비즈니스 규칙**
- 반환 회사 객체의 `benefits`는 비어 있지 않다(D2.6). 등록 회사는 모두 실복지 데이터를 보유하므로 회사페이지용 프리셋 폴백은 불필요하다.
- 검색(FR-D6)과 상세(FR-D7)의 `comp_id`·`comp_nm`·`comp_tp_cd`·`industry_nm`·`logo_nm` 값은 동일 회사에 대해 일치한다.

**추적**: F1·F4·F8 / UC-14(선택)·UC-41(상세 열람)·UC-42(출처)·UC-43(프리필) / D2.5·D2.6·D6.4 / §6.

## [FR-D8] 클라이언트 데이터 모델 `matched.a/b`·`benS.a/b`

**설명**: 비교 툴 SPA가 메모리에 유지하는 런타임 상태의 형태와, API/번들 회사 객체를 이 형태로 정규화(`normalizeCompany`)하는 규칙. 이 상태는 클라이언트에만 존재하며 서버로 전송·저장되지 않는다(NFR16). 영구 저장은 localStorage에만 발생한다(NFR17, UC-36). 런타임 복지 항목(`benS[s][i]`)은 **값 출처(프로버넌스) 축 `value_source`∈{`real`,`preset`,`user`}**를 별도로 유지한다 — 값이 실회사 데이터(`real`)/유형 프리셋(`preset`)/사용자 편집(`user`) 중 어디서 왔는지를 나타낸다. 이 축은 **금액 신뢰도 축 `amt_source`∈{`stated`,`estimated`,`none`}(FR-D5·FR-D9, DEC-2)와 독립된 별개 필드**이며, 둘을 한 필드로 합치지 않는다. `value_source`는 데이터 원천을 요약하는 클라이언트 파생 상태로 DB 컬럼·API 응답 필드가 아니다.

**상세 동작(정상)**
1. 사용자가 회사를 선택하면(UC-14) 클라이언트가 번들 항목 또는 `companies/{id}` 응답(FR-D4/FR-D7)을 `normalizeCompany`로 정규화해 `matched[s]`에 담고, `matched[s].benefits`를 각 항목에 `checked=true`를 붙여 `benS[s]`로 복사한다.
2. 회사 미선택(직접 입력, UC-15)에서 사용자가 유형만 고르면 `benefit_presets[comp_tp_cd]`(FR-D3)를 정규화해 `benS[s]`로 채우고 `matched[s]=null`을 유지한다.
3. 사용자가 복지 체크·금액을 편집하면 `benS[s][i].checked`·`benS[s][i].benefit_amt`가 갱신된다. **금액을 사용자가 편집한 항목은 `benS[s][i].value_source`가 `'user'`로 전이한다**(값 출처=사용자 입력으로 갱신; 금액 신뢰도 `amt_source`(FR-D9)는 이 전이의 대상이 아니다). 비교 계산(FR 비교 리포트)이 `benS.a`·`benS.b`를 입력으로 사용한다(UC-30~35).

**대안·예외**
- 회사 해제(clear) 시 `matched[s]=null`, `benS[s]=[]`, `wsState[s]` 초기화.
- localStorage 저장(UC-36) 시 `matched`(요약)·`benS`·`wsState`·`curPri`·`curSacrifice` 등을 직렬화한다. 프로파일러 상태는 저장 대상에 없다(FR-D10).

**입력·출력 (런타임 상태 구조)**

| 상태 | 타입 | 초기값 | 내용 |
|------|------|--------|------|
| `matched.a` / `matched.b` | 회사 객체(FR-D4) \| null | `null` | 선택된 회사(a=현직/b=이직처). 직접 입력 시 null |
| `benS.a` / `benS.b` | benefit 객체 배열(FR-D5 + 런타임 필드) | `[]` | 비교에 쓰이는 복지 목록(편집 가능) |
| `wsState.a` / `wsState.b` | 객체 `{ot,wage,remote,flex}` | 각 키 `null` | 근무형태 상태(야근/임금유형/재택/유연). `work_style_val`·프리셋에서 초기화 |
| `curPri` | 문자열 | (사용자 선택) | 우선순위(연봉/워라밸/복지 3종, §2-3) |
| `curSacrifice` | 문자열/배열 | (사용자 선택) | 희생요소(§7) |

**FR-D8.1 — `benS[s]` 항목의 런타임 전용 필드**(서버 미전송):

| 필드 | 타입 | 규칙 |
|------|------|------|
| `checked` | bool | 이 항목을 비교에 포함할지. 초기값 = `default_checked_yn`(프리셋) 또는 회사 복지의 경우 `true` |
| `benefit_amt` | int | 사용자 편집으로 변경 가능(초기값은 FR-D5 값을 `Number()` 강제) |
| `value_source` | string ∈ {`real`,`preset`,`user`} | **값 출처(프로버넌스)**: 실회사 복지 데이터=`real`, 직접 입력 유형 프리셋=`preset`, 사용자가 금액을 편집=`user`. 초기화 규칙 FR-D8.2, 편집 전이 상세동작 3. **`amt_source`(금액 신뢰도, FR-D5·FR-D9)와 별개 축**이며, `badge_src_cd=user_report`(큐레이션 시점 제보)와도 무관하다 |

**FR-D8.2 — `normalizeCompany` 정규화 규칙**(API/번들/프리셋을 동일 shape로 통일):

| 대상 필드 | 규칙 |
|-----------|------|
| `benefit_amt` | `Number(v) || 0`(비수치·null → 0으로 계산상 취급) |
| `benefit_ctgr_cd` | 없으면 `'perks'` 폴백 |
| `badge_cd` | 없으면 `'est'` 폴백 |
| `qual_yn` | `!!v`(불리언 강제) |
| `checked` | `v.checked ?? (default_checked_yn ?? true)` |
| `value_source` | 원천이 실회사 복지면 `'real'`, 직접 입력 유형 프리셋(UC-15)이면 `'preset'`으로 초기화. 이후 사용자가 금액을 편집하면 `'user'`로 전이(상세동작 3). 원천(real/preset) 판별의 정확한 근거 필드는 SPEC에서 확정 |

**검증·비즈니스 규칙**
- `matched.a/b`·`benS.a/b`·`wsState`·`curPri`·`curSacrifice`는 클라이언트 메모리와 localStorage에만 존재한다. 어떤 비교 입력·결과도 서버로 전송·저장되지 않는다(NFR16, §2-8).
- 정규화 후 `benS[s]`의 각 항목은 FR-D5 필드 규약을 만족한다(카테고리 폴백·불리언 강제 포함).
- `value_source`(값 출처/프로버넌스, {`real`,`preset`,`user`})와 `amt_source`(금액 신뢰도, {`stated`,`estimated`,`none`}, FR-D5·FR-D9)는 **의미·값 집합이 다른 독립 축**이다. 두 개념을 한 필드로 합치지 않는다. 밴드 계산(FR-D9·밴드 엔진)은 `benS[s][i].amt_source`를 읽고, 값 출처 파이프라인·표기·전이는 `benS[s][i].value_source`를 읽는다.
- `benS[s][i]`는 두 필드를 모두 보유한다: FR-D5에서 상속한 `amt_source`(금액 신뢰도)와 런타임 필드 `value_source`(프로버넌스). `value_source`는 localStorage 직렬화(UC-36) 시 `benS` 항목과 함께 보존된다.
- 회사 객체·복지 문자열은 렌더 시 이스케이프한다(NFR21).

**추적**: F1·F3 / UC-14·UC-15·UC-30~36 / NFR16·NFR17·NFR21·NFR25 / §2-2·§2-8·§7, D2.5.

## [FR-D9] 배지·금액 신뢰도 디커플링·파생 상태

**설명**: 데이터 계약은 **두 개의 독립 신뢰도 축**을 노출한다 — 출처 신뢰도(`badge_cd`)와 금액 신뢰도(`amt_source`). 여기에 만료(`expires_dtm`)를 결합해 클라이언트가 표시 배지·불확실성 밴드를 파생한다(DEC-2, RESEARCH §4; D3.9의 정교화).

**상세 동작(정상)**
1. UI 배지: 클라이언트가 `badge_cd`와 `expires_dtm`으로 3-상태를 파생한다. `official`+미만료 → "공식", `official`+만료 → "만료", 그 외 → "추정"(UC-34·UC-42, 기존 `badgeState`/`badgeLabel` 이식).
2. 불확실성 밴드: 금액 신뢰도(`amt_source`) 기준으로 밴드를 적용하되, 만료 시 확대한다. 출처 배지(`official`)가 곧 금액 정확을 뜻하지 않음을 계약으로 반영한다(DEC-2).
3. 정성 복지(`qual_yn=true`)는 금액 밴드를 적용하지 않고 배지·출처만 표시한다.

**입력**: benefit 객체의 `badge_cd`·`amt_source`·`expires_dtm`·`verified_dtm`·`qual_yn`(FR-D5).

**출력·상태**

**FR-D9.1 — 표시 배지 파생(출처×만료)**:

| `badge_cd` | `expires_dtm` | 파생 상태 | 표시 |
|------------|---------------|-----------|------|
| `official` | 없음/미경과 | `off` | 공식 |
| `official` | 경과(만료) | `stl` | 만료(재확인 필요) |
| `est`(또는 그 외) | 무관 | `est` | 추정 |

**FR-D9.2 — 불확실성 밴드(금액 신뢰도 기준, DEC-2/RESEARCH §4.2)**:

| 조건 | 밴드(±) |
|------|---------|
| `amt_source=stated`(공식 명시 금액) | ±5% |
| `amt_source=estimated`(앵커 추정 금액) | ±20% |
| `expires_dtm` 경과(만료) | ±15% 가산 확대 + "재확인 필요" 표기 |

**검증·비즈니스 규칙**
- 두 축은 독립이다: `badge_cd=official` ∧ `amt_source=estimated` 조합이 유효하며, 이 경우 배지는 "공식"이되 금액 밴드는 ±20%를 유지한다(DEC-2 핵심). 96개 큐레이션 복지가 이 상태로 표기된다(RESEARCH §4).
- **PRD D3.9(배지 기준 밴드: official ±5% / est ±20% / 만료 ±15%)와 RESEARCH §4.2(금액 신뢰도 기준 밴드)의 상충은 미해결 오픈이슈다**(RESEARCH OI-6, `amt_source` 백필 정확도 및 밴드 기준 확정). 본 계약은 두 필드(`badge_cd`, `amt_source`)를 모두 노출하여 어느 기준을 채택하든 데이터가 부족하지 않게 한다. SPEC/TASK 단계에서 밴드 기준을 하나로 확정한다.
- 정성 항목은 금액 밴드 대상이 아니다(FR-D5, `qual_yn=true` ⇒ `amt_source=none`).
- 본 절의 `amt_source`는 **금액 신뢰도**(DEC-2, {`stated`,`estimated`,`none`})만을 뜻한다. 값이 어디서 왔는지(**값 출처=프로버넌스**, {`real`,`preset`,`user`})는 별도 필드 `value_source`(FR-D8)이며, 사용자 금액 편집 시 `value_source=user`로 전이할 뿐 `amt_source`를 바꾸지 않는다. 밴드 산정은 오직 `amt_source`(+만료)만 입력으로 한다. (FR-D9의 "출처 신뢰도"는 복지 존재의 확실성인 `badge_cd`를 가리키며, 값 프로버넌스 `value_source`와 다르다 — 세 축 모두 독립.)

**추적**: F3·F4 / UC-34(밴드)·UC-42(배지·출처) / D3.1·D3.5·D3.9·D3.10 / §4·§7, RESEARCH §4(DEC-2)·OI-6.

## [FR-D10] 프로파일러 키·모델 제거 (명시)

**설명**: 프로파일러(가치관 진단 설문 → 페르소나)는 영구 제외(§2-3, D5)이므로, 데이터 계약의 번들·클라이언트·API 어디에도 프로파일러 데이터가 존재하지 않는다.

**상세 동작**
- 번들: `reference/all` 최상위에서 `profiles`/`questions`/`job_groups` 키를 **제거**한다. 최상위 키는 `company_types`·`benefit_presets`·`companies` 3개로 한정(FR-D1, D5.4).
- 클라이언트: 프로파일러 런타임 상태(`pfResult`, `pfJob`)·데이터 상수(`JOB_GROUPS`, `Q_BASE`, `Q_DESC`)·함수(`pfInit`, `_normalizeQ`, `buildQuestions`, `renderPfIntro` 등)를 제거한다. localStorage 직렬화(`saveDraft`)에서 `pfResult`/`pfJob`을 저장·복원하지 않는다.
- API: `/profiler` 계열 엔드포인트가 존재하지 않는다(API 표면 = health/reference.all/companies.search/companies.{id}만, NFR20).

**입력·출력**: 해당 없음(부재 계약).

**검증·비즈니스 규칙**
- `reference/all` 응답에 `profiles`/`questions`/`job_groups` 키가 나타나면 계약 위반이다(FR-D1 규칙과 동치).
- 비교 판정(vdCard)은 사용자가 직접 선택한 `curPri`/`curSacrifice`만 입력으로 쓰며(FR-D8, UC-33), 페르소나·질문 응답을 입력으로 갖지 않는다.
- 관련 테이블(`TPROFILE`·`TPROFILE_JOB_FIT`·`TJOB_GROUP`·`TJOB`·`TPROFILER_QUESTION`·`TQUESTION_SCENARIO`·`TPROFILER_RESULT`)은 스키마·시드·API에서 참조 불가(D1.8·D5.3).

**추적**: (제외 범위) / UC-33(직접 선택 판정) / NFR20 / D5.1~D5.4·D1.8 / §2-3·§6.

## [FR-D11] 오류·빈 응답·계약 위반 처리

**설명**: 검색·상세·번들 응답의 비정상 상황에서 클라이언트가 크래시하지 않고 사용자에게 상태를 알리도록 계약의 오류·빈 응답 형태를 규정한다(NFR26).

**상세 동작·규칙**

| 상황 | 계약/응답 | 클라이언트 처리 |
|------|-----------|-----------------|
| 검색 결과 0건 | HTTP 200, 빈 배열 `[]` | "결과 없음" 표시 + 직접 입력 안내(UC-13→UC-15). 오류로 취급하지 않음 |
| 검색어 공백/빈 `q` | 요청 억제 또는 빈 배열 | 후보 미표시. 네트워크 낭비 방지 |
| `companies/{id}` 미존재 ID | 오류 응답(예: HTTP 404 또는 `{error:…}`) | "회사 정보를 불러올 수 없습니다" 표시, 선택 취소 상태 유지 |
| `reference/all` 로드 실패 | 네트워크/HTTP 오류 | 오류 상태 표시·재시도 안내, 앱 미크래시(NFR26). 로드 성공 후엔 비교 계산이 오프라인에서도 동작(§7) |
| 응답에 프로파일러 키 존재 | 계약 위반 | 무시(FR-D10). 클라이언트는 해당 키를 사용하지 않음 |
| `benefits` 빈 배열 반환 | 계약 위반(D2.6 위배) | 등록 회사는 모두 실복지 보유 → 빈 배열은 데이터 오류. 클라이언트는 빈 복지로도 크래시하지 않음 |

**검증·비즈니스 규칙**
- 클라이언트는 API 응답을 신뢰하되 방어적으로 처리한다: 배열/객체 부재·널 필드에 대해 폴백(빈 배열·`normalizeCompany` 폴백)을 적용한다(FR-D8).
- 어떤 오류 경로도 사용자 데이터를 서버로 전송하지 않는다(NFR16).

**추적**: F8 / UC-13(결과 없음)·UC-15(직접 입력) / NFR26·NFR16 / D2.6·§6·§7.
