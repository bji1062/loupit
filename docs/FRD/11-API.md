# 읽기 전용 API (FRD)

**목적**: PRD 기능 **F8(읽기 전용 API)**의 GET 5종 엔드포인트(+익명 비교 로그 POST 1종)를 구현 가능한 수준의 **HTTP·전송 계약**으로 확정한다. 각 엔드포인트의 메서드·경로·요청 파라미터·응답 상태코드·오류 응답·캐시 헤더·CORS 정책을 명세하며, **익명 쓰기는 비교 로그 POST 1종뿐이고 인증·서버 측 사용자 저장이 없음**을 API 표면 수준에서 불변식으로 고정한다(SC14 기여 쓰기는 세션·재직 검증을 통과한 요청에 한정, FRD 13).

**참조 상위문서**: 브리프 `CONTEXT_BRIEF.md` §6(읽기 전용 API 표면)·§2(핵심 결정)·§8(인프라·프록시), PRD `08-데이터-요구사항.md`(D1~D6, 특히 D4.4 캐시·D6.4 검색/상세 데이터), USECASE `08-오류와엣지.md`(UC-90·UC-91·UC-93·UC-95)·`01-액터와-개요.md`(UC-A1·UC-A5·§4.1 F8 커버리지).

**상위 추적**: FRD → USECASE → PRD → 브리프. 본 문서(FR-9x)는 **엔드포인트의 전송 계약**을 소유하고, **응답 본문(payload) 스키마**는 데이터 계약 문서(`02-데이터-계약.md`, FR-D1~FR-D11)를 인용한다(중복 정의 금지). 전역 규약은 `01-개요와-FR규칙.md`(FR-01·FR-02·FR-06)를 상속한다.

**FR-ID 대역**: 본 문서는 **FR-9x**(FR-90~FR-96)를 소유한다. 각 FR은 자신을 소비·검증하는 UC(UC-A1·UC-A5·UC-90·UC-91·UC-93·UC-95·UC-10~12 흡수)와 상위 기능 F8, 관련 NFR을 역참조한다.

**전역 전제(모든 FR 공통, `01` §전역 전제 상속)**: 본 문서의 F8 익명 읽기 API에는 로그인/회원/계정/OAuth/JWT/이메일 인증이 없다(§2-1, NFR16) — 비밀번호·OAuth·소셜 로그인은 영구 제외(SC10)이고, 무비밀번호 이메일 코드 로그인·재직 인증은 복지 등록·수정 기여(SC14)에 한정된 **별도 참여 API(FRD 13, FR-10x·11x)**가 소유한다(익명 GET 경로는 무세션·무쿠키 유지, INV-1·INV-9). 프로파일러 영구 제외(§2-3). **익명 경로 서버 측 사용자 데이터 쓰기 없음**(§2-4, NFR20). 사용자 상태는 브라우저 localStorage에만 존재(§2-2, NFR17). 비교 계산은 100% 클라이언트(§2-8). 따라서 본 API의 어떤 엔드포인트도 요청 본문을 사용자 데이터로 저장하지 않으며, 인증·세션·쿠키를 요구하지 않는다.

## [FR-90] 읽기 전용 API 표면·불변식

**설명**: loupit 백엔드가 노출하는 익명 HTTP 표면을 **GET 5종 + 익명 비교 로그 POST 1종**으로 한정하고, 인증·세션·서버 측 사용자 저장이 표면에 존재하지 않음을 불변식으로 고정한다(SC14 기여 쓰기는 세션·재직 검증을 통과한 요청에 한정된 별도 참여 API FRD 13가 열거). `01` FR-01(전역 아키텍처 불변식)의 API 표면 구체화다.

**상세 동작**
- 정상: 라우터에 등록되는 경로는 아래 표(GET 5종 + 익명 비교 로그 POST 1종)뿐이다.

  | # | 메서드 | 경로 | 소유 FR |
  |---|--------|------|---------|
  | 1 | GET | `/api/v1/health` | FR-91 |
  | 2 | GET | `/api/v1/reference/all` | FR-92 |
  | 3 | GET | `/api/v1/companies/search` | FR-93 |
  | 4 | GET | `/api/v1/companies/{comp_id}` | FR-94 |
  | 5 | GET | `/api/v1/comparisons/trending` | 개정 2026-07-14 |
  | 6 | POST | `/api/v1/comparisons/log` | 개정 2026-07-14 |

- **개정(2026-07-14, "실시간 비교 TOP 10")**: 익명 비교 로그 쓰기 1종(`POST /comparisons/log`)과 그 집계 조회 1종(`GET /comparisons/trending`)을 표면에 추가한다. 로그는 **회사쌍 comp_id + 시각뿐** — 사용자 식별자·IP·세션·연봉 등 입력값은 받지도 저장하지도 않는다(`TCOMPARE_LOG` 컬럼 계약이 테스트로 고정). 이 1종 외 쓰기 라우트 0은 계속 유지된다(Tier-0 TS-1).
- 대안(비-GET 메서드): 위 POST 1종 외 경로에 POST/PUT/PATCH/DELETE 요청 시 **405 Method Not Allowed**를 `Allow: GET`(및 HEAD, OPTIONS) 헤더와 함께 반환한다.
- 대안(HEAD): GET 라우트는 HEAD를 함께 처리한다(본문 없이 헤더만 반환, 상태코드·캐시 헤더 동일).
- 금지: 인증 미들웨어·세션 미들웨어·쿠키 발급·`/profiler` 계열·관리자 쓰기·비교 저장(사용자별)/신고/피드 엔드포인트를 라우터에 등록하지 않는다(§2-1·2-3·2-4, FR-D10). 통계 집계는 위 익명 쌍 카운트 1종으로 한정한다.

**입력**: 없음(표면 불변식 자체). 개별 엔드포인트 입력은 FR-91~FR-94.

**출력·상태**
- 서버에 사용자별 상태 0건(무쓰기). 응답은 참조 데이터의 읽기 결과뿐.
- 미등록 경로(`/api/v1/*` 하위 또는 그 밖)는 **404 Not Found**(FR-95).

**검증·비즈니스 규칙**
1. 라우터에 등록된 메서드×경로 조합은 위 표 6행(GET 5종 + 익명 비교 로그 POST 1종, + 자동 HEAD/OPTIONS)뿐이다. 익명 쓰기 메서드 라우트 수 = 1(비교 로그), 그 외 0(NFR20).
2. 어떤 엔드포인트도 요청 본문을 파싱해 저장하지 않는다(GET은 의미상 본문 없음). 서버 측 사용자 데이터 쓰기 0(§2-4).
3. 인증·인가 미들웨어 0. `Authorization`·`Cookie`가 있어도 무시하며 응답이 달라지지 않는다(무상태·비개인화). 광고/제휴 쿠키는 프론트·제3자 스크립트 영역이며 본 API가 발급하지 않는다(NFR17).
4. 결과 크기·비용은 유한 상한을 갖는다(검색 ≤20건 FR-93, 질의 길이 상한 FR-93). 페이지네이션·정렬 파라미터는 제공하지 않는다(MVP 범위).

**추적**: F8 / UC-A5·UC-A1(전제)·전 UC 전제 / NFR16·NFR17·NFR20 / §2-1·2-3·2-4·2-8·§6 / `01` FR-01.

## [FR-91] `GET /api/v1/health` — 헬스체크

**설명**: 운영자·모니터링이 API 프로세스의 **생존·가용성**을 점검하는 경량 엔드포인트. 사용자 데이터를 조회·쓰기하지 않는다(UC-A5).

**상세 동작**
- 정상(라이브니스): 프로세스가 요청을 서빙하면 **HTTP 200**과 경량 JSON을 반환한다. 사용자·회사 데이터 조회 없이 프로세스 생존만 표현한다.
- 대안(레디니스 점검, 선택): 배포 검증용으로 DB 풀 연결을 경량 핑(`SELECT 1` 수준)하는 확장을 허용한다. DB 미가용 시 **503 Service Unavailable**과 `{"status":"degraded"}`를 반환하여 `/api/v1/companies/*`·`/reference/all`이 5xx가 되기 전에 조기 감지되게 한다.
- 예외(미기동·타임아웃): 프로세스 자체가 죽었으면 응답이 없다(연결 거부). 운영자가 systemd/Uvicorn·Nginx 프록시(`/api/v1/*`)를 점검·재기동한다(UC-A5 2a). 사용자 대면 참조 로드 실패는 UC-90(FR-92)에서 별도로 다룬다.

**입력**

| 파라미터 | 위치 | 타입 | 널 | 제약 |
|----------|------|------|----|------|
| (없음) | — | — | — | 쿼리·경로·본문 파라미터 없음 |

**출력·상태**

| 상태 | 조건 | 본문(예) |
|------|------|----------|
| 200 OK | 프로세스 생존(기본 라이브니스) | `{"status":"ok"}` |
| 503 Service Unavailable | (레디니스 확장 시) DB 풀 미가용 | `{"status":"degraded"}` |

- 헤더: `Content-Type: application/json; charset=utf-8`, `Cache-Control: no-store`(항상 실시간 상태 반영, 캐시 금지).

**검증·비즈니스 규칙**
1. 응답은 사용자·회사 식별 정보를 포함하지 않는다(경량, 개인정보 무관, §2-4).
2. 인증 없이 호출 가능하다(모니터링 주체가 사람·자동 감시 모두 허용, UC-A5 액터).
3. `no-store`로 캐시되지 않아 매 호출이 현재 상태를 반영한다.

**추적**: F8(`/api/v1/health`) / UC-A5(헬스체크·가용성 점검) / NFR20(무쓰기·무인증) / §6·§8·§9.

## [FR-92] `GET /api/v1/reference/all` — 참조 번들

**설명**: 비교 툴 SPA가 **부팅 시 1회** 로드하는 전체 참조 데이터 번들 엔드포인트. 검색·비교 입력·계산의 **단일 참조원(REF)**이다(UC-A1, `01` FR-02). 응답 본문 구조는 데이터 계약 **FR-D1**(부팅 번들)이 소유한다.

**상세 동작**
- 정상: SPA 진입 시 프론트가 본 엔드포인트를 1회 호출한다(UC-A1 2단계). 서버는 참조 번들을 **HTTP 200 + `Cache-Control: public, max-age=3600`**으로 반환한다(§6, D4.4). 프론트는 번들을 메모리에 보관하고 이후 회사별 개별 API를 재호출하지 않는다(`01` FR-02, USECASE 01 §4.1).
- 대안(캐시 적중): 브라우저/프록시 캐시가 `max-age`(1시간) 이내면 네트워크 호출 없이 캐시에서 로드한다(UC-A1 3a, UC-90 2a). 실패 화면 없음.
- 예외(로드 실패·5xx·타임아웃): 프론트가 예외를 흡수하여 오류 상태로 전환하고 재시도·안내를 표시한다. 참조 확보 전까지 검색·비교를 제한한다(UC-90, FR-95). 앱은 크래시하지 않는다(NFR26).
- 예외(부분·손상 응답): 최상위 키가 정확히 3종(`company_types`/`benefit_presets`/`companies`)이 아니면 클라이언트가 유효 번들로 취급하지 않고 오류 상태로 처리한다(UC-90 3a, FR-D1·FR-D11).

**입력**

| 파라미터 | 위치 | 타입 | 널 | 제약 |
|----------|------|------|----|------|
| (없음) | — | — | — | 파라미터 없음. 전량 반환 |

**출력·상태**
- HTTP 200, JSON 객체. 본문 스키마 = **FR-D1**: 최상위 키 `company_types[]`(FR-D2), `benefit_presets{type_cd:[…]}`(FR-D3), `companies[{…, aliases[], benefits[]}]`(FR-D4). 회사 복지는 회사별 실데이터가 인라인 포함된다(등록 회사는 모두 실복지 보유 → 회사페이지 프리셋 폴백 불필요, §6, D6.3).
- 프로파일러 키(`profiles`/`questions`/`job_groups`) **부재**(FR-D10, D5.4).
- 헤더: `Cache-Control: public, max-age=3600`(§6·D4.4, NFR3), `Content-Type: application/json; charset=utf-8`.

**검증·비즈니스 규칙**
1. 응답 헤더 `Cache-Control: public, max-age=3600` 필수(브리프 §6 명시 값, NFR3). 데이터 재시드 후 최대 캐시 TTL(1시간) 내 반영(D4.4).
2. 최상위 키는 정확히 3종이며 프로파일러 키가 나타나면 계약 위반(FR-D1·FR-D10).
3. 각 회사의 `benefits`는 비어 있지 않다(등록 회사는 모두 실복지 보유, D2.6).
4. 개인화·인증에 따라 응답이 달라지지 않는다(모든 방문자 동일, 캐시 가능).

**추적**: F8(`reference/all`) → F1·F2·F3 / UC-A1(부팅 로드)·UC-90(로드 실패) / NFR3(캐시)·NFR26(무크래시) / D4.4·D5.4·D6.3 / §6 / `01` FR-02·FR-06 / 본문 스키마 FR-D1~D5.

## [FR-93] `GET /api/v1/companies/search?q=` — 회사 검색

**설명**: 이름·별칭 부분일치(LIKE) 검색 엔드포인트. **최대 20건**의 축소 회사 객체 배열을 반환한다(F1, §6). 응답 항목 스키마는 데이터 계약 **FR-D6**이 소유한다.

**상세 동작**
- 정상: 클라이언트가 사용자 입력을 디바운스하여 `GET /api/v1/companies/search?q=<검색어>`를 호출하고, 반환 항목으로 자동완성·후보 리스트를 표시한다(UC-10~12). 사용자가 항목을 고르면 `comp_id`로 슬롯을 지정한다(UC-14).
- 대안(결과 없음): 매칭이 없으면 **HTTP 200 + 빈 배열 `[]`**을 반환한다. 오류가 아니라 "결과 없음"으로 구분 표시한다(UC-91 3b, FR-D11).
- 대안(번들 폴백): 검색 엔드포인트가 실패해도 이미 로드된 `reference/all` 번들이 `companies[{…, aliases[]}]`를 인라인 보유하므로, 클라이언트는 번들에 대한 이름/별칭 매칭으로 후보를 제시할 수 있다(UC-91 3a, `01` FR-02). 즉 본 엔드포인트는 편의 채널이며 단일 실패점이 아니다.
- 예외(요청 실패·5xx): 프론트가 예외를 흡수하고 크래시하지 않으며 재시도·안내를 표시한다(UC-91, FR-95).

**입력**

| 파라미터 | 위치 | 타입 | 널 | 제약 |
|----------|------|------|----|------|
| `q` | query | string | N(필수) | 검색어. 이름/별칭 LIKE 부분일치(§6). 검증 규칙은 아래 "검증·비즈니스 규칙" 참조 |

**출력·상태**
- HTTP 200, JSON 배열(길이 0~20). 각 항목 = **FR-D6** 축소 투영: `{comp_id:int, comp_nm:string, comp_tp_cd:string, industry_nm:string|null, logo_nm:string|null}`(§6, D6.4).
- `benefits`·`aliases`·`work_style_val`은 검색 응답에 **포함하지 않는다**(상세는 FR-94/FR-D7).
- 헤더: `Content-Type: application/json; charset=utf-8`, `Cache-Control: no-store`(질의 가변·디바운스, 캐시 낭비 방지; FR-96).

**검증·비즈니스 규칙 (q 검증)**

| 입력 상태 | 처리 | 상태코드 |
|-----------|------|----------|
| `q` 미제공(파라미터 없음) | 필수 쿼리 파라미터 누락 → 검증 오류 | 422 (FR-95) |
| `q` 있으나 trim 후 빈 문자열/공백만 | 무질의로 간주, 빈 결과 반환(FR-D11 "빈 배열", 네트워크 낭비 방지) | 200 `[]` |
| `q` trim 후 길이 1~50 | 정상 검색 수행 | 200(≤20건) |
| `q` trim 후 길이 > 50 | 비정상 과길이 질의 거부(LIKE 비용 상한, 방어적 상한) | 422 (FR-95) |

1. 반환 항목 상한 = **20건**(§6). 초과 반환 금지. 페이지네이션·정렬 파라미터 없음(MVP).
2. 검색어는 이름(`comp_nm`)·별칭(`TCOMPANY_ALIAS.ALIAS_NM`) 대상 LIKE 부분일치(§6). SQL은 파라미터 바인딩으로 처리하며 `q`를 SQL에 직접 문자열 결합하지 않는다(주입 방지, NFR 보안 정합).
3. 매칭 0건은 오류가 아니라 빈 배열(FR-D11). 클라이언트는 직접 입력 모드로 안내(UC-13·UC-15).
4. 최대 길이 50은 브리프 미명시의 **방어적 상한**(회사명·별칭은 짧음)이며 SPEC에서 조정 가능. 빈 `q`의 200 `[]` 처리는 FR-D11의 무질의 무해 처리에 정렬한다.

**추적**: F8(`companies/search`)·F1 / UC-10~12(정상 검색 흡수)·UC-91(검색 실패)·UC-13/15(무결과·직접입력) / NFR26 / D6.4 / §6 / 본문 스키마 FR-D6·오류 FR-D11.

## [FR-94] `GET /api/v1/companies/{comp_id}` — 회사 상세

**설명**: 단일 회사 상세 엔드포인트. **실복지가 포함된 완전 회사 객체**를 반환한다(등록 회사는 모두 실복지 보유, §6, D6.4). 응답 본문 스키마는 데이터 계약 **FR-D7**(및 회사 객체 FR-D4)이 소유한다. 미존재 식별자는 **404**로 응답한다(UC-95).

**상세 동작**
- 정상: `GET /api/v1/companies/{comp_id}`가 회사 객체를 **HTTP 200**으로 반환한다. 최소 `work_style_val`·`aliases[]`·`benefits[]`를 포함하며 회사 마스터 필드를 함께 제공한다(FR-D4/FR-D7).
- 정상(실복지 반환): 등록 회사는 모두 실데이터 복지(`TCOMPANY_BENEFIT`)를 보유하므로 회사페이지용 프리셋(`TBENEFIT_PRESET`) 폴백 병합은 불필요하다. `benefits`는 회사별 실복지로 채워지며 배지는 각 항목의 실제 출처(official/est)를 따른다(UC-93). 프리셋은 회사페이지가 아니라 비교 툴 직접 입력 모드(사용자 회사가 등록 회사에 없을 때)에서만 사용된다(UC-15/21).
- 대안(번들 우선): SPA 런타임은 `reference/all` 번들이 동일 회사 객체를 이미 전량 인라인 보유하므로 본 엔드포인트를 개별 호출하지 않을 수 있다(USECASE 01 §4.1, FR-D7). 두 소스의 회사 객체 스키마는 동일하다. 본 엔드포인트는 상세 데이터원·계약 완결성을 위해 제공된다.
- 예외(미존재 id): 존재하지 않는 `comp_id`(등록된 복지 보유 회사 약 96개 밖)는 **404 Not Found**로 응답한다(UC-95 1a). API 경로에서는 액터 A1이며, 프론트는 오류를 흡수하여 크래시하지 않고 "회사 정보를 불러올 수 없습니다"를 표시한다(NFR26, FR-D11).

**입력**

| 파라미터 | 위치 | 타입 | 널 | 제약 |
|----------|------|------|----|------|
| `comp_id` | path | int | N | 회사 PK(양의 정수). 정수 아님·형식 위반 → 422. 형식은 맞으나 미존재 → 404 |

**출력·상태**

| 상태 | 조건 | 본문 |
|------|------|------|
| 200 OK | 존재하는 `comp_id` | FR-D4 회사 객체(`benefits`=회사별 실복지, `aliases[]`·`work_style_val` 포함) |
| 404 Not Found | 존재하지 않는 `comp_id` | 오류 envelope(FR-95) |
| 422 Unprocessable Entity | `comp_id`가 정수 형식 아님 | 검증 오류 envelope(FR-95) |

- 헤더: `Content-Type: application/json; charset=utf-8`, `Cache-Control: public, max-age=3600`(정적-까지-재시드 데이터, D4.4 1시간 반영에 정렬한 파생 기본값; FR-96).

**검증·비즈니스 규칙**
1. 반환 회사 객체의 `benefits`는 **비어 있지 않다**(등록 회사는 모두 실복지 보유, D2.6). 회사페이지(F4/F5)·본 API는 실복지를 반환하며 프리셋 폴백 병합이 없다. 유형 프리셋은 비교 툴 직접 입력 모드(F2 입력 초기값)에서만 쓰인다(UC-15/21).
2. 검색(FR-93)과 상세(FR-94)의 `comp_id`·`comp_nm`·`comp_tp_cd`·`industry_nm`·`logo_nm` 값은 동일 회사에 대해 일치한다.
3. 정성 복지(`qual_yn=true`, `benefit_amt` NULL)는 금액 합산 대상이 아니며 정성 항목으로 표기된다(UC-93 4a, FR-D5).
4. slug는 회사 영문식별자(`COMP_ENG_NM`)를 쓰는 **정적 페이지 경로**의 기준이고, 본 API는 정수 `comp_id`로 조회한다. 별칭·통칭·검색어는 경로가 아니라 검색(FR-93)으로 처리한다(UC-95 3a).

**추적**: F8(`companies/{id}`)·F1·F4 / UC-93(회사 실복지 상세)·UC-95(미존재 404) / NFR26·NFR11(정적 slug canonical은 F4 소유) / D2.5·D2.6·D6.4 / §6 / `01` FR-06 / 본문 스키마 FR-D7·FR-D4.

## [FR-95] 공통 상태코드·오류 응답 계약

**설명**: GET 5종 엔드포인트가 공유하는 HTTP 상태코드 매트릭스와 오류 응답 형태를 규정하여, 비정상 상황에서도 클라이언트가 크래시하지 않고 사용자에게 상태를 알리게 한다(NFR26). 응답 본문의 오류/빈 응답 처리 원칙은 데이터 계약 **FR-D11**과 정합한다.

**상세 동작 — 상태코드 매트릭스**

| 상태코드 | 발생 조건 | 적용 엔드포인트 |
|----------|-----------|-----------------|
| **200 OK** | 정상 응답(검색 0건의 빈 배열 포함) | 전 엔드포인트 |
| **404 Not Found** | 미존재 `comp_id`(FR-94) / 미등록 `/api/v1/*` 경로(FR-90) | companies/{id}, 라우팅 |
| **405 Method Not Allowed** | 등록 경로에 비-GET(POST/PUT/PATCH/DELETE) 요청 | 전 GET 라우트(`Allow: GET` 헤더 포함) |
| **422 Unprocessable Entity** | 파라미터 검증 실패(누락 `q`, 과길이 `q`, 비정수 `comp_id`) | search, companies/{id} |
| **500 Internal Server Error** | 예기치 못한 서버 오류 | 전 엔드포인트 |
| **503 Service Unavailable** | (헬스 레디니스 확장 시) DB 풀 미가용 | health(선택), 데이터 엔드포인트 |

**오류 응답 envelope**
- 오류 본문은 JSON이며 일관된 형태를 갖는다. FastAPI 관례에 정렬한다.

  | 유형 | 형태(예) |
  |------|----------|
  | 명시 오류(404/405/500 등) | `{"detail": "<메시지>"}` |
  | 파라미터 검증 오류(422) | `{"detail": [{"loc": ["query","q"], "msg": "…", "type": "…"}]}` |

- `Content-Type: application/json; charset=utf-8`. 오류 응답은 캐시하지 않는다(`Cache-Control: no-store`).

**검증·비즈니스 규칙**
1. 어떤 상태코드도 앱 크래시로 이어지지 않는다: 클라이언트는 모든 비-2xx를 방어적으로 흡수하고(배열/객체 부재·널 필드 폴백, FR-D8), 사용자에게 친화적 안내를 표시한다(NFR26, FR-D11).
2. 오류 응답은 **사용자 데이터를 서버로 전송·저장하지 않으며**, 응답 본문에 개인 식별 정보를 포함하지 않는다(§2-4, NFR16).
3. 스택 트레이스·내부 경로·SQL 등 민감 정보는 500 응답 본문에 노출하지 않는다(일반 메시지만). 세부는 서버 로그에만 남긴다.
4. 검색 0건은 오류가 아니라 200 빈 배열이며 404/422와 구분된다(FR-93, FR-D11).
5. 미등록 경로 404와 미존재 리소스 404를 클라이언트는 동일하게 "불러올 수 없음"으로 흡수한다(무크래시 우선, FR-D11).

**추적**: F8 / UC-90(reference 실패)·UC-91(검색 실패)·UC-95(404) / NFR26·NFR16 / D2.6 / §6·§7 / 본문 처리 FR-D11.

## [FR-96] CORS·전송 헤더·캐시 정책

**설명**: GET 5종 엔드포인트의 CORS 정책, 공통 응답 헤더, 엔드포인트별 캐시(Cache-Control) 정책을 한곳에 통합한다. 프로덕션은 **동일 오리진**(정적=Nginx, `/api/v1/*`=Uvicorn 프록시, 동일 `jobcho.wiki`)이므로 CORS는 방어적·개발용 성격이 크다(§8).

**상세 동작 — CORS**
- 정상: 브라우저의 GET 요청은 커스텀 요청 헤더·자격증명이 없는 **단순 요청(simple request)**이므로 프리플라이트를 유발하지 않는다. 동일 오리진 프로덕션에서는 CORS 헤더가 필수는 아니다.
- 방어적 허용목록(개발·서브도메인 대비): CORS 미들웨어를 두되 오리진을 **허용목록으로 제한**한다.

  | CORS 항목 | 값 |
  |-----------|----|
  | `Access-Control-Allow-Origin` | `https://jobcho.wiki`(및 필요 시 `https://www.jobcho.wiki`) — 허용목록. **와일드카드 `*`+자격증명 금지** |
  | `Access-Control-Allow-Methods` | `GET, HEAD, OPTIONS` (쓰기 메서드 미포함) |
  | `Access-Control-Allow-Credentials` | 미설정(false). 인증·쿠키가 없으므로 자격증명 허용 불필요(§2-1) |
  | `Access-Control-Allow-Headers` | 불필요(커스텀 요청 헤더 미사용). 요청 시 최소 허용 |
  | 프리플라이트 `OPTIONS` | CORS 미들웨어가 처리(200/204 + 위 헤더) |

**상세 동작 — 캐시(Cache-Control) 정책**

| 엔드포인트 | Cache-Control | 근거 |
|------------|---------------|------|
| `GET /reference/all` | `public, max-age=3600` | **브리프 §6 명시 값**(NFR3). 재시드 후 1h 내 반영(D4.4) |
| `GET /companies/{id}` | `public, max-age=3600` | 파생 기본값. 재시드까지 정적 → D4.4 1h 반영에 정렬(SPEC 조정 가능) |
| `GET /companies/search` | `no-store` | 질의 가변·클라이언트 디바운스·번들 폴백 존재 → 캐시 이득 낮음(파생) |
| `GET /health` | `no-store` | 실시간 상태 반영 필수(FR-91) |

**입력·출력**: 전송 헤더 계약(요청 파라미터 아님).

**검증·비즈니스 규칙**
1. `Access-Control-Allow-Origin`에 `*`를 자격증명과 함께 사용하지 않는다. 자격증명 자체가 없다(무인증, §2-1).
2. `reference/all`의 `public, max-age=3600`은 브리프 명시 값으로 고정(§6·D4.4). 그 외 엔드포인트 캐시 값은 파생 기본값이며 SPEC/TASK에서 확정한다(브리프는 미명시).
3. 모든 JSON 응답 `Content-Type: application/json; charset=utf-8`. 오류·헬스·검색 응답은 `no-store`.
4. 캐시는 방문자·인증에 무관하게 동일 응답을 전제로 하며(비개인화, FR-90), 재시드 데이터 변경은 최대 TTL 내 반영된다(D4.4).

**추적**: F8 / UC-A1(부팅 캐시 적중)·UC-90(캐시 유효 시 무네트워크) / NFR3(캐시)·NFR17(쿠키 미발급) / D4.4 / §6·§8.
