# 05. 비교 계산 엔진 (FRD)

> **이력**: 브랜드 축(성장성·안정성)은 2026-07-20 제거됐다. 우선순위는 연봉·워라밸·복지 3종.
> 경위·파급범위: `docs/ANALYSIS-성장성-용어-2026-07-20.md` §6-1 · `docs/HANDOFF-2026-07-19.md` §K.


**목적**: loupit 비교 리포트(F3)의 **클라이언트 계산 엔진**을 재현 가능한 수준으로 정밀 명세한다. F2 입력(연봉·복지·근무형태·통근·우선순위·희생요소)을 받아 실효연봉·복지 합산·주간 근무시간·야근수당·시간당 가치·워라밸/자율성·통근·복지 불확실성 밴드·우선순위 판정(vdCard)을 **100% 브라우저에서** 산출하는 순수 계산 규칙을 FR-30~FR-39로 확정한다. 모든 수식은 입력 필드·타입·제약, 출력·상태, 검증·비즈니스 규칙과 함께 기술한다. "어떻게 코드로 배치하는가"는 SPEC/TASK로 미룬다.

**참조 상위문서**: FRD `01-개요와-FR규칙.md`(FR-0x 전역 규약·전역 용어·상태 모델), FRD `04-비교입력.md`(FR-2x 입력 소유), USECASE `04-비교리포트.md`(UC-30~UC-36), RESEARCH `benefit-scraping.md`(§4.2 불확실성 밴드·DEC-2). 근거 브리프: §7(비교 알고리즘), §4(데이터 모델), §6(참조 번들). 알고리즘 원형: `/home/ubuntu/job_change/app.js`의 `compare()`·`calc()`·`getBenTotal`·`getSalRange/getOfferRange`·`getWSHours`·`getOTPay`·`renderBenCatCompare`·`renderQualCompare`·`sumBenUncertainty` (로그인·신고·서버저장·프로파일러 코드는 제외).

**상위 추적**: FRD → USECASE(UC-30~36) → PRD(F3) → 브리프. 본 문서(05)는 **FR-3x 대역**을 소유한다(01 §2.3). 각 FR은 충족 UC·상위 F3·관련 NFR을 인용한다.

**범위 경계**: 본 문서는 **엔진(계산 규칙)**만 다룬다. 입력 필드의 수집·검증·초기값·프리셋 폴백 UI는 FR-2x(04)가 소유하고, 본 엔진은 그 결과 상태(`§01-5`)를 **읽기만** 한다. 리포트의 로컬 저장·불러오기·삭제(UC-36)는 전역 규약 FR-07(01)을 따르므로 여기서 재정의하지 않고 인용한다.

**전역 전제(FR-30~39 공통)**: 계산은 클라이언트에서만 수행되며 입력·결과를 서버로 전송·저장하지 않는다(§2-8, NFR16·NFR20·NFR27). 프로파일러 벡터 가중(레거시 `pfResult`)은 엔진에서 **완전히 제거**되어 어떤 계산·판정에도 개입하지 않는다(§2-3, 01 §5.1). 우선순위·희생요소는 사용자 직접 선택값(`curPri`/`curSacrifice`)만 사용한다.

## 1. 엔진 개요

### 1.1 데이터 흐름

```
[입력 상태 §01-5]                     [엔진 순수 계산]                    [리포트 출력]
 salRange(a) ── 연봉 ──┐          FR-32 실효연봉 eff = 연봉+순복지 ──┐
 benS(a/b) ── 복지 ────┼─ FR-31 순복지 net / 카테고리 합산 ─────────┤   FR-30 compare()
 wsState(a/b) ─ 근무 ──┼─ FR-33 주간근무시간 wsHours ───────────────┤     └ 총보상 vs
 com(a/b) ── 통근 ─────┼─ FR-34 야근수당 otPay (포괄/비포괄)────────┼──▶  ├ 시간당 가치
 curPri ── 우선순위 ───┼─ FR-35 시간당 가치 hourly ─────────────────┤     ├ 워라밸·자율성
 curSacrifice ─희생 ───┼─ FR-36 워라밸·자율성 autonomy ─────────────┤     ├ 카테고리 델타/정성
 REF(회사정보) ────────┼─ FR-37 통근 반영 ──────────────────────────┤     ├ 불확실성 밴드
                       ├─ FR-38 불확실성 밴드 (amt_source, DEC-2)────┤     └ vdCard 판정
                       └─ FR-39 우선순위 판정 vdCard(연봉/워라밸/복지) + 희생 비용
```

## 2. 기능 요구사항 (FR-30 ~ FR-39)

각 FR은 순수 함수 하나(또는 밀접한 함수군)에 대응한다. 입력은 `§01-5` 전역 상태에서 조달하며, 슬롯 `s ∈ {a, b}`로 좌우 대칭 계산한다.

### FR-30 비교 엔진 실행 계약 (입력 정규화·실시간 요약·전체 리포트·재계산)

- **제목**: 비교 엔진 실행 계약 — 입력 상태를 정규화하고 실시간 요약(`calc`)과 전체 리포트(`compare`)를 조립·재계산한다.
- **설명**: 엔진의 진입점. `§01-5` 상태를 슬롯별 계산 컨텍스트로 정규화하고, 편집 중에는 경량 요약을, "비교하기" 실행/입력 변경 시에는 전체 리포트를 산출한다. 하위 FR-31~39의 순수 함수를 호출·조립하는 오케스트레이터다.
- **상세 동작**
  - 정상(전체 리포트): (1) 두 슬롯의 연봉 range·복지 합산·근무시간·야근수당·시간당 가치·워라밸/자율성·밴드를 FR-31~38로 산출한다. (2) `curPri`에 따라 FR-39 vdCard를 조립한다. (3) 두 관점(a/b)을 대비하는 리포트 객체를 반환한다. **서버 호출·저장 없음**(NFR16·NFR20).
  - 정상(실시간 요약 `calc`): 편집 중 연봉 range, 순복지(net), 실효연봉 range만 갱신한다(FR-31·FR-32의 부분집합). vdCard·밴드는 계산하지 않아 경량이다.
  - 재계산(UC-35): 입력(연봉·복지·근무형태·통근·`curPri`·`curSacrifice`) 중 하나라도 변경되면 전체 리포트를 다시 산출해 갱신 표시한다. `curPri`만 변경되면 지표 값은 동일하고 vdCard 판정 축만 전환된다(UC-35 분기 2a).
  - 대안/예외:
    - 필수값(연봉) 누락(UC-30 2a / UC-35 1a): 해당 지표를 "미산출"로 두고 부족 항목을 안내한다. 리포트 전체를 무효화하지 않는다.
    - 한 슬롯이 회사 미선택 직접 입력(UC-15): 프리셋 초기값 없이 사용자 입력값만으로 그 슬롯을 계산한다.
    - 로드 후 오프라인: 참조 번들 로드 성공 이후에는 네트워크 없이도 계산이 가능해야 한다(NFR26).
- **입력**

  | 필드 | 타입 | 제약 | 출처 |
  | --- | --- | --- | --- |
  | `salRange(s)` | `{min,max,mid}` 만원 정수 | `min≤max`, 미입력 시 `{0,0,0}` | FR-2x, 아래 규칙 |
  | `benS[s]` | 복지항목[] | 항목 shape `§01-5` | FR-2x |
  | `wsState[s]` | `{ot,wage,remote,flex}` | 각 축 열거값 또는 `null` | FR-2x |
  | `com[s]` | 정수(분) | `≥0`, 편도 통근시간 | FR-2x |
  | `curPri` | `연봉\|워라밸\|복지` | 사용자 직접 선택 | FR-2x |
  | `curSacrifice` | 상동 또는 `null` | `≠ curPri` | FR-2x |
  | `REF` | 참조 번들 | 회사 정보(`work_style_val` 등) 참조 | FR-8x |

  **연봉 range 규칙**(원형 `getSalRange`/`getOfferRange`): 슬롯 a는 `"lo-hi"` 문자열을 파싱해 `{min:lo, max:hi, mid:round((lo+hi)/2)}`. 슬롯 b는 `selectedRate`(%)로 `min/max/mid = round(a값 × (1+rate/100))`(01 §5). 어느 쪽이든 미입력이면 `{0,0,0}`.
- **출력·상태**: 리포트 객체(두 관점 지표 + vdCard + 밴드 범위) 또는 경량 요약(연봉/순복지/실효연봉 range). 클라이언트 메모리에만 존재하며 사용자가 명시적으로 저장할 때만 localStorage에 기록(FR-07).
- **검증·비즈니스 규칙**: (1) 입력·중간값·결과의 서버 전송 0(NFR16·NFR20). (2) 재계산은 클라이언트에서만, INP ≤ 200ms 목표(NFR2). (3) `curPri`는 프로파일러 산출이 아닌 사용자 선택값만 사용(§2-3). (4) `pfResult` 등 프로파일러 상태를 읽지 않는다(01 §5.1). (5) 렌더 시 회사·복지·입력 문자열은 이스케이프(NFR21).
- **추적**: UC-30·UC-35, F3, NFR2·NFR16·NFR20·NFR26·NFR27, §7·§2-8.

### FR-31 복지 합산 — 순복지·카테고리별 합산·정성 분리

- **제목**: 복지 합산 — 체크된 복지의 순복지(net)·전체합(ben)·9카테고리 합산 및 정성 복지 분리.
- **설명**: 각 슬롯의 복지 목록에서 총보상에 더해질 **순복지(net)**와 카테고리별 델타·정성 비교의 기반값을 산출한다. 원형 `getBenTotal`·`getBenByCat`·`renderBenCatCompare`·`renderQualCompare`.
- **상세 동작**
  - 정상: 슬롯 `s`의 `benS[s]`에서 **체크된(checked=true)** 항목만 대상으로:
    - `ben` = Σ `benefit_amt` (체크된 전 항목)
    - `net` = Σ `benefit_amt` (체크됐고 **정성 아님**, `qual_yn=false`) — 실효연봉에 합산되는 순복지
  - 카테고리 합산: 9카테고리(`compensation/flexibility/work_env/time_off/health/family/growth/leisure/perks`)별로 `qual_yn=false` 항목 금액을 합산해 슬롯 간 델타(`sumB − sumA`)를 산출한다(`renderBenCatCompare`).
  - 정성 분리: `qual_yn=true`(금액 NULL) 항목은 금액 합산·델타에서 **제외**하고, 카테고리와 `qual_desc`로 a/b 정성 비교를 별도 구성한다(`renderQualCompare`, UC-32 3a).
  - 대안: 한 슬롯만 특정 카테고리를 보유하면 미보유 슬롯을 0 기준으로 델타 표기(UC-32 1a). 회사 미선택 슬롯은 프리셋 폴백값으로 채워진 복지로 합산(UC-32 1b, FR-06).
- **입력**

  | 필드 | 타입 | 제약 |
  | --- | --- | --- |
  | `benS[s][i].checked` | boolean | 미체크는 전부 제외 |
  | `benS[s][i].benefit_amt` | 정수(만원) 또는 `null` | 정성이면 `null`(0으로 취급) |
  | `benS[s][i].qual_yn` | boolean | `true`=정성(금액 없음) |
  | `benS[s][i].benefit_ctgr_cd` | 9카테고리 열거 | §4 |
  | `benS[s][i].qual_desc` | 문자열 | 정성 설명(짧은 발췌) |
- **출력·상태**: 슬롯별 `{ben, net}`(만원), 카테고리별 `{sumA, sumB, delta}`, 정성 항목 a/b 목록.
- **검증·비즈니스 규칙**: (1) 정성 복지는 순복지·카테고리 금액 합산에 절대 포함하지 않는다(§7, UC-31 1a). (2) 미체크 항목은 어떤 합산에도 들어가지 않는다. (3) 금액 단위는 만원(01 §4). (4) `qual_desc`는 이스케이프 렌더(NFR21).
- **추적**: UC-31·UC-32, F3, NFR21, §4(BENEFIT_CTGR_CD·QUAL_YN·QUAL_DESC_CTNT)·§7.

### FR-32 실효연봉(총보상) 산출

- **제목**: 실효연봉 = 연봉 + 순복지.
- **설명**: 명목 연봉과 구분되는 실질 보상. 슬롯별 연봉 range에 순복지(FR-31 `net`)를 더한다.
- **상세 동작**
  - 정상: `eff(s) = { min: salRange(s).min + net(s), max: salRange(s).max + net(s), mid: salRange(s).mid + net(s) }` (만원). 순복지는 range 세 값에 동일 가산한다(원형 `compare` L996-997).
  - 델타: `effDiffMid = eff(b).mid − eff(a).mid`, `effDiffMin/Max`도 산출해 리포트 보상 비교에 사용.
  - 경고 규칙: 연봉은 b가 높지만(`salDiffMid>0`) 실효 차이가 연봉 차이보다 작고(`effDiffMid<salDiffMid`) a의 복지가 더 많으면(`ben_a>ben_b`), "연봉이 올라도 실질 보상 차이가 줄어든다" 경고를 표기한다(`compare` L1083).
  - 대안: 연봉 미입력 슬롯은 `eff={0,0,0}`이며 해당 지표는 미산출 처리(FR-30).
- **입력**: `salRange(s){min,max,mid}`, `net(s)`(FR-31).
- **출력·상태**: 슬롯별 `eff{min,max,mid}`(만원)와 슬롯 간 델타.
- **검증·비즈니스 규칙**: (1) 순복지만 더하고 **정성 복지·미체크 항목은 더하지 않는다**(FR-31). (2) 표시 시 단일값이 아닌 range(min~max)와 밴드(FR-38)를 함께 제시한다. (3) 단위 만원.
- **추적**: UC-31, F3, NFR26, §7(실효연봉=연봉+순복지)·01 §4(용어).

### FR-33 주간 근무시간 산출

- **제목**: 주간 근무시간 = 야근 빈도(ot) 매핑값.
- **설명**: 시간당 가치·워라밸 판정의 분모가 되는 주당 총 근무시간. 원형 `getWSHours`.
- **상세 동작**
  - 정상: `wsHours(s) = OT_HRS[wsState[s].ot]` = `{low:40, mid:45, high:54}` 중 매핑값. `ot`가 `null`(미선택)이면 `0`(미입력).
  - 구성: 기본 40시간 + 야근분(mid=+5, high=+14). 연장근로 시간 = `max(0, wsHours − 40)`(FR-34·FR-35에서 재사용).
  - 대안: 야근 빈도 미선택 슬롯은 근무시간 기반 비교(시간당 가치·"적게 일하기")를 미산출로 두고 안내(UC-31 2a, UC-33 wlb 분기).
- **입력**: `wsState[s].ot ∈ {low, mid, high, null}`.
- **출력·상태**: `wsHours(s)`(주당 시간, 정수) — 0이면 미입력.
- **검증·비즈니스 규칙**: (1) 매핑은 `OT_HRS` 고정 상수만 사용(임의값 금지). (2) 연간 근무시간은 `wsHours × 52`로 파생한다(FR-35). (3) 40 초과분만 연장근로로 취급(FR-34).
- **추적**: UC-31·UC-33, F3, §7(getWSHours).

### FR-34 야근수당 산출 (포괄/비포괄 임금유형 분기)

- **제목**: 야근수당 = 임금유형 분기 계산 (포괄=미반영 / 비포괄=시급×1.5 기반).
- **설명**: 연장근로에 대한 연간 추가 보상(만원). 임금유형(`wage`)에 따라 분기한다. 원형 `getOTPay`.
- **상세 동작**
  - 분기 규칙:

    | 조건 | `otPay` |
    | --- | --- |
    | `wage='inclusive'`(포괄임금) | **0** — 야근수당이 연봉에 이미 포함, 추가 수당 없음 |
    | `wage='separate'` 이고 `ot='low'` | **0** — 연장근로 없음 |
    | `wage='separate'` 이고 `ot∈{mid,high}` 이고 `sal.mid>0` | 아래 수식 |
    | `sal.mid=0` (연봉 미입력) | 0 |

  - 비포괄 수식(만원/년):
    ```
    hourlyBase = sal.mid × 10000 / 12 / 209      # 통상시급(원): 연봉만원→원, ÷12월, ÷209h
    extraHrs   = OT_HRS[ot] − 40                  # 주당 연장근로시간
    otPay      = round( extraHrs × hourlyBase × 1.5 × 4.33 × 12 / 10000 )
                 #        연장h  ×  통상시급   × 가산1.5 × 4.33주/월 × 12월 ÷ 10000(원→만원)
    ```
    - 슬롯 a는 `sal=salRange(a)`, 슬롯 b는 `sal=getOfferRange`(b 연봉 range)를 사용.
  - 대안/표시: 포괄+야근인 슬롯은 "야근수당 연봉 포함, 추가 0원"으로 명시하고, 시간당 가치 하락 경고를 붙인다(UC-31 2a, `compare` L1088·L1116).
- **입력**

  | 필드 | 타입 | 제약 |
  | --- | --- | --- |
  | `wsState[s].wage` | `inclusive\|separate\|null` | `null`/포괄 → 0 |
  | `wsState[s].ot` | `low\|mid\|high\|null` | `low`/`null` → 0 |
  | `salRange(s).mid` | 정수(만원) | `0`이면 0 반환 |
- **출력·상태**: `otPay(s)`(만원/년, 정수 ≥ 0). 총보상·시간당 가치에 가산된다.
- **검증·비즈니스 규칙**: (1) 포괄임금은 야근수당을 **결코 가산하지 않는다**(핵심 분기). (2) 통상시급 분모는 209h 고정. (3) 가산율 1.5 고정. (4) 음수 불가(연장근로 음수 시 0).
- **추적**: UC-31, F3, §7(getOTPay, 포괄/비포괄 분기).

### FR-35 시간당 실질 가치 산출

- **제목**: 시간당 가치 = (실효연봉 + 야근수당) / 연간 근무시간.
- **설명**: 총보상을 실제 투입 시간으로 나눈 시간당 실질 보상(원). "많이 벌어도 많이 일하면 시간당은 다르다"를 정량화. 원형 `compare` L1105-1110.
- **상세 동작**
  - 정상(양 슬롯 모두 근무시간 입력): 
    ```
    annHrs(s) = wsHours(s) × 52
    hourly(s) = round( (eff(s).mid + otPay(s)) × 10000 / annHrs(s) )   # 원/시간
    hvDiff    = hourly(b) − hourly(a)
    ```
    분자는 총보상(실효연봉+야근수당)을 만원→원 환산(×10000), 분모는 연간 근무시간.
  - 승자 판정: `hvDiff>0 → b`, `<0 → a`, `=0 → 동일`. 차이 %는 `round(|hvDiff| / min(hvA,hvB) × 100)`.
  - 대안: 한쪽이라도 `wsHours=0`(야근 빈도 미입력)이면 시간당 비교를 미산출("근무시간 미입력")로 둔다(UC-31 2a).
  - 포괄임금 경고: 비교 대상이 포괄임금+야근이면 "야근이 늘수록 시간당 실질 가치 하락"을 표기(`compare` L1116).
- **입력**: `eff(s).mid`, `otPay(s)`, `wsHours(s)`.
- **출력·상태**: 슬롯별 `hourly(s)`(원/시간, 정수), 슬롯 간 차이·%·승자.
- **검증·비즈니스 규칙**: (1) 연간 근무시간은 `wsHours × 52`로 고정 환산(FR-33). (2) 분모 0(미입력) 시 산출하지 않는다(0 나눗셈 방지). (3) 총액 우위와 시간당 우위가 엇갈릴 수 있으며 이를 vdCard가 대비 제시(FR-39).
- **추적**: UC-31, F3, §7(시간당 가치)·NFR26.

### FR-36 워라밸·자율성 요인 산출

- **제목**: 자율성 점수 = 재택 절약(REMOTE_SAVE) + 유연근무 + 무제한휴가.
- **설명**: 근무시간 외 워라밸 요인을 점수화. 재택은 통근 절약 가치(만원)로, 유연근무·무제한휴가는 가산 점수로 반영. 원형 `compare` L1049-1050.
- **상세 동작**
  - 정상:
    ```
    autonomy(s) = REMOTE_SAVE[remote(s)]                 # none:0 partial:72 hybrid:120 free:180 (만원)
                + (flex(s) ≠ 'none' ? 50 : 0)            # 유연근무(시차/탄력) 가산
                + (unlimitedPTO(s) ? 80 : 0)             # 무제한/자율 휴가 가산
    autoWin = autonomy(a)>autonomy(b) ? a : autonomy(b)>autonomy(a) ? b : 동일
    ```
    - `remote`·`flex`는 `wsState[s]`에서, `unlimitedPTO`는 회사 `work_style_val`(REF, `matched[s]`)에서 읽는다.
  - 요인 라벨: 재택 수준(`REMOTE_LABELS`), 유연근무 유형(`FLEX_LABELS`), 자율 휴가 유무를 승자 슬롯 근거로 표시한다.
  - 대안: 재택·유연·무제한휴가 미체크/미보유 슬롯은 해당 가산 없이 계산(UC-31 3a). 회사 미선택 슬롯은 `unlimitedPTO` 참조 불가 → 해당 가산 0.
- **입력**

  | 필드 | 타입 | 제약 |
  | --- | --- | --- |
  | `wsState[s].remote` | `none\|partial\|hybrid\|free` | REMOTE_SAVE 키 |
  | `wsState[s].flex` | `none\|stagger\|flexible` | `none` 외 +50 |
  | `matched[s].work_style_val.unlimitedPTO` | boolean | 회사 선택 시만 참조 |
- **출력·상태**: 슬롯별 `autonomy(s)`(가중 합), 승자, 근거 요인 목록.
- **검증·비즈니스 규칙**: (1) 재택 절약 값은 `REMOTE_SAVE` 고정 상수. (2) 유연 +50·무제한휴가 +80 고정. (3) 이 점수는 워라밸 vdCard의 "시간 자율성" 관점(FR-39) 입력이며, 근무시간(FR-33)의 "적게 일하기" 관점과 **별개 축**으로 다룬다.
- **추적**: UC-31·UC-33, F3, §7(REMOTE_SAVE·유연·무제한휴가)·§4(WORK_STYLE_VAL).

### FR-37 통근 시간·비용 반영

- **제목**: 통근 — 편도 통근시간 비교 및 재택에 의한 통근 절약 반영.
- **설명**: 통근은 (a) 재택 근무를 통한 **연간 통근 절약 금액**(REMOTE_SAVE, FR-36)으로 자율성에 반영되고, (b) 슬롯 간 **편도 통근시간** 비교로 워라밸 요약에 노출된다.
- **상세 동작**
  - 정상(금전 반영): 재택 수준이 높을수록 통근이 줄어드는 가치를 `REMOTE_SAVE[remote]`(만원/년)로 자율성 점수에 가산한다(FR-36). 즉 통근 절감의 경제 가치는 재택 축을 통해 계산에 들어간다.
  - 정상(시간 비교): `com(a)` vs `com(b)`(편도 분)를 워라밸 나머지 기준 요약에 "통근 A분 vs B분"으로 대비 표시한다(`compare` L1276). 야근 빈도가 미입력이라 근무시간 비교가 불가할 때 통근시간이 워라밸 대체 지표가 된다(UC-31 3a).
  - 대안: 통근시간 미입력(`com=0`) 슬롯은 통근 비교를 생략한다.
- **입력**: `com[s]`(정수 분, 편도, ≥0), `wsState[s].remote`(REMOTE_SAVE 매핑).
- **출력·상태**: 슬롯 간 통근시간 대비 표시, 통근 절약 금액(REMOTE_SAVE 경유 자율성 반영).
- **검증·비즈니스 규칙**: (1) 통근시간은 표시·워라밸 보조 지표이며 총보상 금액 합산에는 직접 들어가지 않는다(통근 절감은 REMOTE_SAVE로만 금전화). (2) 통근시간은 분 단위 비음수.
- **추적**: UC-31, F3, §7(통근)·FR-36.

### FR-38 복지 불확실성 밴드 (amt_source 기반, DEC-2)

- **제목**: 불확실성 밴드 — 금액 신뢰도(amt_source)별 ± 오차를 항목별로 산정·합산 (stated ±5% / estimated ±20% / 만료 +15% 확대).
- **설명**: 리포트의 복지 금액·총보상을 **단일값이 아닌 범위**로 제시하기 위한 오차 밴드. **DEC-2**에 따라 밴드는 출처 배지(`badge_cd=official/est`)가 아니라 **금액 신뢰도(`amt_source`)**를 기준으로 결정한다. 원형 `sumBenUncertainty`(단, 계수 기준을 DEC-2로 교체).
- **상세 동작**
  - 밴드 계수(항목별):

    | 조건 | 밴드 계수 | 비고 |
    | --- | :---: | --- |
    | `amt_source='stated'` (공식 명시 금액) | **0.05** (±5%) | RESEARCH §4.2 |
    | `amt_source='estimated'` (앵커 추정 금액) | **0.20** (±20%) | 96개 다수가 여기 해당 |
    | `amt_source='none'` (정성 복지, 금액 NULL) | **0** | 금액 없음 → 밴드 미적용(FR-31) |
    | `EXPIRES_DTM` 경과(만료) | **+0.15 가산** | 위 기준 계수에 **더함**(확대) |

  - 산정:
    ```
    isExpired(b) = expires_dtm != null AND Date(expires_dtm) < now
    band(b)      = base(amt_source) + (isExpired(b) ? 0.15 : 0)
                   # 예) stated+만료 = 0.05+0.15 = 0.20 ;  estimated+만료 = 0.20+0.15 = 0.35
    sumBand(s)   = Σ over (checked AND qual_yn=false) [ benefit_amt × band(b) ]   # ± 만원
    총보상 범위   = [ total(s) − sumBand(s),  total(s) + sumBand(s) ]
    ```
  - **DEC-2 핵심**: 96개 복지는 출처 배지가 `official`이더라도 **추정 금액 항목은 `amt_source=estimated`이므로 ±20% 밴드를 유지**한다. "공식 출처지만 금액은 추정" 상태가 계산·UI에 정직하게 반영된다.
  - 만료 표기: 만료 항목은 밴드 +15% 확대와 함께 "재확인 필요" 표기를 노출한다(UC-34 2a, RESEARCH §4.2).
  - 대안: 정성 항목(`amt_source=none`)·미체크 항목은 밴드 합산에서 제외하고 배지·출처만 표기(UC-34 3a). 프리셋 폴백 항목은 그 항목의 `amt_source`/만료 상태로 밴드 산정(UC-34 1a).
- **입력**

  | 필드 | 타입 | 제약 |
  | --- | --- | --- |
  | `benS[s][i].amt_source` | `stated\|estimated\|none` | 밴드 기준(배지와 분리) |
  | `benS[s][i].expires_dtm` | 날짜 또는 `null` | 경과 시 +0.15 |
  | `benS[s][i].benefit_amt` | 정수(만원) 또는 `null` | 정성이면 제외 |
  | `benS[s][i].checked / qual_yn` | boolean | 체크·비정성만 합산 |
- **출력·상태**: 슬롯별 `sumBand(s)`(± 만원), 총보상 범위, 항목별 밴드 계수. UI는 배지(공식/추정/만료)와 함께 밴드를 표기.
- **검증·비즈니스 규칙**: (1) 밴드는 **금액 신뢰도(amt_source) 기준**이며 출처 배지(`badge_cd`)와 **무관**하다(DEC-2, 디커플링). 이는 01 §FR-05의 "공식/추정/만료" 스케치를 정교화한 것으로, `official` 배지라도 `estimated` 금액이면 ±20%를 유지한다. (2) 만료는 기준 계수에 **가산(+0.15)**하는 확대이며 치환이 아니다. (3) 정성 복지엔 밴드를 적용하지 않는다(금액 부재). (4) 밴드는 항목별로 산정 후 합산(단일 % 일괄 적용 금지). (5) F7 데이터 정확성 면책조항과 정합해야 한다(UC-34).
- **추적**: UC-34, F3(F7 정합), RESEARCH §4.2·DEC-2, §7·§4(BADGE_CD·EXPIRES_DTM·amt_source).

### FR-39 우선순위 판정(vdCard) + 희생요소 포기비용

- **제목**: 우선순위(curPri) 기반 판정 카드(vdCard) — 연봉·워라밸·복지 3축 분기 + 희생요소 포기비용 정량화.
- **설명**: 사용자가 직접 선택한 우선순위 축으로 두 관점을 평가해 "누가 유리 / 무엇을 택할지"를 요약하고, 희생요소를 선택했으면 그 포기 비용을 정량화한다. 원형 `compare`의 vdCard 헬퍼(L1019) + 축별 분기(L1030~L1077) + sacrifice(L1253~). **프로파일러 가중(pfResult)은 제거**되어 판정은 오직 `curPri`/`curSacrifice`와 지표값에만 근거한다.
- **상세 동작**
  - vdCard 구조: 각 판정은 두 관점(p1, p2)을 가지며, 각 관점마다 승자 슬롯과 근거, 그리고 "무엇이 중요하면 어느 쪽" 선택 문구를 제시한다.
  - **축별 분기**(`curPri`):

    | curPri | 관점 p1 | 관점 p2 | 승자 규칙 | 선택 문구 |
    | --- | --- | --- | --- | --- |
    | **연봉**(salary) | 총액 = `total(s)=eff.mid+otPay` | 시간당 가치 = `hourly(s)`(FR-35) | 각 관점 더 큰 값이 유리 | 총액 중요 → 총액 승자 / 시간효율 → 시간당 승자 |
    | **워라밸**(wlb) | 적게 일하기 = `wsHours(s)` 적은 쪽 | 시간 자율성 = `autonomy(s)`(FR-36) 큰 쪽 | p1 근무시간 적은 슬롯, p2 자율성 높은 슬롯 | 근무시간 vs 시간자유 대비 |
    | **복지**(benefits) | 복지 항목 = `ben(s)` 큰 쪽 | 총보상 포함 = `total(s)` 큰 쪽 | 각 큰 값 유리 | `|total 차이|<1,200만`이면 "복지 만족도 우선" |

  - **희생요소 포기비용**(`curSacrifice` 선택 시):

    | curSacrifice | 포기 비용 정량화 |
    | --- | --- |
    | 연봉 | `|total(b)−total(a)|` → 연/월/일 환산 "이 금액을 포기할 수 있나요?" |
    | 워라밸 | `|wsHours(a)−wsHours(b)|` 주당 → 연간 시간·근무일 "이 추가 근무를 감수?" |
    | 복지 | `|ben(a)−ben(b)|` 만원 "이 복지 차이를 포기?" |

  - 나머지 기준 요약: `curPri`·`curSacrifice`가 아닌 축들을 간단 승자 요약으로 표기(`compare` L1268).
  - 대안/예외:
    - 우선순위 미선택(UC-33 1a): 판정 전 우선순위 직접 선택을 요구하고 F2로 안내(프로파일러 없음).
    - 근소 차이(UC-33 3a): 우열이 근소함을 표기해 과잉 단정을 피한다.
- **입력**: `curPri`, `curSacrifice`, `total(s)`(FR-32·34), `hourly(s)`(FR-35), `wsHours(s)`(FR-33), `autonomy(s)`(FR-36), `ben(s)`(FR-31).
- **출력·상태**: vdCard(두 관점 승자·근거·선택 문구), 희생 포기비용 박스, 나머지 기준 요약. 클라이언트 렌더만.
- **검증·비즈니스 규칙**: (1) 판정은 전적으로 클라이언트 계산이며 서버 판정 호출 0(NFR20, F3 수용기준 4). (2) 판정 입력은 `curPri`/`curSacrifice`(사용자 직접 선택)와 지표값뿐 — **프로파일러 벡터·페르소나 가중 없음**(§2-3, 01 §5.1). (3) `curSacrifice ≠ curPri`. (4) 렌더 문자열 이스케이프(NFR21).
- **추적**: UC-33, F3, NFR20·NFR21, §2-3·§7(vdCard·curPri/curSacrifice).

## 3. 통합 계산 파이프라인 (compare 의사코드)

FR-30이 조립하는 전체 리포트 산출 순서다. 순수 함수만 호출하며 부수효과(서버 호출·저장)는 없다.

```text
function compare(state):                                    # FR-30
  if not hasRequired(state.salRange):                       # 연봉 필수 (UC-30 2a)
      return 안내("연봉 입력 필요")
  for s in {a, b}:
      sal[s]   = salRange(s)                                # FR-30 (getSalRange/getOfferRange)
      {ben[s], net[s]} = benTotal(s)                        # FR-31
      eff[s]   = { sal[s].* + net[s] }                      # FR-32
      wsH[s]   = OT_HRS[ws[s].ot] or 0                       # FR-33
      otPay[s] = otPayCalc(s)                                # FR-34 (포괄=0 / 비포괄=시급×1.5)
      total[s] = eff[s].mid + otPay[s]                       # FR-34/32
      hourly[s]= (wsH[s]>0) ? round(total[s]*1e4/(wsH[s]*52)) : null   # FR-35
      auto[s]  = REMOTE_SAVE[ws[s].remote] + (ws[s].flex!=none?50:0)
                                          + (unlimitedPTO(s)?80:0)      # FR-36
      band[s]  = Σ benefit_amt × ( base(amt_source) + (expired?0.15:0) )  # FR-38 (DEC-2)
  catDelta   = benCatCompare(a,b)                           # FR-31 (9카테고리)
  qualCmp    = qualCompare(a,b)                             # FR-31 (정성)
  commute    = commuteCompare(com[a], com[b])               # FR-37
  vd         = vdCard(curPri, {total,hourly,wsH,auto,ben})  # FR-39 (연봉·워라밸·복지 3축)
  sacCost    = curSacrifice ? sacrificeCost(curSacrifice, ...) : none   # FR-39
  return Report{ eff, total(±band), hourly, auto, catDelta, qualCmp, commute, vd, sacCost }
  # 서버 전송/저장 없음 (NFR16·NFR20). 저장은 사용자 명시적 요청 시 localStorage 한정 (FR-07)
```

경량 실시간 요약 `calc()`은 위에서 `sal / net / eff` 만 산출하고 나머지는 생략한다(FR-30).
