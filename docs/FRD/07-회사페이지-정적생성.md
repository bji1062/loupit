# 회사 상세 정적 생성 (FRD)

**목적**: F4(회사 상세 페이지 ~96개, 복지 데이터가 있는 회사)의 **빌드타임 정적 HTML 생성**을 필드·규칙 단위로 정밀 명세한다. 구체적으로 (1) 복지 데이터가 있는 회사(~96개)에 대한 정적 페이지 생성 파이프라인(정적 산출·JS 없는 본문·서버 무렌더), (2) 페이지 URL·slug 규칙(영문 식별자 기반), (3) 본문 구성(기업정보·근무형태·복지표·배지·출처·검증/만료·데이터정확성 면책), (4) SEO 메타데이터(고유 `<title>`·`meta description`·OpenGraph·JSON-LD Organization·자기참조 canonical), (5) `sitemap.xml`·`robots.txt` 등록, (6) "이 회사로 비교하기" CTA와 비교 툴 프리필 파라미터, (7) 콘텐츠 위주 광고/제휴 슬롯, (8) 생성물 검증·404 정합(등록 회사는 모두 실복지 보유 → 회사페이지 프리셋 폴백 불필요)을 확정한다. 본 문서는 **"빌드타임에 무엇을 어떤 규칙으로 생성·표시하는가"**를 규정하며, 회사·복지 데이터의 **계약(스키마·필드)은 재정의하지 않고** FRD `02-데이터-계약.md`(FR-D4·FR-D5·FR-D9)와 브리프 §4를 인용한다.

**상위 추적**: FRD → USECASE → PRD → 브리프. 상위 근거 = USECASE `05-회사상세페이지.md`(UC-40·UC-41·UC-42·UC-43·UC-44), PRD `05-기능-요구사항.md`(F4, 연동 F2·F6·F7), PRD `07-비기능-요구사항.md`(NFR1·NFR2·NFR5·NFR6·NFR7·NFR8·NFR9·NFR10·NFR11·NFR12·NFR13·NFR15·NFR19·NFR21·NFR24·NFR29), PRD `08-데이터-요구사항.md`(D1.2·D1.6·D1.7·D2.3·D2.5·D3·D4.1·D4.2), 브리프 §2-6(광고 배치)·§2-7(SEO 정적 페이지)·§4(데이터 모델)·§5(보존 실데이터)·§8(빌드타임 파이썬 생성 스크립트). 전역 규약·용어는 FRD `01-개요와-FR규칙.md`(FR-01·FR-04·FR-05·FR-06), 데이터 필드 계약은 FRD `02-데이터-계약.md`(FR-D2·FR-D3·FR-D4·FR-D5·FR-D9)를 인용하며 재정의하지 않는다.

**범위 경계**: 본 문서는 F4의 **정적 페이지 생성물과 그 표시·연동 규칙**만 다룬다. (1) 비교 입력의 상세 필드·검증은 F2(FR-2x)가, 리포트 산출은 F3(FR-3x/FR-4x)이 소유하며, 본 문서는 CTA가 넘기는 **프리필 파라미터의 형태·의미**만 규정한다(FR-57). (2) 인기 비교 조합 페이지(회사쌍)의 생성은 F5(FR-6x 계열 담당 문서, UC-50~54)가 소유하고, 본 문서는 회사 상세 URL만 `sitemap.xml`에 기여한다(FR-56). (3) 광고/제휴 슬롯의 **콘텐츠·컴포넌트**는 F6(FR-6x, 문서 07-광고·제휴·정책)이 소유하고, 본 문서는 정적 페이지 내 **슬롯 배치·고정 크기 예약·"광고" 표기 부착**만 규정한다(FR-58). (4) 읽기 전용 API(`companies/{id}`·`reference/all`)의 응답 계약은 F8(FR-8x)이 소유하며, 정적 생성기는 빌드타임에 동일 데이터 소스(DB/시드)를 사용하되 **런타임 API를 호출하지 않는다**(NFR1). 로그인/회원/프로파일러/서버측 사용자 쓰기(영구 제외, §2-1·§2-3·§2-4)는 어떤 생성·표시·연동 경로에도 등장하지 않는다(FR-01). 정적 페이지 열람·CTA 진입 전 과정에서 사용자 데이터의 서버 저장·전송이 없으며(NFR16), 프리필 상태는 URL 파라미터·클라이언트에만 존재한다(§2-2).

**FR-ID 대역**: 본 문서는 **FR-5x 대역**(FR-50~FR-59)을 소유한다(작성 지시). 부여한 ID는 재사용하지 않는다(안정 ID). 본 문서의 헤드라인 커버 UC는 **UC-40·UC-41·UC-42·UC-43·UC-44**다. 하위 문서(FLOW/WIREFRAME/SPEC/TASK)가 이 ID를 인용한다.

## 공통 규약 (전 FR 적용)

- **빌드타임 생성·런타임 무렌더**: 회사 상세 ~96개는 파이썬 생성 스크립트가 **빌드타임에 정적 HTML로 산출**하고 Nginx가 파일을 직접 서빙한다. 요청 시 Uvicorn 호출·DB 쿼리 0건이며 서버 사이드 렌더링이 없다(NFR1, §2-7·§8). 데이터 갱신은 재시드 → 재생성(재빌드)로만 이뤄진다(D4.2). 런타임 쓰기 경로 없음(§2-4).
- **JS 없는 본문(비-JS 렌더)**: 기업정보·근무형태·복지표·배지·출처·면책 등 **본문 콘텐츠는 최종 HTML 마크업에 완성된 상태로 포함**된다. JS 실행 없이 브라우저·크롤러가 본문을 그대로 표시·색인할 수 있어야 한다(NFR12·NFR24, UC-44). JS는 광고 슬롯 주입 등 **향상(enhancement)** 목적으로만 쓰이며 본문 표시/CTA 이동을 막지 않는다(FR-58, CTA는 순수 `<a href>`).
- **데이터 소스**: 각 페이지 데이터는 브리프 §4/§5의 큐레이션 데이터에서 온다 — 기업정보=`TCOMPANY`(+`TCOMPANY_TYPE`), 복지=`TCOMPANY_BENEFIT` **실데이터**. 회사페이지는 복지 실데이터가 있는 회사(~96개)만 생성하므로 **등록 회사는 모두 실복지를 보유하며, 회사페이지용 프리셋 폴백은 불필요**하다(§4·§5, 2026-07-09 결정). `TBENEFIT_PRESET`는 비교 툴의 직접 입력 모드(사용자 회사가 등록 회사에 없을 때 유형 기반 기본 복지 템플릿)에서만 쓰인다(UC-15/21). 정적 페이지의 복지 데이터 소스·표시 규약은 `companies/{id}`(FR-D7)와 **동일**한 실데이터 기준이다.
- **단위·포맷**: 복지 금액은 **만원 단위 정수**이며(§4, FR-04) 표시 시 한국어 "만/억" 포맷으로 렌더한다(예: `1200` → "1,200만원"). 정성 복지(`QUAL_YN=Y`)는 금액 없이 설명(`QUAL_DESC_CTNT`)만 표시한다(FR-D5).
- **렌더 안전(XSS 방지)**: 회사·복지 문자열(`COMP_NM`·`INDUSTRY_NM`·`BENEFIT_NM`·`NOTE_CTNT`·`QUAL_DESC_CTNT`·별칭·`BADGE_SRC_URL_CTNT`)은 빌드타임에 **HTML 이스케이프**하여 마크업에 삽입한다. 원문 문자열의 무이스케이프 삽입을 금지한다(NFR21). 출처 URL은 `http`/`https` 스킴으로 제한하고 속성값을 이스케이프한다(FR-54).
- **접근성·시맨틱**: 각 페이지는 시맨틱 HTML5 landmark(`header`/`nav`/`main`/`footer`)와 단일 `<h1>`(회사 정식명) 이하 위계, 복지·카테고리 표는 `<table>`로 구성한다. 배지는 색상만이 아니라 **텍스트 라벨**로도 의미를 전달하고, 의미 있는 이미지에 `alt`를 부여한다(NFR13·NFR15).
- **레이아웃 안정**: 광고/이미지 슬롯은 고정 크기를 예약하고 폰트는 `font-display`로 로드하여 레이아웃 이동을 억제한다(CLS ≤ 0.1, NFR5). 정적 본문은 LCP ≤ 2.5s 목표를 저해하지 않는다(NFR2).
- **한국어·인코딩**: 모든 페이지는 `<html lang="ko">`·`<meta charset="utf-8">`로 서빙하고 UTF-8로 인코딩한다(NFR29).
- **미전송·로컬 한정**: 열람·CTA 전 과정에서 사용자 데이터의 서버 전송·저장이 없다(NFR16). 프리필 등 상태는 URL 파라미터·클라이언트에만 존재한다(§2-2, FR-57).

## [FR-50] 회사 상세 정적 페이지 빌드타임 생성 파이프라인(~96개)

**설명**: 복지 데이터가 있는 회사(~96개, D2.3) 각각에 대해 빌드타임 파이썬 생성 스크립트가 독립 URL을 갖는 정적 HTML 페이지를 산출한다. 본문은 JS 실행 없이 완성된 상태로 배포되며, 요청 시 서버 렌더링·DB 조회가 없다.

**상세 동작**

- 정상 흐름
  1. 생성 스크립트가 데이터 소스(재시드된 DB 또는 시드 산출물)에서 회사 목록과 회사별 데이터(기업정보 + 복지[실데이터] + 근무형태 + 별칭 + 배지·출처·검증/만료)를 읽는다(§4/§5). 등록 회사는 모두 실복지를 보유하므로 회사페이지 프리셋 폴백은 없다.
  2. 회사마다 slug(FR-51)를 확정하고, `<head>` 메타(FR-55) + 기업정보·근무형태(FR-52) + 복지표·배지·면책(FR-53·FR-54) + CTA(FR-57) + 광고 슬롯 자리(FR-58)를 **완성된 HTML 문자열**로 렌더해 파일로 기록한다.
  3. ~96개 파일과 `sitemap.xml` 엔트리(FR-56)를 산출하고, 생성물 검증(FR-59)을 통과하면 배포 대상으로 확정한다.
  4. Nginx가 정적 파일을 직접 서빙한다. 열람 요청은 백엔드(Uvicorn)·DB에 도달하지 않는다(NFR1).
- 대안/예외
  - 데이터 재갱신: 시드/스크래핑 → DB 재시드 → 전체 재생성(재빌드)로만 반영한다. 개별 페이지의 런타임 부분 갱신·쓰기 경로는 없다(D4.2, §2-4).
  - 생성 대상 외 회사(등록 ~96개 밖): 페이지를 생성하지 않는다. 해당 URL은 존재하지 않으며 404로 처리된다(FR-59, UC-40 2a·UC-44 2a).

**입력**

| 입력 | 소스 | 타입/제약 |
| --- | --- | --- |
| 회사 목록 | `TCOMPANY` 등록 ~96행(D2.6) | 각 `{COMP_ID, COMP_ENG_NM(unique), COMP_NM(unique), COMP_TP_ID, INDUSTRY_NM, LOGO_NM, WORK_STYLE_VAL(JSON), CAREERS_BENEFIT_URL}` |
| 회사별 복지 | `TCOMPANY_BENEFIT` 실데이터(등록 회사는 모두 보유) | 복지 항목 배열(FR-D5 필드) |
| 기업유형 특성 | `TCOMPANY_TYPE` | `{COMP_TP_CD, COMP_TP_NM, GROWTH_LABEL_NM, STABILITY_SCORE_NO}` |
| 사이트 상수 | 빌드 설정 | 사이트 오리진 `https://jobcho.wiki`, 기본 OG 이미지 경로, AdSense 플레이스홀더 client id(§8, NFR22) |

**출력·상태**

- 산출물: `company/{slug}/index.html` 형태의 정적 파일 ~96개(경로 규약 FR-51) + `sitemap.xml`·`robots.txt`(FR-56).
- 각 파일은 자기완결적 HTML로, 외부 JS 없이 본문이 렌더된다. 서버·DB 상태 변화 0건.

**검증·비즈니스 규칙**

1. 생성 개수 = `TCOMPANY` 등록 행 수(≈96, D2.6). 누락·초과 0(FR-59가 검증).
2. 각 페이지 본문(기업정보·복지·근무형태)이 초기 HTML에 존재해야 한다(비-JS 렌더, NFR12·NFR24). JS 무효화 상태에서 본문 가독성 검증(§10 생성물 검증 테스트).
3. 정적 경로 응답에 백엔드 호출·DB 쿼리 0건(NFR1).
4. 산출물·저장소에 시크릿(DB 자격증명·PAT·배포 토큰) 미포함, AdSense client id는 플레이스홀더(NFR22).

**추적**: UC-40·UC-44 / F4 / NFR1·NFR12·NFR24·NFR29·NFR22 / §2-7·§8·D2.3·D2.6·D4.2.

## [FR-51] 회사 상세 URL·slug 규칙(영문 식별자 기반)

**설명**: 각 회사 상세 페이지의 URL·파일 경로를 회사 영문 식별자(`COMP_ENG_NM`)에서 파생한 안정적 ASCII slug로 확정한다. 한글 정식명은 URL 경로가 아니라 페이지 콘텐츠(제목·본문·메타)에 사용한다.

**상세 동작**

- 정상 흐름
  1. slug는 `COMP_ENG_NM`(영문식별자, unique, D1.2)에서 파생한다: 소문자화 → 영숫자 외 문자를 하이픈으로 치환 → 연속 하이픈 축약 → 양끝 하이픈 제거.
  2. URL = `https://jobcho.wiki/company/{slug}`, 파일 경로 = `company/{slug}/index.html`. 이 URL이 canonical(FR-55)·sitemap 엔트리(FR-56)·CTA 없는 직접 진입 대상이 된다.
  3. 한글 정식명(`COMP_NM`)은 `<h1>`·`<title>`·`description`·OG·JSON-LD의 텍스트 값으로 쓰되(FR-52·FR-55), URL 경로에는 넣지 않는다(퍼센트 인코딩·공유·크롤 호환성 확보).
- 대안/예외
  - 별칭(한글/영문, `TCOMPANY_ALIAS`)은 URL을 만들지 않는다. 검색 매칭·`alternateName`(FR-55) 용도로만 쓴다.
  - slug 충돌(서로 다른 `COMP_ENG_NM`이 동일 slug로 축약)·빈 slug는 **생성 실패**로 처리하고 데이터 수준에서 `COMP_ENG_NM`을 교정해야 한다(FR-59가 빌드 실패로 검출).

**slug 파생(의사코드)**

```
def slug_of(comp_eng_nm: str) -> str:
    s = comp_eng_nm.strip().lower()
    s = re.sub(r'[^a-z0-9]+', '-', s)   # 비영숫자(공백·기호) → 하이픈
    s = re.sub(r'-{2,}', '-', s)        # 연속 하이픈 축약
    s = s.strip('-')                    # 양끝 하이픈 제거
    if not s:                           # 빈 slug 금지
        raise BuildError('empty slug for %r' % comp_eng_nm)
    return s
# 전역: len({slug_of(c.eng) for c in companies}) == len(companies)  (충돌 0, FR-59)
```

**입력**

| 필드 | 타입 | 제약 |
| --- | --- | --- |
| `COMP_ENG_NM` | 문자열 | unique(D1.2). ASCII 영숫자·하이픈으로 축약 가능해야 함 |
| `COMP_NM` | 문자열 | unique. 콘텐츠 텍스트 전용(URL 미사용) |

**출력·상태**

- slug 문자열(정규식 `^[a-z0-9]+(-[a-z0-9]+)*$`)과 그로부터의 URL·파일 경로. 회사당 정확히 1개.

**검증·비즈니스 규칙**

1. slug는 `COMP_ENG_NM`에서 결정적으로 파생되며 회사 간 유일하다(충돌 0, FR-59).
2. 재빌드 간 동일 `COMP_ENG_NM`은 동일 slug를 산출한다(안정 URL, 색인 유지). `COMP_ENG_NM`을 바꾸면 URL이 바뀌므로 데이터 변경 시 주의한다.
3. slug 패턴 위반·빈 값·충돌은 빌드 실패(FR-59).

**추적**: UC-40 / F4 / NFR11·NFR12 / §4·D1.2·D1.3 / FR-55(canonical)·FR-56(sitemap).

## [FR-52] 기업정보·근무형태 본문 구성

**설명**: 페이지 본문 상단에 기업정보 섹션(정식명·산업·기업유형·로고·성장 라벨·고용안정성)과 근무형태 섹션(`WORK_STYLE_VAL` 플래그)을 JS 없이 렌더된 정적 마크업으로 구성한다.

**상세 동작**

- 정상 흐름
  1. **기업정보 섹션**: `<h1>`에 정식명(`COMP_NM`), 이어서 산업(`INDUSTRY_NM`)·기업유형명(`COMP_TP_NM`)·로고 표식(`LOGO_NM`)을 표시한다. 기업유형 특성값이 있으면 성장 라벨(`GROWTH_LABEL_NM`)·고용안정성 점수(`STABILITY_SCORE_NO`, 1~100)를 함께 노출한다(FR-D2·FR-D4, UC-41 1).
  2. **근무형태 섹션**: `WORK_STYLE_VAL`(JSON)의 5개 키(remote·flex·unlimitedPTO·refreshLeave·overtime, D1.6)를 사람이 읽는 라벨로 표기한다.
- 대안/예외
  - 기업유형 특성값 미보유(1a): 성장 라벨·안정성 점수를 생략하고 정식명·산업·유형·로고만 표시한다(허위 표기 금지, UC-41 1a).
  - `WORK_STYLE_VAL`의 특정 키가 없거나 `false`(UC-41 4a): 해당 근무형태를 "미해당/미제공"으로 표기하거나 노출하지 않는다. 근거 없는 "제공"으로 표기하지 않는다(허위 표기 금지).
  - `LOGO_NM`은 로고 약어(§4)이므로 텍스트/모노그램 표식으로 렌더한다. 존재하지 않는 로고 이미지 URL을 생성하지 않는다.

**근무형태 키 표시 규약(WORK_STYLE_VAL, D1.6)**

| 키 | 의미(라벨) | true | false/누락 |
| --- | --- | --- | --- |
| `remote` | 재택근무 | "재택근무 제공" | 미표기/"미제공" |
| `flex` | 유연근무 | "유연근무 제공" | 미표기/"미제공" |
| `unlimitedPTO` | 무제한 휴가 | "무제한 휴가" | 미표기 |
| `refreshLeave` | 리프레시 휴가 | "리프레시 휴가" | 미표기 |
| `overtime` | 야근(잦음) | "야근 있음" 고지 | 미표기 |

**입력**: `COMP_NM`·`INDUSTRY_NM`·`LOGO_NM`(TCOMPANY), `COMP_TP_NM`·`GROWTH_LABEL_NM`·`STABILITY_SCORE_NO`(TCOMPANY_TYPE, FR-D2), `WORK_STYLE_VAL`(JSON, D1.6). 모든 문자열은 이스케이프 대상(FR-54 규약·NFR21).

**출력·상태**: 기업정보·근무형태 섹션의 정적 HTML 마크업(단일 `<h1>`, 시맨틱 위계). 서버 상태 변화 없음.

**검증·비즈니스 규칙**

1. `<h1>`은 페이지당 1개(회사 정식명)이며 heading 위계가 논리적이다(NFR13).
2. 근무형태는 데이터 근거(`WORK_STYLE_VAL`)가 있는 값만 "제공"으로 표기한다(허위 표기 금지, UC-41 4a).
3. 미보유 특성값은 생략하고 페이지를 렌더 실패시키지 않는다(UC-41 1a).
4. 본문은 비-JS 상태에서 표시된다(NFR24).

**추적**: UC-41 / F4 / NFR13·NFR15·NFR24 / §4·D1.6·FR-D2·FR-D4.

## [FR-53] 복지 섹션(카테고리별 복지표·금액·정성) 구성

**설명**: 회사 복지를 9개 카테고리(D1.7)로 그룹화한 복지표로 구성한다. 회사페이지는 복지 실데이터가 있는 회사만 생성하므로 등록 회사의 `TCOMPANY_BENEFIT` 실데이터로 항목을 채운다(프리셋 폴백 불필요). 각 항목은 명칭과 금액(또는 정성 설명)을 표시한다.

**상세 동작**

- 정상 흐름
  1. 회사 실데이터 복지 목록(FR-D5)을 카테고리(`BENEFIT_CTGR_CD` 9종: compensation/flexibility/work_env/time_off/health/family/growth/leisure/perks, D1.7)로 그룹화하여 `<table>`(또는 카테고리별 목록)로 렌더한다(UC-41 2).
  2. 항목별로 명칭(`BENEFIT_NM`)과, 금액형은 금액(`BENEFIT_AMT`, 만원 → "만/억" 포맷, FR-04), 정성형(`QUAL_YN=Y`)은 금액 대신 정성 설명(`QUAL_DESC_CTNT`)을 표시한다(UC-41 3, FR-D5).
  3. 각 항목에는 배지·출처·검증/만료 표식(FR-54)이 결합되어 노출된다.
  4. 정렬은 `SORT_ORDER_NO`(있으면)를 따르고, 없으면 카테고리 순서를 따른다.
- 대안/예외
  - 회사페이지는 복지 실데이터가 있는 회사(~96개)만 생성하므로, 실데이터 미보유로 인한 프리셋 폴백 케이스는 발생하지 않는다(등록 정책, 2026-07-09). `TBENEFIT_PRESET`는 비교 툴 직접 입력 모드(UC-15/21) 전용이다.
  - 금액이 정성적이라 NULL(UC-41 3a, `BENEFIT_AMT` NULL): 금액 표기를 생략하고 정성 설명(`QUAL_DESC_CTNT`)·비고(`NOTE_CTNT`)로 대체한다.
  - 등록 회사는 복지 실데이터를 보유하므로 복지 목록이 비지 않는다(회사페이지는 복지 있는 회사만 생성). 프리셋이 비는 유형(예: freelance)의 처리는 비교 툴 직접 입력 모드(UC-15/21) 소관이며 회사페이지에는 등장하지 않는다.

**입력(복지 항목, FR-D5)**

| 필드 | 타입 | 표시 규칙 |
| --- | --- | --- |
| `BENEFIT_NM` | 문자열 | 항목명(이스케이프) |
| `BENEFIT_AMT` | 정수(만원)\|null | 금액형은 "만/억" 포맷, 정성형은 생략(FR-04) |
| `BENEFIT_CTGR_CD` | 문자열(9종) | 카테고리 그룹 키(D1.7) |
| `QUAL_YN`/`QUAL_DESC_CTNT` | 여부/문자열 | 정성 여부·설명(금액 대체) |
| `NOTE_CTNT` | 문자열\|null | 비고(이스케이프) |
| `SORT_ORDER_NO` | 정수\|null | 정렬 |
| (amt_source) | real | 회사페이지는 실데이터만 사용(프리셋 폴백 없음, FR-54) |

**출력·상태**: 카테고리별 복지표의 정적 HTML(`<table>` 시맨틱). 비어있지 않은 복지 목록(실데이터, D2.6). 서버 상태 변화 없음.

**검증·비즈니스 규칙**

1. 복지 데이터 소스·표시 규약은 `companies/{id}`(FR-D7)와 동일한 실데이터 기준이다(회사페이지 프리셋 폴백 없음; `TBENEFIT_PRESET`는 비교 툴 직접 입력 모드 전용).
2. 등록 ~96개 회사 각각이 실데이터로 **비어있지 않은 복지 목록**을 갖는다(등록 정책상 모두 실복지 보유, D2.6, FR-59가 검증).
3. 정성 복지는 금액 없이 설명만 표시한다(FR-04·FR-D5).
4. 모든 복지 문자열은 이스케이프하여 삽입한다(NFR21, FR-54 규약).
5. 복지 본문은 비-JS 상태에서 표시된다(NFR24).

**추적**: UC-41(복지 확인)·UC-41 2a·3a / F4 / NFR13·NFR21·NFR24 / §4·§5·D1.7·D2.5·D2.6·FR-06·FR-D5.

## [FR-54] 복지 배지·출처·검증/만료·데이터정확성 면책 표시

**설명**: 각 복지 항목에 신뢰도 배지(공식/추정/만료)와 출처·검증/만료 정보를 시각·텍스트로 구분해 표시하고, 페이지에 데이터 정확성 면책 고지를 노출한다(F7 면책조항과 정합).

**상세 동작**

- 정상 흐름
  1. 배지 상태를 전역 규약(FR-05·FR-D9)으로 파생한다: `BADGE_CD='official'`이고 `EXPIRES_DTM` 미도래면 **공식(official)**, `EXPIRES_DTM` 경과면 **만료(stale)**, 그 외/`est`는 **추정(est)**. 배지는 색상 + 텍스트 라벨(예: "공식 확인"/"추정"/"만료·재확인 필요")로 표기한다(UC-42 1·2, NFR15).
  2. 항목에 출처(`BADGE_SRC_CD`: scrape_official/scrape_fallback/ai_parse/manual/user_report)·출처 URL(`BADGE_SRC_URL_CTNT`)이 있으면 출처를 확인할 수 있게 링크로 렌더한다(UC-42 3).
  3. 검증일(`VERIFIED_DTM`)·만료일(`EXPIRES_DTM`)이 있으면 신선도/만료 여부를 표시한다(UC-42 4, D4.3).
  4. 페이지에 **데이터 정확성 면책 고지**를 노출한다: 복지·연봉·근무조건 데이터는 참고용이며 실제와 다를 수 있고 최종 확인 책임은 사용자에게 있음을 고지하고, 정책 페이지(F7 면책조항)로 연결한다(D4.1).
- 대안/예외
  - 회사페이지는 실데이터가 있는 회사만 생성하므로 프리셋 폴백 항목은 등장하지 않는다(등록 정책). 프리셋 배지 표기는 비교 툴 직접 입력 모드(UC-15/21) 소관이다.
  - 출처 URL 없음(UC-42 3a): 출처 링크 없이 배지만 표시한다.
  - 만료 경과(UC-42 4a, `EXPIRES_DTM` < 빌드시각): "만료·재확인 필요"로 표기(정확성 한계 고지, D4.3·F7 정합).

**배지 파생·표시(FR-05·FR-D9 인용)**

| 파생 상태 | 조건 | 표시 라벨(텍스트) |
| --- | --- | --- |
| 공식(official) | `BADGE_CD=official` ∧ 미만료 | "공식 확인" |
| 만료(stale) | `EXPIRES_DTM` 경과(배지 무관) | "만료·재확인 필요" |
| 추정(est) | 그 외 / `BADGE_CD=est` | "추정" |

**입력**: `BADGE_CD`·`BADGE_SRC_CD`·`BADGE_SRC_URL_CTNT`·`VERIFIED_DTM`·`EXPIRES_DTM`(FR-D9·D3), 빌드 기준시각(만료 판정용), 면책 문구·정책 페이지 URL(F7).

**출력·상태**: 항목별 배지 라벨·(있으면) 출처 링크·검증/만료 표식 + 페이지 데이터정확성 면책 블록. 정적 HTML.

**검증·비즈니스 규칙**

1. 배지 파생 규약은 FR-05/FR-D9와 동일해야 한다(공식/만료/추정 3파생, 배지 밴드 계수는 비교 계산 소유라 여기서 재정의하지 않음).
2. 배지는 색상만이 아니라 텍스트로도 의미를 전달한다(NFR15).
3. 출처 URL(`BADGE_SRC_URL_CTNT`)은 `http`/`https` 스킴으로 제한하고 속성값을 이스케이프한다(NFR21).
4. 면책 고지는 모든 회사 페이지에 노출되며 F7 면책조항과 모순되지 않는다(D4.1).

**추적**: UC-42 / F4(F7 면책 정합) / NFR15·NFR21 / D3·D4.1·D4.3·FR-05·FR-D9.

## [FR-55] SEO 메타데이터·구조화 데이터

**설명**: 각 회사 상세 페이지 `<head>`에 회사별 고유 `<title>`·`meta description`, OpenGraph 태그, JSON-LD Organization 구조화 데이터, 자기참조 canonical을 포함한다. SEO는 수익의 뼈대다(§2-7).

**상세 동작**

- 정상 흐름
  1. **고유 title/description**: 회사 정식명(`COMP_NM`)·산업(`INDUSTRY_NM`)을 반영해 페이지마다 중복 없는 `<title>`과 `meta description`을 생성한다(NFR6). 예: title `"{COMP_NM} 복지·연봉·근무조건 | jobcho.wiki"`, description은 유형·산업·대표 복지 요약(정적 텍스트).
  2. **OpenGraph**: `og:title`·`og:description`·`og:type`(=`website`)·`og:url`(= canonical URL)·`og:image`(사이트 기본 공유 이미지 경로)를 포함한다(NFR7). 회사별 로고 이미지 자산이 없으므로 `og:image`는 **사이트 기본 OG 이미지**를 사용하며, 존재하지 않는 회사 이미지 URL을 창작하지 않는다(§4 `LOGO_NM`=약어).
  3. **JSON-LD Organization**(NFR8·§4 TCOMPANY): `@type=Organization`, `name=COMP_NM`, `url=canonical URL`, `alternateName=[COMP_ENG_NM, 별칭…]`, `industry=INDUSTRY_NM`. 데이터에 없는 필드(로고 이미지·주소·전화 등)는 넣지 않는다(파싱 오류 0·창작 금지).
  4. **자기참조 canonical**: `<link rel="canonical" href="{canonical URL}">`로 자신의 URL(FR-51)을 가리킨다(NFR11).
- 대안/예외
  - 별칭 없음: `alternateName`은 `COMP_ENG_NM`만 포함하거나 생략한다.
  - description 길이 초과: 검색 스니펫 권장 길이로 절단하되 회사명은 유지한다.

**JSON-LD 예시(형태 참고)**

```json
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "<COMP_NM>",
  "url": "https://jobcho.wiki/company/<slug>",
  "alternateName": ["<COMP_ENG_NM>", "<alias1>", "…"],
  "industry": "<INDUSTRY_NM>"
}
```

**입력**: `COMP_NM`·`COMP_ENG_NM`·`INDUSTRY_NM`·별칭(`TCOMPANY_ALIAS`), slug/canonical URL(FR-51), 사이트 오리진·기본 OG 이미지 경로(빌드 상수).

**출력·상태**: `<head>` 내 title·description·OG·JSON-LD(`<script type="application/ld+json">`)·canonical 태그. 페이지별 고유.

**검증·비즈니스 규칙**

1. title·description은 페이지 간 중복이 없고 회사명을 반영한다(NFR6, FR-59가 중복 0 검증).
2. `og:url`·canonical은 동일한 자기 URL(FR-51)을 가리킨다(NFR11).
3. JSON-LD는 구조화 데이터 테스트를 통과하고 파싱 오류 0이며, 데이터에 없는 필드를 창작하지 않는다(NFR8).
4. 메타·구조화 데이터의 문자열은 이스케이프·유효 JSON으로 삽입한다(NFR21).

**추적**: UC-40·UC-44 / F4 / NFR6·NFR7·NFR8·NFR11 / §2-7·§4 / FR-51(canonical URL).

## [FR-56] `sitemap.xml`·`robots.txt` 등록

**설명**: 빌드타임에 회사 상세 ~96개 URL을 포함한 `sitemap.xml`을 생성(회사 URL 기여)하고, 사이트 루트에 `robots.txt`를 제공하며 sitemap 위치를 명시한다.

**상세 동작**

- 정상 흐름
  1. 각 회사의 canonical URL(FR-51)을 `sitemap.xml`의 `<url><loc>` 엔트리로 등재한다(NFR9). `lastmod`는 빌드/재시드 시각을 사용한다.
  2. `sitemap.xml`은 회사 ~96개 + (다른 문서가 소유하는) 조합·정책·랜딩 URL을 함께 담는 **사이트 전체 sitemap**의 일부다. 본 문서는 **회사 URL ~96개 엔트리의 완전성**을 보장하고(누락 0), 조합/정책 URL은 각 담당(F5/F7)이 기여한다.
  3. `robots.txt`는 사이트 루트에 배치하며 크롤링 정책 + `Sitemap: https://jobcho.wiki/sitemap.xml` 라인을 포함한다(NFR10).
- 대안/예외
  - 생성 대상 외(등록 ~96개 밖) 회사: sitemap에 등재하지 않는다(존재하지 않는 URL, FR-59·UC-44 2a).

**입력**: 회사 slug/canonical URL ~96개(FR-51), 빌드 시각, 사이트 오리진.

**출력·상태**: `sitemap.xml`(회사 ~96개 URL 포함)·`robots.txt`(Sitemap 라인 포함). 정적 파일.

**검증·비즈니스 규칙**

1. 회사 ~96개 URL 누락 0(생성물 검증 테스트, NFR9·§10, FR-59).
2. sitemap의 회사 URL은 canonical URL(FR-51)과 정확히 일치한다(NFR11 정합).
3. `robots.txt`에 `Sitemap:` 라인이 존재하고 유효 위치를 가리킨다(NFR10).
4. 콘텐츠는 HTTPS(jobcho.wiki)로 서빙된다(NFR12·NFR23).

**추적**: UC-44 / F4 / NFR9·NFR10·NFR12 / §2-7·§8 / FR-51·FR-59.

## [FR-57] "이 회사로 비교하기" CTA·비교 툴 프리필 파라미터

**설명**: 각 회사 상세 페이지에 "이 회사로 비교하기" CTA를 배치하고, 클릭 시 비교 툴(SPA, F2)로 해당 회사를 프리필하는 URL 파라미터를 정의한다. CTA는 JS 없이도 동작하는 순수 링크다.

**상세 동작**

- 정상 흐름
  1. CTA는 순수 `<a href>` 링크로 비교 툴 진입 URL을 가리킨다: `/{비교툴경로}?prefill={COMP_ENG_NM}&slot=a`. JS 실행 없이도 이동 가능하다(NFR24, UC-44 4a와 정합).
  2. 비교 툴이 진입하면 참조 번들(REF, `reference/all`, FR-02)을 로드하고, `prefill` 값(`COMP_ENG_NM`)으로 `REF.companies`에서 회사를 해석해 지정 슬롯(기본 `a`=현직)에 프리필한다(UC-43 2·3). 프리필 대상은 등록 회사이므로 실복지 실데이터를 사용한다(프리셋 폴백 무관).
  3. 프리필된 회사의 유형·산업·복지(실데이터)가 비교 입력(F2)의 초기값으로 채워진다(UC-43 4). 프리필 대상은 등록 회사이므로 실데이터를 보유하며, 비교 툴의 프리셋 기반 직접 입력은 사용자가 등록 외 회사를 입력할 때만 적용된다(UC-15/21).
- 대안/예외
  - `slot` 파라미터가 `b`면 이직처 슬롯에 프리필한다(허용 값 `a`/`b`, 기본 `a`).
  - `prefill` 값이 `REF`에서 해석되지 않음: 프리필을 건너뛰고 비교 툴을 정상 진입시킨다(무크래시, FR-9x·FR-06). 상세 페이지의 프리필은 초기값일 뿐 고정이 아니다(UC-43 5a).
  - 참조 번들 로드 실패(UC-43 3a): 비교 툴이 재시도·오류 안내를 표시(FR-02·FR-9x).

**프리필 파라미터 계약**

| 파라미터 | 값 | 기본 | 의미 |
| --- | --- | --- | --- |
| `prefill` | `COMP_ENG_NM`(FR-51 소스) | (필수) | 프리필 대상 회사 식별자. `REF.companies[].comp_eng_nm`로 해석 |
| `slot` | `a` \| `b` | `a` | 프리필 슬롯(현직 a / 이직처 b) |

**입력**: 회사 `COMP_ENG_NM`(빌드타임), 비교 툴 진입 경로(빌드 상수).

**출력·상태**: 정적 페이지 내 CTA 링크(URL 파라미터 포함). 클릭 시 상태는 URL 파라미터·클라이언트(REF·`matched`)에만 존재하며 서버로 전송·저장되지 않는다(NFR16, §2-2).

**검증·비즈니스 규칙**

1. CTA는 JS 없이 이동 가능한 순수 링크다(NFR24). 본문 색인에 CTA가 영향을 주지 않는다(UC-44).
2. `prefill`은 `COMP_ENG_NM`(URL slug와 동일 소스, FR-51)이며 비교 툴이 `REF`로 해석한다(FR-02·FR-D4).
3. 프리필 실패는 비교 툴을 크래시시키지 않고 프리필만 생략한다(FR-06·FR-9x, UC-43 4a·5a).
4. 파라미터·프리필 상태의 서버 전송·저장 0(NFR16, FR-01).

**추적**: UC-43 / F4(F2 비교 입력 연동) / NFR16·NFR24 / §2-2·FR-02·FR-06·FR-D4 / 연계 UC-14(슬롯 지정)·UC-A1(참조 로드).

## [FR-58] 콘텐츠 위주 광고/제휴 슬롯 배치·"광고" 표기

**설명**: 회사 상세 페이지에 콘텐츠 위주 광고 배치 정책(§2-6)에 따라 AdSense·제휴(affiliate) 슬롯 자리를 배치하고, 모든 광고·제휴 요소에 "광고" 표기를 부착한다. 슬롯 콘텐츠·컴포넌트 로직은 F6(문서 07-광고·제휴·정책)이 소유하며 본 문서는 정적 페이지 내 배치·예약·표기만 규정한다.

**상세 동작**

- 정상 흐름
  1. 회사 상세는 콘텐츠 페이지이므로 광고 배치 대상이다(§2-6). 본문(기업정보·복지·근무형태)을 해치지 않는 위치(예: 섹션 사이·본문 하단)에 광고/제휴 슬롯 자리를 배치한다.
  2. 각 슬롯은 **고정 크기를 예약**하여 삽입 시 레이아웃 이동을 억제한다(CLS ≤ 0.1, NFR5).
  3. 모든 광고·제휴 요소에 "광고" 라벨을 노출한다(공정위 표시광고 규정, NFR19).
  4. AdSense 스니펫은 플레이스홀더 client id를 사용한다(NFR22). 광고는 클라이언트 스크립트로 주입되며 **본문 색인에 영향을 주지 않는다**(UC-44 4a).
- 대안/예외
  - JS/광고 스크립트 미실행(크롤러·JS 차단): 예약된 슬롯이 비어도 본문은 정적으로 표시·색인된다(NFR24, UC-44 3a·4a).
  - 광고 쿠키: 개인화 광고 쿠키 동의·개인정보처리방침은 F6/F7이 소유(NFR18), 본 문서는 슬롯 배치·표기만 규정.

**입력**: 광고 슬롯 규격(고정 크기), AdSense 플레이스홀더 client id, affiliate 컴포넌트 자리(F6 계약).

**출력·상태**: 정적 페이지 내 광고/제휴 슬롯 마크업(고정 크기 예약 + "광고" 표기). 본문과 분리.

**검증·비즈니스 규칙**

1. 모든 광고·제휴 요소에 "광고" 표기가 노출된다(NFR19).
2. 슬롯은 고정 크기를 예약해 CLS ≤ 0.1을 저해하지 않는다(NFR5).
3. 광고 스크립트 미실행 시에도 본문 표시·색인이 유지된다(NFR24, UC-44).
4. AdSense client id는 플레이스홀더이며 시크릿 미포함(NFR22).

**추적**: UC-40 / F4(F6 광고·제휴 연동) / NFR5·NFR19·NFR24 / §2-6 / 연계 NFR18(동의는 F6/F7 소유).

## [FR-59] 생성물 검증·404 정합·복지 완전성(엣지)

**설명**: 빌드타임 생성물의 정합성을 검증하는 규칙을 확정한다(생성 개수·slug 유일성·메타 중복·sitemap 완전성·비-JS 본문·복지 완전성). 생성 대상 외 회사 URL의 404 정합을 규정한다.

**상세 동작**

- 정상 흐름
  1. 생성 스크립트 완료 시 생성물 검증 테스트(§10)를 실행한다: (a) 페이지 개수 = 등록 회사 수(≈96, D2.6), (b) slug 유일·패턴 준수(FR-51), (c) 페이지별 title/description 중복 0(NFR6), (d) sitemap의 회사 URL 누락 0(NFR9), (e) 각 페이지 본문이 초기 HTML에 존재(비-JS, NFR12·NFR24), (f) JSON-LD 파싱 오류 0(NFR8), (g) 등록 ~96개 회사 각각 비어있지 않은 복지 목록(실데이터, 등록 정책상 모두 실복지 보유, D2.6).
  2. 하나라도 실패하면 빌드를 실패로 처리하고 배포하지 않는다.
- 대안/예외
  - 생성 대상 외(등록 ~96개 밖) 회사 URL 요청: 정적 파일이 없으므로 404로 처리한다. 해당 URL은 색인·열람·sitemap 대상에서 제외된다(UC-40 2a·UC-44 2a).
  - slug 충돌·빈 slug(FR-51): 빌드 실패로 검출(데이터 교정 필요).
  - 등록 회사는 복지 실데이터를 보유하므로 복지 목록이 비는 케이스는 회사페이지에 등장하지 않는다(회사페이지는 복지 있는 회사만 생성). 프리셋이 비는 freelance 등 케이스는 비교 툴 직접 입력 모드(UC-15/21) 소관이다(FR-53).

**입력**: 생성물(HTML ~96개·sitemap·robots), 회사 데이터(D2.6 기준값), 검증 기준(§10).

**출력·상태**: 검증 통과/실패 판정. 통과 시 배포 대상 확정, 실패 시 빌드 중단(무배포).

**검증·비즈니스 규칙(정량)**

| 검증 항목 | 기준 | 근거 |
| --- | --- | --- |
| 페이지 개수 | = 등록 회사 수(≈96) | D2.6·FR-50 |
| slug | 유일·`^[a-z0-9]+(-[a-z0-9]+)*$` | FR-51 |
| title/description 중복 | 0 | NFR6·FR-55 |
| sitemap 회사 URL 누락 | 0 | NFR9·FR-56 |
| 비-JS 본문 존재 | 전 페이지 | NFR12·NFR24 |
| JSON-LD 파싱 오류 | 0 | NFR8·FR-55 |
| 복지 목록 비어있지 않음 | ~96/~96(실데이터) | D2.6·FR-53 |

**추적**: UC-40 2a·UC-41 2a·UC-44 2a / F4 / NFR1·NFR6·NFR8·NFR9·NFR12·NFR24 / §10·D2.5·D2.6 / FR-50·FR-51·FR-53·FR-55·FR-56.
