# 01. 개요와 FR 규칙 (FRD)

**목적**: loupit FRD(기능 명세서) 문서군의 진입점으로서 (1) FRD의 목적·범위를 확정하고, (2) 기능 요구사항 식별자 `FR-ID`의 명명·대역 규칙을 정의하며, (3) `FR → UC → F(→ NFR)` 추적성 원칙을 세우고, (4) 모든 FRD 문서가 공유하는 **전역 용어**와 **전역 클라이언트 상태 모델**을 단일하게 정의한다. 본 문서는 FR-0x 대역의 **전역(횡단) 기능 요구사항**을 직접 소유한다.

**참조 상위문서**: PRD `05-기능-요구사항.md`(F1~F8), USECASE `USECASE.md`·`01-액터와-개요.md`(UC-ID 체계·UC-A1~UC-A5). 근거 브리프 섹션: §2(핵심 결정), §3(MVP 범위), §4(데이터 모델), §6(읽기 전용 API), §7(비교 알고리즘), §9(문서 규칙).

**상위 추적**: FRD → USECASE → PRD → 브리프. 본 문서의 각 FR은 자신이 충족을 돕는 UC-*와 상위 기능 F#를 인용하고, 관련 품질기준 NFR#(PRD `07-비기능-요구사항.md`)를 함께 참조한다. 하위 문서(FLOW/WIREFRAME/SPEC/TASK)는 본 문서가 확정한 FR-ID·용어·상태 모델을 그대로 인용한다.

**전역 전제(모든 FR 공통)**: 로그인/회원/계정/소셜피드 없음(§2-1, NFR16), 프로파일러(가치관 진단 설문) 영구 제외(§2-3), 서버 측 사용자 데이터 쓰기 없음(§2-4, NFR20). 사용자 상태는 브라우저 localStorage에만 저장(§2-2, NFR17). 비교 계산은 100% 클라이언트(§2-8). 따라서 어떤 FR에도 로그인/회원가입/프로파일러/서버 쓰기 기능은 존재하지 않는다.

## 1. FRD 목적과 범위

### 1.1 목적
FRD는 USECASE가 정의한 각 유즈케이스(UC-*)를 **충족시키기 위해 시스템이 제공해야 하는 기능**을 구현 이전 수준에서 정밀하게 명세한다. 각 FR은 "무엇을(What) 어떤 규칙으로" 제공하는지를 정의하며, "어떻게(How) 구현하는지"의 코드·파일 배치는 SPEC/TASK 단계로 미룬다. FRD는 UC의 개괄적 수용기준을 **입력 필드·타입·제약, 출력·상태, 검증·비즈니스 규칙**의 검증 가능한 명세로 구체화한다.

### 1.2 범위 (In-Scope)
FRD가 명세하는 대상은 PRD의 MVP 기능 F1~F8과 이를 지지하는 전역 규칙이다.

| 대상 | 상위 F | 소유 FRD 문서 |
| --- | --- | --- |
| 전역 규칙·용어·상태 모델(횡단) | 전체 | 01 (본 문서) |
| 데이터 계약(번들·응답 스키마·클라이언트 모델, 횡단) | F1~F8 지지 | 02 |
| 회사 검색 + 선택 | F1 | 03 |
| 비교 입력 | F2 | 04 |
| 비교 계산 엔진 | F3 | 05 |
| 리포트 표현·로컬 저장 | F3 | 06 |
| 회사 상세 페이지 ~96개(복지 데이터 보유 회사, 정적) | F4 | 07 |
| 인기 비교 조합 페이지(정적) | F5 | 08 |
| 광고/제휴 슬롯 | F6 | 09 |
| 정책·고지 페이지 | F7 | 10 |
| 읽기 전용 API(전송 계약) | F8 | 11 |
| 오류·엣지(횡단) | F1~F5·F8 | 12 |

### 1.3 범위 밖 (Out-of-Scope, 명세 금지)
아래는 제품 범위에서 제외이므로 어떤 FR에도 등장하지 않는다. FRD는 이들을 명세하지 않는다.

- 로그인/회원가입/OAuth/JWT/이메일 인증/회사 이메일 인증(§2-1).
- **프로파일러**(가치관 진단 설문 → 페르소나 산출) 및 그 상태·데이터·화면(§2-3, 영구 제외).
- 서버 측 사용자 데이터 쓰기(비교 저장·신고·피드·통계 집계 등)·쓰기 엔드포인트·인증 미들웨어(§2-4, NFR20). *(개정 2026-07-14: "실시간 비교 TOP 10"용 익명 회사쌍 비교 로그 1종은 예외 — FRD 11 §API 표면 개정 참조. 사용자 식별자·입력값 저장은 계속 금지.)*
- 관리자 화면(복지 승격 등) 및 그에 딸린 쓰기 흐름.
- (2차 이후) 복지 카테고리 가이드 콘텐츠, 회사 랭킹/리스트 페이지, 사용자 계정·저장(§3).

## 2. FR-ID 명명 규칙

### 2.1 형식
기능 요구사항 식별자는 두 형식을 쓴다.
1. **숫자 대역** `FR-<두자리: 십단위 대역·일련번호>`(예: `FR-01`, `FR-13`, `FR-90`) — 앞자리는 **십단위 대역**으로 담당 문서·상위 기능을 지시하고 뒷자리는 대역 내 일련번호다.
2. **letter 대역** `FR-<대문자 letter·일련번호>`(예: `FR-D1`, `FR-E1`) — 특정 상위 F 하나에 귀속되지 않고 여러 기능을 가로지르는 **횡단 계약·규약 대역**에 부여한다. 현재 letter 대역은 **FR-Dx**(데이터 계약, 문서 02)와 **FR-Ex**(오류·엣지, 문서 12) 둘이며, 이 형식은 정식으로 인정된다.

본 문서(01)는 **FR-0x 대역(전역·횡단 규칙)**을 소유한다.

### 2.2 규칙
1. **접두사 `FR-`**: FRD 항목임을 표시하며, 상위 식별자 `F1`~`F8`(PRD 기능), `UC-*`(유즈케이스), `NFR*`(비기능)와 명확히 구분한다.
2. **대역 = 담당 문서(1:1)**: 각 FRD 상세 문서가 자기 대역(숫자 십단위 또는 letter)을 1:1로 소유한다(2.3 표). FR-ID 앞부분만 보고 담당 문서와 상위 기능을 추적할 수 있다. 숫자 대역은 USECASE의 `UC-<십단위>` 체계와 대체로 **정렬**되나(FR-1x↔UC-1x=F1, FR-2x↔UC-2x=F2, FR-3x↔UC-3x=F3), **F3이 계산 엔진(05, FR-3x)과 표현·로컬저장(06, FR-4x) 두 문서로 분할**되면서 그 이후 대역은 한 칸씩 밀린다(FR-5x=F4↔UC-4x, FR-6x=F5↔UC-5x, FR-7x=F6·FR-8x=F7↔UC-6x, FR-9x=F8, FR-Ex↔UC-9x). 따라서 자릿수 정렬은 참고 지표일 뿐이며, **각 FR이 명시 인용하는 UC가 권위 있는 연결**이다(§3.2-1).
3. **일련번호는 대역 내 정수**로 `x0`(또는 `x1`)부터 증가한다. 담당 문서가 부여한다.
4. **안정 ID·중복 금지·재사용 금지**(브리프 §9): 한 번 부여한 FR-ID는 의미가 바뀌어도 재사용하지 않는다. 폐기 시 번호를 비우고 새 번호를 부여한다.
5. **F8은 두 문서로 계약된다**: USECASE에서 F8은 전용 십단위 대역 없이 소비 맥락별 UC로 흡수되지만(USECASE 01 §4.1), FRD는 API 계약을 구현 가능한 수준으로 확정해야 하므로, **응답 본문(payload) 스키마는 FR-Dx(문서 02 데이터 계약)**가, **엔드포인트 전송 계약(메서드·경로·상태코드·캐시·CORS)은 FR-9x(문서 11 읽기 전용 API)**가 소유한다(중복 정의 금지). FR-9x의 각 항목은 자신을 소비·검증하는 UC(UC-A1·UC-A5·UC-10~12 흡수·UC-90·UC-91·UC-93·UC-95)를 역참조한다.
6. **F6·F7은 분리 대역이다**: 광고·제휴(F6)는 **FR-7x(문서 09)**, 정책·고지 페이지(F7)는 **FR-8x(문서 10)**가 각각 소유한다(문서 분리에 따라 대역도 분리). 앞선 초안이 예약(미사용)으로 두었던 FR-7x·FR-8x는 이로써 실제 배정되었으며, 예약 상태의 대역은 없다.

### 2.3 대역 배정

| 대역 | 영역 | 상위 F | 정렬되는 UC 대역 | 담당 문서 |

## 3. FR → UC → F (→ NFR) 추적성 원칙

### 3.1 추적 방향
추적 사슬은 상향(근거)·하향(구체화) 양방향이다.

```
브리프(단일 진실 소스)
  └─ PRD: 기능 F1~F8 / 비기능 NFR1~NFR29
       └─ USECASE: UC-A1~A5, UC-10~UC-95  (USECASE→PRD)
            └─ FRD: FR-0x·FR-Dx·FR-1x…FR-9x·FR-Ex  (FRD→USECASE, 각 FR은 UC·F·NFR 인용)
                 └─ FLOW → WIREFRAME → SPEC → TASK
```

### 3.2 원칙
1. **모든 FR은 최소 하나의 UC와 하나의 F를 인용**한다(추적 필드 필수). UC는 "이 기능이 어느 사용자 상호작용을 충족하는가", F는 "어느 상위 기능에 속하는가"를 고정한다.
2. **품질 기준은 NFR로 인용**한다. 성능·SEO·접근성·개인정보·보안·PE·자원·한국어 요구는 각 FR이 관련 NFR#(예: NFR16·NFR17·NFR20·NFR25·NFR26)를 참조하여 검증 조건을 상속한다. FRD는 NFR을 재정의하지 않고 인용만 한다.
3. **커버리지 완전성**: 모든 UC(UC-A1~UC-95)는 하나 이상의 FR로 커버되어야 한다. F8의 4개 엔드포인트는 FR-8x가 소비 UC와 함께 커버한다(USECASE 01 §4.1의 흡수 매핑 준수).
4. **역참조 일관성**: 하위 문서가 FR-ID를 인용할 때, 해당 FR이 인용한 UC·F·NFR과 모순되면 안 된다(예: F3 계산 FR이 "서버 판정 호출"을 요구하면 §2-8·NFR20 위반 — 금지).
5. **인용 표기**: 추적 필드는 `UC-*`, `F#`, `NFR#`, 필요 시 `§브리프섹션`을 나열한다.

### 3.3 예시(형식 참고)
> **FR-32**(문서 05 소유) 실효연봉(총보상) 산출 — 충족 UC-31, 상위 F3, 관련 NFR26(로드 후 오프라인 계산 가능)·§7(실효연봉=연봉+순복지).

## 4. 전역 용어 정의 (Glossary)

모든 FRD 문서는 아래 용어를 동일한 의미로 사용한다. 코드 식별자는 영어(§9), UI·설명은 한국어다.

| 용어 | 정의 | 근거 |
| --- | --- | --- |
| **현직 a / 슬롯 a** | 비교의 기준이 되는 현재 직장. 상태에서 접미/키 `a`로 표기. | §7, PRD F1 |
| **이직처 b / 슬롯 b** | 비교 대상인 이직 후보 직장. 키 `b`로 표기. | §7, PRD F1 |
| **슬롯(slot)** | 회사·조건을 담는 두 자리(a/b). 각 슬롯은 회사 선택(선택적)과 직접 입력 값을 가진다. | UC-14·UC-15 |
| **만원 단위** | 모든 금액 값(연봉·복지 금액·계산 결과)의 기본 단위는 **만원**이다. 예: `benefit_amt=360` → 360만원. 데이터 모델 금액 컬럼과 동일 단위(§4). | §4, PRD F2 |
| **실효연봉(총보상)** | 연봉 + 순복지(금전 환산 복지 합). 명목 연봉과 구분되는 실질 보상. | §7, F3 |
| **순복지(net benefit)** | 체크된 복지 중 **금전 환산 가능**(정성 아님) 항목의 금액 합. 정성 복지는 제외. | §7, `getBenTotal` |
| **정성 복지(qual)** | 금액으로 환산하기 어려운 복지. 상태 필드 `qual_yn=true`이며 금액(`benefit_amt`)이 없다(NULL). 총보상 합산에서 제외되고 별도 비교로 표시. | §4(QUAL_YN), §7 |
| **복지 카테고리(9종)** | `compensation`/`flexibility`/`work_env`/`time_off`/`health`/`family`/`growth`/`leisure`/`perks`. 복지 항목 분류 기준. | §4(BENEFIT_CTGR_CD) |
| **배지(badge)** | 복지 금액의 신뢰 상태 표시. 파생 상태 3종: **공식**(official, 유효)·**만료**(stale, 공식이었으나 만료)·**추정**(est). `badge_cd`(official/est)와 `expires_dtm` 유효성으로 결정. | §4(BADGE_CD), §7 |
| **불확실성 밴드(band)** | 복지 금액의 **금액 신뢰도**(02 FR-D9의 `amt_source`=stated/estimated/none)에 적용하는 ± 오차 범위. **명시 금액(stated) ±5% / 추정 금액(estimated) ±20% / 만료(`expires_dtm` 경과) +15% 가산**. 출처 배지(공식/추정)와 **무관**하며 `official` 배지라도 금액이 추정이면 ±20%를 유지한다(DEC-2 디커플링). 체크된 금전 복지의 밴드 절대값을 합산해 총보상 신뢰 범위로 제시. | §7, RESEARCH §4.2·DEC-2, 02 FR-D9·05 FR-38 |
| **amt_source(금액 출처·provenance)** | 각 복지 항목 금액이 **어디서 왔는가**의 provenance 표기. 값 영역: `real`(회사 실데이터, 배지 공식/추정 포함) · `preset`(기업유형 프리셋 폴백값) · `user`(사용자가 직접 입력·수정). 데이터 모델의 `BADGE_SRC_CD`(scrape_official/scrape_fallback/ai_parse/manual/user_report)와 폴백 규약(§5, UC-93)을 클라이언트 관점으로 요약한 마커. "실데이터 vs 프리셋 폴백 vs 사용자수정" 구분과 출처 표시의 근거가 된다. **주의**: 불확실성 밴드는 이 provenance가 아니라 **금액 신뢰도**(데이터 계약 02 FR-D9의 `amt_source`=stated/estimated/none)를 기준으로 하는 별개 축이다(DEC-2 디커플링, FR-05). | §4(BADGE_SRC_CD), §5, UC-93 |
| **프리셋 폴백(preset fallback)** | 비교 툴 **직접 입력 모드**에서 사용자가 등록 회사(복지 실데이터 보유, ~96개)에 없는 회사를 입력할 때 선택한 **기업유형 기본 복지**(TBENEFIT_PRESET)로 초기값을 채우는 규약. amt_source=`preset`. 등록 회사는 모두 실복지를 보유하므로 회사페이지에는 폴백이 불필요하다. | §4, UC-93·UC-15 |
| **참조 번들(REF)** | 부팅 시 `GET /api/v1/reference/all`로 1회 로드하는 전체 참조 데이터(회사·별칭·복지 인라인, 기업유형, 프리셋). 검색·입력·계산의 단일 참조원. | §6, UC-A1 |
| **우선순위(curPri)** | 사용자가 **직접 선택**하는 비교 판정 축. 값 영역: **연봉/워라밸/복지**. 프로파일러 산출이 아니라 사용자 선택이다(§2-3). | §3(F2), §7 |
| **희생요소(curSacrifice)** | 사용자가 감수하기로 직접 선택하는 축(우선순위와 다른 값, 선택 안 함 허용). 판정(vdCard) 입력. | §7, UC-25·UC-33 |
| **근무형태(wsState)** | 재택(remote)·유연(flex)·야근(ot)·임금유형(wage=포괄/비포괄) 4축의 슬롯별 선택 상태. 워라밸·야근수당 계산 입력. | §7, UC-23 |
| **판정 카드(vdCard)** | 우선순위·희생요소 기준으로 "누가 유리한가/무엇을 택할지"를 요약한 리포트 출력 요소. | §7, UC-33 |
| **"광고" 표기** | 모든 광고·제휴 영역에 부착하는 라벨(공정위 표시광고 규정 준수). | §2-5, NFR19 |

## 5. 전역 클라이언트 상태 변수 개요

비교 툴(F1~F3)은 서버 세션 없이 **클라이언트 메모리 상태 + localStorage**만으로 동작한다(§2-2·§2-8, NFR16·NFR17). 아래는 모든 FRD 문서가 공유하는 전역 상태 모델이다. 세부 조작(설정·검증)은 각 담당 FRD가 명세하고, 여기서는 **형태·소유·수명**만 확정한다.

### 5.1 프로파일러 상태 제거 (변경 금지)
프로파일러가 영구 제외(§2-3)되므로, 레거시(`/home/ubuntu/job_change/app.js`)에 존재하던 **프로파일러 계열 상태는 loupit 상태 모델에 존재하지 않는다.**

- 제거: `pfShuffled`, `pfCur`, `pfAnswers`, **`pfResult`**, `pfJob`(가치관 설문 진행·결과·직군 상태).
- 파급: 우선순위는 프로파일러 산출값이 아니라 **`curPri` 사용자 직접 선택**으로만 설정된다. 어떤 FR도 `pfResult` 등 프로파일러 상태를 읽거나 쓰지 않는다. 참조 번들(REF)에도 프로파일러 키(`profiles`/`job_groups`/`questions`)가 없다(§6, UC-A1).

### 5.2 지속성 규약
`localStorage` 지속성은 **로컬 한정·서버 미전송**이다(NFR16·NFR17). 저장 실패(사생활 모드·차단·용량 초과)는 try/catch로 흡수하며 비교 기능은 정상 동작하고 복원만 생략한다(NFR25, UC-92). 크로스 디바이스 동기화는 없다.

## 6. 전역 기능 요구사항 (FR-0x)

아래 FR-01~FR-08은 특정 화면 하나에 귀속되지 않고 여러 기능을 가로지르는 **전역 불변식·규약**이다. 하위 FR(FR-Dx·FR-1x~FR-9x·FR-Ex)은 이들을 위반할 수 없다.

### FR-01 전역 아키텍처 불변식(로그인·서버 쓰기·프로파일러 부재)
- **설명**: 모든 기능은 비로그인·서버 무쓰기·클라이언트 계산·읽기 전용 API라는 4대 제약 위에서 동작한다.
- **상세 동작**
  - 정상: 사용자는 계정·인증 없이 검색(F1)·입력(F2)·리포트(F3)를 수행하고, 백엔드는 GET 4종만 응답한다(FR-8x). 비교 계산은 브라우저에서 수행된다.
  - 대안/금지: 어떤 흐름도 로그인 게이트·회원가입·회사 이메일 인증·서버 저장/신고/피드/집계·프로파일러 설문을 트리거하지 않는다. 레거시의 해당 경로는 이식하지 않는다.
- **입력**: 없음(제약 자체).
- **출력·상태**: 서버에 사용자별 상태 0건. 클라이언트 상태만 존재(§5).
- **검증·규칙**: (1) API 표면은 `health`·`reference/all`·`companies/search`·`companies/{id}` GET로 한정, 쓰기 엔드포인트·인증 미들웨어 0(NFR20). (2) 비교 입력값의 서버 전송 0(네트워크 검증, NFR16). (3) 프로파일러 화면·상태·데이터 0(§2-3).
- **추적**: UC-A1·UC-A5·전 UC 전제, F1~F8, NFR16·NFR17·NFR20, §2-1·2-3·2-4·2-8.

### FR-02 참조 데이터 단일 소스 로드(부팅)
- **설명**: 비교 툴은 부팅 시 참조 번들(REF)을 1회 로드하여 검색·입력·계산의 단일 참조원으로 삼는다.
- **상세 동작**
  - 정상: SPA 진입 → `GET /api/v1/reference/all` 1회 호출 → 번들을 메모리에 보관 → 이후 검색(F1)·프리셋 폴백(F2)·계산(F3)이 REF만 참조(회사별 개별 API 재호출 없음).
  - 대안: 브라우저/프록시 캐시 유효 시 네트워크 재호출 없이 캐시 사용(NFR3). 로드 실패(네트워크·5xx) 시 재시도·오류 안내를 표시하고 참조 확보 전까지 검색·비교를 제한(FR-9x, UC-90).
- **입력**: 없음(부팅 트리거).
- **출력·상태**: `REF = {company_types[], benefit_presets{type_cd:[…]}, companies[{…, aliases[], benefits[]}]}`. 프로파일러 키 없음.
- **검증·규칙**: 응답 헤더 `Cache-Control: public, max-age=3600`(NFR3). 번들 최상위 키가 정확히 3종이고 `profiles`/`job_groups`/`questions` 부재. 로드 실패가 앱 크래시를 유발하지 않음(NFR26).
- **추적**: UC-A1, UC-90, F8 → F1·F2·F3, NFR3·NFR26, §6.

### FR-03 전역 클라이언트 상태 모델 준수
- **설명**: 모든 기능은 §5가 정의한 상태(`matched`/`benS`/`wsState`/`curPri`/`curSacrifice`/`selectedRate`/`REF`)만을 공용 상태로 사용한다.
- **상세 동작**
  - 정상: F1이 `matched`를, F2가 `benS`/`wsState`/`curPri`/`curSacrifice`/`selectedRate`를 채우고, F3이 이를 읽어 리포트를 산출한다.
  - 대안/금지: 프로파일러 상태(`pfResult` 등)를 도입·참조하지 않는다(§5.1). 상태 형태는 §5 표를 벗어날 수 없다.
- **입력**: F1·F2 사용자 조작.
- **출력·상태**: §5 표의 shape에 부합하는 상태 객체.
- **검증·규칙**: (1) `curPri`는 사용자 직접 선택으로만 설정(프로파일러 산출 금지). (2) `curSacrifice ≠ curPri` 또는 `null`. (3) 금액 필드 단위는 만원. (4) 정성 복지(`qual_yn=true`)는 `benefit_amt` 없음.
- **추적**: UC-A2·UC-20~UC-25·UC-30~UC-36, F2·F3, §2-3·§7.

### FR-04 금액·단위 전역 규약
- **설명**: 모든 금액 값과 그 출처(amt_source)의 표현 규칙을 통일한다.
- **상세 동작**
  - 정상: 연봉·복지 금액·계산 결과를 만원 단위 정수로 다루고 표시 시 "만/억"으로 포맷한다. 각 복지 금액에 amt_source(`real`/`preset`/`user`)를 부여한다.
  - 대안: 정성 복지는 금액 없이(`null`) 카테고리·설명만 보유. 사용자가 프리셋 값을 수정하면 해당 항목 amt_source가 `user`로 전이.
- **입력**: 회사 실데이터·프리셋·사용자 입력 금액.
- **출력·상태**: 만원 단위 정수 금액 + amt_source 마커.
- **검증·규칙**: (1) 단위는 데이터 모델 금액 컬럼(§4)과 동일한 만원. (2) 음수·비수치 입력 거부(구체 검증은 F2 FR). (3) provenance `amt_source`(real/preset/user)는 "실데이터 vs 프리셋 폴백 vs 사용자수정" 표시의 근거가 된다. 불확실성 밴드는 이와 별개로 **금액 신뢰도**(02 FR-D9 `amt_source`=stated/estimated/none)를 기준으로 한다(FR-05, DEC-2).
- **추적**: UC-20·UC-22·UC-32, F2·F3, §4·§7.

### FR-05 복지 배지·불확실성 밴드 전역 규약
- **설명**: 복지 금액의 신뢰도를 **두 개의 독립(디커플링) 축**으로 일관되게 표현한다 — (a) **출처 배지**: 이 회사가 이 복지를 제공한다는 사실의 확실성(공식/만료/추정), (b) **불확실성 밴드**: 금액 수치가 공식 명시값인지 앵커 추정값인지에 따른 ± 오차. 두 축은 **분리**되며 밴드는 배지가 아니라 **금액 신뢰도**를 기준으로 한다(DEC-2·RESEARCH §4.2).
- **상세 동작**
  - 배지(출처) 산출: `badge_cd='official'`이고 `expires_dtm` 미도래면 **공식**, 만료면 **만료(재확인 필요)**, 그 외/`est`는 **추정**.
  - 밴드(금액 신뢰도) 산출: 밴드는 출처 배지와 **무관**하게 **금액 신뢰도**(데이터 계약 02 FR-D9의 `amt_source`=`stated`/`estimated`/`none`)를 기준으로 정한다. 체크된 금전 복지에 **명시 금액(`stated`) ±5% / 추정 금액(`estimated`) ±20%**를 적용하고, 만료(`expires_dtm` 경과) 항목은 기준 계수에 **+15% 가산 확대**한다. 항목별 밴드 절대값을 합산해 총보상 신뢰 범위(±만원)를 제시한다.
  - **디커플링 핵심**: 96개 큐레이션 복지처럼 출처 배지가 `official`이더라도 금액이 앵커 추정(`amt_source=estimated`)이면 밴드는 ±20%를 유지한다. "공식 출처지만 금액은 추정" 상태가 계산·UI에 정직하게 반영된다(DEC-2).
  - 대안: 정성 복지(`amt_source=none`, 금액 NULL)·미체크 항목은 밴드 합산에서 제외한다.
- **입력**: `benS` 항목의 `badge_cd`, `expires_dtm`, 금액 신뢰도(02 FR-D9 `amt_source`=stated/estimated/none), `benefit_amt`, `checked`, `qual_yn`.
- **출력·상태**: 항목별 배지 라벨(공식/추정/만료) + 리포트 총보상의 ± 절대값(만원).
- **검증·규칙**: (1) 밴드 계수는 **금액 신뢰도 기준**으로 명시 0.05·추정 0.20으로 고정하고, 만료는 기준 계수에 **+0.15 가산**(치환 아님)한다. **출처 배지(`badge_cd`)와 무관**하다(DEC-2 디커플링). (2) 밴드는 항목별 산정 후 합산하며 단일 % 일괄 적용을 금지한다. (3) 배지·밴드 규약은 F7 데이터 정확성 면책조항과 정합해야 한다(UC-34·UC-42·UC-51). 세부 산식은 05 FR-38이 소유하며 본 규약과 일치한다.
- **추적**: UC-34·UC-42(·UC-51 F7 정합), F3, RESEARCH §4.2·DEC-2, 02 FR-D9·05 FR-38, §7·§4(BADGE_CD·EXPIRES_DTM·amt_source).

### FR-06 기업유형 프리셋 폴백 전역 규약
- **설명**: 비교 툴에서 사용자가 등록 회사(복지 실데이터 보유, ~96개)에 없는 회사를 직접 입력할 때(**직접 입력 모드**) 기업유형 기본 복지로 초기값을 채운다. 등록 회사는 모두 실복지를 보유하므로 회사 상세·조합·`companies/{id}` 응답에는 회사페이지용 프리셋 폴백이 불필요하다.
- **상세 동작**
  - 정상: 비교 툴 직접 입력 모드에서 사용자가 등록되지 않은 회사를 입력하고 기업유형만 선택하면, 그 `comp_tp_cd`에 해당하는 `benefit_presets`로 복지를 채운다(amt_source=`preset`). 등록 회사(~96개)를 선택하면 실데이터 복지를 사용한다(amt_source=`real`).
  - 대안: 프리셋도 비어있는 유형(예: freelance)은 복지 목록이 빌 수 있으며, 사용자가 직접 추가·입력한다.
- **입력**: 사용자가 선택한 `comp_tp_cd`, 등록 회사의 `benefits[]`(실데이터), `benefit_presets{type_cd}`.
- **출력·상태**: 등록 회사=실데이터 복지 목록, 미등록(직접 입력) 회사=프리셋 초기값 복지 목록.
- **검증·규칙**: 폴백은 F2(비교 툴 직접 입력 모드 초기값)에서만 적용된다. 등록 회사는 모두 실복지를 보유하므로 F4/F5(정적 페이지)·F8(`companies/{id}`)에는 회사페이지용 프리셋 폴백 병합이 불필요하다(UC-93·UC-15·UC-21).
- **추적**: UC-93·UC-15·UC-21, F2·F3, §4·§5·§6.

### FR-07 로컬 지속성(localStorage) 전역 규약
- **설명**: 사용자 상태는 브라우저 localStorage에만 저장하며 서버로 전송하지 않는다.
- **상세 동작**
  - 정상: 비교 입력·"최근 비교"를 localStorage에 저장하고 재방문 시 복원 제시.
  - 대안: localStorage 비활성·용량 초과 시 저장을 생략하고 비교는 정상 동작, 복원만 미제공. 저장 기록 없음 시 복원 항목 미표시.
- **입력**: F2·F3 사용자 상태(`§5`).
- **출력·상태**: 로컬 저장 레코드(서버 미전송). 크로스 디바이스 동기화 없음.
- **검증·규칙**: 저장 실패를 try/catch로 흡수(무크래시). 서버 전송 0(NFR16). 저장소는 localStorage 한정, 쿠키·서버 세션 미사용(광고 쿠키 제외, NFR17).
- **추적**: UC-A2·UC-36·UC-92, F2·F3, NFR16·NFR17·NFR25, §2-2.

### FR-08 FR-ID 명명·추적성 규약(메타)
- **설명**: 모든 FRD 문서는 §2 명명 규칙과 §3 추적성 원칙을 따른다.
- **상세 동작**
  - 정상: 각 FR은 지정 대역에서 안정 ID를 부여받고, UC·F(·NFR)를 인용하며, 하위 문서가 이를 역참조한다.
  - 대안/금지: ID 중복·재사용·대역 이탈 금지. 인용한 UC·F·NFR과 모순되는 명세 금지(FR-01 위반 금지 포함).
- **입력**: 없음(문서 규약).
- **출력·상태**: 추적 가능한 FR 카탈로그(본 문서 §7 인덱스).
- **검증·규칙**: (1) 모든 UC가 ≥1 FR로 커버(§3.2-3). (2) 각 FR에 추적 필드 존재. (3) 대역-담당문서 1:1 유지(§2.3).
- **추적**: 전 UC·전 F, 브리프 §9.
