# 오류·엣지 처리 (FRD)

**목적**: PRD 비기능 요구사항 NFR24·NFR25·NFR26이 요구하고 USECASE UC-90~UC-95가 흐름으로 정의한 **비정상·엣지 상황**을, 시스템(클라이언트 SPA + 정적 서빙 + 읽기 전용 API)이 제공해야 하는 **횡단 기능 요구사항(FR-Ex)**으로 구현 이전 수준에서 정밀하게 명세한다. 대상은 (1) 참조 번들 `reference/all` 로드 실패의 재시도·오류 UX, (2) 회사 검색 API 실패, (3) localStorage 사용 불가(사생활 모드·차단·용량 초과), (4) 미등록 회사(비교 툴 직접 입력)의 기업유형 프리셋 템플릿(등록 회사는 모두 실복지 보유 → 회사페이지 프리셋 폴백 불필요), (5) JavaScript 비활성·크롤러의 정적 본문 접근(점진적 향상), (6) 존재하지 않는 회사 slug·잘못된 URL의 404, (7) 이들 전체를 관통하는 사용자 안내 메시지·무크래시 검증 규약이다. 각 FR은 **트리거·기대 동작·사용자 메시지·무크래시**를 검증 가능하게 고정한다.

**상위 추적**: 본 문서는 FRD → USECASE → PRD → 브리프 체인에 속한다. 상위 근거 = USECASE `08-오류와엣지.md`(UC-90~UC-95), PRD `07-비기능-요구사항.md`(핵심 **NFR25·NFR26**, 보조 NFR1·NFR3·NFR5·NFR11·NFR12·NFR16·NFR17·NFR20·NFR21·NFR24), 브리프 §2-2(localStorage)·§2-4(서버 무쓰기)·§2-8(클라이언트 계산)·§4(데이터 모델·프리셋)·§5(보존 데이터·프리셋)·§6(읽기 전용 API·캐시)·§7(비교 알고리즘)·§8(정적 서빙 인프라). 전역 규칙·용어·상태 모델은 FRD `01-개요와-FR규칙.md`(FR-01·FR-02·FR-03·FR-06·FR-07, §4 용어, §5 상태 모델), 데이터 계약은 FRD `02-데이터-계약.md`(FR-D1·FR-D3·FR-D4·FR-D6·FR-D7·FR-D8·FR-D11)를 인용한다. 관련 화면 UX는 검색 빈/오류 상태 FR-13(FRD 03), 회사 상세·조합 정적 생성 FR-5x·FR-6x(FRD 07·08)와 연계한다.

**범위 경계**: 본 문서는 **비정상·엣지 처리의 횡단 규약**만 소유한다. 정상(해피패스) 흐름은 각 담당 FRD가 소유한다 — 검색·선택 F1(FR-1x, FRD 03), 비교 입력 F2(FR-2x, FRD 04), 비교 리포트 F3(FR-3x/FR-4x, FRD 05·06), 회사 상세·조합 정적 생성 F4·F5(FR-5x·FR-6x, FRD 07·08), 읽기 전용 API 계약 F8(FR-Dx, FRD 02). 본 문서는 그 정상 명세가 **실패·부재·비활성 조건에서 어떻게 방어·폴백·안내되는가**를 고정하며, 담당 FR과 모순되지 않는다. 화면 문구·컴포넌트 레이아웃의 픽셀 수준은 WIREFRAME/SPEC로 미룬다. 로그인/회원/계정/프로파일러(가치관 진단 설문)/서버 측 사용자 데이터 쓰기 저장은 제품 범위에서 **영구 제외**이므로(브리프 §2-1·§2-3·§2-4, FR-01) 어떤 오류·엣지 흐름에도 등장하지 않는다.

**FR-ID 대역**: 본 문서는 **FR-Ex 대역**(FR-E1~FR-E7)을 소유한다. 이는 특정 화면 하나에 귀속되지 않고 F1~F5·F8을 가로지르는 **오류·엣지(횡단)** 대역으로, USECASE의 UC-9x 대역과 정렬된다(데이터 계약 FR-Dx와 동일한 letter-suffix 명명 관례를 따른다). 부여한 ID는 재사용하지 않는다(안정 ID, 브리프 §9). 하위 문서(FLOW/WIREFRAME/SPEC/TASK)가 이 ID를 인용한다.

## 공통 규약 (전 FR 적용)

USECASE 08 "공통 전제 1~3"을 FRD 검증 규칙으로 고정한다. 아래는 FR-E1~FR-E7 전체에 적용되는 불변식이다.

- **무크래시 불변식(NFR26)**: 어떤 오류·엣지 상황에서도 앱은 **미처리 예외·흰 화면(white screen)을 남기지 않는다**. 모든 실패 경로는 예외를 흡수(try/catch·방어적 파싱)한 뒤 정의된 오류 상태·안내로 전이한다. "크래시"란 (a) 미처리 예외로 스크립트 실행이 중단되어 이후 인터랙션이 죽거나, (b) 콘텐츠 없는 빈 화면이 남는 상태를 말하며, 두 경우 모두 금지된다.
- **오류 ≠ 무결과 구분**: 정상 응답의 "빈 결과"(예: 검색 0건, 저장분 없음)와 "실패"(네트워크·타임아웃·5xx·파싱 실패)는 **서로 다른 상태·문구**로 구분한다. 실패를 무결과로 오인해 잘못된 다음 행동으로 넘기지 않는다(FR-13·FR-D11 정합).
- **서버 무쓰기(NFR16·NFR20, FR-01)**: 어떤 오류·엣지·재시도 경로도 사용자 입력·상태를 서버로 **쓰기 전송하지 않는다**. 사용자 상태는 브라우저 메모리와 localStorage에만 존재한다(§2-2·§2-4).
- **로드 후 오프라인 계산 가능(NFR26, §2-8)**: 참조 번들(REF)이 한 번 로드된 뒤에는 비교 계산이 100% 클라이언트 사이드이므로, 네트워크가 끊겨도 검색(번들 폴백 매칭, FR-E2)·비교 입력·계산은 수행 가능하다.
- **렌더 안전(NFR21)**: 모든 오류·안내 메시지와 그에 삽입되는 회사명·검색어·복지 문자열은 `textContent`/이스케이프로 삽입하며 `innerHTML` 직접 삽입을 금지한다(오류 경로에서도 XSS 차단).
- **한정된 재시도(무한 루프 금지)**: 재시도는 사용자 트리거(예: "다시 시도" 버튼)를 기본으로 하며, 자동 재시도를 두더라도 **횟수·지연이 한정**되어야 한다. 무한 재시도·무한 로딩 스피너·응답 없는 대기 상태를 남기지 않는다.

## [FR-E1] `reference/all` 로드 실패·재시도·오류 상태 처리

**설명**: 비교 툴(SPA)이 부팅 시 1회 로드하는 참조 번들(REF)의 로드가 실패하거나 손상 응답이 도착했을 때, 앱을 크래시시키지 않고 오류 상태로 전이하여 **재시도 수단**을 제공하고, 참조가 확보되기 전까지 검색·비교를 제한한다. FR-02(부팅 로드)·FR-D1(번들 계약)·FR-D11(오류 계약)의 실패 경로를 클라이언트 동작으로 확정한다. UC-90을 담당한다.

**상세 동작**
- 정상 전제: 부팅 시 `GET /api/v1/reference/all`을 **1회** 호출한다(FR-02, UC-A1). 응답 헤더 `Cache-Control: public, max-age=3600`으로 1시간 내 재방문은 브라우저/프록시 캐시가 흡수한다(NFR3, 아래 분기 2a).
- 실패(로드 오류): 네트워크 오류·타임아웃·5xx로 요청이 실패하면 프론트가 예외를 흡수하고 **오류 상태**로 전이한다(무크래시). 오류 안내와 "다시 시도" 재시도 수단을 표시한다.
- 실패(손상·부분 응답): HTTP 200이라도 JSON 파싱 실패, 또는 번들 최상위 키(`company_types`/`benefit_presets`/`companies`) 누락·형 불일치면 **유효 번들로 취급하지 않고** 오류 상태로 처리한다(분기 3a).
- 기능 제한: 참조가 확보되기 전까지 검색(F1)·비교 입력(F2)·비교 계산(F3)을 비활성(disabled) 상태로 두고, 그 이유("데이터를 불러오지 못해 비교를 시작할 수 없습니다")를 표시한다(UC-A1 예외 2a, UC-90 기본 흐름 5).
- 재시도 성공: 사용자가 "다시 시도"로 재요청하여 유효 번들을 받으면 `REF`를 메모리에 보관하고(프로파일러 키 부재 검증, FR-D10) 정상 흐름(UC-A1)으로 복귀하여 검색·비교를 활성화한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 유효 캐시 존재(`max-age` 1시간 이내) | 네트워크 호출 없이 캐시 번들로 부팅. 오류 화면 없음(NFR3, UC-90 분기 2a) |
| 3a | 부분·손상 응답(불완전 JSON, 키 누락, 형 불일치) | 유효 번들로 취급하지 않고 오류 상태로 처리. 최상위 3키 유무·타입을 검증(FR-D1) |
| 5a | 직전 방문에서 이미 REF가 메모리에 로드됨 | 로드 성공 후에는 네트워크가 끊겨도 계산 가능(NFR26, 공통 규약 4). 재부팅 시엔 캐시(2a)로 흡수 |
| 6a | 재시도 반복 실패 | 오류 안내를 유지하고 크래시하지 않음. 정적 콘텐츠 페이지(F4/F5)로 이탈 경로를 안내(UC-90 분기 6a) |

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 부팅 트리거 | 이벤트 | SPA 진입 시 1회. 사용자 입력 없음 |
| 번들 응답 | JSON \| 오류 신호 | `{company_types[], benefit_presets{}, companies[]}` 또는 네트워크/HTTP 오류/파싱 실패 |
| 재시도 액션 | 사용자 클릭 | "다시 시도". 트리거당 재요청 1회 |

**출력·상태**
- 앱 상태 ∈ `{부팅중, 준비완료(REF 보관), 오류(재시도 대기)}`. 오류 상태에서 검색·입력·계산은 비활성.
- 성공 시 `REF = {company_types[], benefit_presets{type_cd:[…]}, companies[{…, aliases[], benefits[]}]}`(메모리 보관, FR-02·FR-D1).
- 서버·localStorage에 어떤 사용자 상태도 기록하지 않는다(로드 단계, NFR16).

**검증·비즈니스 규칙**
- (R1) 부팅 요청은 1회이며(FR-02), 재요청은 캐시 무효화 후 재방문 또는 사용자 재시도로만 발생한다. 무한 자동 재시도 금지(공통 규약: 한정된 재시도).
- (R2) 유효 번들 판정 = 최상위 키 `company_types`(배열)·`benefit_presets`(객체)·`companies`(배열)가 모두 존재하고, `profiles`/`job_groups`/`questions` 프로파일러 키가 부재해야 한다(FR-D1·FR-D10). 하나라도 위배하면 오류 처리.
- (R3) 로드 실패·손상 응답이 미처리 예외·흰 화면을 유발하지 않는다(NFR26, 공통 규약: 무크래시).
- (R4) 참조 확보 전 검색·비교를 제한하되, 그 사유를 사용자에게 명시한다(UC-90 기본 흐름 4·5).
- (R5) 성공 후에는 REF가 검색·프리셋 템플릿(직접 입력)·계산의 단일 참조원이 되며(FR-02·FR-E2·FR-E4), 네트워크 단절 시에도 계산이 가능하다(NFR26).

**추적**: UC-90 / F8 → F1·F2·F3 / FR-02·FR-D1·FR-D10·FR-D11 / NFR26·NFR3·NFR16 / 브리프 §6·§2-8.

## [FR-E2] 회사 검색 API 실패·번들 폴백 매칭

**설명**: 회사 검색 요청(`GET /api/v1/companies/search?q=`)이 실패했을 때, 앱을 크래시시키지 않고 오류를 무결과와 구분하여 안내하며, **참조 번들(REF)이 이미 로드되어 있으면 번들에 대한 클라이언트 이름/별칭 매칭으로 검색을 계속**할 수 있게 한다. 검색 빈/오류 화면 UX 자체는 FR-13이 소유하며, 본 FR은 실패의 **횡단 처리 규약**(오류/무결과 구분·번들 폴백·무크래시)을 고정한다. UC-91을 담당한다.

**상세 동작**
- 정상 전제: 사용자가 검색어를 입력하면 프론트가 `GET /api/v1/companies/search?q=`를 호출한다(FR-10·FR-D6, 이름/별칭 LIKE, 최대 20건).
- 실패(요청 오류): 네트워크 오류·타임아웃·5xx로 실패하면 예외를 흡수하고(무크래시) 검색 실패 안내("검색을 불러오지 못했습니다")와 재시도 수단을 표시한다. 이는 0건 무결과 화면과 문구·행동이 다르다(FR-13, 공통 규약: 오류 ≠ 무결과).
- 대안(번들 폴백 매칭): REF가 이미 메모리에 로드되어 있으면(FR-02, FR-E1 성공 후), 전용 검색 엔드포인트 실패와 무관하게 `REF.companies[]`의 `comp_nm` + `aliases[]`에 대해 클라이언트 부분일치 매칭을 수행하여 후보(최대 20건)를 제시할 수 있다(§6 "검색의 단일 참조원", UC-91 분기 3a). 폴백 매칭 결과는 검색 응답과 동일한 축소 투영 `{comp_id, comp_nm, comp_tp_cd, industry_nm, logo_nm}`으로 FR-12에 전달한다.
- 대안(재시도): 사용자가 "다시 시도"로 재검색하여 성공하면 후보 리스트를 표시하고 슬롯 지정(FR-14)으로 이어간다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | REF 번들이 이미 메모리에 로드됨(FR-02) | 번들의 `comp_nm`+`aliases[]` 클라이언트 매칭으로 후보를 제시(폴백). 전용 엔드포인트 없이도 검색 지속(UC-91 분기 3a) |
| 3b | 요청 성공·결과 0건(HTTP 200 `[]`) | 실패가 아니라 "결과 없음"으로 구분 표시. 오류 안내를 띄우지 않음(재검색·직접 입력 안내는 FR-13, UC-91 분기 3b) |
| 5a | 재시도 반복 실패 & REF 미로드 | 오류 안내 유지·무크래시. 검색 없이는 슬롯 지정을 진행할 수 없음을 안내(UC-91 분기 5a) |

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 검색어 `q` | string | FR-10 전달값(`trim()`, ≥1자) |
| 검색 응답 | 배열 \| 오류 신호 | `[{…}]`(성공) / `[]`(무결과) / 예외·5xx(실패) |
| REF 로드 여부 | boolean | 폴백 매칭 가용 조건(FR-02) |

**출력·상태**
- 검색 실패 시: `오류` 상태(재시도 안내) 또는 REF 로드 시 `폴백 후보` 리스트. 무결과(200 `[]`)와 문구·행동을 구분.
- `matched[s]`는 후보 선택 전까지 미변경. 서버·localStorage 쓰기 없음(NFR16).
- 앱은 크래시하지 않으며 다른 슬롯·기능은 정상 동작(NFR26).

**검증·비즈니스 규칙**
- (R1) 검색 실패(예외·5xx)와 무결과(200 `[]`)를 서로 다른 상태·문구로 구분한다(FR-13·FR-D11, 공통 규약).
- (R2) REF 로드 상태에서 검색 엔드포인트가 실패해도, 번들 폴백 매칭으로 검색을 계속할 수 있어야 한다(§6, UC-91 분기 3a). 폴백 매칭은 대소문자 무시 부분일치, `comp_id` 기준 중복 제거, 최대 20건 상한을 FR-11·FR-D6과 동일하게 적용한다.
- (R3) 검색어·입력값을 서버로 쓰기 전송하지 않는다(NFR16·NFR20).
- (R4) 검색 실패가 미처리 예외·흰 화면을 유발하지 않는다(NFR26). 폴백 후보 렌더 문자열은 이스케이프한다(NFR21).

**추적**: UC-91 / F1 → FR-D6·FR-D11·FR-02 / NFR26·NFR16·NFR21 / 브리프 §6·§4(TCOMPANY_ALIAS)·§2-8.

## [FR-E3] localStorage 사용 불가·용량 초과 흡수

**설명**: 브라우저 localStorage가 사용 불가(사생활/시크릿 모드, 저장소 차단 설정, 용량 초과 `QuotaExceededError`)한 상황에서, 저장·복원 실패를 예외로 흡수하여 **비교 기능은 정상 동작**시키고 "최근 비교"만 저장·복원되지 않게 한다. FR-07(로컬 지속성 전역 규약)의 실패 경로를 확정한다. UC-92를 담당한다.

**상세 동작**
- 정상 전제: 프론트는 비교 입력 상태(`benS`·`wsState`·`curPri`·`curSacrifice`·`selectedRate`·`matched` 요약)와 "최근 비교"를 localStorage에 저장하고, 재방문 시 복원 제시한다(FR-07, UC-A2, 서버 전송 없음).
- 실패(저장 예외): 사생활 모드·저장소 차단·용량 초과로 `setItem`이 예외를 던지면, 안전 래퍼(예: `safeStore.set`)가 `try/catch`로 흡수하고 오류 없이 진행한다. 저장 실패는 사용자 흐름을 막지 않으며 별도 오류 팝업을 강제하지 않는다(NFR25).
- 실패의 파급: 저장이 불가하므로 세션 간 지속성(재방문 복원 및 "최근 비교")이 제공되지 않는다. 그러나 현재 세션의 비교 입력·계산·리포트(F2/F3)는 **메모리 상태만으로 정상 수행**된다(§5 상태는 세션 내 메모리에 존재, UC-92 기본 흐름 5).
- 복원 실패(저장분 없음): 재방문 시 복원 시도했으나 유효 저장분이 없으면(불가·비어있음) 복원 항목을 표시하지 않고, 비교는 새로 입력하여 정상 수행한다(UC-A2 분기 3a).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 용량 초과(`QuotaExceededError`)만 발생 | 신규 저장을 생략하거나, 오래된 "최근 비교"를 정리 후 재저장을 1회 시도할 수 있음. 실패해도 비교 기능은 영향 없음(NFR25) |
| 4a | 재방문 복원 시도, 저장분 없음 | 복원 항목 미표시(UC-A2 분기 3a). 비교는 새로 입력 |
| 5a | 사용자가 결과의 기기 간 보관을 기대 | 서버 저장·크로스 디바이스 동기화는 제공하지 않음(PRD-CTX-4, §2-2). localStorage 외 서버 전송 없음 |

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 저장 대상 상태 | 객체 | `§5` 상태의 로컬 저장 표현("최근 비교" 요약 포함). 서버 미전송 |
| 저장소 응답 | 성공 \| 예외 | `QuotaExceededError`·`SecurityError`(사생활/차단) 등 |
| 복원 트리거 | 재방문 | 유효 저장분 유무를 검사 |

**출력·상태**
- 저장 성공 시: 로컬 저장 레코드 갱신. 저장 실패 시: 저장 생략(무오류 진행), 세션 내 상태는 메모리에 유지.
- 비교 입력·계산·리포트는 정상 동작하고 앱은 크래시하지 않는다. "최근 비교"만 미저장·미복원.
- 서버 전송·저장은 어떤 경우에도 발생하지 않는다(NFR16·NFR17).

**검증·비즈니스 규칙**
- (R1) 모든 localStorage 접근(`getItem`/`setItem`/`removeItem`)은 `try/catch`로 감싸 예외를 흡수한다. 저장 실패가 비교 흐름을 중단·크래시시키지 않는다(NFR25, 공통 규약: 무크래시).
- (R2) 저장 불가 시 비교 기능(F2/F3)은 메모리 상태만으로 정상 수행되며, 미제공되는 것은 세션 간 복원·"최근 비교" 지속뿐이다(§5.2).
- (R3) 저장 실패를 위한 별도 차단성 오류 화면을 강제하지 않는다(사용자 흐름 무중단). 필요 시 비침투적 안내만 허용.
- (R4) 저장소는 localStorage 한정이며 쿠키·서버 세션에 사용자 상태를 두지 않는다(광고 쿠키 제외, NFR17). 서버 전송 0(NFR16).

**추적**: UC-92 / F2·F3 / FR-07 / NFR25·NFR16·NFR17 / PRD-CTX-4·PRD-NEED-5 / 브리프 §2-2·§7.

## [FR-E4] 미등록 회사 직접 입력의 기업유형 프리셋 템플릿

**설명**: 등록 회사(복지 실데이터가 있는 ~96개 — 회사 상세 F4·조합 F5·`companies/{id}` API F8의 대상)는 **모두 실데이터 복지를 보유**하므로, 회사 페이지·API·정적 생성 경로에는 기업유형 프리셋 폴백 병합이 **불필요**하다(실복지 없는 회사는 등록·생성 대상이 아니며, 미존재 `comp_id`는 404 — FR-E6). 기업유형 프리셋(TBENEFIT_PRESET)의 폴백 역할은 **비교 툴의 직접 입력 모드**로 한정되어, 사용자가 비교하려는 회사가 등록 ~96개에 없어 직접 입력할 때 그 회사의 기업유형(COMP_TP_ID)에 대응하는 기본 복지 프리셋으로 입력값을 채운 **편집 가능한 템플릿**을 제공한다. FR-06(프리셋 전역 규약)·FR-D3·FR-D4가 규약을 소유하며, 본 FR은 **미등록 회사 직접 입력이라는 엣지에서의 프리셋 채움과 사용자 편집 병합**을 확정한다. UC-93을 담당한다.

**상세 동작**
- 등록 회사(실복지 보유): 등록 ~96개 회사는 모두 실데이터 복지를 보유하므로, 검색·선택으로 이 회사를 지정하면 실데이터 복지를 그대로 사용한다(amt_source=`real`, 배지 공식/추정). 별도 프리셋 폴백 없음.
- 프리셋 채움(미등록 회사 직접 입력): 사용자가 비교하려는 회사가 등록 ~96개에 없어 직접 입력하면, 선택한 기업유형 `comp_tp_cd`에 해당하는 `benefit_presets{type_cd:[…]}`로 입력값을 채운다(amt_source=`preset`, 배지 주로 `est`). 프리셋 항목(`benefit_nm`·`benefit_amt`·`benefit_ctgr_cd`·`badge_cd`·`default_checked_yn`)이 비교 입력의 편집 가능한 기본값이 된다(FR-06·FR-D3).
- 병합(사용자 편집): 프리셋으로 채운 뒤 사용자가 특정 항목을 직접 수정하면 사용자 입력값을 우선하고, 수정하지 않은 항목만 프리셋 기본값을 유지한다(항목 키 기준 병합, UC-93 분기 3a).
- 정성 복지 처리: 정성 복지(`qual_yn=true`, `benefit_amt=null`)는 금액 합산(순복지)에서 제외하고 정성 항목으로 별도 표기한다(§4 QUAL_DESC_CTNT). 프리셋 항목에도 동일 적용(FR-04·FR-D5).
- 적용 경로: 프리셋 채움은 비교 입력 F2에서 미등록 회사 직접 입력 시 초기값 채움(FR-14)에만 적용된다. 등록 회사 대상인 `companies/{id}` API(F8)·회사 상세/조합 정적 생성(F4/F5)에는 실복지 데이터만 사용되며 프리셋 폴백을 병합하지 않는다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | `companies/{id}` API 응답 경로 | 등록 회사는 실복지 보유 → 프리셋 폴백 병합 없이 실데이터 `benefits[]`를 반환. 미존재 `comp_id`는 404(FR-D7·FR-E6). 클라이언트는 빈/오류 응답에도 크래시하지 않음 |
| 1b | 정적 상세·조합 페이지 생성 경로(빌드타임) | 복지 실데이터가 있는 회사만 페이지 생성 → 프리셋 폴백 불필요. 요청 시 DB 조회 없이 정적 서빙(NFR1) |
| 3a | 프리셋 채움 후 사용자가 일부 항목 직접 수정 | 사용자 입력 우선, 미수정 항목만 프리셋 기본값 유지(병합) |
| 4a | 프리셋이 비어있는 유형(예: `freelance`) | 직접 입력 기본값이 빌 수 있음. 빈 상태로도 크래시하지 않으며, 사용자가 직접 추가·입력(FR-06 대안·FR-17) |

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 직접 입력 회사 유형 `comp_tp_cd` | enum(6종) | 프리셋 조회 키(`large`/`startup`/`mid`/`foreign`/`public`/`freelance`) |
| 사용자 입력 복지 | 배열 | 프리셋 기본값 + 사용자 편집(수정·추가·삭제) |
| 프리셋 `benefit_presets{type_cd}` | 배열 | 유형별 기본 복지. 비어있을 수 있음(4a) |

**출력·상태**
- 프리셋 기본값 + 사용자 편집으로 구성된 직접 입력 복지 목록. 각 항목에 amt_source(`preset`/사용자 편집) 마커 부여(FR-04).
- 정성 항목은 금액 합산 제외·별도 표기. 서버 사용자 저장 없음.

**검증·비즈니스 규칙**
- (R1) 직접 입력 병합 우선순위 = 사용자 편집값 > 프리셋 기본값(`preset`). 사용자가 수정한 항목은 프리셋으로 덮어쓰지 않는다(UC-93 분기 3a).
- (R2) 프리셋 폴백은 비교 툴 직접 입력 경로에만 적용된다. 등록 회사 대상 `companies/{id}` API·정적 상세/조합 생성에는 실복지만 사용하며 프리셋 폴백을 병합하지 않는다(등록 회사는 실복지 보유, UC-93 분기 1a·1b).
- (R3) 정성 복지(`qual_yn=true`)는 금액이 없으며(NULL) 순복지 합산에서 제외한다(FR-04·FR-D5).
- (R4) 프리셋도 비어있는 유형은 빈 기본값을 허용하고, 비교 입력·계산 모두 빈 복지로 크래시하지 않는다(FR-D11, NFR26).

**추적**: UC-93 / F2·F3·F4·F5·F8 / FR-06·FR-D3·FR-D4·FR-D7·FR-D11 / NFR1·NFR26 / 브리프 §4(TBENEFIT_PRESET)·§5·§6.

## [FR-E5] 비-JS·크롤러의 정적 본문 접근(점진적 향상)

**설명**: JavaScript를 실행하지 않는 방문자(브라우저 JS 비활성)·검색 크롤러가 정적 콘텐츠 페이지(회사 상세 F4·인기 조합 F5·정책 F7)에 접근했을 때, 초기 HTML만으로 **본문 콘텐츠가 가독·색인**되게 하고, 인터랙션(비교 CTA)·광고 슬롯은 JS 향상 요소로 분리한다. UC-94를 담당하며, 정적 생성의 본문 구성 자체는 FR-5x·FR-6x(FRD 07·08)가 소유한다.

**상세 동작**
- 정상: A2(JS 비활성)/A3(크롤러)가 정적 콘텐츠 페이지 URL을 요청하면 Nginx가 **완성된 정적 HTML**을 반환한다(§8, 요청 시 서버 렌더링·DB 조회 없음 — NFR1).
- 본문 가독: JS 없이도 초기 HTML에 본문이 존재해 읽힌다 — 회사 상세(기업정보·복지·근무형태), 조합(두 회사 핵심 비교), 정책(정책 문안). 크롤러(A3)는 본문을 색인하고 사용자(A2)는 본문을 열람한다(NFR12·NFR24).
- 인터랙션 분리: "이 회사로 비교하기" CTA 등 인터랙션은 JS 향상 요소이며, JS 미실행 시에도 본문 열람은 유지된다(NFR24 "인터랙션만 JS 향상"). CTA는 링크(정적 이동)로도 최소 접근 가능하게 두는 것을 권장한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 5a | 비교 툴(SPA, F1~F3)에 JS 비활성으로 접근 | 비교 툴은 클라이언트 계산(§2-8) 기반이라 JS가 필요. 비-JS 본문 가독 보장 대상은 콘텐츠 페이지(F4/F5/F7)에 한정된다(NFR24 범위, UC-94 분기 5a) |
| 3a | 광고/제휴 슬롯(F6)이 JS 미실행으로 미로드 | 광고/제휴는 JS 향상 요소로 미표시될 수 있으나 본문 가독에는 영향 없음. 슬롯은 고정 크기 예약으로 레이아웃 이동 방지(NFR5, UC-94 분기 3a) |
| 2a | 요청 페이지가 존재하지 않음 | 404로 처리(FR-E6, UC-95) |

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 요청 URL | 정적 경로 | 배포된 회사 slug·조합·정책 경로 |
| 실행 환경 | JS 미실행 | 크롤러 또는 JS 비활성 브라우저 |

**출력·상태**
- 초기 HTML에 포함된 본문(가독·색인 가능). 서버 사용자 저장 없음.
- 인터랙션·광고는 JS 실행 환경에서만 향상 적용. 슬롯 자리는 예약되어 CLS를 유발하지 않는다.

**검증·비즈니스 규칙**
- (R1) 정적 콘텐츠 페이지의 본문(회사 정보·복지·근무형태 / 조합 비교 / 정책 문안)은 초기 HTML에 존재하여 JS 없이 가독·색인된다(NFR12·NFR24). 본문을 JS로만 주입하지 않는다.
- (R2) 인터랙션(비교 CTA)·광고/제휴 슬롯은 점진적 향상 요소로, 미실행 시에도 본문 열람이 유지된다(NFR24).
- (R3) 광고/이미지 슬롯은 고정 크기로 예약하여 미로드 시에도 레이아웃 이동을 방지한다(NFR5, CLS ≤ 0.1).
- (R4) 정적 경로는 요청 시 백엔드·DB 호출 0으로 Nginx가 직접 서빙한다(NFR1).

**추적**: UC-94 / F4·F5·F7 / NFR24·NFR12·NFR1·NFR5 / PRD-SEG-3·PRD-CTX-2 / 브리프 §2-7·§8.

## [FR-E6] 미존재 slug·잘못된 URL·미존재 ID(404·not-found)

**설명**: 존재하지 않는 회사 slug·미배포 조합·잘못된 URL 요청, 그리고 `companies/{id}` API의 미존재 식별자 요청을 **404/not-found**로 일관 처리하여 색인에서 제외하고, 앱을 크래시시키지 않으며, 유효 페이지로의 이동 경로를 안내한다. 중복 경로(조합 역순 등)는 canonical로 정규화한다. UC-95를 담당한다.

**상세 동작**
- 정상 전제: 정적 회사 상세 페이지는 복지 데이터가 있는 등록 회사(~96개, 존재하는 slug)에 한해, 인기 조합 페이지는 사전 정의된 회사쌍 목록에 한해 생성된다(F4/F5, §5). slug는 회사 영문식별자(COMP_ENG_NM, §4)를 기준으로 한 정적 경로다.
- 정적 404: 요청 URL이 미생성 slug·미배포 조합을 가리키면 Nginx가 404를 반환한다(§8). 404 응답은 **안내 페이지**(유효한 회사 상세·인기 조합·랜딩으로의 이동 링크 포함)를 제공하여 탐색이 이어지게 한다.
- 크롤러 색인 제외: 크롤러(A3)는 404 URL을 색인에서 제외한다(UC-A3 분기 4a).
- API not-found: `GET /api/v1/companies/{id}`에 미존재 `comp_id`가 오면 not-found로 응답한다(예: HTTP 404 또는 `{error:…}`, FR-D7·FR-D11). 프론트는 오류를 흡수해 "회사 정보를 불러올 수 없습니다"를 표시하고 슬롯을 미선택 상태로 유지하며 크래시하지 않는다(NFR26).
- 중복 경로 정규화: 존재하는 조합의 역순 URL 등 중복 경로는 자기 자신을 가리키는 canonical로 단일 정규화한다(NFR11). 미존재 조합은 404로 처리한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | `companies/{id}` API에 미존재 `comp_id` 요청(A1) | not-found로 응답(FR-D7·FR-D11). 프론트는 흡수·안내·무크래시, 슬롯 미선택 유지(UC-95 분기 1a) |
| 3a | 별칭·검색어 형태의 잘못된 slug | slug는 COMP_ENG_NM 기준. 별칭·통칭·검색어는 slug가 아니라 검색(FR-10, `companies/search`)으로 처리(UC-95 분기 3a) |
| 5a | 존재하는 조합의 역순 URL 등 중복 경로 | canonical로 단일 정규화(NFR11). 미존재 조합은 404(UC-95 분기 5a) |

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 요청 URL | 정적 경로 | slug=COMP_ENG_NM 기준. 미생성·오타·미배포 조합 가능 |
| `comp_id` | int | API 경로. 미존재 식별자 가능 |

**출력·상태**
- 미존재 정적 경로 → HTTP 404 + 안내 페이지(유효 페이지 이동 링크). 색인 제외 대상.
- 미존재 API 식별자 → not-found 응답. 프론트는 안내·무크래시, `matched[s]` 미변경.
- 중복 경로 → canonical 정규화. 서버 사용자 저장 없음.

**검증·비즈니스 규칙**
- (R1) 정적 페이지는 등록된 ~96개 회사(복지 데이터 보유)·정의된 조합에 한해 존재하며, 그 밖의 경로는 404로 응답한다(F4/F5, §5).
- (R2) API 미존재 `comp_id`는 not-found로 응답하고, 클라이언트는 이를 흡수해 크래시하지 않는다(FR-D7·FR-D11, NFR26).
- (R3) slug는 COMP_ENG_NM 기준의 정적 경로이며, 별칭·검색어는 slug가 아니라 검색으로 처리한다(§4).
- (R4) 중복 URL(조합 역순 등)은 단일 canonical로 정규화한다(NFR11). 404 페이지는 유효 페이지로의 이동 경로를 제공한다.

**추적**: UC-95 / F4·F5·F8 / FR-D7·FR-D11 / NFR26·NFR11·NFR1 / PRD-SEG-3·PRD-CTX-2 / 브리프 §4(COMP_ENG_NM)·§5.

## [FR-E7] 오류·빈 상태 사용자 안내·무크래시 검증 규약(횡단)

**설명**: FR-E1~FR-E6 전 실패 경로에 대해, 각 트리거가 **정의된 사용자 안내 메시지·복구 행동**을 표시하고 **미처리 예외·흰 화면을 남기지 않음**을 검증 가능한 카탈로그로 고정한다. 오류/무결과 구분·렌더 안전·서버 무쓰기를 횡단 규칙으로 확정한다. UC-90~UC-95 전체의 무크래시·안내 요구(NFR26)를 검증 조건으로 소유한다.

**상세 동작**
- 각 실패 트리거는 아래 카탈로그의 (a) 사용자 메시지(의미 고정, 문안은 SPEC/WIREFRAME에서 카피 조정 가능), (b) 복구 행동, (c) 무크래시 단언을 만족한다.

**오류·빈 상태 안내 카탈로그**

| 트리거 | 소유 FR | 사용자 메시지(예) | 복구 행동 | 무크래시 단언 |
|--------|---------|-------------------|-----------|---------------|
| `reference/all` 로드 실패 | FR-E1 | "데이터를 불러오지 못했습니다. 다시 시도해 주세요." | "다시 시도" 재요청 / 정적 페이지 이탈 안내 | 예외 흡수, 검색·비교 비활성화(사유 표시), 흰 화면 없음 |
| `reference/all` 손상·부분 응답 | FR-E1 | (동일: 데이터 불러오기 실패) | "다시 시도" | 유효 번들 아님으로 처리, 크래시 없음 |
| 검색 API 실패 | FR-E2 | "검색을 불러오지 못했습니다." | "다시 시도" / REF 로드 시 번들 폴백 매칭 | 무결과와 구분, 다른 슬롯·기능 정상 |
| 검색 결과 0건 | FR-E2·FR-13 | "검색 결과가 없습니다." | 다른 검색어 재검색 / 직접 입력(FR-17) | 오류 아님(정상 빈 상태) |
| localStorage 저장·복원 실패 | FR-E3 | (차단성 오류 없음, 비침투적 안내만) | 비교는 정상 진행, "최근 비교"만 미저장 | 저장 예외 흡수, 비교 흐름 무중단 |
| 미등록 회사 직접 입력 | FR-E4 | (오류 아님) 기업유형 프리셋 기반 기본 복지 표시 | 프리셋 템플릿 채움, 사용자 편집 가능 | 빈 복지에도 크래시 없음 |
| API 미존재 `comp_id` | FR-E6 | "회사 정보를 불러올 수 없습니다." | 슬롯 미선택 유지 / 재검색 | not-found 흡수, 크래시 없음 |
| 미존재 slug·URL | FR-E6 | (404 안내 페이지) "페이지를 찾을 수 없습니다." | 유효 페이지(회사·조합·랜딩) 이동 링크 | 색인 제외, 앱 무크래시 |

- 렌더 안전: 모든 메시지·삽입 문자열(회사명·검색어·복지)은 `textContent`/이스케이프로 삽입한다(NFR21).
- 서버 무쓰기: 위 어떤 경로도 사용자 입력·상태를 서버로 전송하지 않는다(NFR16·NFR20).

**입력**

| 필드 | 타입 | 제약 |
|------|------|------|
| 실패 트리거 | enum | 카탈로그 행 중 하나(FR-E1~FR-E6 원천) |
| 문맥 데이터 | 문자열 | 메시지에 삽입될 회사명·검색어 등(이스케이프 대상) |

**출력·상태**
- 트리거별 사용자 안내(메시지 + 복구 행동)와 정의된 상태 전이. 흰 화면·미처리 예외 없음.
- 서버·localStorage로의 사용자 데이터 쓰기 전송 0(NFR16).

**검증·비즈니스 규칙**
- (R1) 카탈로그의 모든 트리거는 사용자 가시 메시지와 복구 행동을 갖는다(빈 실패·묵음 실패 금지). 단, localStorage 실패는 차단성 오류 화면 없이 비침투적으로 처리한다(NFR25, FR-E3).
- (R2) 오류(실패)와 빈 상태(정상 무결과·데이터 부재)를 문구·행동으로 구분한다(공통 규약).
- (R3) 모든 실패 경로에서 미처리 예외·흰 화면이 남지 않는다(NFR26). 이는 실패 주입 테스트(네트워크 실패·손상 응답·저장소 예외 주입)로 검증한다.
- (R4) 메시지·삽입 문자열은 이스케이프하며(NFR21), 한국어·UTF-8로 표시한다(NFR29). 어떤 오류 경로도 서버 쓰기 전송을 하지 않는다(NFR16·NFR20).

**추적**: UC-90~UC-95 / F1~F5·F8 / FR-D11·FR-E1~FR-E6 / NFR26·NFR21·NFR16·NFR20·NFR25·NFR29 / 브리프 §6·§7·§2-4.
