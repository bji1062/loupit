# 참여 (USECASE)

**목적**: 로그인 없는 익명 열람·비교(A1·A2) 위에, 회사 복지 데이터의 **등록·수정(기여)**을 담당하는 **인증 기여자(A6)**의 상호작용을 SC14(참여, 2026-07-21 증분)에 근거하여 표준 유즈케이스로 정의한다. 무비밀번호 이메일 코드 로그인(UC-70·71), 계정 관리·탈퇴(UC-72), 회사 도메인 이메일 재직 인증(UC-73·74), 복지 항목 등록·수정(UC-75·76), 편집 이력 열람(UC-77)을 다룬다.

**상위 추적**: 본 문서는 USECASE→PRD 체인에 속하며, 각 UC는 PRD 범위 SC14와 비기능 요구사항 NFR16·NFR17·NFR20·NFR30·NFR31을 인용한다. 기능 계약은 FRD 13(참여 API), 상세 구현 계약은 SPEC 13(SP-AUTH)이 소유한다. 액터 A6는 [01-액터와-개요.md](01-액터와-개요.md) §2 정의를 따르며, 로그인·세션·해시·PII 계약은 아키텍처 불변식 INV-1·INV-8·INV-9(SPEC 01)에 근거한다.

**범위 경계**: 이 문서는 SC14 기여(로그인·재직 인증·복지 등록/수정·이력)만 다룬다. 익명 열람·비교(F1~F5, UC-1x~UC-6x)는 **로그인 없이 그대로**이며 본 문서 대상이 아니다. 비밀번호·OAuth·소셜 로그인·소셜피드는 제품 범위에서 영구 제외이므로(SC10) 어떤 흐름에도 등장하지 않는다 — 로그인은 **무비밀번호 이메일 6자리 코드** 방식뿐이다. 세션·재직 검증은 **미들웨어가 아니라 라우트 의존성**으로 표현되어 익명 GET 경로는 무세션·무쿠키로 유지된다(INV-1·INV-9). 서버가 저장하는 회원 PII는 로그인 이메일 + 닉네임뿐이고, 회사 이메일·인증 코드·세션 토큰은 원문 저장 0(해시/HMAC만, 검증 직후 원문 파기, INV-8·NFR30).

## UC 인덱스

| UC-ID | 제목 | 주 액터 | 관련 범위 |
|-------|------|---------|-----------|
| UC-70 | 이메일 코드 로그인 요청(6자리 코드 발송) | A6 | SC14 |
| UC-71 | 코드 입력·로그인 완료(세션 발급·닉네임 자동 생성) | A6 | SC14 |
| UC-72 | 마이페이지 — 닉네임 변경·인증 목록·로그아웃·탈퇴 | A6 | SC14 |
| UC-73 | 회사 도메인 이메일로 재직 인증(코드 발송→검증) | A6 | SC14 |
| UC-74 | 도메인 미등록 회사의 수동 재직 승인 요청 | A6(요청) / A5(승인) | SC14 |
| UC-75 | 복지 항목 등록(재직 인증 보유자) | A6 | SC14 |
| UC-76 | 복지 항목 수정(동시성 충돌 처리) | A6 | SC14 |
| UC-77 | 복지 편집 이력 열람(공개, 익명 가능) | A1·A2·A6 | SC14 |

**공통 사전조건(로그인 필요 UC: UC-70~76)**

1. 익명 열람·비교(F1~F5)는 로그인 없이 그대로 가능하다. 로그인은 **복지 등록·수정 기여**를 하려는 사용자에게만 요구된다(SC14, PRD-NEED 익명 우선).
2. 로그인은 **무비밀번호 이메일 6자리 코드** 방식뿐이다. 비밀번호·OAuth·소셜 로그인은 존재하지 않는다(SC10). 서버가 저장하는 PII는 로그인 이메일 + 닉네임뿐이다(INV-8).
3. 세션은 불투명 랜덤 토큰(DB엔 SHA-256 해시만) + `HttpOnly·Secure·SameSite=Lax·Path=/api/v1` 쿠키(loupit_sid) 1종으로 유지되며, 미들웨어가 아니라 라우트 의존성으로 검증된다(INV-9). 쓰기 요청은 커스텀 헤더(`X-Loupit-Client`)를 동반한다(CSRF 방어, SP-AUTH).
4. 인증 코드 발송·검증에는 상한이 있다: 메일 발송 rate limit + 재전송 쿨다운, 코드 시도 상한(5회)·만료(10분)(NFR31).

**공통 사후조건 제약**: 회사 이메일·인증 코드·세션 토큰의 **원문은 서버에 저장되지 않는다**(해시/HMAC만, 검증 직후 파기, INV-8·NFR30). 계정 존재 여부는 응답으로 노출되지 않는다(균일 응답, NFR31). 익명 비교 로그(TCOMPARE_LOG)의 4컬럼 익명 계약은 불변이다(INV-8).

## [UC-70] 이메일 코드 로그인 요청(6자리 코드 발송)

**액터**: A6 인증 기여자(로그인 전 상태 — 익명 방문자 A1이 기여를 시작하며 전환).

**사전조건**
- 공통 사전조건 1~4.
- 사용자가 복지 등록·수정(UC-75/76) 또는 재직 인증(UC-73)을 시도해 로그인 진입점(GNB "로그인" 또는 편집 CTA)에 도달했다.

**기본 흐름**
1. 사용자가 로그인 화면에서 **개인정보 수집·이용 동의**를 확인하고 이메일 주소를 입력한다.
2. 서버가 6자리 코드를 생성해 코드의 **해시**만 만료시각(+10분)과 함께 저장하고(원문 미저장), 해당 이메일로 코드를 발송한다.
3. 서버가 계정 유무와 무관하게 **균일한 성공 응답(204)**을 반환한다(계정 열거 차단, NFR31).
4. 사용자에게 코드 입력 화면과 재전송 쿨다운 안내가 표시된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 동의 미체크 | 코드 발송을 진행하지 않고 동의 필요 안내. 동의는 로그인 진행의 전제(개인정보보호법) |
| 2a | 발송 rate limit·재전송 쿨다운 초과 | 발송을 억제하고 잠시 후 재시도 안내(NFR31). 응답은 균일 유지 |
| 3a | 유효하지 않은 이메일 형식 | 형식 오류 안내(발송 없음). 계정 유무는 노출하지 않음 |

**사후조건**: 코드 해시가 만료시각과 함께 저장되고 이메일이 발송된다(성공 시). 코드 원문·이메일-계정 매핑 노출 없음.

**추적**: SC14 / A6 / NFR30·NFR31 / FRD 13(로그인 코드 발송) / SP-AUTH / 연계 UC-71(코드 입력).

## [UC-71] 코드 입력·로그인 완료(세션 발급·닉네임 자동 생성)

**액터**: A6 인증 기여자.

**사전조건**
- 공통 사전조건 1~4, UC-70으로 코드가 발송된 상태.

**기본 흐름**
1. 사용자가 수신한 6자리 코드를 입력한다.
2. 서버가 저장된 코드 해시·만료·시도횟수와 대조해 검증한다.
3. 검증 성공 시 서버가 계정을 조회하거나(기존) **신규 생성**한다. 신규면 닉네임을 자동 생성한다(`직장인-{랜덤6}`, 2단계 가입 흐름 제거).
4. 서버가 불투명 세션 토큰을 발급하고 **해시만 저장**한 뒤, `HttpOnly·Secure·SameSite=Lax` 세션 쿠키를 설정한다(세션 ~30일, INV-9).
5. 사용자에게 로그인 완료와 닉네임(신규 여부 포함)이 표시되고, 진입 이전 맥락(편집/인증)으로 복귀한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 코드 불일치 | 시도횟수 증가·오류 안내. 상한(5회) 초과 시 해당 코드 무효화(NFR31) |
| 2b | 코드 만료(+10분 경과) | 만료 안내 후 UC-70 재요청 유도 |
| 3a | 신규 사용자 | 닉네임 자동 생성 후 마이페이지(UC-72)에서 변경 가능 |

**사후조건**: 세션이 발급되고(쿠키+DB 해시), 사용자는 로그인 상태가 된다. 코드는 소비 처리(재사용 불가). PII는 이메일 + 닉네임뿐.

**추적**: SC14 / A6 / INV-8·INV-9·NFR16·NFR30 / FRD 13(로그인) / SP-AUTH / 연계 UC-70, UC-72.

## [UC-72] 마이페이지 — 닉네임 변경·인증 목록·로그아웃·탈퇴

**액터**: A6 인증 기여자(로그인 상태).

**사전조건**
- 공통 사전조건 1~4, 유효한 세션 보유.

**기본 흐름**
1. 사용자가 마이페이지에 진입한다(세션 의존성으로 인가).
2. 현재 닉네임·보유 재직 인증 목록(회사·만료일)·상태를 확인한다.
3. 필요 시 닉네임을 변경한다(고유성 검증).
4. 로그아웃 시 세션이 폐기(쿠키 삭제 + DB 세션 무효화)된다.
5. 탈퇴 시 **로그인 이메일이 파기(NULL)**되고 전 세션이 폐기된다. **닉네임과 편집 이력은 존치**(공개 이력의 무결성)되며, 이 존치 사실은 약관·가입 동의에 고지된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 세션 없음·만료 | 인가 실패(401 상당) → 로그인(UC-70) 유도. 마이페이지 미노출 |
| 3a | 닉네임 중복 | 변경을 거부하고 다른 닉네임 요청 |
| 5a | 탈퇴 후 재로그인 | 같은 이메일로 재로그인 시 신규 계정으로 취급(이전 이메일 매핑 파기됨). 과거 편집 이력의 닉네임은 그대로 남음 |

**사후조건**: 계정 상태가 갱신된다. 탈퇴 시 이메일 원문이 파기되고 이력·닉네임은 남는다.

**추적**: SC14 / A6 / INV-8·NFR16 / FRD 13(마이페이지·탈퇴) / SP-AUTH·SP-POL(약관 T5 고지) / 연계 UC-73(인증), UC-77(이력).

## [UC-73] 회사 도메인 이메일로 재직 인증(코드 발송→검증)

**액터**: A6 인증 기여자(로그인 상태).

**사전조건**
- 공통 사전조건 1~4, 유효한 세션 보유.
- 인증 대상 회사가 등록 회사(~96)에 존재하고, 해당 회사의 이메일 도메인이 등록되어 있다(TCOMPANY_EMAIL_DOMAIN). 미등록이면 UC-74로 분기.

**기본 흐름**
1. 사용자가 인증할 회사를 검색·선택한다.
2. 회사 이메일 주소를 입력한다. 서버가 입력 도메인이 해당 회사 등록 도메인과 일치하는지 확인한다.
3. 일치 시 서버가 6자리 코드를 생성해 **코드 해시**만 저장하고 그 회사 이메일로 발송한다.
4. 사용자가 코드를 입력한다. 서버가 검증한다.
5. 검증 성공 시 서버가 재직 인증 레코드를 생성한다: 회사 이메일은 **원문을 파기**하고 **HMAC 해시만** 보관(중복 인증 차단용), 인증 방법=domain, 만료=+365일(TEMPLOY_VERIFICATION).
6. 사용자에게 해당 회사 재직 인증 완료가 표시되고, 그 회사 복지 편집(UC-75/76) 권한이 활성화된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 입력 도메인이 회사 등록 도메인과 불일치 | 인증 진행 거부(422). 필요 시 UC-74(수동 승인) 안내 |
| 2b | 회사 도메인 미등록 | domain 방식 불가 → UC-74(수동 승인)로 유도 |
| 4a | 코드 불일치·만료·시도 초과 | UC-71과 동일 상한 규칙(5회·10분, NFR31) |
| 5a | 같은 회사 이메일 HMAC이 타 계정에 이미 존재 | 중복 인증 차단(409). 한 회사 이메일 = 한 계정 |
| 5b | 이미 인증 보유 | 재인증 불필요 안내(409 already_verified). 만료 임박 시 갱신 |

**사후조건**: 해당 회사 재직 인증이 생성되고, 회사 이메일 원문은 저장되지 않는다(HMAC만). 편집 권한이 회사 단위로 부여된다.

**추적**: SC14 / A6 / INV-8·NFR30·NFR31 / FRD 13(재직 인증) / SP-AUTH / 연계 UC-74(폴백), UC-75·76(편집 권한).

## [UC-74] 도메인 미등록 회사의 수동 재직 승인 요청

**액터**: A6 인증 기여자(요청) / A5 빌드·운영자(승인·거부).

**사전조건**
- 공통 사전조건 1~4, 유효한 세션 보유.
- 대상 회사의 이메일 도메인이 미등록이거나(그룹사 공유 도메인 등) domain 인증이 불가하다.

**기본 흐름**
1. 사용자가 대상 회사에 대해 수동 승인 요청 폼에서 소명(재직 증빙 설명)을 제출한다(TEMPLOY_VRF_REQUEST, 상태=pending).
2. 운영자(A5)가 운영자 CLI(`python -m server.ops`)로 대기 큐를 확인한다.
3. 운영자가 요청을 승인 또는 거부한다.
4. 승인 시 서버가 해당 회사 재직 인증을 생성한다(인증 방법=manual, 만료=+365일). 거부 시 상태=rejected로 기록한다.
5. 사용자가 마이페이지(UC-72)에서 처리 결과를 확인한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 동일 회사에 대한 pending 요청 중복 | 기존 요청 유지, 중복 생성 방지 |
| 3a | 남용·허위 의심 | 운영자가 거부(사유 기록). 반달리즘 장벽 유지 |
| 4a | 운영자 웹 패널 부재 | v1은 웹 관리 패널 없이 운영자 CLI로만 처리(기존 시드 스크립트 관례) |

**사후조건**: 승인 시 manual 방식 재직 인증이 생성되어 편집 권한이 부여된다. 결정 주체·시각이 기록된다.

**추적**: SC14 / A6·A5 / INV-8 / FRD 13(수동 승인) / SP-AUTH·SP-INFRA(운영자 CLI) / 연계 UC-73(domain 폴백), UC-75·76.

## [UC-75] 복지 항목 등록(재직 인증 보유자)

**액터**: A6 인증 기여자(해당 회사 재직 인증 보유).

**사전조건**
- 공통 사전조건 1~4, 유효한 세션 + **해당 회사** 재직 인증 보유(UC-73/74). 미보유 로그인 사용자는 "재직 인증하고 편집하기" CTA로 유도.

**기본 흐름**
1. 사용자가 회사 복지 페이지에서 "복지 추가"로 등록 폼을 연다.
2. 복지 표시명·카테고리(9종)·금액 또는 정성(토글)·비고·편집 요약을 입력한다.
3. 서버가 세션·해당 회사 재직 인증을 라우트 의존성으로 검증하고, 커스텀 헤더(`X-Loupit-Client`)를 확인한다(CSRF).
4. 서버가 복지 행을 삽입한다. 사용자 편집은 배지 시맨틱을 강제한다: `BADGE_CD='verified'`(재직자 확인), `AMT_SOURCE_CD='estimated'`, `BADGE_SRC_CD='user_report'`, 만료=+18개월. 정성이면 금액 NULL.
5. 같은 트랜잭션에서 편집 이력(TBENEFIT_EDIT_LOG, EDIT_TYPE=create, before=∅→after 스냅샷·편집자·시각)이 append된다.
6. 응답으로 갱신된 복지 목록이 반환되어 화면이 즉시 재렌더된다. 참조 캐시가 무효화된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 세션 없음·해당 회사 재직 인증 없음 | 401/403 → 로그인·재직 인증 유도(편집 차단) |
| 3b | 커스텀 헤더 부재(크로스 오리진 위조 시도) | 거부(CSRF 방어, nginx 게이트 + 앱 이중 검사) |
| 4a | 일일 편집 상한(30건) 초과 | 차단·안내(남용 방지, NFR31 계열) |
| 4b | 중복 복지(동일 회사·코드) | 신규 등록 거부, 수정(UC-76)으로 유도 |

**사후조건**: 복지 항목이 verified 배지로 추가되고, 편집 이력이 append된다. 정적 페이지 반영은 야간 재생성으로 전파(신선도 고지).

**추적**: SC14 / A6 / INV-1·INV-8 / FRD 13(복지 등록) / SP-AUTH·SP-DB(TBENEFIT_EDIT_LOG) / 연계 UC-73(권한), UC-76·77.

## [UC-76] 복지 항목 수정(동시성 충돌 처리)

**액터**: A6 인증 기여자(해당 회사 재직 인증 보유).

**사전조건**
- 공통 사전조건 1~4, 유효한 세션 + 해당 회사 재직 인증 보유.
- 수정 대상 복지 항목이 존재한다.

**기본 흐름**
1. 사용자가 행별 "편집"으로 인라인 편집 폼(표시명·금액/정성 토글·비고·편집 요약)을 연다. 폼은 읽은 행의 수정시각(`base_dtm`)을 함께 보유한다.
2. 사용자가 값을 수정해 제출한다(세션·재직 인증·커스텀 헤더 검증).
3. 서버가 `base_dtm`이 현재 행의 수정시각과 일치하는지 확인한다(낙관적 동시성).
4. 일치 시 서버가 행을 갱신하고 같은 트랜잭션에서 편집 이력(EDIT_TYPE=update, before→after 스냅샷)을 append한다. official 행을 사용자가 수정하면 배지가 verified로 강등된다.
5. 응답으로 갱신된 복지 목록이 반환되어 즉시 재렌더된다. 참조 캐시 무효화.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | `base_dtm` 불일치(다른 사용자가 먼저 수정) | 409 충돌 + **현재 행 동봉** 반환. 사용자에게 현재값 diff를 보여주고 재편집 유도 |
| 2a | 세션·재직 인증·헤더 결여 | UC-75의 3a/3b와 동일(차단) |
| 4a | 일일 편집 상한 초과 | 차단·안내 |

**사후조건**: 복지 항목이 갱신되고 before→after 이력이 남는다. 충돌 시 데이터 손상 없이 현재값으로 재조정된다.

**추적**: SC14 / A6 / INV-1·INV-8 / FRD 13(복지 수정·409) / SP-AUTH·SP-DB / 연계 UC-75, UC-77.

## [UC-77] 복지 편집 이력 열람(공개, 익명 가능)

**액터**: A1 익명 방문자·A2 검색 유입 사용자·A6 인증 기여자(누구나 열람).

**사전조건**
- 없음(공개). 로그인 불필요 — 익명도 열람 가능하다(나무위키식 투명성).

**기본 흐름**
1. 사용자가 회사 복지 페이지에서 "편집 이력"을 연다.
2. 서버가 해당 회사의 편집 이력을 최신순으로 반환한다(닉네임 조인, 페이지네이션). 응답은 `no-store`.
3. 사용자가 누가·언제·무엇을(before→after) 편집했는지 확인한다. 모든 출력은 이스케이프(XSS 방어, NFR21).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 이력 없음 | 빈 이력 안내 |
| 2b | 탈퇴한 기여자의 이력 | 닉네임은 존치되어 그대로 표시(이메일은 이미 파기, UC-72) |
| 3a | 반달리즘 의심 편집 | before→after 스냅샷이 롤백·검증 재료가 된다(운영자 CLI로 삭제 시 EDIT_TYPE=delete 기록) |

**사후조건**: 편집 이력이 공개 열람된다. 열람 과정에서 서버 사용자 저장은 발생하지 않는다.

**추적**: SC14 / A1·A2·A6 / NFR21 / FRD 13(편집 이력 조회) / SP-AUTH·SP-DB(TBENEFIT_EDIT_LOG) / 연계 UC-75·76(편집), UC-42(배지 정합).
