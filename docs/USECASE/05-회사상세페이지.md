# 회사 상세 페이지 (USECASE)

**목적**: 검색엔진에서 유입된 사용자가 회사 상세 페이지(정적, 복지 데이터가 있는 회사 ~96개)에서 기업정보·복지·근무형태를 열람하고, 복지 배지(공식/추정)를 확인하며, "이 회사로 비교하기" CTA로 비교 툴에 진입하는 상호작용을, 그리고 검색 크롤러가 JS 없이 본문을 색인하는 동작을 F4(회사 상세 페이지 ~96개, 복지 데이터가 있는 회사)에 근거하여 표준 유즈케이스로 정의한다.

**상위 추적**: 본 문서는 USECASE→PRD 체인에 속하며, 각 UC는 PRD 기능 요구사항 F4(연동 지점에서 F2·F8)와 타깃 사용자 세그먼트 PRD-SEG-3(회사 복지 정보 검색 유입자)·페르소나 PRD-PER-3·니즈 PRD-NEED-4·맥락 PRD-CTX-2를 인용한다. 데이터 근거는 브리프 §4(데이터 모델)·§5(보존 실데이터), 정적 생성·SEO 근거는 §2-7·§8을 인용한다. 액터 정의(A2 검색유입 사용자, A3 크롤러/봇, A5 빌드/운영자)는 01-액터와-개요를 따른다.

**범위 경계**: 이 문서는 F4(회사 상세 페이지 열람·CTA·크롤)만 다룬다. 상세 페이지는 빌드타임에 생성된 정적 HTML이며(§8, UC-A4 산출물), 본문은 JS 실행 없이 렌더된 상태로 배포된다. CTA 이후의 비교 입력 상세는 F2(UC-2x), 리포트 산출은 F3에서 다룬다. 프로파일러와 익명 경로 서버 사용자저장은 제품 범위에서 영구 제외이며, 로그인·회원은 이 익명 회사 상세 흐름에 등장하지 않는다(로그인·재직 인증은 복지 등록·수정 기여 SC14 한정 — 09-참여.md; 브리프 §2, PRD 04-범위 SC9·SC10·SC14).

**공통 사전조건(모든 UC 적용)**

1. 대상 회사의 상세 페이지가 빌드타임 파이썬 생성 스크립트로 정적 HTML로 생성·배포되어 있다(§8, UC-A4 완료). 본문(기업정보·복지·근무형태)은 JS 실행 없이 완성된 상태다.
2. 페이지는 랜딩을 경유하지 않고 검색엔진 결과(SERP)에서 직접 진입 가능한 독립 URL이다(PRD-CTX-2).
3. 사용자는 가입·로그인 없이 접근하며, 열람 과정에서 서버로의 사용자 데이터 저장은 발생하지 않는다(브리프 §2-1·§2-2, PRD-NEED-1).

**공통 사후조건 제약**: 열람·CTA 진입 전 과정에서 서버 측 사용자 데이터 쓰기가 없다. 회사 프리필 등 상태는 클라이언트(URL 파라미터·localStorage)에만 존재한다(브리프 §2).

**데이터 출처 요약(§4/§5)**: 기업정보는 TCOMPANY(COMP_NM 정식명, COMP_ENG_NM 영문식별자, INDUSTRY_NM 산업, LOGO_NM 로고, COMP_TP_ID→TCOMPANY_TYPE)와 TCOMPANY_TYPE(COMP_TP_CD 유형코드, COMP_TP_NM 유형명). 근무형태는 TCOMPANY.WORK_STYLE_VAL(JSON: remote/flex/unlimitedPTO/refreshLeave/overtime). 복지는 TCOMPANY_BENEFIT(회사 실데이터). 등록 회사는 모두 실복지를 보유하므로 회사페이지 프리셋 폴백은 불필요하다(§5 등록 정책: 복지 데이터가 있는 회사만 등록). TBENEFIT_PRESET(기업유형 프리셋)은 비교 툴 직접 입력 모드용이다(§4).

## [UC-40] 검색엔진에서 회사 상세 페이지 직접 유입·열람

**액터**: PRD-SEG-3(회사 복지 정보 검색 유입자) / 페르소나 PRD-PER-3(정보 탐색자 "수아") / 01문서 A2 검색엔진 유입 사용자.

**사전조건**
- 공통 사전조건 1~3.
- 대상 회사 상세 페이지가 sitemap.xml에 포함되어 검색 색인된 상태(UC-A3 결과), 사용자가 SERP에서 해당 URL을 클릭.

**기본 흐름**
1. 사용자가 검색엔진 결과에서 회사 상세 페이지 URL을 클릭해 직접 진입한다(랜딩 경유 없음, PRD-CTX-2).
2. Nginx가 빌드타임 생성된 정적 HTML을 반환한다.
3. 본문이 JS 실행 없이 렌더된 상태로 즉시 표시된다: 회사 정식명(COMP_NM)·산업(INDUSTRY_NM)·기업유형(COMP_TP_NM)·로고(LOGO_NM), 복지 요약, 근무형태.
4. 사용자가 기업정보·복지·근무형태 섹션을 열람한다(상세 확인은 UC-41).
5. 페이지 내 "이 회사로 비교하기" CTA(UC-43 진입점)와 콘텐츠 페이지 광고/제휴 슬롯(F6, §2-6)이 함께 노출된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 요청 URL의 회사 페이지 미존재(404) | 해당 URL은 색인·열람 대상에서 제외(UC-A3 4a와 정합). 복지 데이터가 있는 회사(~96개)만 정적 생성 대상이며, 그 외 회사는 상세 페이지가 없음 |
| 3a | 모바일 진입(PRD-CTX-1) | 모바일 우선 레이아웃으로 기업정보·복지·근무형태·CTA를 표시(§8) |
| 5a | 사용자가 열람만 하고 이탈 | 비교로 전환하지 않아도 콘텐츠·광고 노출 목적(F4/F6)은 달성. 서버 저장 없음 |

**사후조건**
- 회사 상세 정보가 사용자에게 표시되고, 비교 전환 경로(CTA, UC-43)가 제공된다.
- 공통 사후조건 제약(서버 사용자저장 없음).

**추적**: F4 / PRD-SEG-3, PRD-PER-3 / PRD-NEED-4·PRD-CTX-2 / §2-7(SEO), §8(정적 생성), §4/§5(데이터).

## [UC-41] 기업정보·복지·근무형태 확인

**액터**: PRD-SEG-3 / A2 검색엔진 유입 사용자.

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 회사 상세 페이지에 진입해 열람 중(UC-40).

**기본 흐름**
1. 사용자가 **기업정보** 섹션을 확인한다: 정식명(COMP_NM), 산업(INDUSTRY_NM), 기업유형(COMP_TP_NM), 로고(LOGO_NM). 기업유형은 코드·명칭(COMP_TP_CD·COMP_TP_NM)만 노출하며 파생 지표는 표시하지 않는다(§4 TCOMPANY_TYPE).
2. 사용자가 **복지** 섹션을 확인한다. 복지 항목은 카테고리(§4 BENEFIT_CTGR_CD 9종: compensation/flexibility/work_env/time_off/health/family/growth/leisure/perks)로 구성되어 표시된다.
3. 각 복지 항목은 명칭(BENEFIT_NM)과 함께, 금액형은 금액(BENEFIT_AMT, 만원 단위 §4), 정성형(QUAL_YN=Y)은 금액 대신 정성 설명(QUAL_DESC_CTNT)으로 표시된다.
4. 사용자가 **근무형태** 섹션을 확인한다: WORK_STYLE_VAL(JSON)의 재택(remote)·유연근무(flex)·무제한휴가(unlimitedPTO)·리프레시휴가(refreshLeave)·야근(overtime) 플래그가 표기된다.
5. 사용자가 열람 후 비교 전환(UC-43) 또는 이탈을 선택한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 등록 회사는 모두 실복지 보유(TCOMPANY_BENEFIT) — 복지 없는 회사는 상세 페이지 자체가 생성되지 않음(§5 등록 정책) | 회사페이지 프리셋 폴백은 불필요. TBENEFIT_PRESET은 비교 툴 직접 입력 모드용(§4) |
| 3a | 금액이 정성적이라 NULL(BENEFIT_AMT NULL) | 금액 표기 생략, 정성 설명(QUAL_DESC_CTNT)·비고(NOTE_CTNT)로 대체 |
| 4a | WORK_STYLE_VAL의 특정 키가 없거나 false | 해당 근무형태 플래그를 "미해당/미제공"으로 표기하거나 노출하지 않음(허위 표기 금지) |
| ~~1a~~ | ~~기업유형 특성값 미보유~~ | **폐기(2026-07-20)**: 브랜드 축(성장성·안정성) 제거로 성장 라벨(GROWTH_LABEL_NM)·안정성 점수(STABILITY_SCORE_NO) 자체가 사라져 미보유 분기가 성립하지 않는다. 기업유형은 코드·명칭(COMP_TP_CD·COMP_TP_NM)만 노출한다. 경위: `docs/ANALYSIS-성장성-용어-2026-07-20.md` §6-1 |

**사후조건**
- 회사의 기업정보·복지(카테고리별)·근무형태가 열람된 상태.
- 공통 사후조건 제약.

**추적**: F4 / PRD-SEG-3, PRD-PER-3 / PRD-NEED-4 / §4(TCOMPANY·TCOMPANY_TYPE·TCOMPANY_BENEFIT), §5.

## [UC-42] 복지 배지(공식/추정)·출처 확인

**액터**: PRD-SEG-3 / A2 검색엔진 유입 사용자.

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 복지 섹션을 열람 중(UC-41), 각 복지 항목에 배지 데이터(§4 TCOMPANY_BENEFIT.BADGE_CD 등)가 부여되어 있다.

**기본 흐름**
1. 사용자가 복지 항목별로 신뢰도 배지를 확인한다: **공식(official)** 또는 **추정(est)**(§4 BADGE_CD).
2. 배지 유형에 따라 항목의 데이터 근거가 구분되어 표시된다.

   | 배지(BADGE_CD) | 의미 | 표시 취지 |
   |----------------|------|-----------|
   | official | 공식 확인된 복지 | 회사 공식 출처 기반으로 확인된 항목 |
   | est | 추정 복지 | 추정 기반으로 확정 아님을 사용자에게 고지 |

3. 항목에 출처 정보(BADGE_SRC_CD: scrape_official/scrape_fallback/ai_parse/manual/user_report, BADGE_SRC_URL_CTNT 출처 URL)가 있으면 출처를 확인할 수 있다.
4. 항목에 검증일(VERIFIED_DTM)·만료일(EXPIRES_DTM)이 있으면 데이터 최신성/만료 여부를 확인할 수 있다.
5. 사용자가 배지로 각 복지 정보의 확실성을 판단한 뒤 열람을 이어가거나 비교 전환(UC-43)한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 출처 URL(BADGE_SRC_URL_CTNT) 없음 | 출처 링크 없이 배지만 표시 |
| 4a | 만료일 경과(EXPIRES_DTM < 현재) | 만료·재확인 필요 항목임을 표기(정확성 한계 고지, F7 면책조항과 정합) |

**사후조건**
- 각 복지 항목의 공식/추정 구분과(있으면) 출처·검증/만료 정보가 열람된 상태.
- 공통 사후조건 제약.

**추적**: F4(배지 표기) / PRD-SEG-3 / PRD-NEED-4 / §4(TCOMPANY_BENEFIT.BADGE_CD·BADGE_SRC_CD·BADGE_SRC_URL_CTNT·VERIFIED_DTM·EXPIRES_DTM), F7(데이터 정확성 면책과 정합).

## [UC-43] "이 회사로 비교하기" CTA로 비교 툴 진입(회사 프리필)

**액터**: PRD-SEG-3 / A2 검색엔진 유입 사용자(전환 시 01문서 A1 동작으로 이행).

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 회사 상세 페이지에서 정보를 열람한 뒤 비교를 원하는 상태(UC-40/41/42 경유).

**기본 흐름**
1. 사용자가 페이지 내 "이 회사로 비교하기" CTA를 클릭한다.
2. 클라이언트가 해당 회사 식별자(COMP_ID/COMP_ENG_NM)를 담아 비교 툴(SPA, F2)로 진입한다.
3. 비교 툴이 참조 번들을 로드하고(UC-A1, §6 reference/all), 해당 회사를 비교 슬롯(현직 a / 이직처 b 중 하나)에 프리필한다.
4. 프리필된 회사의 유형·산업·복지(실데이터)가 비교 입력(F2)의 초기값으로 채워진다. 상세 페이지 회사는 모두 등록된 실복지 보유 회사이므로 프리셋 폴백이 아니라 실데이터로 프리필된다(F4 수용기준 3, UC-14 1a와 정합).
5. 사용자가 나머지 슬롯을 검색(F1)으로 지정하고 비교 입력·리포트(F2/F3)로 진행한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 참조 번들 로드 실패(UC-A1 2a) | 재시도/오류 안내 후 참조 데이터 확보 전까지 비교 기능 제한 |
| 5a | 사용자가 프리필 회사를 다른 회사로 교체 | 슬롯 재지정(UC-14 2a) — 상세 페이지 회사 프리필은 초기값일 뿐 고정이 아님. 교체 대상이 등록 회사면 실데이터, 미등록이면 F2 직접 입력 모드(프리셋 템플릿, §4)로 진행 |

**사후조건**
- 상세 페이지의 회사가 비교 슬롯에 프리필된 상태로 비교 입력(F2)이 시작된다.
- 프리필·선택 상태는 클라이언트에만 존재하며 서버 사용자저장이 없다(브리프 §2).

**추적**: F4(CTA), F2(비교 입력 연동) / PRD-SEG-3→(전환)PRD-SEG-1·PRD-SEG-2 / PRD-CTX-2(검색 유입→비교 전환) / 연계 UC-14(슬롯 지정)·UC-A1(참조 로드), §6.

## [UC-44] 검색 크롤러의 JS 없는 정적 본문 크롤·색인

**액터**: A3 검색 크롤러/봇(Googlebot 등). 유입 기반은 A2·PRD-SEG-3.

**사전조건**
- 공통 사전조건 1(정적 페이지 배포).
- 회사 상세 페이지가 sitemap.xml에 등재되어 있다(UC-A4 산출).

**기본 흐름**
1. 크롤러가 회사 상세 페이지 URL을 요청한다(sitemap 또는 링크 경유).
2. Nginx가 빌드타임 생성된 정적 HTML을 반환한다.
3. 본문 콘텐츠(기업정보·복지·근무형태)가 JS 실행 없이 HTML 마크업에 이미 포함되어 있어, 크롤러가 스크립트 렌더링 없이 본문을 그대로 파싱한다(§8: 빌드타임 정적 생성, 빌드 없는 프론트지만 상세 페이지는 사전 렌더된 HTML).
4. 크롤러가 회사명·복지·근무형태 텍스트를 색인한다.
5. 색인된 페이지가 SERP에 노출되어 A2(검색엔진 유입 사용자)의 직접 유입(UC-40) 기반을 형성하고, AdSense 승인 콘텐츠 기반이 된다(§2-7).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 페이지 미존재(404) | 해당 URL을 색인에서 제외(UC-A3 4a와 정합) |
| 3a | 크롤러가 JS를 실행하지 않음 | 본문이 정적 HTML에 포함되어 있으므로 색인 내용 손실 없음(핵심 요구: JS 없는 본문 크롤 가능) |
| 4a | 광고/제휴 슬롯(F6)은 클라이언트 스크립트로 주입 | 광고 영역이 JS로 채워져도 본문 색인에는 영향 없음(본문은 정적) |

**사후조건**
- 회사 상세 페이지 본문이 검색 색인된다.
- A2의 직접 유입 경로(UC-40)와 AdSense 콘텐츠 기반이 형성된다.

**추적**: F4 / A3 / PRD-CTX-2·PRD-SEG-3 / §2-7(SEO), §8(정적 생성) / 상위 연계 UC-A3(사이트맵 색인)·UC-A4(생성·배포).
