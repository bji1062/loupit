# 검색과 회사 선택 (USECASE)

**목적**: loupit 비교 툴에서 비교 대상 회사를 찾아 두 슬롯(현직 a / 이직처 b)에 지정하는 사용자 상호작용을, F1(회사 검색 + 선택)에 근거하여 표준 유즈케이스로 정의한다.

**상위 추적**: 본 문서는 USECASE→PRD 체인에 속하며, 각 UC는 PRD 기능 요구사항 F1(그리고 연동 지점에서 F2/F4)과 타깃 사용자 세그먼트 PRD-SEG-* / 니즈 PRD-NEED-*를 인용한다. 데이터·API 근거는 브리프 §4(데이터 모델)·§6(읽기 전용 API)를 인용한다.

**범위 경계**: 이 문서는 F1(검색·선택)만 다룬다. 조건 입력의 상세(연봉·복지·우선순위 등)는 F2 유즈케이스, 리포트 산출은 F3 유즈케이스에서 다룬다. 로그인/회원/프로파일러/서버 사용자저장은 제품 범위에서 영구 제외이므로 어떤 흐름에도 등장하지 않는다(브리프 §2, PRD 04-범위 SC9·SC10).

**공통 사전조건(모든 UC 적용)**

1. SPA 초기 로드가 완료되어 검색 API(`GET /api/v1/companies/search?q=`, §6) 및 부팅 시 로드한 참조 번들(`reference/all`, §6)에 접근 가능하다.
2. 사용자는 가입·로그인 없이 접근한다(브리프 §2, PRD-NEED-1).
3. 비교 입력(F2) 컨텍스트에서 회사를 지정할 슬롯이 두 개(현직 a / 이직처 b) 제공된다.

**공통 사후조건 제약**: 검색·선택 전 과정에서 서버로의 사용자 데이터 저장이 발생하지 않는다. 슬롯 지정 상태는 클라이언트에만 존재한다(브리프 §2, F1 수용기준 4).

## [UC-10] 회사명 검색

**액터**: PRD-SEG-1(이직 고려 직장인), PRD-SEG-2(신입/경력 오퍼 비교자). 보조: PRD-SEG-3(회사 상세에서 비교로 전환한 사용자).

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 검색을 실행할 대상 슬롯(현직 a 또는 이직처 b)을 정한 상태.

**기본 흐름**
1. 사용자가 회사를 지정할 대상 슬롯(a 또는 b)을 선택한다.
2. 검색 입력란에 회사 정식명의 일부 또는 전체를 입력한다.
3. 클라이언트가 `GET /api/v1/companies/search?q={입력어}`를 호출한다(§6).
4. 서버가 이름/별칭 LIKE 매칭으로 최대 20건의 후보를 `{comp_id, comp_nm, comp_tp_cd, industry_nm, logo_nm}` 형태로 반환한다(§6, F1 주요 출력).
5. 후보 리스트가 회사명·기업유형·산업·로고와 함께 표시된다.
6. 사용자가 후보 중 하나를 선택한다 → UC-14로 연결.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 입력어가 공백 또는 빈 값 | 검색을 호출하지 않고 후보 리스트를 비운 상태로 유지 |
| 4a | 매칭이 20건을 초과 | 상위 20건만 반환한다(§6 상한). 사용자에게 글자를 더 입력해 좁히도록 안내 |
| 4b | 매칭 0건 | UC-13(검색결과 없음 처리)으로 전이 |
| 3b | 검색 API 응답 실패·네트워크 오류 | 무결과와 구분되는 오류 상태를 표시하고 재시도 또는 직접 입력(UC-15) 경로를 안내 |

**사후조건**
- 후보 리스트가 렌더링된다(선택 전). 후보 선택 시 UC-14의 사후조건이 적용된다.
- 공통 사후조건 제약(서버 사용자저장 없음).

**추적**: F1 / PRD-SEG-1, PRD-SEG-2, PRD-SEG-3 / PRD-NEED-1·PRD-NEED-2 / §6(검색 API), §4(TCOMPANY).

## [UC-11] 별칭·영문 통칭 검색

**액터**: PRD-SEG-1, PRD-SEG-2, PRD-SEG-3.

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 회사 정식명을 정확히 모르거나 약칭·영문·그룹 통칭으로 입력하려는 상태.

**기본 흐름**
1. 사용자가 대상 슬롯(a 또는 b)을 지정한다.
2. 약칭·영문 표기·통칭 중 하나를 검색 입력란에 입력한다(F1 설명: 별칭 매칭으로 약칭·영문·통칭을 정식 회사로 연결).
3. 클라이언트가 `GET /api/v1/companies/search?q={입력어}`를 호출한다.
4. 서버가 별칭 테이블(§4 TCOMPANY_ALIAS의 ALIAS_NM) LIKE 매칭으로, 입력한 별칭에 연결된 정식 회사를 후보로 반환한다.
5. 후보는 정식명(`comp_nm`)으로 표시되어, 별칭 입력으로도 사용자가 정식 회사를 식별한다.
6. 사용자가 후보를 선택한다 → UC-14로 연결.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 4a | 서로 다른 별칭이 동일 회사로 수렴 | 동일 회사는 정식명 1건으로 후보에 표기(중복 제거). §4 unique(COMP_ID, ALIAS_NM)로 한 회사 내 별칭 중복은 없음 |
| 4b | 별칭 매칭 0건 | UC-13(검색결과 없음 처리)으로 전이 |
| 2a | 정식명 일부를 입력 | 이름 매칭 경로(UC-10)와 동일 API로 처리되며 이름·별칭 매칭 결과가 함께 후보에 포함 |

**사후조건**
- 별칭·영문·통칭 입력이 정식 회사 후보로 연결되어 표시된다. 선택 시 UC-14의 사후조건 적용.
- 공통 사후조건 제약.

**추적**: F1 / PRD-SEG-1, PRD-SEG-2, PRD-SEG-3 / PRD-NEED-1·PRD-NEED-4 / §4(TCOMPANY_ALIAS), §6.

## [UC-12] 자동완성·부분일치 검색

**액터**: PRD-SEG-1, PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 검색 입력란에 글자를 순차적으로 입력하는 상태(모바일 우선, PRD-CTX-1: 타이핑 부담 최소화).

**기본 흐름**
1. 사용자가 대상 슬롯(a 또는 b)을 지정한다.
2. 검색 입력란에 글자를 순차 입력한다(정식명 또는 별칭의 일부).
3. 입력이 갱신될 때마다 클라이언트가 `GET /api/v1/companies/search?q={현재 입력어}`를 호출한다.
4. 서버가 이름·별칭에 대한 LIKE 부분일치로, 입력 문자열을 부분으로 포함하는 회사를 최대 20건 후보로 반환한다(§6: LIKE 검색, 최대 20건).
5. 입력 글자가 늘어날수록 후보가 좁혀져 표시된다.
6. 사용자가 좁혀진 후보에서 하나를 선택한다 → UC-14로 연결.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 입력이 계속 변경됨 | 후보 리스트는 항상 최신 입력어 기준으로 갱신된다(이전 입력어 기준 응답은 반영하지 않음) |
| 4a | 부분일치 후보가 20건 초과 | 상위 20건만 표시하고 추가 입력으로 좁히도록 안내(§6 상한) |
| 4b | 부분일치 후보 0건 | UC-13(검색결과 없음 처리)으로 전이 |
| 2a | 입력어가 공백/빈 값으로 되돌아감 | 후보 리스트를 비운 상태로 초기화 |

**사후조건**
- 입력 변화에 따라 후보 리스트가 실시간 갱신된 상태로 표시된다. 선택 시 UC-14 사후조건 적용.
- 공통 사후조건 제약.

**추적**: F1 / PRD-SEG-1, PRD-SEG-2 / PRD-CTX-1(모바일 우선) / §6(LIKE·최대 20건).

## [UC-13] 검색결과 없음 처리

**액터**: PRD-SEG-1, PRD-SEG-2, PRD-SEG-3.

**사전조건**
- 공통 사전조건 1~3.
- UC-10/UC-11/UC-12의 검색 실행 결과, 이름·별칭 매칭 후보가 0건.

**기본 흐름**
1. 입력한 검색어에 대해 이름·별칭 매칭이 0건으로 반환된다.
2. 후보 리스트 대신 "검색 결과 없음" 빈 상태를 표시한다.
3. 사용자에게 두 가지 대안을 안내한다: (a) 다른 검색어·별칭으로 재검색, (b) 직접 입력 모드로 진행(UC-15).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 사용자가 철자·약칭을 수정해 재검색 | UC-10/UC-11/UC-12로 재진입 |
| 3b | 사용자가 직접 입력을 선택 | UC-15(회사 미선택 시 직접 입력 모드)로 전이 |
| 1a | 검색 API 오류로 0건처럼 보이는 경우 | 무결과 상태와 오류 상태를 구분해 표시하고 재시도를 유도(무결과로 오인해 직접 입력으로 넘기지 않음) |

**사후조건**
- 대상 슬롯은 회사 미선택 상태로 비어 있다.
- 재검색 대기 또는 직접 입력(UC-15) 진입 중 하나로 이어진다.
- 공통 사후조건 제약.

**추적**: F1 / PRD-SEG-1, PRD-SEG-2, PRD-SEG-3 / §6.

## [UC-14] 비교 대상 회사 선택 (현직 a / 이직처 b)

**액터**: PRD-SEG-1, PRD-SEG-2, PRD-SEG-3.

**사전조건**
- 공통 사전조건 1~3.
- UC-10/UC-11/UC-12에서 후보 리스트가 표시되어 있고, 회사를 반영할 대상 슬롯(a 또는 b)이 지정되어 있다.

**기본 흐름**
1. 사용자가 후보 리스트에서 회사 1건을 선택한다.
2. 선택한 회사가 지정 슬롯(현직 a 또는 이직처 b)에 반영된다.
3. 선택 회사의 기업유형(`comp_tp_cd`)·산업·복지 데이터가 비교 입력(F2)에 연동되어, 실데이터 복지 또는 기업유형 프리셋 폴백으로 복지 항목 초기값이 채워진다(F1 수용기준 2, §4 TCOMPANY_BENEFIT / TBENEFIT_PRESET).
4. 나머지 슬롯에 대해 UC-10~UC-12를 반복하여 서로 다른 회사를 지정한다(F1 수용기준 3: a·b 두 슬롯에 서로 다른 회사 지정 가능).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | F4 회사 상세의 "이 회사로 비교하기" CTA로 진입 | 검색 없이 해당 회사가 슬롯에 사전 지정된 상태로 시작하며(F4 연계), 기본 흐름 2단계에 합류. 주로 PRD-SEG-3의 전환 경로 |
| 2a | 이미 회사가 지정된 슬롯을 다시 선택 | 기존 선택을 새 선택으로 교체 |
| 3a | 선택 회사에 실데이터 복지가 없음 | 기업유형 프리셋으로 폴백해 초기값을 채운다(§4/§5, F2 수용기준 5) |
| 4a | a·b에 동일 회사를 지정 | 두 슬롯에 서로 다른 회사를 지정하는 것이 기본이므로, 다른 회사 선택을 안내한다(비교는 서로 다른 회사를 전제) |

**사후조건**
- 지정 슬롯에 회사가 반영되고, 해당 회사의 유형·산업·복지 초기값이 비교 입력(F2)에 연동된다.
- 선택 상태는 클라이언트에만 존재하며 서버로의 사용자 데이터 저장이 없다(브리프 §2, F1 수용기준 4).

**추적**: F1(선택), F2(입력 연동), F4("이 회사로 비교하기" CTA 진입) / PRD-SEG-1, PRD-SEG-2, PRD-SEG-3 / PRD-NEED-2 / §4, §6.

## [UC-15] 회사 미선택 시 직접 입력 모드

**액터**: PRD-SEG-1, PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.
- 검색 결과가 없거나(UC-13) 사용자가 회사 선택을 건너뛴 상태로, 대상 슬롯이 비어 있다.

**기본 흐름**
1. 사용자가 회사를 선택하지 않고 "직접 입력"으로 진행을 선택한다.
2. 해당 슬롯은 회사 연동 없이 비교 입력(F2)으로 진입한다.
3. 회사가 지정되지 않았으므로 실데이터·프리셋 자동 초기값 없이, 연봉·복지 체크/금액·근무형태·통근시간 등 조건을 사용자가 직접 입력한다(F2).
4. 입력한 조건이 F3 비교 리포트 생성의 입력으로 전달된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 한 슬롯은 회사 선택, 다른 슬롯은 직접 입력 | 슬롯별 독립 구성으로 혼합을 허용(한 슬롯은 UC-14, 다른 슬롯은 직접 입력) |
| 2a | 이후 회사를 검색·선택하기로 변경 | UC-10~UC-12로 전이하여 슬롯에 회사를 지정한다. 회사 선택 시 복지 초기값이 채워져 직접 입력값을 대체할 수 있음을 안내 |

**사후조건**
- 회사 미지정 슬롯에 사용자 직접 입력값이 채워진 상태로 비교 입력(F2)이 구성된다.
- 서버로의 사용자 데이터 저장이 없다(브리프 §2).
