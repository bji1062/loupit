# 비교 리포트 (USECASE)

**목적**: loupit 비교 툴이 F2 입력을 받아 100% 클라이언트 사이드에서 비교 리포트를 생성·조회하고, 총보상·시간조정 가치·워라밸·카테고리별 복지 델타·우선순위 판정·복지 불확실성 밴드를 사용자가 확인하며, 입력 변경 시 재계산하고, 최근 비교를 브라우저 localStorage에 저장·불러오기·삭제하는 사용자 상호작용을 F3(클라이언트 비교 리포트 생성)에 근거하여 표준 유즈케이스로 정의한다.

**상위 추적**: 본 문서는 USECASE→PRD 체인에 속하며, 각 UC는 PRD 기능 요구사항 F3(그리고 연동 지점에서 F2·F6·F7)과 타깃 사용자 세그먼트 PRD-SEG-* / 페르소나 PRD-PER-* / 니즈 PRD-NEED-* / 이용 맥락 PRD-CTX-*를 인용한다. 계산·밴드·데이터 근거는 브리프 §7(비교 알고리즘)·§4(데이터 모델)·§6(읽기 전용 API)를 인용한다.

**범위 경계**: 이 문서는 F3(비교 리포트 생성·조회·재계산·로컬 저장)만 다룬다. 회사 검색·선택은 F1(02 문서), 조건·우선순위·희생요소 입력의 상세는 F2(03 문서)에서 다룬다. 로그인/회원/계정/프로파일러(가치관 진단 설문)/서버 사용자 데이터 쓰기 저장은 제품 범위에서 영구 제외이므로 어떤 흐름에도 등장하지 않는다(브리프 §2-1·§2-3·§2-8, PRD 04-범위 SC9·SC10). 모든 계산은 클라이언트에서 수행되고 서버로 전송·저장되지 않는다(§2-8). 사용자 상태의 유일한 지속 저장소는 브라우저 localStorage다(§2-2).

**UC-ID 대역**: 본 문서는 F3에 **UC-3x** 대역(UC-30~UC-36)을 사용한다. 검색(F1)이 UC-1x 대역을 쓰는 것과 동일한 십단위 대역 규칙이며, 부여한 번호는 재사용하지 않는다(안정 ID).

**공통 사전조건(모든 UC 적용)**

1. F2 비교 입력이 구성되어 a(현직)·b(이직처) 각 슬롯에 조건(연봉, 복지 체크/금액, 근무형태 재택·유연·야근·임금유형, 통근시간)과 우선순위(curPri)·희생요소(curSacrifice)가 존재한다(§7 입력 정의).
2. SPA 부팅으로 참조 번들(`reference/all`, §6)이 로드되어 기업유형 상승률(GROWTH_RATE_VAL)·안정성(STABILITY_SCORE_NO)·프리셋을 참조 가능하다(UC-A1).
3. 사용자는 가입·로그인 없이 접근하며(PRD-NEED-1), 모든 비교 계산은 클라이언트에서 수행된다(§2-8).

**공통 사후조건 제약(모든 UC 적용)**: 리포트 생성·조회·재계산 전 과정에서 계산 입력·결과가 서버로 전송·저장되지 않는다(§2-8). 지속 저장은 사용자가 명시적으로 최근 비교를 남기는 경우에 한해 브라우저 localStorage에만 발생한다(§2-2, UC-36). 기존 로그인 게이트·신고버튼·서버 저장호출은 리포트 경로에 존재하지 않는다(§7, F3 수용기준 4).

## [UC-30] 비교 리포트 생성·조회

**액터**: PRD-SEG-1(이직 고려 직장인), PRD-SEG-2(신입/경력 오퍼 비교자). 페르소나 PRD-PER-1(3년차 개발자 "지원")·PRD-PER-2(취업 준비생 "민재").

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 a·b 두 슬롯에 대한 F2 입력을 마친 상태(회사 선택 또는 직접 입력, UC-14/UC-15).

**기본 흐름**
1. 사용자가 F2 입력을 마치고 "비교하기"를 실행한다.
2. 클라이언트가 `compare()`로 a·b 각 관점의 리포트를 생성한다(§7). 서버 호출·저장은 발생하지 않는다.
3. 각 관점에 대해 총보상(실효연봉=연봉+순복지), 시간조정(시간당) 가치, 워라밸·자율성, 카테고리별 복지 델타, 우선순위 기반 판정(vdCard)을 산출한다(§7 출력 정의).
4. 두 관점(a/b)을 대비하는 형태로 리포트를 화면에 표시한다.
5. 사용자가 리포트를 조회한다(각 지표의 상세 확인은 UC-31~UC-34).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 필수 입력(예: 연봉) 누락 | 리포트를 생성하지 않고 부족 항목을 안내하며 F2 입력으로 복귀 |
| 3a | 한 슬롯이 회사 미선택 직접 입력(UC-15)이라 프리셋 초기값 없음 | 사용자가 직접 입력한 값만으로 해당 슬롯을 계산한다 |
| 4a | 비교 결과 화면 진입 | 입력 화면과 달리 결과 화면에는 절제된 광고 슬롯이 노출될 수 있다(§2-6, F6). 광고 슬롯 상세는 F6(06 문서) |
| 5a | 사용자가 재방문·재진입 | 최근 비교가 있으면 UC-36으로 저장분을 불러와 조회 상태로 복원한다 |

**사후조건**
- a/b 비교 리포트가 클라이언트에 렌더링·조회된다.
- 공통 사후조건 제약(서버 전송·저장 없음). 사용자가 원하면 UC-36으로 localStorage 저장.

**추적**: F3 / PRD-SEG-1, PRD-SEG-2 / PRD-PER-1, PRD-PER-2 / PRD-NEED-2(두 직장 한 화면 비교) / §7(compare 출력), §6(참조 데이터), §2-8, §2-6.

## [UC-31] 총보상·시간조정 가치·워라밸·자율성 확인

**액터**: PRD-SEG-1, PRD-SEG-2. 페르소나 PRD-PER-1(연봉 상승분이 복지·야근 감소를 상쇄하는지 판단).

**사전조건**
- 공통 사전조건 1~3.
- UC-30으로 a/b 리포트가 생성·표시되어 있다.

**기본 흐름**
1. 사용자가 총보상(실효연봉=연봉+순복지) 항목을 확인한다. 순복지는 카테고리별 복지 합산(`getBenTotal`)에서 도출된다(§7).
2. 시간조정(시간당) 가치를 확인한다: 주간근무시간(`getWSHours`)과 야근수당(`getOTPay`, 임금유형 포괄/비포괄 분기)을 반영한 시간당 가치(§7).
3. 워라밸·자율성 요인을 확인한다: 재택절약(REMOTE_SAVE)·유연근무·무제한휴가 반영(§7).
4. a/b 두 관점의 각 지표 값과 차이를 대비하여 확인한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 임금유형=포괄(wage) | 야근수당이 급여에 포함된 것으로 처리(`getOTPay` 포괄 분기). 비포괄이면 야근시간 기반 별도 수당으로 산정(§7) |
| 3a | 재택·유연·무제한휴가 미체크 | 해당 워라밸·자율성 가산 없이 계산한다 |
| 1a | 복지 항목이 정성적(QUAL_YN=Y, BENEFIT_AMT NULL) | 총보상 금액 합산에서 제외하고, 정성 항목은 UC-32의 정성 복지 비교로 별도 표기한다(§4) |

**사후조건**
- 사용자가 두 관점의 총보상·시간당 가치·워라밸·자율성 지표를 확인한 상태.
- 공통 사후조건 제약.

**추적**: F3 / §7(실효연봉·getBenTotal·getWSHours·getOTPay·REMOTE_SAVE) / §4(BENEFIT_AMT 만원 단위, QUAL_YN) / PRD-SEG-1, PRD-SEG-2 / PRD-PER-1.

## [UC-32] 카테고리별 복지 델타·정성 복지 확인

**액터**: PRD-SEG-1, PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.
- UC-30으로 a/b 리포트가 생성·표시되어 있다.

**기본 흐름**
1. 사용자가 리포트의 복지 비교 영역에서 9개 카테고리(compensation/flexibility/work_env/time_off/health/family/growth/leisure/perks)별 a·b 복지 합산을 확인한다(`renderBenCatCompare`, §4 BENEFIT_CTGR_CD 9종, §7).
2. 각 카테고리에서 a와 b의 금액 차이(델타)를 확인한다.
3. 정성적 복지(QUAL_YN=Y, 금액 없음)는 금액 델타 대신 정성 비교로 대비 표시된다(`renderQualCompare`, §4 QUAL_DESC_CTNT).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 한 슬롯만 특정 카테고리 복지 보유 | 델타를 보유 슬롯 금액 기준으로 표기(미보유 슬롯은 0 기준) |
| 1b | 회사 미선택 슬롯의 복지가 유형 프리셋 폴백값 | 프리셋(§4 TBENEFIT_PRESET)으로 채워진 값으로 카테고리 합산·델타를 계산한다 |
| 3a | 카테고리에 정성 항목만 있고 금액 없음 | 금액 델타 대신 정성 설명(QUAL_DESC_CTNT)을 a/b 대비로 표시한다 |

**사후조건**
- 사용자가 9개 카테고리별 복지 델타와 정성 복지의 a/b 대비를 확인한 상태.
- 공통 사후조건 제약.

**추적**: F3 / §4(BENEFIT_CTGR_CD 9종, QUAL_YN/QUAL_DESC_CTNT, TBENEFIT_PRESET) / §7(renderBenCatCompare/renderQualCompare) / PRD-SEG-1, PRD-SEG-2.

## [UC-33] 우선순위 기반 판정(vdCard) 확인

**액터**: PRD-SEG-1, PRD-SEG-2. 페르소나 PRD-PER-2(브랜드보다 워라밸을 우선순위로 직접 선택해 선택지 좁히기).

**사전조건**
- 공통 사전조건 1~3.
- UC-30으로 a/b 리포트가 생성되어 있고, F2에서 우선순위(curPri)·희생요소(curSacrifice)가 사용자에 의해 직접 선택된 상태(§2-3).

**기본 흐름**
1. 클라이언트가 사용자가 직접 선택한 우선순위(curPri: 연봉/워라밸/복지/브랜드)와 희생요소(curSacrifice)를 판정 입력으로 사용한다(§2-3, §7). 프로파일러 설문은 존재하지 않는다.
2. 선택 축에 따라 두 관점을 평가한다: 연봉→총보상·시간당 가치, 워라밸→워라밸·자율성 요인, 복지→카테고리별 복지 합산, 브랜드→기업유형 상승률(GROWTH_RATE_VAL)·안정성(STABILITY_SCORE_NO).
3. "누가 유리 / 무엇을 택할지" 판정 카드(vdCard)를 표시한다(§7).
4. 사용자가 선택 우선순위 축을 기준으로 한 판정 근거를 확인한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 우선순위 미선택 | 판정에 앞서 우선순위 직접 선택을 요구하고 F2로 안내한다. 프로파일러가 없으므로 판정에는 사용자 직접 선택이 필요하다(§2-3) |
| 2a | 브랜드 우선순위인데 한 슬롯이 회사 미선택(직접 입력, UC-15) | 기업유형 상승률·안정성 참조가 없어 브랜드 축 판정을 제한하고, 회사 선택 또는 다른 축 사용을 안내한다 |
| 3a | 두 관점이 선택 축에서 근소한 차이 | 판정 카드가 우열이 근소함을 표기하여 과잉 단정을 피한다 |

**사후조건**
- 선택 우선순위 기준의 판정 카드(vdCard)가 표시된 상태.
- 판정은 전적으로 클라이언트 계산이며 서버 판정 호출이 없다(§7, F3 수용기준 4).

**추적**: F3 / §2-3(우선순위 직접 선택), §7(vdCard, curPri/curSacrifice), §6(기업유형 상승률·안정성) / PRD-NEED-3(중요 축 직접 선택) / PRD-SEG-1, PRD-SEG-2 / PRD-PER-2.

## [UC-34] 복지 불확실성 밴드(배지) 이해

**액터**: PRD-SEG-1, PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.
- UC-30으로 리포트가 표시되어 복지 금액·총보상이 산출되어 있다.

**기본 흐름**
1. 리포트의 복지 금액·총보상 옆에 배지가 표기된다: 공식(official), 추정(est), 만료(EXPIRES_DTM 경과)(§4 BADGE_CD, EXPIRES_DTM).
2. 배지에 따라 불확실성 밴드(±)가 적용된다: official ±5%, est ±20%, 만료 ±15%(§7).
3. 사용자가 배지·밴드를 통해 복지 금액의 신뢰수준을 이해한다(추정치는 밴드 폭이 넓음).
4. 밴드가 총보상 등 금액 지표를 단일값이 아닌 범위로 표시하는 데 반영된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 실데이터 복지 없이 기업유형 프리셋으로 채워진 항목 | 프리셋 배지(§4 TBENEFIT_PRESET BADGE_CD) 기준으로 밴드를 표기한다 |
| 2a | 한 항목이 만료(EXPIRES_DTM 경과) 상태 | 만료 밴드(±15%) 규칙을 적용한다 |
| 3a | 정성 항목(QUAL_YN=Y, 금액 없음) | 금액 밴드를 적용하지 않고 배지·출처(BADGE_SRC_CD)만 표기한다 |

**사후조건**
- 사용자가 복지 배지와 불확실성 밴드의 의미를 이해하고, 리포트 수치가 정확한 단일값이 아니라 밴드 범위임을 인지한 상태.
- 공통 사후조건 제약.

**추적**: F3 / §7(밴드 규칙 official ±5% / est ±20% / 만료 ±15%), §4(BADGE_CD, EXPIRES_DTM, BADGE_SRC_CD) / F7(데이터 정확성 면책조항과 정합) / PRD-SEG-1, PRD-SEG-2.

## [UC-35] 입력 변경 시 리포트 재계산

**액터**: PRD-SEG-1, PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.
- UC-30으로 리포트가 조회되고 있고, 사용자가 F2 입력을 조정할 수 있는 상태.

**기본 흐름**
1. 사용자가 조회 중 F2 입력(연봉, 복지 체크/금액, 근무형태, 통근시간, 우선순위, 희생요소) 중 하나 이상을 변경한다.
2. 클라이언트가 변경된 입력으로 리포트를 재계산한다(`compare()`, 실시간 요약 `calc()`, §7). 서버 호출은 발생하지 않는다.
3. 총보상·시간당 가치·워라밸·자율성·카테고리별 복지 델타·판정(vdCard)·불확실성 밴드가 갱신된 값으로 다시 표시된다.
4. 사용자가 갱신된 a/b 비교 결과를 확인한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 우선순위(curPri)만 변경 | 판정 축이 바뀌어 vdCard가 재산출된다(다른 지표 값은 동일, 판정만 축 전환, UC-33) |
| 2b | 회사 교체·슬롯 재지정(UC-14) | 새 회사의 실데이터 또는 유형 프리셋 복지로 초기값이 바뀌고 전체 지표를 재계산한다 |
| 1a | 변경으로 필수값(예: 연봉)이 비워짐 | 해당 지표를 미산출로 표시하고 입력 보완을 안내한다(리포트 전체를 무효화하지 않음) |

**사후조건**
- 변경된 입력을 반영한 최신 리포트가 표시된 상태.
- 재계산은 클라이언트에서만 발생하며 서버 저장이 없다(§2-8). 공통 사후조건 제약.

**추적**: F3 / §7(compare 리포트 생성, calc 실시간 요약) / F2(입력 변경 연동) / PRD-SEG-1, PRD-SEG-2 / §2-8.

## [UC-36] 최근 비교 로컬 저장·불러오기·삭제 (localStorage)

**액터**: PRD-SEG-1, PRD-SEG-2. 이용 맥락 PRD-CTX-3(로그인·개인정보 제공 원치 않음)·PRD-CTX-4(1회성·짧은 세션, 재방문 시 계정 복원 없음).

**사전조건**
- 공통 사전조건 1~3.
- UC-30으로 비교 리포트가 생성·조회된 상태.
- 브라우저 localStorage 사용이 가능하다.
- **서버 계정·서버 저장은 존재하지 않으며(§2-1), 저장은 오직 브라우저 localStorage에만 이루어진다(§2-2). 저장·불러오기·삭제 전 과정이 로그인 없이 기기 내에서만 동작한다.**

**기본 흐름**
1. 사용자가 조회한 비교를 "최근 비교"로 저장한다 → 클라이언트가 비교 입력과 결과 요약을 브라우저 localStorage에 기록한다(서버 전송 없음, §2-2). 이는 공통 로컬 저장 유즈케이스 UC-A2를 F3 리포트 맥락에서 구체화한 것이다.
2. 사용자가 재방문·재진입 시 최근 비교 목록을 불러온다 → localStorage에서 읽어 리포트를 복원 제시한다(UC-30 조회 상태로 재진입).
3. 사용자가 특정 최근 비교 항목을 삭제한다 → 해당 localStorage 항목을 제거하고 목록에서 뺀다.
4. UI는 저장·불러오기·삭제가 로그인·서버 통신 없이 이 기기 내에서만 유지됨을 명확히 고지한다(크로스 디바이스 동기화 없음).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | localStorage 비활성/용량 초과 | 저장을 생략한다. 비교·조회·재계산은 정상 동작하되 저장·복원만 제공하지 않는다(UC-A2 대안과 정합) |
| 2a | 저장된 최근 비교 없음 | 복원 목록을 표시하지 않는다(빈 상태) |
| 2b | 복원 후 최신 참조로 갱신 원함 | 복원 항목은 저장 시점의 사용자 입력을 기준으로 표시하며, 사용자가 원하면 UC-35 재계산으로 갱신한다 |
| 3a | 사용자가 최근 비교 전체 삭제 | 모든 항목을 제거하여 목록을 비운다 |

**사후조건**
- 최근 비교가 해당 기기의 localStorage에만 저장·유지·삭제된다. 서버 저장·크로스 디바이스 동기화는 없다(PRD-CTX-4).
- 로그인·회원 없이 로컬에서만 상태가 유지되며, 서버로의 사용자 데이터 저장이 발생하지 않는다(§2-1, §2-2).

**추적**: F3 / UC-A2(공통 로컬 저장·복원) / PRD-NEED-5(최근 비교 기기 내 유지) / PRD-CTX-3, PRD-CTX-4 / §2-1, §2-2.
