# 오류와 엣지케이스 (USECASE)

**목적**: loupit의 정상 경로가 아닌 **비정상·엣지 상황**을 표준 유즈케이스로 정의한다. 대상은 (1) 참조 번들 `reference/all` 로드 실패, (2) 회사 검색 API 실패, (3) localStorage 사용 불가(사생활 모드·차단·용량 초과), (4) 상세 복지가 없는 회사의 기업유형 프리셋 폴백, (5) JavaScript 비활성·크롤러의 정적 콘텐츠 접근(점진적 향상), (6) 존재하지 않는 회사 slug·잘못된 URL의 404 처리다. 각 UC는 "오류를 사용자에게 알리되 앱이 크래시하지 않는다"(NFR26)와 "저장 실패를 흡수하고 비교는 정상 동작한다"(NFR25)를 검증 가능한 흐름으로 구체화한다.

**상위 추적**: 본 문서는 USECASE→PRD 체인에 속한다. 각 UC는 PRD 기능 요구사항(주로 F8 읽기 전용 API, F1 검색, F2 비교 입력, F3 비교 리포트, F4 회사 상세, F5 인기 조합, F7 정책)과 비기능 요구사항(07-비기능-요구사항 **NFR24·NFR25·NFR26**, 보조로 NFR3·NFR5·NFR11·NFR12), 타깃 사용자(PRD-SEG-*/PRD-PER-*/PRD-CTX-*/PRD-NEED-*)를 인용한다. 실패·폴백·정적 서빙 근거는 브리프 §6(API·캐시 표면)·§4/§5(데이터 모델·프리셋 폴백)·§7(클라이언트 비교)·§8(정적 서빙 인프라)·§2-2(localStorage)·§2-8(클라이언트 계산)을 인용한다.

**범위 경계**: 이 문서는 **비정상·엣지 처리**만 다룬다. 정상(해피패스) 흐름은 각 담당 문서에서 다룬다 — 검색·슬롯 지정 F1(UC-10~UC-15, 02 문서), 비교 입력 F2(UC-20~UC-25, 03 문서), 비교 리포트 F3(UC-30~UC-36, 04 문서), 회사 상세 F4(UC-40~UC-44, 05 문서), 인기 조합 F5(UC-50~UC-54, 06 문서), 시스템 공통 UC-A1~UC-A4(01 문서). 로그인/회원/계정/프로파일러(가치관 진단 설문)/서버 측 사용자 데이터 쓰기 저장은 제품 범위에서 **영구 제외**이므로 어떤 오류·엣지 흐름에도 등장하지 않는다(브리프 §2-1·§2-3·§2-4, PRD 04-범위 SC9·SC10).

**UC-ID 대역**: 본 문서는 **UC-9x** 대역(UC-90~UC-95)을 사용한다. 이는 특정 기능 하나에 귀속되지 않고 여러 기능을 가로지르는 **오류·엣지 처리** 대역이며, F1~F5 상세 문서가 쓰는 십단위 대역(F1=1x, F2=2x, F3=3x, F4=4x, F5=5x)과 겹치지 않는다. 부여한 번호는 재사용하지 않는다(안정 ID).

## [UC-90] 참조 번들 `reference/all` 로드 실패·오류 상태 처리

**액터**: A1 익명 방문자(비교 툴 SPA 사용자, PRD-SEG-1·PRD-SEG-2). A2가 CTA로 비교 툴에 전환 진입한 경우 동일.

**사전조건**
- 공통 전제 1~3.
- 사용자가 비교 툴(SPA)에 진입하여 부팅 시 `GET /api/v1/reference/all`을 호출한다(UC-A1 기본 흐름).
- 요청이 정상 응답하지 못하는 상황이다(네트워크 오류·타임아웃·5xx). 브라우저/프록시 캐시에 유효한 번들이 없다(최초 방문 또는 `max-age` 1시간 경과).

**기본 흐름**
1. 프론트가 부팅 시 `GET /api/v1/reference/all`을 1회 호출한다(UC-A1 2단계).
2. 요청이 네트워크 오류·타임아웃·5xx 등으로 실패한다.
3. 프론트가 예외를 흡수하여 앱을 크래시시키지 않고 **오류 상태**로 전환한다(NFR26, 공통 전제 1).
4. 사용자에게 **오류 안내와 재시도 수단**을 표시한다(NFR26).
5. 참조 데이터가 확보되기 전까지 검색(F1)·비교 입력(F2)·비교 계산(F3)을 **제한**한다(UC-A1 예외 2a: 참조 확보 전 기능 제한).
6. 사용자가 재시도하여 성공하면 프론트가 번들(`company_types[]`·`benefit_presets{…}`·`companies[…]`)을 메모리에 보관하고 정상 흐름(UC-A1)으로 복귀한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 유효 캐시 존재(`max-age` 1시간 이내) | 네트워크 호출 없이 캐시 번들로 부팅한다. 실패 화면 없음(NFR3, UC-A1 3a) |
| 3a | 부분·손상 응답(불완전 JSON, 최상위 키 누락) | 손상 응답을 유효 번들로 취급하지 않고 오류 상태로 처리한다. 번들 최상위 키(`company_types`/`benefit_presets`/`companies`) 유무를 검증한다(§6) |
| 5a | 직전 방문에서 번들이 이미 로드되어 메모리에 있음 | 비교 계산은 클라이언트 사이드이므로 로드 성공 후에는 네트워크가 끊겨도 계산이 가능하다(NFR26, 공통 전제 3) |
| 6a | 재시도 반복 실패 | 오류 안내를 유지하고 앱은 크래시하지 않는다(NFR26). 사용자는 정적 콘텐츠 페이지(F4/F5) 열람으로 이탈할 수 있다 |

**사후조건**
- `reference/all` 실패가 오류 상태·안내로 전환되고 앱이 크래시하지 않은 상태. 서버 사용자 저장 없음.
- 재시도 성공 시 UC-A1 정상 흐름으로 복귀하여 검색·비교가 가능해진다.

**추적**: F8(`reference/all`) → F1·F2·F3 / NFR26(API 실패 안내·무크래시), NFR3(캐시) / PRD-SEG-1, PRD-SEG-2 / UC-A1 / §6, §2-8.

## [UC-91] 회사 검색 API 실패 처리

**액터**: A1 익명 방문자(PRD-SEG-1·PRD-SEG-2).

**사전조건**
- 공통 전제 1~3.
- 사용자가 검색어를 입력하여 `GET /api/v1/companies/search?q=`를 호출한다(F1, UC-10~UC-15).
- 검색 요청이 실패하는 상황이다(네트워크 오류·타임아웃·5xx).

**기본 흐름**
1. 사용자가 검색창에 회사명 또는 별칭 일부를 입력한다(F1).
2. 프론트가 `GET /api/v1/companies/search?q=`를 호출한다(§6, 이름/별칭 LIKE, 최대 20건).
3. 요청이 네트워크 오류·타임아웃·5xx로 실패한다.
4. 프론트가 예외를 흡수하고 앱을 크래시시키지 않는다(NFR26, 공통 전제 1).
5. 사용자에게 검색 실패 안내와 재시도 수단을 표시한다(NFR26).
6. 재시도가 성공하면 후보 리스트(최대 20건: `{comp_id, comp_nm, comp_tp_cd, industry_nm, logo_nm}`)를 표시하고 슬롯 지정(UC-14, 현직 a / 이직처 b)으로 이어간다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | `reference/all` 번들이 이미 메모리에 로드됨(UC-A1) | 번들은 `companies[{…, aliases[]}]`를 인라인 포함하므로(§6), 전용 검색 엔드포인트가 실패해도 로드된 번들에 대한 클라이언트 이름/별칭 매칭으로 후보를 제시할 수 있다(§6, UC-A1 4단계 "검색의 단일 참조원") |
| 3b | 요청은 성공했으나 결과 0건 | 실패가 아니라 "결과 없음"으로 구분 표시한다(별칭 미매칭). 오류 안내를 띄우지 않는다 |
| 5a | 재시도 반복 실패 & 번들 미로드 | 오류 안내를 유지하고 크래시하지 않는다. 검색 없이는 슬롯 지정을 진행할 수 없음을 안내한다 |

**사후조건**
- 검색 API 실패가 안내되고 앱이 크래시하지 않은 상태. 서버 사용자 저장 없음.
- 번들이 로드되어 있으면 클라이언트 매칭으로 검색을 이어갈 수 있다.

**추적**: F8(`companies/search`) · F1 / NFR26 / PRD-SEG-1, PRD-SEG-2, PRD-NEED-1(가입 없이 즉시 이용) / UC-A1 · UC-10~UC-15(02 문서) / §6.

## [UC-92] localStorage 사용 불가(사생활 모드·차단·용량 초과) 처리

**액터**: A1 익명 방문자(PRD-SEG-1·PRD-SEG-2).

**사전조건**
- 공통 전제 1~3.
- 사용자가 비교를 수행하거나 리포트를 조회한 뒤 "최근 비교"를 저장/복원하려 한다(UC-A2).
- 브라우저 localStorage가 사용 불가한 상황이다(사생활/시크릿 모드, 브라우저 저장소 차단 설정, 용량 초과 `QuotaExceededError`).

**기본 흐름**
1. 사용자가 비교를 수행하거나 리포트를 조회한다(F2/F3).
2. 프론트가 "최근 비교" 항목을 localStorage에 저장 시도한다(UC-A2 2단계, 서버 전송 없음).
3. 사생활 모드·저장소 차단·용량 초과로 저장이 실패한다.
4. 프론트가 저장 실패를 `try/catch`로 흡수하고 오류 없이 진행한다(NFR25).
5. 비교 입력·계산·리포트(F2/F3)는 정상 동작하며, "최근 비교"만 저장·복원되지 않는다(NFR25, UC-A2 예외 2a).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 4a | 재방문 시 복원 시도했으나 저장분 없음 | 복원 항목을 표시하지 않는다(UC-A2 3a). 비교는 새로 입력하여 정상 수행한다 |
| 3a | 용량 초과(`QuotaExceeded`)만 발생 | 신규 저장을 생략하거나 오래된 "최근 비교"를 정리 후 재저장할 수 있으며, 실패해도 비교 기능은 영향받지 않는다(NFR25) |
| 5a | 사용자가 결과의 기기 간 보관을 기대 | 서버 저장·크로스 디바이스 동기화는 제공하지 않는다(PRD-CTX-4). localStorage 외 서버 전송은 없다(§2-2) |

**사후조건**
- localStorage 불가 상황에서도 비교 기능이 정상 동작하고 앱이 크래시하지 않은 상태. "최근 비교"만 미저장.
- 서버 전송·저장은 발생하지 않는다.

**추적**: F2 · F3 / NFR25(localStorage 실패 흡수·비교 정상 동작) / PRD-SEG-1, PRD-SEG-2, PRD-NEED-5(기기 내 최근 비교 유지), PRD-CTX-4(1회성·재방문 복원 없음) / UC-A2 / §2-2, §7.

## [UC-93] 상세 복지 없는 회사의 기업유형 프리셋 폴백

**액터**: 데이터 소비 A1(비교 입력·계산 시), A2(회사 상세 열람 시). 정적 페이지 생성 시 A5 빌드/운영자.

**사전조건**
- 공통 전제 1~3.
- 대상 회사에 회사별 실데이터 복지(TCOMPANY_BENEFIT)가 없다(상세 복지 SQL 96개 대상 밖의 회사, §5).
- 회사의 기업유형(TCOMPANY.COMP_TP_ID)에 대응하는 기본 복지 프리셋(TBENEFIT_PRESET)이 존재한다(§4).

**기본 흐름**
1. 시스템이 대상 회사의 복지를 조회·구성한다(`companies/{id}` 응답, 또는 비교 입력 F2 초기값 채움, 또는 정적 페이지 생성 F4/F5).
2. 회사별 실데이터 복지(TCOMPANY_BENEFIT)가 없음을 확인한다.
3. 회사의 기업유형(COMP_TP_ID)에 대응하는 TBENEFIT_PRESET로 폴백하여 기본 복지를 채운다(§4, §6 "프리셋 폴백 병합").
4. 프리셋 항목(BENEFIT_NM·BENEFIT_AMT·BENEFIT_CTGR_CD·BADGE_CD·DEFAULT_CHECKED_YN)으로 화면·리포트·정적 페이지의 복지를 구성한다.
5. 비교 입력(F2)에서는 프리셋 초기값이 채워지고 사용자가 이를 수정할 수 있다(UC-21/UC-22 연동, F2 수용기준 5).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | `companies/{id}` API 응답 경로 | 프리셋 폴백이 병합되어 실데이터 없는 회사도 복지를 반환한다(§6, F8 수용기준 4) |
| 1b | 정적 상세·조합 페이지 생성 경로(빌드타임, A5) | 실데이터 없는 회사는 프리셋으로 폴백해 복지를 채운다(UC-A4 3단계, F4 수용기준 2) |
| 3a | 실데이터가 일부 항목만 존재 | 존재하는 실데이터를 우선 사용하고, 누락된 항목만 프리셋으로 보완한다(병합) |
| 4a | 정성적 복지(QUAL_YN=Y, BENEFIT_AMT NULL) 포함 | 금액 합산에서 제외하고 정성 항목으로 표기한다(§4 QUAL_DESC_CTNT). 프리셋·실데이터 공통 규칙 |

**사후조건**
- 실데이터 복지가 없는 회사도 기업유형 프리셋으로 복지가 채워져 비교·열람·정적 페이지가 정상 구성된 상태. 서버 사용자 저장 없음.

**추적**: F2 · F3 · F4 · F5 / F8(`companies/{id}` 프리셋 폴백 병합) / PRD-SEG-1, PRD-SEG-2, PRD-SEG-3 / UC-A4(빌드 폴백) · UC-52 예외 4b(비교 툴 폴백) / §4(TBENEFIT_PRESET), §5(보존 데이터·폴백), §6.

## [UC-94] JS 비활성·크롤러의 정적 콘텐츠 접근(점진적 향상)

**액터**: A2 검색엔진 유입 사용자(JavaScript 비활성 환경, PRD-SEG-3), A3 검색 크롤러/봇.

**사전조건**
- 공통 전제 1~3.
- 회사 상세(F4)·인기 조합(F5)·정책(F7) 정적 페이지가 배포되어 있다(UC-A4 완료 상태).
- 방문자·크롤러가 JavaScript를 실행하지 않는다(브라우저 JS 비활성 설정 또는 크롤러 특성).

**기본 흐름**
1. A2/A3가 정적 콘텐츠 페이지 URL을 요청한다.
2. Nginx가 완성된 정적 HTML을 반환한다(§8, 요청 시 서버 렌더링·DB 조회 없음 — NFR1).
3. JS 없이도 초기 HTML에 본문(회사: 기업정보·복지·근무형태 / 조합: 두 회사 핵심 비교 / 정책: 정책 문안)이 존재하여 가독된다(NFR12, NFR24).
4. 크롤러(A3)는 본문을 색인하고(UC-A3), 사용자(A2)는 본문을 열람한다.
5. 비교 CTA 등 인터랙션은 JS 향상 요소이며, JS 미실행 시에도 본문 열람은 유지된다(NFR24: "인터랙션만 JS 향상").

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 5a | 비교 툴(SPA, F1~F3)에 JS 비활성으로 접근 | 비교 툴은 클라이언트 계산(§2-8) 기반이라 JS가 필요하다. 정적 콘텐츠 페이지와 달리 인터랙션이 동작하지 않으며, 비-JS 본문 가독 보장 대상은 콘텐츠 페이지(F4/F5/F7)에 한정된다(NFR24 범위) |
| 3a | 광고/제휴 슬롯(F6)이 JS 미실행으로 미로드 | 광고/제휴는 JS 향상 요소로 미표시될 수 있으나 본문 콘텐츠 가독에는 영향이 없다. 슬롯은 고정 크기 예약으로 레이아웃 이동을 방지한다(NFR5) |
| 2a | 요청 페이지가 존재하지 않음 | 404로 처리한다(UC-95, UC-A3 4a) |

**사후조건**
- JS 미실행 상태에서도 정적 콘텐츠 페이지 본문이 가독·색인 가능한 상태. 서버 사용자 저장 없음.
- A3 색인이 A2의 검색 유입 기반을 형성한다(§2-7).

**추적**: F4 · F5 · F7 / NFR24(비-JS 본문 가독·인터랙션만 JS 향상), NFR12(비-JS 크롤러 가독), NFR1(정적 서빙), NFR5(슬롯 CLS 예약) / PRD-SEG-3, PRD-CTX-2(검색 유입) / A2 · A3 / UC-A3 / §2-7, §8.

## [UC-95] 존재하지 않는 회사 slug·잘못된 URL(404) 처리

**액터**: A2 검색엔진 유입 사용자(잘못된 URL로 진입), A3 검색 크롤러/봇. API 경로에서는 A1 익명 방문자.

**사전조건**
- 공통 전제 1~3.
- 정적 회사 상세 페이지는 회사 200개(존재하는 slug)에 한해, 인기 조합 페이지는 사전 정의된 회사쌍 목록에 한해 생성된다(F4/F5, §5).
- 요청 URL이 존재하지 않는 회사 slug(오타·미생성) 또는 미배포 조합을 가리킨다.

**기본 흐름**
1. 사용자(A2) 또는 크롤러(A3)가 존재하지 않는 회사 slug URL을 요청한다.
2. 해당 정적 페이지가 배포되어 있지 않다(200개 회사 목록·인기 조합 목록 밖).
3. Nginx가 404를 반환한다(UC-A3 예외 4a, UC-50 예외 3a와 동일 규칙: 존재하는 페이지만 존재).
4. 크롤러(A3)는 해당 URL을 색인에서 제외한다(UC-A3 4a).
5. 사용자(A2)는 404 안내를 받고 유효한 페이지(회사 상세·인기 조합·랜딩)로 이동할 수 있다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | `companies/{id}` API에 미존재 `comp_id` 요청(A1) | 존재하지 않는 식별자는 정상 데이터가 아니므로 not-found로 응답하고, 프론트는 오류를 흡수하여 크래시하지 않는다(NFR26 정합, F8) |
| 3a | 별칭·검색어 형태의 잘못된 slug | slug는 회사 영문식별자(COMP_ENG_NM, §4)를 기준으로 한 정적 경로이며, 별칭·통칭·검색어는 slug가 아니라 검색(F1, `companies/search`)으로 처리한다 |
| 5a | 존재하는 조합의 역순 URL 등 중복 경로 | canonical로 단일 정규화한다(NFR11). 미존재 조합은 404로 처리한다 |

**사후조건**
- 존재하지 않는 URL이 404로 처리되어 색인 제외·앱 무크래시가 성립한 상태. 서버 사용자 저장 없음.
- 유효 페이지로의 이동 경로가 안내되어 탐색이 이어질 수 있다.

**추적**: F4 · F5 / F8(`companies/{id}` not-found), NFR26(무크래시), NFR11(canonical 정규화) / PRD-SEG-3, PRD-CTX-2 / A2 · A3 / UC-A3 예외 4a · UC-50 예외 3a / §4(COMP_ENG_NM), §5.
