# 비교 입력 (USECASE)

**목적**: loupit 비교 툴에서 두 직장(현직 a / 이직처 b)의 조건을 사용자가 직접 입력하는 상호작용을, F2(비교 입력)에 근거하여 표준 유즈케이스로 정의한다. 연봉·연봉범위, 복지 체크/금액, 근무형태(재택/유연/야근/임금유형), 통근시간, 우선순위 직접 선택, 그리고 선택 회사의 복지 프리셋 자동 채움·수정을 다룬다.

**상위 추적**: 본 문서는 USECASE→PRD 체인에 속하며, 각 UC는 PRD 기능 요구사항 F2(그리고 회사 지정 연동 지점에서 F1, 결과 산출 지점에서 F3)와 타깃 사용자 세그먼트 PRD-SEG-* / 니즈 PRD-NEED-* / 페르소나 PRD-PER-*를 인용한다. 입력 항목 정의·계산 연동은 브리프 §7(비교 알고리즘), 데이터·프리셋은 §4(데이터 모델), 광고 최소화 정책은 §2-6을 인용한다.

**범위 경계**: 이 문서는 F2(비교 조건 입력)만 다룬다. 비교 대상 회사를 찾아 슬롯에 지정하는 상호작용은 F1 유즈케이스(UC-10~UC-15), 입력을 받아 실효연봉·시간가치·워라밸·판정을 산출하는 리포트 생성은 F3 유즈케이스에서 다룬다. 우선순위는 프로파일러(가치관 진단) 없이 **사용자가 직접 선택**한다(§2-3, F2 수용기준 1). 로그인/회원/프로파일러/서버 사용자저장은 제품 범위에서 영구 제외이므로 어떤 흐름에도 등장하지 않는다(브리프 §2, PRD 04-범위 SC9·SC10).

- 입력 화면에서는 광고를 최소화한다(브리프 §2-6, F2 수용기준 5).

## [UC-20] 연봉·연봉범위 입력

**액터**: PRD-SEG-1(이직 고려 직장인), PRD-SEG-2(신입/경력 오퍼 비교자). 참고 페르소나 PRD-PER-1("지원": 연봉 상승분이 복지·야근 감소를 상쇄하는지 판단).

**사전조건**
- 공통 사전조건 1~3.
- 연봉을 입력할 대상 슬롯(현직 a / 이직처 b)이 구성되어 있다.

**기본 흐름**
1. 사용자가 슬롯 a(현직)와 b(이직처) 각각에 연봉을 입력한다. 금액 단위는 만원이다(§4 금액 단위 규칙).
2. 단일 금액 대신 연봉 범위(하한~상한)를 입력할 수 있다(§7 입력: 연봉 범위, 함수 `getSalRange`/`getOfferRange`).
3. 입력된 연봉/범위는 F3에서 실효연봉(연봉 + 순복지) 산출의 기준값으로 사용된다(§7 계산: 실효연봉).
4. 입력 즉시 실시간 요약(`calc`)에 반영되어 대략적 비교가 갱신된다(§7: `calc()`=실시간 요약).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 이직처(b)를 오퍼 범위로 입력 | 오퍼 범위(`getOfferRange`)로 하한~상한 지정 가능 |
| 2b | 범위 하한 > 상한 | 하한 ≤ 상한이 되도록 정정을 안내한다 |
| 1a | 연봉 미입력(빈 값) | 해당 슬롯은 연봉 미입력 상태로 두고, 실효연봉 계산은 입력 시점까지 보류한다 |
| 1b | 음수·비정상 값 입력 | 0 이상의 유효 금액만 허용한다 |

**사후조건**
- a/b 연봉(또는 범위)이 입력 상태 객체에 반영되어 F3 실효연봉 입력으로 전달된다.
- 공통 사후조건 제약(서버 사용자저장 없음, 입력 화면 광고 최소화).

**추적**: F2(연봉 입력), F3(실효연봉 계산 연동) / PRD-SEG-1, PRD-SEG-2 / PRD-PER-1 / PRD-NEED-2 / §7(연봉 범위·실효연봉·`getSalRange`/`getOfferRange`), §4(만원 단위).

## [UC-21] 선택 회사의 복지 프리셋 자동 채움·수정

**액터**: PRD-SEG-1, PRD-SEG-2. 보조: PRD-SEG-3(회사 상세 F4의 "이 회사로 비교하기" CTA로 전환한 사용자).

**사전조건**
- 공통 사전조건 1~3.
- 대상 슬롯에 F1(UC-14)로 회사가 지정된 상태(직접 입력 모드 UC-15는 예외 흐름에서 처리).

**기본 흐름**
1. 슬롯에 회사가 지정되면(UC-14) 해당 회사의 복지 데이터가 F2 복지 항목의 초기값으로 자동 채워진다(F1 수용기준 2, F2 수용기준 5).
2. 실데이터 복지가 있는 회사는 `TCOMPANY_BENEFIT` 값(`BENEFIT_NM`, `BENEFIT_AMT`(만원), `BENEFIT_CTGR_CD`, `BADGE_CD`)으로 항목을 채운다(§4).
3. 실데이터가 없으면 기업유형 프리셋 `TBENEFIT_PRESET`으로 폴백하고, `DEFAULT_CHECKED_YN`에 따라 각 항목의 기본 체크 상태를 설정한다(§4, F2 수용기준 5, §5 유형 프리셋 폴백).
4. 자동 채워진 항목은 배지(`BADGE_CD`: official=공식 / est=추정)로 출처 신뢰도를 표시한다(§4).
5. 사용자가 자동값을 그대로 두거나, 개별 항목의 체크/금액을 수정한다 → 수정 상세는 UC-22로 연결된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 실데이터가 일부 항목만 존재 | 존재하는 항목은 실데이터로, 나머지는 기업유형 프리셋 또는 미채움으로 초기화한다(병합 폴백, §4/§6) |
| 5a | 사용자가 자동값(금액/체크)을 수정 | 수정한 항목의 배지는 사용자 입력 기준으로 전환하여, 공식/추정 배지의 근거를 사용자 값이 대체한다. 만료(`EXPIRES_DTM`) 항목도 사용자가 수정할 수 있다 |
| 1a | 직접 입력 모드 슬롯(UC-15) | 회사 미지정이므로 자동 채움 없이 빈 상태에서 사용자가 복지를 직접 입력한다(UC-22) |
| 1b | 슬롯의 회사를 다른 회사로 교체(UC-14 2a) | 새 회사의 실데이터/프리셋으로 복지 초기값을 재설정하며, 기존 사용자 수정값이 재설정으로 대체됨을 안내한다 |

**사후조건**
- 슬롯의 복지 항목이 실데이터 또는 기업유형 프리셋으로 초기화되고, 사용자 수정이 반영된 상태가 된다.
- 확정된 복지 값은 F3 카테고리별 델타·순복지 계산의 입력이 된다.
- 공통 사후조건 제약.

**추적**: F2(프리셋 자동 채움·수정), F1(UC-14 회사 지정 연동), F3(복지 입력 연동) / PRD-SEG-1, PRD-SEG-2, PRD-SEG-3 / PRD-NEED-2 / §4(`TCOMPANY_BENEFIT`/`TBENEFIT_PRESET`/`BADGE_CD`/`DEFAULT_CHECKED_YN`), §6(프리셋 폴백 병합).

## [UC-22] 복지 항목 체크·금액 조정

**액터**: PRD-SEG-1, PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.
- 복지 항목이 카테고리별로 표시된 상태(UC-21 자동 채움 이후, 또는 직접 입력 모드 UC-15).

**기본 흐름**
1. 복지 항목이 9개 카테고리로 구성되어 표시된다: compensation / flexibility / work_env / time_off / health / family / growth / leisure / perks(§4 `BENEFIT_CTGR_CD`, F2 수용기준 3).
2. 사용자가 각 항목의 체크 여부(비교에 포함/제외)를 지정한다.
3. 금액이 있는 정량 항목은 금액(만원)을 입력·조정한다(§4 금액 단위 규칙).
4. 정성적 항목(`QUAL_YN`=Y)은 금액 없이 체크만 하며, 정성 설명(`QUAL_DESC_CTNT`)으로 표시된다(§4: 정성적이면 `BENEFIT_AMT`는 NULL).
5. 체크/금액 변경은 카테고리별 복지 합산과 순복지 계산(F3)의 입력으로 반영된다(§7 `getBenTotal`, `renderBenCatCompare`, `renderQualCompare`).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 항목 체크 해제 | 해당 항목을 복지 합산에서 제외한다 |
| 3a | 정량 항목인데 금액 미입력 | 0 또는 미반영으로 처리한다(입력 시 합산 반영) |
| 3b | 프리셋 자동값을 수정 | UC-21 수정 경로와 동일하게 처리되며, 해당 항목 배지는 사용자 입력 기준으로 전환된다 |
| 1a | a/b 동일 카테고리 비교 | 카테고리별 델타 산출은 F3에서 수행하며, 본 UC는 각 슬롯의 값 지정까지 담당한다 |

**사후조건**
- a/b 각 슬롯의 복지 항목 체크/금액이 9개 카테고리별로 확정된다.
- 확정 값은 F3의 순복지·카테고리 델타 계산 입력으로 전달된다.
- 공통 사후조건 제약.

**추적**: F2(복지 체크·금액), F3(카테고리 델타·순복지 연동) / PRD-SEG-1, PRD-SEG-2 / PRD-NEED-2 / §4(`BENEFIT_CTGR_CD` 9종·`QUAL_YN`·`QUAL_DESC_CTNT`), §7(`getBenTotal`·`renderBenCatCompare`·`renderQualCompare`).

## [UC-23] 근무형태 입력 (재택/유연/야근/임금유형)

**액터**: PRD-SEG-1, PRD-SEG-2. 참고 페르소나 PRD-PER-1("지원": 야근 감소·재택이 처우 판단의 핵심).

**사전조건**
- 공통 사전조건 1~3.

**기본 흐름**
1. 사용자가 슬롯 a/b 각각의 근무형태를 입력한다(§7 입력: 야근 ot, 재택 remote, 유연 flex, 임금유형 wage).
2. 재택(remote): 재택 여부/수준을 입력한다 → F3에서 재택절약(`REMOTE_SAVE`) 워라밸 요인 입력이 된다(§7).
3. 유연(flex): 유연근무 여부를 입력한다 → F3 자율성 요인 입력이 된다(§7).
4. 야근(overtime/ot): 야근 수준을 입력한다 → F3 야근수당(`getOTPay`) 산정 입력이 된다(§7).
5. 임금유형(wage): 포괄임금 / 비포괄을 선택한다 → F3 `getOTPay`의 포괄/비포괄 분기 입력이 된다(§7: wage=포괄/비포괄).
6. 근무형태 입력은 F3의 주간근무시간(`getWSHours`)·야근수당(`getOTPay`)·워라밸·자율성 요인 계산으로 전달된다(§7).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 슬롯에 회사가 지정됨 | 회사의 `WORK_STYLE_VAL`(§4 JSON: remote/flex/unlimitedPTO/refreshLeave/overtime)로 근무형태 초기값을 제안하고, 사용자가 수정할 수 있다 |
| 5a | 임금유형 미지정 | `getOTPay` 분기 기본값 적용 전까지 야근수당 산정을 보류/기본 처리한다 |
| 2a | 무제한휴가(unlimitedPTO)·리프레시휴가(refreshLeave) 등 `WORK_STYLE_VAL` 항목 | 해당 항목을 F3 워라밸 요인 입력에 반영한다(§7 무제한휴가 요인) |

**사후조건**
- a/b 근무형태(재택/유연/야근/임금유형)가 입력 상태 객체에 반영된다.
- 반영 값은 F3 시간조정 가치·워라밸·자율성 계산 입력으로 전달된다.
- 공통 사후조건 제약.

**추적**: F2(근무형태 입력), F3(시간가치·워라밸 연동) / PRD-SEG-1, PRD-SEG-2 / PRD-PER-1 / PRD-NEED-2 / §7(`getWSHours`·`getOTPay` 포괄/비포괄 분기·`REMOTE_SAVE`), §4(`WORK_STYLE_VAL`).

## [UC-24] 통근시간 입력

**액터**: PRD-SEG-1, PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.

**기본 흐름**
1. 사용자가 슬롯 a/b 각각의 통근시간을 입력한다(§7 입력: 통근시간).
2. 통근시간은 F3의 시간조정(시간당) 가치·워라밸 요인 계산 입력으로 전달된다(§7 출력: 시간조정 가치, 워라밸).
3. 입력 즉시 실시간 요약(`calc`)에 반영되어 대략적 비교가 갱신된다(§7 `calc()`).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 통근시간 미입력 | 통근시간 미반영 상태로 두고, 입력 시점에 계산에 반영한다 |
| 1b | 재택(UC-23)과의 관계 | 재택 비중이 통근 부담을 낮추는 반영은 F3 계산에서 처리하며, 본 UC는 통근시간 값 지정까지 담당한다 |

**사후조건**
- a/b 통근시간이 입력 상태 객체에 반영되어 F3 시간가치·워라밸 계산 입력으로 전달된다.
- 공통 사후조건 제약.

**추적**: F2(통근시간 입력), F3(시간가치·워라밸 연동) / PRD-SEG-1, PRD-SEG-2 / PRD-NEED-2 / §7(통근시간 입력·시간조정 가치).

## [UC-25] 우선순위 직접 선택 (연봉/워라밸/복지/브랜드)

**액터**: PRD-SEG-1, PRD-SEG-2. 대표 페르소나 PRD-PER-2("민재": 브랜드보다 워라밸을 우선순위로 직접 선택).

**사전조건**
- 공통 사전조건 1~3.
- 가치관 진단 설문(프로파일러)은 존재하지 않으며, 사용자가 우선순위를 직접 선택한다(§2-3, F2 수용기준 1, SC9 영구 제외).

**기본 흐름**
1. 사용자가 비교 우선순위를 연봉 / 워라밸 / 복지 / 브랜드 중에서 직접 선택한다(§7 입력: `curPri`). 프로파일러 설문 없이 사용자가 직접 고른다(§2-3, F2 수용기준 1).
2. 사용자가 희생요소(`curSacrifice`)를 직접 선택한다 — 무엇을 감수할지 지정한다(§7 입력, F2 수용기준 2; F2에서 우선순위·희생요소는 함께 직접 선택되는 입력).
3. 선택한 우선순위·희생요소는 F3의 우선순위 기반 "누가 유리/무엇을 택할지" 판정(`vdCard`) 입력으로 전달된다(§7 출력, F2 수용기준 2).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 우선순위 미선택 | 사용자가 직접 선택하는 것이 전제이므로, 판정 전 우선순위 선택을 요구한다(§2-3) |
| 1b | 우선순위·희생요소 변경 | 변경 시 F3의 우선순위 기반 판정(`vdCard`)이 재계산됨을 입력 변경으로 반영한다 |
| 1c | 프로파일러(자동 추천)를 기대하는 사용자 | 해당 기능은 영구 제외(SC9, 비타깃)이며, 직접 선택으로 대체됨을 안내한다(§2-3) |

**사후조건**
- 우선순위(`curPri`)와 희생요소(`curSacrifice`)가 확정되어 F3 판정(`vdCard`) 입력으로 전달된다.
- 공통 사후조건 제약(서버 사용자저장 없음).

**추적**: F2(우선순위·희생요소 직접 선택), F3(우선순위 판정 `vdCard` 연동) / PRD-SEG-1, PRD-SEG-2 / PRD-PER-2 / PRD-NEED-3 / §2-3(프로파일러 제외·직접 선택), §7(`curPri`·`curSacrifice`·우선순위 판정).
