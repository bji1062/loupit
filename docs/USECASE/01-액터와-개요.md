# 01. 액터와 개요

**목적**: loupit USECASE 문서군의 진입점으로서 (1) 시스템과 상호작용하는 **액터(A1~A5)**를 정의하고, (2) 전체 유즈케이스의 **마스터 인덱스(UC-ID→제목→관련 F)**를 제공하며, (3) **UC-ID 명명 규칙**을 확정한다. 본 문서는 공통·시스템 수준 유즈케이스(UC-A 대역)를 표준 형식으로 직접 소유한다.

**참조 브리프 섹션**: §2(로그인/프로파일러 제거·localStorage), §3(MVP F1~F8), §6(읽기 전용 API), §7(비교 알고리즘 입출력), §8(빌드타임 정적 생성·인프라).

**상위 추적**: 본 문서는 PRD의 기능 요구사항(F1~F8)과 타깃 사용자(PRD-SEG-*/PRD-PER-*/PRD-CTX-*)를 인용한다(USECASE→PRD). 하위 문서(02~)의 UC는 본 문서가 정한 액터·명명 규칙을 그대로 따른다.

**전역 전제(모든 액터·UC 공통)**: 로그인/회원/계정/소셜피드 없음(SC10, §2-1), 프로파일러(가치관 진단 설문) 없음(SC9, §2-3), 서버 측 사용자 데이터 쓰기 저장 없음(§2-4). 사용자 상태는 브라우저 localStorage에만 저장(§2-2). 비교 계산은 100% 클라이언트(§2-8). 따라서 본 문서군 어디에도 로그인/회원가입/프로파일러/서버 쓰기 유즈케이스는 존재하지 않는다.

## 1. 개요

loupit은 로그인 없이 한국 회사의 복지·연봉·근무조건을 비교하는 클라이언트 사이드 도구(F1~F3)이자, 검색 색인을 노린 정적 콘텐츠 페이지 모음(F4~F5)이다(§1). 상호작용 주체는 사람 사용자 두 부류(비교 목적 방문자 / 검색 유입 열람자)와 자동화·외부 주체 세 부류(검색 크롤러, 광고·제휴 시스템, 빌드·운영 주체)로 나뉜다. 본 문서는 이 다섯 액터를 확정하고, 이후 문서가 상세화할 유즈케이스의 전체 지도를 제시한다.

## 2. 액터 정의

액터 ID는 `A1..` 체계로 부여한다(UC-ID와 구분: 3장 참조). 유형은 사람(end-user)·자동화(bot)·외부시스템(3rd party)·운영(build/ops)으로 분류한다.

| ID | 액터 | 유형 | 정의 | 대응 PRD 세그먼트·맥락 | 주 관련 기능 |
|----|------|------|------|------------------------|--------------|
| **A1** | 익명 방문자 | 사람(end-user) | 로그인 없이 비교 툴(SPA)을 사용하는 최종 사용자. 계정·서버 저장 없이 검색·입력·리포트를 수행하고, 상태는 localStorage에만 남긴다. | PRD-SEG-1(이직 고려 직장인), PRD-SEG-2(오퍼 비교자), PRD-CTX-1(모바일)·CTX-3(비로그인) | F1, F2, F3 (F6·F7 노출·열람) |
| **A2** | 검색엔진 유입 사용자 | 사람(end-user) | 검색엔진 결과(SERP)에서 정적 회사 상세(F4)·인기 조합(F5) 페이지로 **직접** 진입한 사용자. 열람 후 CTA를 통해 A1 동작(비교)으로 전환될 수 있다. 랜딩을 경유하지 않는다. | PRD-SEG-3(복지 정보 검색 유입자), PRD-CTX-2(검색 유입) | F4, F5 (전환 시 F1·F2) |
| **A3** | 검색 크롤러/봇 | 자동화(bot) | Googlebot 등 검색엔진 크롤러. sitemap.xml과 정적 페이지를 요청·색인하여 A2의 유입 기반을 만든다. 사람이 아니며 입력·계산을 하지 않는다. | PRD-CTX-2(SEO 유입 전제), §2-7(SEO) | F4, F5 (sitemap 색인) |
| **A4** | 광고·제휴 시스템 (3rd party) | 외부시스템 | Google AdSense(자동광고+수동 슬롯)와 `affiliate.json` 기반 제휴 대상. 콘텐츠 페이지의 광고/제휴 슬롯에 콘텐츠를 공급·노출하는 외부 주체. "광고" 표기 대상이다. | 수익화(§2-5), 배치 정책(§2-6) | F6 |
| **A5** | 빌드/운영자 | 운영(build/ops) | 빌드타임 정적 생성 스크립트를 실행하고 정적 자산·sitemap·정책 페이지를 배포하며 읽기 전용 API를 운영하는 주체(사람 또는 CI). 런타임 사용자 쓰기가 아니라 배포·생성만 수행한다. | 인프라·스택(§8), 문서·생성 규칙(§9) | F4, F5, F7, F8 (생성·배포·운영) |

### 2.1 액터-세그먼트 관계

- **A1과 A2는 모두 비로그인 사람 사용자**이며, **진입점과 1차 의도**로 구분한다: A1은 비교 툴에 직접/전환 진입(비교 의도, PRD-SEG-1·2), A2는 검색엔진에서 정적 페이지로 진입(정보 열람 의도, PRD-SEG-3). PRD-SEG-3이 "이 회사로 비교하기" CTA로 A1 동작으로 전환되는 흐름을 전제한다(03-타깃-사용자 §2·§5).
- 페르소나 대응: PRD-PER-1/PER-2 → A1, PRD-PER-3 → A2.
- A3·A4·A5는 사람 세그먼트에 대응하지 않는 자동화·외부·운영 주체다.

## 3. UC-ID 명명 규칙

**형식**: 기능 상세 UC는 `UC-<십단위 대역번호>`, 시스템 공통 UC는 `UC-A<일련번호>` (예: `UC-10`, `UC-25`, `UC-63`, `UC-A1`).

1. **접두사 `UC-`**: USECASE 항목임을 표시하며, PRD의 기능 ID(`F1`~`F8`)·사용자 ID(`PRD-SEG-*` 등)와 구분한다.
2. **십단위 숫자 대역**: 각 기능 상세 문서(02~08)가 자기 대역을 1:1로 소유한다(아래 표). F1=`UC-1x`, F2=`UC-2x`, F3=`UC-3x`, F4=`UC-4x`, F5=`UC-5x`, F6·F7=`UC-6x`, 오류·엣지(횡단)=`UC-9x`. 대역 내 번호는 담당 문서가 `x0`부터 증가시키며 부여한다. 대역과 담당 문서가 일대일이라 UC-ID만 보고 상세 위치를 추적할 수 있다.
3. **시스템 공통 `UC-A<n>`**: 특정 기능 하나에 귀속되지 않고 여러 기능을 가로지르는 부팅·로컬저장·색인·빌드·헬스체크 유즈케이스는 본 문서(01)가 `UC-A1`부터 소유한다. 숫자 대역 대신 문자 `A`를 쓰는 이유는 이들이 어느 기능 대역에도 속하지 않는 횡단 유즈케이스임을 표시하기 위함이다.
4. **일련번호**: 대역 내 정수로 증가한다. 중복 금지, 한 번 부여한 번호는 재사용하지 않는다(안정 ID, 브리프 §9).
5. **읽기 전용 API(F8)는 전용 대역을 두지 않는다**: 4개 엔드포인트가 소비 맥락별 UC로 흡수된다(4장 F8 커버리지 표 참조).
6. **액터 ID는 별도 체계 `A1..`**(`UC-` 접두사 없음)로 UC-ID와 구분한다. 즉 `A1`(액터)과 `UC-A1`(공통 시스템 유즈케이스)은 서로 다른 식별자다.

**대역 배정**:

|:---:|------|--------|-----------|
| **UC-Ax** | 공통·시스템(부팅·로컬저장·색인·빌드·헬스체크) | F8·F2·F3·F4·F5 | 본 문서(01) |
| **UC-1x** | 회사 검색·선택 | F1 | 02 |
| **UC-2x** | 비교 입력 | F2 | 03 |
| **UC-3x** | 비교 리포트 | F3 | 04 |
| **UC-4x** | 회사 상세 페이지 | F4 | 05 |
| **UC-5x** | 인기 비교 조합 | F5 | 06 |

### 4.1 F8(읽기 전용 API) 커버리지

F8은 전용 십단위 대역을 두지 않는다. 브리프 §6·F8이 정의한 4개 엔드포인트는 아래처럼 소비 맥락별 UC로 흡수되며, 정상경로와 오류·엣지가 모두 커버된다.

| 엔드포인트 | 정상경로 UC | 오류·엣지 UC |
|-----------|-------------|--------------|
| `GET /api/v1/reference/all` | UC-A1 (SPA 부팅 시 번들 로드) | UC-90 (로드 실패·오류 상태) |
| `GET /api/v1/companies/search?q=` | UC-10~UC-12 (검색이 이 API를 호출) | UC-91 (검색 API 실패) |
| `GET /api/v1/companies/{id}` | 전용 정상경로 UC 없음 — SPA는 `reference/all` 번들로 회사 상세(별칭·복지 인라인)를 이미 전량 보유(§6)하므로 런타임에 개별 호출하지 않는다. 응답 의미(프리셋 폴백 병합)는 UC-93이 소유. | UC-93 (프리셋 폴백 병합), UC-95 (존재하지 않는 id·404) |
| `GET /api/v1/health` | UC-A5 (운영 헬스체크) | UC-A5 예외 흐름 (미기동·5xx·타임아웃) |

## 5. UC-A: 공통·시스템 유즈케이스

특정 기능 하나에 귀속되지 않고 여러 기능을 가로지르는 시스템 수준 유즈케이스를 표준 형식으로 정의한다.

### UC-A1 참조 데이터 번들 로드(SPA 부팅)

- **액터**: A1 익명 방문자(주). A2가 비교 툴로 전환 진입하는 경우 동일.
- **사전조건**: 사용자가 비교 툴(SPA)을 로드한다. `/api/v1/reference/all`이 응답 가능하다. 프로파일러 관련 키(profiles/job_groups/questions)는 번들에 없다(§6).
- **기본 흐름**:
  1. 사용자가 비교 툴 화면(SPA)에 진입한다.
  2. 프론트가 `GET /api/v1/reference/all`을 1회 호출한다.
  3. 서버가 참조 번들(`company_types[]`, `benefit_presets{type_cd:[…]}`, `companies[{…, aliases[], benefits[]}]`)을 `Cache-Control: public, max-age=3600`으로 반환한다.
  4. 프론트가 번들을 클라이언트 메모리에 보관하고 검색(F1)·비교 입력(F2)·비교 계산(F3)의 단일 참조원으로 사용한다.
- **대안·예외 흐름**:
  - 3a. 브라우저/프록시 캐시가 유효(max-age 이내)하면 네트워크 호출 없이 캐시에서 로드한다.
  - 2a. 응답 실패(네트워크 오류·5xx): 프론트가 재시도하거나 오류 안내를 표시하고, 참조 데이터 확보 전까지 검색·비교 기능을 제한한다.
- **사후조건**: 참조 번들이 클라이언트에 로드되어 후속 검색·비교가 가능하다. 서버에 사용자별 저장이 발생하지 않는다.
- **추적**: F8(reference/all), F1·F2·F3(참조 소비), PRD-SEG-1·PRD-SEG-2, §6.

### UC-A2 최근 비교 기록 로컬 저장·복원(localStorage)

- **액터**: A1 익명 방문자.
- **사전조건**: 브라우저 localStorage 사용이 가능하다. 서버 계정·서버 저장은 존재하지 않는다(§2-1·§2-2).
- **기본 흐름**:
  1. 사용자가 비교를 수행하거나 리포트를 조회한다(F2/F3).
  2. 프론트가 "최근 비교" 항목을 브라우저 localStorage에 저장한다(서버 전송 없음).
  3. 사용자가 재방문·재진입하면 프론트가 localStorage에서 최근 비교를 읽어 화면에 복원 제시한다.
- **대안·예외 흐름**:
  - 2a. localStorage 비활성/용량 초과: 저장을 생략하고, 비교 기능은 정상 동작하되 복원만 제공하지 않는다.
  - 3a. 저장된 기록 없음: 복원 항목을 표시하지 않는다.
- **사후조건**: 최근 비교가 해당 기기 내에만 유지된다. 서버 저장·크로스 디바이스 동기화는 없다(PRD-CTX-4).
- **추적**: F2·F3, PRD-NEED-5, PRD-CTX-4, §2-2.

### UC-A3 검색 크롤러의 정적 페이지·사이트맵 색인

- **액터**: A3 검색 크롤러/봇.
- **사전조건**: 회사 상세(F4)·인기 조합(F5) 정적 페이지와 sitemap.xml이 배포되어 있다(UC-A4 완료 상태).
- **기본 흐름**:
  1. 크롤러가 `sitemap.xml`을 요청한다.
  2. 서버(Nginx 정적)가 sitemap 항목(회사 200개 + 인기 조합)을 반환한다.
  3. 크롤러가 각 정적 페이지 URL을 요청한다.
  4. 서버가 완성된 정적 HTML(회사: 기업정보·복지·근무형태 / 조합: 두 회사 핵심 비교 정보)을 반환한다.
  5. 크롤러가 콘텐츠를 색인하여 검색 유입과 AdSense 승인 콘텐츠 기반을 형성한다.
- **대안·예외 흐름**:
  - 4a. 페이지 미존재(404): 해당 URL을 색인에서 제외한다.
- **사후조건**: 정적 페이지가 검색 색인된다. A2(검색엔진 유입 사용자)의 유입 경로가 형성된다.
- **추적**: F4·F5, §2-7(SEO), PRD-CTX-2, PRD-SEG-3.

### UC-A4 빌드타임 정적 페이지·사이트맵 생성·배포

- **액터**: A5 빌드/운영자.
- **사전조건**: 참조 데이터(회사 200개, 기업유형·프리셋, 회사별 복지)와 인기 회사쌍 목록이 준비된다. 빌드타임 파이썬 생성 스크립트(§8) 실행 환경이 갖춰진다.
- **기본 흐름**:
  1. 운영자가 빌드타임 정적 생성 스크립트를 실행한다.
  2. 스크립트가 회사 200개 상세 페이지(F4)와 인기 조합 페이지(F5)를 정적 HTML로 생성한다.
  3. 실데이터 복지가 없는 회사는 기업유형 프리셋으로 폴백하여 복지를 채운다(§4 TBENEFIT_PRESET).
  4. 스크립트가 sitemap.xml을 생성·갱신한다.
  5. 운영자가 생성물과 정책 페이지(F7)를 Nginx 정적 경로에 배포한다.
- **대안·예외 흐름**:
  - 2a. 회사 데이터 누락·불일치: 해당 회사 생성을 건너뛰거나 프리셋으로 폴백하고 로그를 남긴다.
- **사후조건**: 정적 자산과 sitemap이 배포되어 크롤러(A3) 색인과 사용자(A2) 열람 준비가 완료된다. 런타임 서버 쓰기는 발생하지 않는다.
- **추적**: F4·F5·F7, §8, §2-7.

### UC-A5 읽기 전용 API 헬스체크·가용성 운영 점검

- **액터**: A5 빌드/운영자(주). 외부 모니터링 주체(사람 또는 자동 감시)를 포함한다.
- **사전조건**: 읽기 전용 API(Uvicorn)가 배포되어 `/api/v1/health`가 라우팅된다(§6·§9). 인증 미들웨어·사용자 쓰기 경로는 없다(§2-4).
- **기본 흐름**:
  1. 운영자 또는 모니터링이 `GET /api/v1/health`를 호출한다.
  2. 서버가 사용자 데이터 조회·쓰기 없이 API 생존 상태를 나타내는 경량 응답을 반환한다.
  3. 운영자가 응답으로 API 가용성과 배포 정상 여부를 확인한다.
- **대안·예외 흐름**:
  - 2a. API 미기동·5xx·타임아웃: 운영자가 서비스(systemd/Uvicorn)와 프록시(Nginx `/api/v1/*`) 상태를 점검·재기동한다. (사용자 대면 참조 로드 실패는 UC-90에서 별도로 다룬다.)
- **사후조건**: API 가용성이 확인된다. 사용자별 서버 저장·쓰기는 발생하지 않는다(§2-4).
- **추적**: F8(`/api/v1/health`), §6(읽기 전용 API 표면), §9(인프라·프록시), 연계 UC-90(참조 로드 실패 처리).
