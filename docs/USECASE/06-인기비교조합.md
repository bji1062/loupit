# 인기 비교 조합 (USECASE)

**목적**: loupit의 인기 비교 조합 페이지(예: 삼성전자 vs SK하이닉스)에서, 검색엔진으로 유입한 사용자가 정적 조합 페이지를 열람하고, 빌드타임에 사전 생성된 두 회사의 핵심 비교 요약을 확인하며, "비교 툴에서 열기" 경로로 진입해 본인 조건으로 클라이언트 재계산까지 이어지는 사용자 상호작용을 F5(인기 비교 조합 페이지)에 근거하여 표준 유즈케이스로 정의한다.

**상위 추적**: 본 문서는 USECASE→PRD 체인에 속하며, 각 UC는 PRD 기능 요구사항 F5(그리고 전환·연동 지점에서 F2·F3·F4·F6)와 타깃 사용자 세그먼트 PRD-SEG-* / 페르소나 PRD-PER-* / 니즈 PRD-NEED-* / 이용 맥락 PRD-CTX-*를 인용한다. 생성·데이터·계산 근거는 브리프 §8(빌드타임 정적 생성)·§4/§5(데이터 모델·보존 데이터)·§7(비교 알고리즘)·§2-6/§2-7(광고 배치·SEO)을 인용한다.

**범위 경계**: 이 문서는 F5(인기 조합 페이지 유입·열람·요약 확인·비교 툴 진입)만 다룬다. 조합 페이지·사이트맵을 실제로 생성·배포하는 빌드 동작은 공통 유즈케이스 UC-A4, 크롤러 색인은 UC-A3에서 다룬다. 회사 검색·슬롯 지정은 F1(UC-10~UC-15, 02 문서), 비교 조건 입력은 F2(UC-20~UC-25, 03 문서), 비교 리포트 산출·재계산은 F3(UC-30~UC-36, 04 문서), 광고/제휴 슬롯 상세는 F6에서 다룬다. 로그인/회원/계정/프로파일러(가치관 진단 설문)/서버 사용자 데이터 쓰기 저장은 제품 범위에서 영구 제외이므로 어떤 흐름에도 등장하지 않는다(브리프 §2-1·§2-3·§2-8, PRD 04-범위 SC9·SC10).

**UC-ID 대역**: 본 문서는 F5에 **UC-5x** 대역(UC-50~UC-54)을 사용한다. 검색(F1)이 UC-1x, 비교 입력(F2)이 UC-2x, 비교 리포트(F3)가 UC-3x 대역을 쓰는 것과 동일한 십단위 대역 규칙이며, 부여한 번호는 재사용하지 않는다(안정 ID).

**공통 사전조건(모든 UC 적용)**

1. 대상 인기 회사쌍의 정적 조합 페이지가 빌드타임에 생성·배포되어 있고 sitemap.xml에 포함되어 있다(UC-A4 완료 상태, §8, §2-7).
2. 사용자는 가입·로그인 없이 접근하며(PRD-NEED-1, PRD-CTX-3), 랜딩을 경유하지 않고 조합 페이지를 독립 진입점으로 사용할 수 있다(PRD-CTX-2).
3. 조합 페이지에 노출되는 비교 요약은 빌드타임 생성 스크립트가 두 회사의 참조·복지 데이터(§4/§5, 등록 회사는 모두 실복지 보유이므로 실데이터 기반)로 사전 산출한 것이며, 방문자별 개인 입력(연봉·우선순위 등)은 포함하지 않는다.

**공통 사후조건 제약(모든 UC 적용)**: 조합 페이지 열람·요약 확인·비교 툴 진입 전 과정에서 서버로의 사용자 데이터 쓰기 저장이 발생하지 않는다(§2-1·§2-4). 비교 요약은 런타임 서버 계산이 아니라 빌드타임 정적 산출물이며(§8), 방문자 본인 조건을 반영한 계산은 비교 툴 진입 후 100% 클라이언트에서 수행된다(§2-8). 페이지 열람은 로그인·회원 없이 모바일 우선 레이아웃에서 동작한다(PRD-CTX-1).

## [UC-50] 검색엔진에서 인기 비교 조합("A vs B") 페이지 유입·열람

**액터**: PRD-SEG-3(회사 복지 정보 검색 유입자, 페르소나 PRD-PER-3 "수아"). 보조: PRD-SEG-2(신입/경력 오퍼 비교자 — F5를 주 진입점으로 사용, 03-타깃-사용자 §2). 액터 유형상 A2(검색엔진 유입 사용자).

**사전조건**
- 공통 사전조건 1~3.
- 사용자가 검색엔진 결과(SERP) 또는 공유 링크에서 특정 회사쌍의 조합 페이지 URL로 진입한다.

**기본 흐름**
1. 크롤러가 사전에 색인한 정적 조합 페이지(UC-A3)가 검색 결과에 노출된다.
2. 사용자가 "A vs B"(예: 삼성전자 vs SK하이닉스) 형태의 결과를 클릭해 조합 페이지로 직접 진입한다(랜딩 경유 없음, PRD-CTX-2).
3. Nginx가 완성된 정적 HTML(두 회사의 핵심 비교 정보)을 반환한다(§8).
4. 사용자가 모바일 우선 레이아웃에서 두 회사의 조합 비교 페이지를 열람한다.
5. 사용자가 사전 생성 비교 요약을 확인하거나(UC-51), 비교 툴 진입(UC-52)·내부 탐색(UC-53)으로 이어간다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 요청 URL의 조합 페이지가 미배포·미존재 | 404를 반환한다(색인 대상에서도 제외, UC-A3 4a). 존재하는 조합만 페이지가 존재한다(사전 정의된 인기 회사쌍 목록 기준, §3 F5) |
| 2a | 공유 링크·북마크로 직접 진입 | 조합 페이지는 독립 진입점이므로 동일하게 정적 HTML을 열람한다(랜딩 불필요) |
| 4a | 콘텐츠 페이지로서 광고/제휴 슬롯 노출 대상 | 조합 페이지는 광고 배치 대상이다(§2-6). 광고 노출은 UC-54, 슬롯 상세는 F6 |

**사후조건**
- 사용자가 인기 조합 페이지의 두 회사 핵심 비교 정보를 열람한 상태.
- 서버 사용자 저장 없음(공통 사후조건 제약). 이후 UC-51/UC-52/UC-53으로 진행 가능.

**추적**: F5 / PRD-SEG-3, PRD-SEG-2 / PRD-PER-3 / PRD-NEED-4(검색으로 회사 정보 열람) / PRD-CTX-1, PRD-CTX-2 / UC-A3(크롤러 색인)·UC-A4(생성·배포) / §8, §2-7.

## [UC-51] 빌드타임 사전 생성 비교 요약 확인

**액터**: PRD-SEG-3(정보 열람 의도), PRD-SEG-2(오퍼 비교 의도). A2.

**사전조건**
- 공통 사전조건 1~3.
- UC-50으로 조합 페이지가 열람되고 있다.

**기본 흐름**
1. 사용자가 조합 페이지에 사전 표시된 두 회사(A/B)의 핵심 비교 요약을 확인한다(F5 수용기준 2).
2. 요약은 빌드타임 생성 스크립트가 각 회사의 참조·복지 데이터(§4/§5: 기업정보·근무형태 WORK_STYLE_VAL·복지)로 산출한 정적 값이다(§8, UC-A4). 방문자 개인 입력(연봉·우선순위)은 반영되지 않는다.
3. 사용자가 두 회사의 조합 비교 정보를 통해 대략적 처우·복지·근무형태 차이를 파악한다.
4. 개인 조건을 반영한 실효연봉·시간가치·우선순위 판정을 원하면 비교 툴 진입(UC-52)으로 안내된다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 2a | 조합 대상은 등록 회사(모두 실복지 보유)만 페어링 | 두 회사 모두 실데이터로 요약을 생성한다. 등록 회사는 실복지를 보유하므로 회사페이지 프리셋 폴백은 불필요하다(§4/§5, UC-A4) |
| 2b | 정성적 복지(QUAL_YN=Y, 금액 없음) 포함 | 금액 요약에서 제외하고 정성 항목으로 표기한다(§4 QUAL_DESC_CTNT) |
| 3a | 사용자가 요약을 자신의 상황으로 오해할 여지 | 요약이 개인 입력 없는 사전값 기준임을 명시하고, 복지 데이터 정확성 한계는 면책조항(F7)과 정합하게 고지한다(§3 F7) |

**사후조건**
- 사용자가 사전 생성 비교 요약(사전값 기준)을 확인한 상태.
- 요약은 정적 산출물이며 서버 런타임 계산·사용자 저장이 없다(§8, 공통 사후조건 제약).

**추적**: F5 / PRD-SEG-3, PRD-SEG-2 / PRD-NEED-4, PRD-NEED-2(두 직장 비교) / §8(빌드타임 생성), §4/§5(회사·복지 데이터), F7(데이터 정확성 면책 정합) / UC-A4.

## [UC-52] 조합 페이지에서 비교 툴 진입·본인 조건 재계산

**액터**: 진입 시 PRD-SEG-3(A2), 비교 툴 진입 후 PRD-SEG-1/PRD-SEG-2(A1)로 전환. 페르소나 PRD-PER-2(취업 준비생 "민재": 오퍼 비교), PRD-PER-3(정보 탐색자 "수아": 열람 후 비교로 전환).

**사전조건**
- 공통 사전조건 1~3.
- UC-50/UC-51로 조합 페이지를 열람 중이며, 페이지가 "비교 툴에서 열기" 진입 경로를 제공한다(F5 수용기준 2).

**기본 흐름**
1. 사용자가 조합 페이지의 "비교 툴에서 열기"(해당 A/B 조합) 경로를 실행한다.
2. 비교 툴(SPA)이 로드되며 참조 번들(`reference/all`)을 로드한다(UC-A1).
3. 조합의 두 회사가 슬롯 a(현직)·b(이직처)에 사전 지정된 상태로 진입한다(F1 슬롯 지정 UC-14 연동).
4. 사용자가 본인 조건(연봉, 복지 체크/금액, 근무형태 재택·유연·야근·임금유형, 통근시간, 우선순위 직접 선택)을 입력한다(F2, UC-20~UC-25).
5. 클라이언트가 본인 조건으로 비교 리포트를 재계산한다(F3, `compare()`, UC-30). 서버 호출·저장은 발생하지 않는다(§2-8).
6. 사용자가 실효연봉·시간조정 가치·워라밸·카테고리별 복지 델타·우선순위 판정(vdCard)을 확인한다(UC-31~UC-33).

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 사용자가 슬롯 회사를 교체하려 함 | 검색(UC-10)·슬롯 재지정(UC-14)으로 회사를 변경한 뒤 재계산한다 |
| 4a | 본인 조건을 입력하지 않고 바로 진행 | 조합 기본값(두 회사의 실복지 사전값)으로 리포트를 산출하고, 이후 UC-35 재계산으로 개인 조건을 반영하도록 안내한다 |
| 4b | 슬롯에 지정된 조합 회사의 복지 초기값 | 조합의 두 회사는 등록 회사(실복지 보유)이므로 실복지 데이터로 초기값이 채워지고, 사용자가 수정 가능하다(§4/§5, UC-21/UC-22) |
| 6a | 사용자가 결과를 기기 내 보관하려 함 | 최근 비교를 브라우저 localStorage에만 저장한다(UC-36/UC-A2, 서버 저장 없음) |

**사후조건**
- 두 회사 조합이 비교 툴에서 본인 조건으로 재계산된 리포트로 표시된 상태.
- 모든 재계산은 클라이언트에서 수행되며 서버 전송·저장이 없다(§2-8, 공통 사후조건 제약). PRD-SEG-3 → PRD-SEG-1/PRD-SEG-2로의 전환이 성립한다.

**추적**: F5 → F2·F3 (F1 슬롯 연동) / PRD-SEG-3 → PRD-SEG-1, PRD-SEG-2 / PRD-PER-2, PRD-PER-3 / PRD-NEED-2, PRD-NEED-3(중요 축 직접 선택) / UC-A1(번들 로드)·UC-14·UC-20~UC-25·UC-30 / §7, §2-8.

## [UC-53] 관련 인기 조합·회사 상세로 내부 탐색

**액터**: PRD-SEG-3(A2). 보조: PRD-SEG-2.

**사전조건**
- 공통 사전조건 1~3.
- UC-50으로 조합 페이지를 열람 중이며, 페이지에 관련 조합·회사 상세로 이어지는 내부 링크가 있다.

**기본 흐름**
1. 사용자가 조합 페이지에서 관련 인기 조합(다른 회사쌍) 또는 개별 회사 상세 페이지로 이어지는 내부 링크를 확인한다.
2. 사용자가 링크를 따라 다른 정적 페이지로 이동한다.
3. 회사 상세로 이동하면 해당 회사의 기업정보·복지·근무형태를 열람하고, "이 회사로 비교하기" CTA로 비교 진입할 수 있다(F4).
4. 다른 인기 조합으로 이동하면 UC-50/UC-51 흐름을 그 조합에 대해 반복한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 3a | 회사 상세 페이지로 이동 | 회사 상세 유즈케이스(F4)로 이어진다. 상세 열람·CTA는 F4 문서에서 다룬다 |
| 4a | 대상 조합 페이지가 미존재 | 링크를 노출하지 않는다(존재하는 조합만 내부 링크 대상, §3 F5) |
| 1a | 관련 링크 없음 | 내부 탐색 링크 영역을 표시하지 않는다 |

**사후조건**
- 사용자가 관련 조합·회사 상세로 이동하여 탐색을 이어간 상태.
- 내부 링크는 검색 색인·유입 확대(§2-7)와 콘텐츠 순회에 기여한다. 서버 사용자 저장 없음.

**추적**: F5 (F4 연동) / PRD-SEG-3, PRD-SEG-2 / PRD-NEED-4 / PRD-CTX-2 / §2-7(SEO 내부 링크) / UC-A3(색인).

## [UC-54] 조합 페이지 광고·제휴 노출("광고" 표기)

**액터**: 노출 대상 PRD-SEG-3, PRD-SEG-2(A2, 열람자). 공급 주체 A4(광고·제휴 시스템).

**사전조건**
- 공통 사전조건 1~3.
- UC-50으로 조합 페이지를 열람 중이며, 조합 페이지가 콘텐츠 페이지로서 광고 배치 대상이다(§2-6, F5 수용기준 4).

**기본 흐름**
1. 조합 페이지 렌더링 시 AdSense 자동광고·수동 슬롯과 `affiliate.json` 기반 제휴 컴포넌트가 노출된다(§2-5, F6).
2. 모든 광고/제휴 영역에 "광고" 표기가 부착된다(§2-5, 공정위 표시광고 규정).
3. 사용자가 콘텐츠(비교 요약)와 함께 광고/제휴를 열람하며, 필요 시 제휴 링크를 클릭한다.

**대안·예외 흐름**

| 분기 | 조건 | 처리 |
|------|------|------|
| 1a | 제휴 데이터 구성·교체 | 제휴 링크는 `affiliate.json` 데이터로 구성되어 코드 수정 없이 교체 가능하다(§2-5) |
| 2a | 광고/제휴 고지 확인 | "광고" 표기·고지는 광고/제휴 고지 정책 페이지(F7)와 정합한다(§2-5) |
| 3a | 광고 슬롯 배치·렌더링 상세 | 슬롯 렌더링·클릭 유즈케이스는 F6(광고·제휴)에서 다룬다 |

**사후조건**
- 조합 페이지(콘텐츠 페이지)에 "광고" 표기가 부착된 광고/제휴가 노출된 상태.
- 광고 노출은 콘텐츠 열람을 해치지 않는 범위에서 배치되며(§2-6 콘텐츠 페이지 정책), 서버 사용자 저장은 없다.

**추적**: F5 → F6 / PRD-SEG-3, PRD-SEG-2 / §2-5(수익화·"광고" 표기), §2-6(콘텐츠 페이지 광고 배치), F7(광고/제휴 고지 정합) / A4(광고·제휴 시스템).
